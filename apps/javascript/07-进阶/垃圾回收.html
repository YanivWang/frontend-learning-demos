垃圾回收：可达性与常见泄漏

请打开 DevTools Console 查看可达性、闭包保活和 WeakMap 的输出示例。

知识点要点

  • JavaScript 会在创建对象、数组、函数和闭包时自动分配内存，并由垃圾回收器在对象不再需要时回收；开发者不能可靠地手动触发标准 GC，只能管理引用关系。
  • 现代 GC 的核心判断是可达性：从全局对象、当前调用栈、闭包环境、活动 DOM 和宿主对象出发能访问到的对象都是可达对象，不会被回收。
  • 标记清除会从根对象出发标记所有可达对象，再清理未标记对象，所以即使两个对象互相引用，只要整体从根不可达，也可以回收；这比单纯引用计数更能处理循环引用。
  • 本页的闭包示例展示了 reader 会保留 bigData，直到 reader = null 断开最后一条强引用；真实泄漏常来自全局缓存、未清理定时器、事件监听和长期闭包。
  • WeakMap 的键和 WeakSet 的值是弱持有，不会阻止对象被 GC，适合给 DOM 节点或对象实例挂元数据；代价是不可枚举、不能依赖它判断对象是否已经被回收。

面试考点

  • JavaScript 的垃圾回收为什么通常按可达性判断？ 结论是引擎只能安全回收从根对象不可达的对象。原因是“对象是否还会被业务使用”无法被精确判定，可达性是可实现的近似模型。边界是对象不可达后也不保证立刻释放，GC 时机由引擎根据内存压力和性能策略决定。
  • 标记清除为什么能处理循环引用？ 结论是它看的是整组对象是否能从根访问到，而不是对象之间是否互相引用。原因是循环中的对象如果整体不可达，就不会被标记，清除阶段可以一起回收。项目中循环引用本身不一定泄漏，真正的问题是循环中的某个对象仍被全局、DOM、监听器或闭包持有。
  • WeakMap 和 Map 在内存保留上的关键区别是什么？ 结论是 Map 强持有键和值，WeakMap 不会因为键存在于集合中而阻止键对象被回收。原因是 WeakMap 的键是弱引用，键被回收后对应值也可成为回收候选。边界是 WeakMap 只能用对象或非注册 Symbol 作键，且不能枚举键集合。
  • 项目里哪些引用最容易造成对象无法回收？ 结论是生命周期比对象更长的引用最危险，例如全局缓存、模块单例、定时器、事件监听、闭包回调和 DOM 节点引用。原因是这些引用会把本该随页面片段释放的数据留在可达图里。排查时应结合 Heap snapshot 的 retainers 找到具体引用链。
