Actions 与 useActionState

下方 #root 为交互演示区：在表单中输入名称并提交，观察 isPending 禁用态与错误提示；请打开 DevTools Console 查看 Action 调用日志。

知识点要点

  • React 19 的 Action 是用于处理异步提交的函数，可绑定到表单 action，也可和 transition 体系配合，让提交状态进入 React 的调度模型；不同于普通 onSubmit 需手动 preventDefault 和维护 loading。
  • useActionState(action, initialState) 返回 [state, formAction, isPending]；本页把 formAction 传给 <form action>，由浏览器提交事件触发 Action。
  • Action 函数接收 previousState 和 formData，返回下一次 state 或 Promise；这让失败信息、成功结果和服务端返回值都能成为显式状态。
  • isPending 表示当前提交未完成，可用于禁用输入和按钮，避免重复提交；失败路径要返回可展示的错误，而不是只在控制台打印。
  • 本页校验空名称和长度不足两个失败路径；与手写 useState pending 相比，Action 把「提交中 + 结果状态」标准化，减少样板代码。

面试考点

  • React 19 Action 与普通 submit handler 有什么区别？ 结论是 Action 把异步提交结果纳入 React 的状态模型，可直接绑定 form action。原因是 useActionState 能直接返回下一次 state 和 pending 状态，减少手写 loading / error 样板。边界是高度定制的表单库仍可能保留自己的提交管线。
  • previousState 参数有什么价值？ 结论是它让下一次提交能基于上一次结果计算。原因是表单提交常需要保留错误、计数、服务端返回 ID 或局部状态。本页只返回错误字符串，复杂项目可返回对象，例如 { ok, message, fieldErrors }。
  • isPending 为什么适合处理重复提交和加载态？ 结论是用它禁用按钮、展示「提交中」，并防止重复提交。原因是异步 Action 期间用户可能连续点击或修改输入。边界是禁用输入会影响用户修正内容，实际产品要根据提交耗时和可取消性设计交互。
  • 表单校验失败时 Action 应该返回什么状态？ 结论是校验失败应作为 Action 返回 state 展示给用户。原因是表单失败是正常业务结果，不应只靠异常或控制台。本页空名称返回「请输入名称」，长度不足返回「名称至少 2 个字符」。
  • useActionState 与 useState 手写 pending 如何选？ 结论是表单提交场景优先 useActionState，非表单或高度定制流程可继续手写。原因是 Action 与 form action、FormData、pending 天然集成；简单本地状态或第三方表单库不一定需要 Action 模型。
