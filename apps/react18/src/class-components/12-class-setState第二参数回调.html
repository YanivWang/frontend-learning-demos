setState 第二参数回调与 state 读取时机

请打开 DevTools Console 对比 setState 前后与 callback 内的 count 值；点击下方按钮观察 DOM 更新提示区；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • this.setState(updater, callback) 的第二参数会在本次更新提交到 DOM 之后执行，因此可以读取最新 this.state ，也可以安全读取本组件已更新后的 DOM。
  • setState 是请求更新，不是同步赋值。React 会批处理多个更新，所以在同一个事件处理函数里调用后立刻读 this.state ，通常仍会看到旧值。
  • 如果下一步逻辑依赖"更新完成后"的状态或 DOM，可用第二参数 callback；如果要响应任意一次 props/state 变化并比较前后值，更适合写在 componentDidUpdate(prevProps, prevState) 。
  • 如果新 state 依赖旧 state，尤其是连续点击或同一事件内多次更新，应使用函数式 updater： this.setState(prev => ...) 。第二参数 callback 不能替代函数式 updater 的"基于最新队列状态计算"。
  • React 18 对更多来源的更新做自动批处理，class 维护代码里更不要依赖同步读取 state。确实需要同步刷新 DOM 的少数场景可了解 flushSync ，但本 demo 重点是 class 的 callback 语义。

面试考点

  • setState 第二参数 callback 什么时候执行？ 结论是更新被提交后执行。原因是 React 先把更新排队、计算并提交 DOM，再调用 callback；项目里常用于滚动到新元素、聚焦输入框或读取更新后的尺寸。
  • 为什么 setState 后同步读取 this.state 可能还是旧值？ 结论是 setState 不是立即修改实例字段。原因是 React 会批处理更新并在之后渲染提交；边界是不要把同步读取结果当作下一次更新依据。
  • callback、componentDidUpdate 和函数式 updater 如何选？ 结论是 callback 处理当前这次更新后的副作用， componentDidUpdate 处理跨多次更新的比较逻辑，函数式 updater 处理基于旧 state 计算新 state。原因是三者介入的阶段不同；连续自增必须用函数式 updater。
  • React 18 自动批处理对 class 读取 state 有什么影响？ 结论是异步来源中的更新也更可能被批处理，代码更不应依赖同步读到新 state。维护旧 class 组件时，读取最新值放在 callback 或生命周期里更稳。
  • 连续点击或连续 setState 时如何避免丢更新？ 结论是用函数式 updater，不要用对象式快照。原因是同批次多次对象式 setState({ count: this.state.count + 1 }) 都基于同一旧值；callback 只保证"本次更新提交后"的执行，不能串联多次更新的计算逻辑。
