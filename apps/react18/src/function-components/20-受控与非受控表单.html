受控组件与非受控组件

请打开 DevTools Console 查看提交数据输出；下方 #root 为 React 挂载演示区。

知识点要点

  • 受控组件把表单当前值保存在 React state 中， value / checked 与 onChange 成对出现；页面展示和业务状态始终来自同一数据源。
  • 非受控组件让 DOM 自己保存输入值，React 只用 defaultValue 设置初始值，并在提交或需要时通过 ref 读取。
  • 需要实时校验、联动禁用按钮、展示错误提示、根据输入发请求时，受控组件更容易维护；一次性备注、文件选择或集成第三方表单库时，非受控也很常见。
  • value 传了字符串但没有同步更新的 onChange ，输入框会被锁住；从 undefined 切到字符串也可能触发受控/非受控切换警告。
  • 用户输入必须按文本处理。React 默认会转义文本插值，但提交前仍要做空值校验、长度限制和服务端校验；非受控 ref 也要防止节点为空。

面试考点

  • 受控组件和非受控组件的核心区别是什么？ 结论是数据源不同：受控值来自 React state，非受控值来自 DOM。原因是受控组件每次输入都走 state 更新，非受控组件只在需要时读取 DOM 当前值。项目里可校验、联动、回显的业务字段通常受控，简单一次性字段可以非受控。
  • 为什么 value 没有 onChange 会让输入框不可编辑？ 结论是 React 会把输入框值固定为传入的 value 。原因是用户输入后如果 state 不更新，下一次渲染仍把旧值写回 DOM。边界是只读字段应显式加 readOnly ，可编辑字段必须更新 state。
  • 业务表单为什么通常优先受控？ 结论是受控组件更容易做实时校验、联动和提交状态管理。原因是 React state 持有完整表单快照，错误提示、禁用按钮和派生字段都能同步计算。项目里大型表单可借助表单库优化渲染，但核心仍是明确数据源。
  • 非受控表单提交时要注意哪些失败路径？ 结论是要处理 ref 为空、空字符串、长度限制和服务端校验失败。原因是 DOM 当前值绕过了 React state 的实时约束。示例里空用户名会显示错误，备注从 ref 读取前也要确认节点存在。
