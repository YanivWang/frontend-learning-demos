Import Maps 模块映射

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • <script type="importmap"> 提供一张模块说明符映射表，让浏览器把 import "utils/greet.js" 解析到真实 URL；本页用 utils/ 前缀映射到本地目录。
  • imports 既可以做精确映射，也可以用以 / 结尾的 key 做路径前缀映射；映射后的资源仍按 JavaScript module 加载，必须满足同源/CORS、正确 MIME 类型和模块语法要求。
  • import map 必须在依赖它的 <script type="module"> 执行前出现；已经解析过的模块不会因为后续 import map 变化而重新解析。
  • 它不负责转译、tree-shaking、代码分割、压缩和旧浏览器兼容，因此大型生产应用通常仍用 bundler；Import Maps 更适合 demo、微前端依赖钉住、CDN 依赖别名和无构建渐进迁移。
  • 常见坑是把 Import Maps 当作 Node 解析或 npm 安装替代品：浏览器只按 URL 加载文件，不会自动寻找 package.json、目录入口或扩展名。

面试考点

  • Import Maps 如何让浏览器识别 bare specifier？ 结论是通过 imports 把非 URL 形式的模块说明符映射成绝对或相对 URL。原因是浏览器原生 ESM 不会像 Node 或 bundler 一样查找 npm 包入口，只能加载可解析的 URL。项目里可用它给 CDN 依赖、本地工具模块或微前端共享依赖做显式别名。
  • importmap 为什么必须在 module 脚本前声明？ 结论是依赖映射的模块解析前必须已经有 import map。原因是模块图在加载时就会解析说明符，迟到的映射无法回头修改已解析模块。实际页面里应把 import map 放在第一个相关 type="module" 脚本之前。
  • Import Maps 与 bundler 在生产项目里如何分工？ 结论是 Import Maps 解决运行时模块说明符到 URL 的映射，bundler 解决构建期优化和兼容。原因是 Import Maps 不做转译、压缩、tree-shaking 或代码分割。生产中可在现代浏览器、微前端或无构建页面里使用，但仍要评估浏览器支持、缓存策略和回退方案。
