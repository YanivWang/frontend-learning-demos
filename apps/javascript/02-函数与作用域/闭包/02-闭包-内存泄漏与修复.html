闭包导致的内存泄漏与修复手段

请阅读本复习区，并在下方演示区点击按钮观察 Console 输出；4 秒后事件监听会自动解绑。

知识点要点

  • 闭包本身是正常机制，不等于内存泄漏。只有当闭包已经不再有业务价值，却仍被全局变量、定时器、事件监听器或缓存持有时，它引用的外层对象才会持续可达。
  • 示例中 outer 返回的 inner 引用了大数组 arr；把 inner = null 后，如果没有其他引用，闭包链和数组就可以等待垃圾回收。
  • 非严格模式下未声明变量赋值会创建全局属性，导致对象生命周期被拉长；使用严格模式、ES modules 或构建工具默认严格环境能减少这类问题。
  • 定时器和事件监听器经常持有回调，回调又持有组件状态、DOM 或大对象。页面切换、组件卸载、弹窗销毁时要 clearInterval、removeEventListener 或使用框架生命周期清理。
  • 现代垃圾回收能处理不可达的循环引用；真正需要关注的是「从根对象仍然可达」的引用链。排查时可用浏览器 Memory 面板查看 Retainers。

面试考点

  • 闭包一定会导致内存泄漏吗？ 先结论：不会，闭包只是让函数保留外层环境。原因是只要闭包仍有用，保留变量就是正确行为；只有无用闭包仍从根对象可达时才是泄漏。项目里常见于组件卸载后遗留的回调、全局缓存和未停止的轮询。
  • 为什么把闭包变量置为 null 能帮助释放内存？ 先结论：它断开了当前变量到闭包函数的强引用。原因是垃圾回收按可达性判断，对象不可达后才有机会回收。边界是如果同一个闭包还被别的变量、DOM 监听器或定时器持有，单独置空一个引用没有效果。
  • 定时器和事件监听为什么容易和闭包一起造成泄漏？ 先结论：宿主环境会保存回调，回调闭包又保存外层状态。原因是 setInterval 不清理会一直运行，DOM 监听器不解绑会保留监听函数。项目中在 React effect 清理函数、Vue 卸载钩子或原生页面销毁逻辑中集中释放。
  • 前端项目里如何定位和修复闭包相关内存问题？ 先结论：先复现增长，再找可达引用链，最后释放生命周期不匹配的引用。原因是内存泄漏本质是对象仍被 Retainers 持有。实践中用 Chrome Memory 快照比较、Performance monitor 观察趋势，并修复 timer、listener、全局数组和缓存。
