并发渲染与 Transitions 面试

下方 #root 有两组演示：先试「useTransition」输入框，再试「useDeferredValue」输入框；快速输入数字（如 12）观察输入即时回显、列表滞后与 isPending；请打开 DevTools Console 查看日志。

知识点要点

  • 并发渲染不是把 React 变成多线程，而是让渲染工作可被中断、恢复或丢弃；用户输入这类紧急更新可以先提交，列表筛选、路由切换这类过渡更新可延后。
  • startTransition 标记低优先级更新，useTransition 在此基础上返回 isPending；本页第一组演示把 query 紧急更新、deferredQuery 放进 transition。
  • useDeferredValue(value) 返回值的延迟版本，适合「已有 state，派生 UI 想延后」的场景；本页第二组只需一个 query，列表消费 useDeferredValue(query)，无需手写第二个 state。
  • 版本边界：React 18 已提供并发根和 useTransition；React 19 的 Actions、表单 Action、useOptimistic 等常和 transition 一起解释异步交互状态；createRoot 是并发前提，但并发更新需 opt-in（transition / deferredValue）。
  • Transition 不会减少慢计算本身的耗时，也不会替代防抖、虚拟列表、缓存或 Web Worker；如果渲染或计算太重，仍要降低工作量。
  • 受控输入的 value 更新必须保持紧急，不能只放进 transition；否则输入会滞后，甚至出现用户输入和 UI 回显不一致的体验问题。

面试考点

  • 并发渲染是不是多线程？它真正改变了什么？ 结论是不是多线程，主要改变的是 React 调度渲染工作的方式。原因是 React 可以把非紧急渲染拆开、暂停并让更高优先级输入先更新。边界是业务代码仍运行在 JS 主线程，重计算仍会阻塞，需要单独优化。
  • startTransition 与普通 setState 的优先级差异是什么？ 结论是普通更新默认紧急，transition 更新可被打断。原因是 React 认为输入回显、点击反馈应优先，列表筛选或路由内容可以稍后显示。项目中常把搜索关键字立刻更新，把结果列表、图表或路由内容放进 transition。
  • useTransition、useDeferredValue 和防抖分别适合什么场景？ 结论是 useTransition 用在你能控制更新发起点时，useDeferredValue 用在延后消费某个值时，防抖用于减少事件或请求次数。原因是前两者是 React 调度语义，防抖是业务频率控制。输入联想通常会结合防抖请求和 transition 渲染结果。
  • 为什么 transition 不能包住受控输入本身？ 结论是输入框的 value 更新必须保持同步响应。原因是用户输入属于紧急交互，延迟回显会破坏受控组件体验。正确做法是立刻更新输入值，再把昂贵的派生列表或页面内容放进 transition。
