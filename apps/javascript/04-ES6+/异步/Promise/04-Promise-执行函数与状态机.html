Promise 的执行函数（executor）与状态机

请打开 DevTools Console 查看输出。

知识点要点

  • new Promise(executor) 创建 Promise 时会立刻同步调用 executor；executor 不是等到 then 注册后才执行。
  • Promise 状态机只有三种状态：pending、fulfilled、rejected。resolve 和 reject 只能把 pending 推到终态。
  • 状态一旦 settled 就不可逆，后续再次 resolve 或 reject 会被忽略；这保证同一个异步结果不会被消费多次或前后矛盾。
  • executor 内同步抛错会等价于 reject(error)；但 setTimeout 等异步回调里的异常不会被 Promise 构造器自动捕获，应在回调里显式 try...catch 后 reject。
  • 只有在把回调型 API 转成 Promise 时才手写 new Promise；如果已有 Promise，直接 return 或组合它，避免 Promise constructor antipattern。

面试考点

  • Promise executor 什么时候执行，then 回调什么时候执行？ executor 在构造 Promise 的当下同步执行，then/catch 回调在状态 settled 后作为微任务执行。项目里不要把 executor 当懒执行函数；想懒执行应封装成“返回 Promise 的函数”。
  • Promise 状态为什么一旦 settled 就不能再改变？ 结论是 fulfilled 或 rejected 后不可逆，后续决议会被忽略。原因是 Promise 表示一次异步结果，消费者需要稳定的一致状态；否则多个回调可能看到互相冲突的结果。
  • executor 里同步 throw 和异步回调 throw 有什么区别？ executor 同步 throw 会被 Promise 构造器捕获并转成 rejection；异步回调已经脱离 executor 调用栈，throw 不会自动变成当前 Promise 的 rejection。实际包装回调 API 时，应在异步回调内部捕获异常并调用 reject。
  • 什么时候应该手写 new Promise？ 当你要把事件、计时器、Node 风格回调等非 Promise API 包成 Promise 时才手写。已有 Promise 的场景直接返回、链式组合或使用 async/await，否则容易多包一层并制造错误处理盲区。
