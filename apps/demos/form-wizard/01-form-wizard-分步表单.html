分步表单向导与逐步校验

这个 demo 用原生 DOM 实现三步表单，重点观察：step 如何驱动视图、每步如何校验、汇总页如何安全展示用户输入。

知识点要点

  • 状态拆分： step 只表示当前处于第几步， data 保存已填写的业务值。两者分开后，返回上一步不会丢数据，也不会把 DOM 当成唯一数据源。
  • 逐步校验： 点击下一步时先把当前字段写回 data ，再用表单控件的 checkValidity() / reportValidity() 触发 HTML 约束校验；不通过就停留在当前步骤。
  • 失败路径： 空姓名、非法邮箱、空城市都应该阻止继续；真实项目还要处理接口失败、重复提交、刷新恢复、离开页面前确认等边界。
  • 安全展示： 汇总页展示的是用户输入，示例用 textContent 写入 pre ，避免把 <img onerror=...> 这类字符串当 HTML 解析。
  • 取舍： 分步表单适合字段多、需要分阶段确认的流程；字段少或强依赖上下文时，普通长表单往往更直接，减少步骤切换成本。

面试考点

  • 分步表单为什么要把 step 和表单数据分开管理？ 结论是 step 控制视图，data 保存业务状态。这样返回上一步、重新渲染或最终汇总都能从同一份数据恢复，不依赖临时 DOM。边界是复杂表单还需要草稿保存、脏状态提示和跨步骤依赖校验。
  • 下一步按钮为什么必须先做当前步骤校验？ 结论是不能让非法数据进入后续步骤。原因是后续步骤可能依赖前一步值，提前拦截能减少错误传播；项目里通常同时做前端约束校验和服务端最终校验。
  • 如何避免汇总页把用户输入当 HTML 渲染？ 展示纯文本时用 textContent 或创建文本节点，不把输入拼到 innerHTML 。原因是 innerHTML 会解析字符串，用户输入可能变成 DOM 注入；只有可信且净化后的 HTML 才适合渲染为标记。
  • 分步表单和普通长表单如何取舍？ 结论是字段多、流程性强、每步有独立目标时用分步表单；字段少或需要整体对照时用长表单。边界是分步会增加导航、状态恢复和校验设计成本。
  • 真实项目里分步表单还要补哪些失败路径？ 至少要处理空输入、格式错误、接口失败、重复点击提交、刷新后恢复和离开页面确认。这个 demo 只演示前端约束校验和汇总展示，真实提交仍需后端兜底。
