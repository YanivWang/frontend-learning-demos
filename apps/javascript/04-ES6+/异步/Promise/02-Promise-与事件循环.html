Promise + async/await 与事件循环（宏任务/微任务）

请打开 DevTools Console 查看输出。

知识点要点

  • new Promise(executor) 会立刻同步执行 executor，所以示例里的 console.log("1")、console.log("2") 会早于 then 回调。
  • then、catch、finally 这类 Promise 反应回调会进入微任务队列，等当前同步调用栈清空后执行。
  • await 会计算右侧表达式并暂停 async 函数，后续代码作为恢复执行的微任务排队；即使 await 的是已 fulfilled Promise 或普通值，也会发生这次让出。
  • 浏览器事件循环在一个任务结束后会先处理微任务，再取下一个任务；因此 Promise/await 回调通常早于 setTimeout(..., 0)。
  • 过度拆分 await 会制造更多微任务切换，可能让两个流程交错执行并引入竞态；只有后续代码确实依赖结果时才需要 await。

面试考点

  • Promise executor、then、setTimeout 的执行顺序如何判断？ 先跑当前同步代码，包括 Promise executor；再清空 Promise/await 产生的微任务；最后才进入 setTimeout 这类后续任务。项目排查“为什么日志顺序不对”时，先把同步、微任务、任务三类分开标记。
  • await 后面的代码为什么会晚于同步代码执行？ 因为 await 会让 async 函数暂停，恢复执行被排成微任务。即使等待的是普通值，也不会继续留在当前同步调用栈里执行；这就是 async 函数能让出控制权的原因。
  • 微任务和宏任务在浏览器事件循环中如何调度？ 每轮事件循环取一个任务执行，任务结束后执行微任务检查点，清空当前可执行微任务，再进入渲染机会和下一任务。边界是微任务如果持续追加微任务，可能推迟计时器和渲染。
  • 为什么不建议滥用不必要的 await？ 多余的 await 会把代码拆成更多微任务，增加调度成本，也可能让并发流程交错产生竞态。只需要返回 Promise 给调用方时，可以直接 return；需要在当前函数捕获错误或改善栈信息时再 return await。
