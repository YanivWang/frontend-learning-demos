Vue2 与 Vue3 差异

在浏览器中操作下方交互演示；复习要点与面试答法见本复习区。

知识点要点

  • 本页点表格行可高亮当前话题；下方展示对比维度。响应式是最常被问到的底层差异：Vue2 主要依赖 Object.defineProperty ，对新增属性、删除属性、数组索引等场景有历史限制；Vue3 基于 Proxy 与 Reflect ，追踪对象操作更完整（深讲见 01-Proxy响应式原理 ）。
  • API 风格不是“Vue3 只能写 Composition API”。Vue3 仍支持 Options API，但 Composition API 更适合按业务能力组织逻辑、抽取 composable，并改善大型组件中 data/methods/watch 分散的问题。
  • createApp() 改变了应用实例边界。插件、全局组件、mixin、config 等应用级配置挂在 app 上，不再像 Vue2 的全局 Vue.use() 那样容易污染多个根实例。
  • 模板与组件能力也有明显变化：Vue3 支持 Fragment 多根节点、 Teleport 、 Suspense （实验性），组件 v-model 默认改为 modelValue / update:modelValue （见 05-script-setup编译宏 ）。
  • 点击不同行时 activeTopic 会更新，便于对照面试答法。现代选型上，新项目通常优先 Vue3；维护 Vue2 老项目时要结合生态依赖、IE 兼容、构建链和迁移成本判断，不能只把 API 名称机械替换。

面试考点

  • 面试中如何概括 Vue2 到 Vue3 的差异？ 结论是按响应式、API 组织、应用创建、模板能力和生态迁移五条线讲最清楚。原因是这些变化分别影响底层更新、代码组织、全局配置隔离、组件表达能力和实际升级成本。项目场景里还要补充当前是新项目选型还是老项目渐进迁移。
  • Vue3 的 Composition API 是不是替代 Options API？ 结论是不是，Vue3 保留 Options API，Composition API 是新增的逻辑组织方式。原因是 Options API 对简单页面仍直观，而 Composition API 更适合复用状态逻辑、类型推导和大型组件拆分。团队迁移时可以在 Vue3 中混用，但要避免一个组件里无规划地混杂两套组织方式。
  • createApp 相比 new Vue 改进了什么？ 结论是它把应用级能力挂到独立 app 实例上，减少全局污染。原因是插件、全局组件和配置不再修改同一个全局 Vue 构造器。多应用挂载、微前端或测试隔离场景下，这个边界比 Vue2 更清晰。
  • Vue2 项目迁移 Vue3 时最容易踩哪些兼容性问题？ 结论是不要只看响应式，还要核对 v-model 、事件/attrs 透传、过滤器移除、生命周期命名、插件版本和构建工具链。原因是 Vue3 有明确的破坏性变更，很多问题出在组件契约和生态包上。实际项目通常先跑迁移构建或兼容版本，再分模块替换高风险 API。
