瀑布流布局（CSS column-count，无需 JS）

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • column-count 来自 CSS 多列布局，它把一个内容流分成多列；column-gap 控制列间距，本页用媒体查询在窄屏减少列数。
  • break-inside: avoid 告诉浏览器尽量不要把单个卡片拆分到两列之间，适合图片卡片、商品卡片和信息流块级内容。
  • 多列布局实现的是“内容按列流动”的瀑布流近似效果，视觉上会形成高低错落；但它不是 JS 瀑布流常见的“每次放到当前最短列”算法。
  • CSS 方案优点是零脚本、响应式成本低、DOM 顺序保持不变；缺点是视觉阅读顺序可能按列而不是按行，键盘/读屏顺序仍按 DOM 走，需要产品上接受这种顺序差异。
  • 动态图片高度、无限滚动、懒加载、拖拽排序和虚拟列表场景通常需要 JS 计算或专门布局库；CSS 多列更适合静态内容墙和简单营销页。

面试考点

  • column-count 瀑布流的原理是什么？ 结论是利用 CSS 多列布局把同一内容流分成多列，卡片高度不同后形成类似瀑布流的视觉。原因是浏览器会负责列宽、列间距和内容分片。边界是它不是最短列优先算法，不能精确控制每张卡片落在哪一列。
  • break-inside: avoid 能解决什么问题？ 结论是它能提示浏览器尽量不要在卡片内部断列。原因是多列布局本来允许内容跨列流动。边界是这是避免断裂的布局约束，不是强制瀑布流算法，特别高的内容、打印分页和不同浏览器仍要验证。
  • CSS 多列瀑布流和 JS 算法怎么选？ 结论是静态或轻量内容墙优先 CSS，多数据、排序、懒加载、虚拟化和精确列高控制选 JS。原因是 CSS 成本低但控制力有限，JS 能按最短列或业务规则放置元素。项目里还要考虑图片加载后高度变化导致的重排。
  • 为什么瀑布流要关注可访问性顺序？ 结论是视觉顺序和 DOM 顺序可能不完全一致。原因是多列布局会按列呈现内容，但键盘 tab、读屏和搜索引擎通常仍依据 DOM。业务上如果卡片顺序表达时间线或优先级，要确认这种视觉排列不会误导用户。
