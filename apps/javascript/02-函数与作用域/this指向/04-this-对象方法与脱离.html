方法被「赋值出去」后 this 的脱离

请打开 DevTools Console 查看输出。

知识点要点

  • 调用位置决定 this：obj.fn() 是「通过对象属性引用调用」，所以 this 指向 obj。
  • 方法脱离：const fn = obj.fn; fn() 只保留函数值，不保留原来的 base object，因此会按普通函数调用规则绑定 this。
  • 常见场景：把方法传给 setTimeout、事件监听、Promise 回调、数组方法，或从对象中解构方法，都可能让 this 脱离原对象。
  • 修复方式：bind(obj) 适合提前固定回调；() => obj.fn() 适合保留最新对象引用；也可以把方法改成接收参数，减少对 this 的依赖。
  • 严格模式边界：非严格普通函数脱离后 this 可能是全局对象，严格模式下是 undefined，所以错误表现不完全一样。

面试考点

  • 为什么 obj.fn() 有 this，而 const fn = obj.fn; fn() 会丢 this？ 结论是 this 不是函数定义时固定的，而是普通函数调用时由调用表达式决定。obj.fn() 有对象作为 base，赋值后只剩函数值，原对象信息丢失。
  • 哪些真实场景容易发生 this 脱离？ 事件回调、定时器、Promise 回调、数组迭代回调、解构方法和把类方法传给子组件都常见。边界是箭头函数没有自己的 this，它会捕获外层词法 this。
  • bind 和箭头函数解决 this 脱离有什么区别？ bind 创建一个固定 this 的新函数，适合稳定回调；箭头包装 () => obj.fn() 每次执行时重新读取 obj.fn。项目里要看是想固定当前函数，还是想保留对象的最新方法。
  • 解构对象方法时为什么要警惕 this？ const { fn } = obj 和赋值给变量一样，会丢掉对象调用上下文。若方法内部读 this，解构后直接调用可能报错或读到错误对象。
  • 如何从设计上减少 this 依赖？ 优先把依赖作为参数传入，或使用闭包保存状态；类组件/面向对象代码中再用 bind、类字段箭头函数或构造器绑定。这样回调传递时更不容易出错。
