ES Module import/export 静态导入与动态导入

请打开 DevTools Console 查看输出。 需通过 http(s) 本地服务访问 （如 npm run dev / VitePress）， file:// 无法加载模块。

知识点要点

  • <script type="module"> 会把脚本按模块解析：自动启用严格模式，默认延迟执行，并且每个模块只会按 URL 实例化与执行一次。
  • 静态 import 只能出现在模块顶层，导入的是只读 live binding；导出方更新绑定后，导入方能观察到新值，但不能直接给导入名重新赋值。
  • 动态 import() 是表达式并返回 Promise，可放在事件回调、条件分支或懒加载路径里；失败时要按异步错误处理，不能假设网络与模块解析一定成功。
  • 浏览器按 URL 解析模块说明符，相对路径通常要写完整文件名（如 ./moduleA.js ）；裸模块名需要 import maps 或构建工具处理。
  • 直接用 file:// 打开本页会被浏览器模块加载安全策略拦截；本地学习应通过 Vite/VitePress 或静态 HTTP 服务器访问。

面试考点

  • ES Module 和 CommonJS 的核心区别是什么？ 结论是 ESM 以静态声明为主，导入 live binding，适合浏览器原生加载与构建期分析；CommonJS 以运行时 require 为主，常见导出值是对象属性读取。边界是 Node.js、打包器会提供兼容层，但浏览器原生 ESM 仍按 URL 与 CORS 规则加载。
  • 静态 import 与动态 import() 如何选？ 默认用静态导入，因为依赖关系清晰，便于预加载、类型检查和 tree shaking；需要按路由、权限或用户操作拆包时再用动态导入。项目里动态导入必须处理 Promise reject，例如模块路径错误、网络失败或被 CSP 拦截。
  • 为什么本页不能直接用 file:// ？ 浏览器模块脚本按安全模型从 URL 拉取依赖， file: 下相对模块加载常被禁止或行为不一致。实际开发应通过本地 HTTP 服务访问，并把模块路径、MIME 类型和跨域头配置正确。
