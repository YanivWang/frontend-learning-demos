UMD 插件模式与多模块环境兼容

请打开 DevTools Console 查看 Record 插件在浏览器全局模式下的导出与调用。

知识点要点

  • UMD（Universal Module Definition）不是 ECMAScript 标准语法，而是一种库发布包装模式：外层立即执行函数接收 root 和 factory ，按环境把 factory() 的返回值交给 AMD、CommonJS 或浏览器全局变量。
  • 本页的 plugin.js 在普通 <script> 中会走浏览器全局分支，把插件暴露为 Record ；如果运行在 AMD 加载器中会调用 define(factory) ，在 CommonJS 环境中会写入 module.exports 。
  • UMD 的价值是兼容老式加载器和直接 script 引入；现代新项目优先使用 ESM 的 import / export ，库通常通过 exports 字段同时提供 ESM、CJS 或 UMD 构建产物。
  • 工程局限 ：环境探测过宽时，页面上存在 AMD loader 会优先进入 AMD 分支；严格模式、ESM 或 worker 场景里 this 未必是 window ，更稳妥的模板应使用 globalThis 或由打包器生成。UMD 无法获得 ESM 的静态分析、tree-shaking 和异步模块图能力。
  • 浏览器全局分支会占用全局命名空间，真实项目应使用明确命名、版本化构建和无副作用初始化，避免覆盖已有变量或在加载脚本时立即执行重逻辑。

面试考点

  • UMD 为什么能让同一份插件支持多种模块系统？ 结论是 UMD 把业务代码放进 factory，再由外层包装根据当前环境选择导出方式。原因是 AMD 使用 define，CommonJS 使用 module.exports，普通浏览器脚本只能挂到全局对象。边界是它只能兼容加载方式，不能让代码自动获得 ESM 的静态分析、tree-shaking 和异步模块图能力。
  • UMD 与 ESM、CommonJS 怎么选？ 新业务代码和现代库入口优先选 ESM，因为语法标准化、可静态分析，也更利于构建优化。CommonJS 主要用于 Node.js 旧生态或兼容包；UMD 适合需要 CDN script 标签、AMD 加载器或老项目同时消费的发布产物。项目里通常源码写 ESM，再由构建工具输出多格式。
  • 为什么常见 UMD 模板先判断 define.amd？ 结论是为了在存在 AMD 加载器的浏览器页面中走异步模块注册，而不是误判成全局变量模式。原因是 AMD 的 define 是显式加载器协议，CommonJS 分支通常面向 Node 或打包环境。边界是不同模板检测顺序可能不同，库作者必须测试目标运行环境。
  • 浏览器全局模式有哪些风险？ 结论是全局导出最容易发生命名冲突和加载顺序问题。原因是所有 script 共享同一个全局命名空间，后加载脚本可能覆盖前面的对象。项目中应给全局变量取稳定前缀、避免加载时副作用，并在文档中明确 UMD 包只面向兼容场景。
