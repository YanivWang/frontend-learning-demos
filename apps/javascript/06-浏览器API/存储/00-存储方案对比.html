浏览器存储方案对比

阅读本复习区后，在下方表格对照四种方案，并用按钮写入/读取/清理观察实际行为。

知识点要点

  • Cookie 的核心特点是会随匹配 Domain/Path 的请求自动发送，适合服务端需要读取的少量状态；登录态应由服务端通过 Set-Cookie 配置 HttpOnly、Secure、SameSite，前端脚本不能设置 HttpOnly。
  • sessionStorage 以标签页为生命周期，关闭页面后释放；localStorage 持久到手动清理或浏览器策略清理，二者都是同步字符串 key-value API。
  • localStorage / sessionStorage 适合少量偏好、草稿和非敏感状态，不适合大对象、高频写入或敏感 token；大量同步读写会阻塞主线程，隐私模式还可能直接抛错。
  • IndexedDB 是异步、事务型、同源隔离的浏览器数据库，可存结构化对象和 Blob，更适合离线数据、大文件缓存、复杂索引；localforage 这类库通常是在它之上提供 Promise 封装和降级。
  • 浏览器存储没有“永久安全”的承诺：配额、驱逐策略、第三方上下文限制、用户清理和私密浏览都会影响可用性，业务写入必须捕获 QuotaExceededError、SecurityError 等失败路径。
  • 运行环境： localStorage / sessionStorage / Cookie 在 file:// 与 http(s):// 一般可用；隐私模式或禁用存储时可能抛 SecurityError。本页不涉及需 HTTPS 的持久化 API（如 navigator.storage.persist()）。

面试考点

  • 登录态、偏好设置和离线缓存分别选哪种存储？ 结论是登录态通常用服务端设置的安全 Cookie，偏好设置可用 localStorage，离线结构化数据优先 IndexedDB。原因是它们分别对应自动随请求发送、前端同步读取、异步大容量存储三类需求；边界是敏感 token 不应放进可被 XSS 读取的 Web Storage。
  • localStorage 和 sessionStorage 生命周期有什么区别？ 结论是前者跨会话持久，后者限定在当前页面会话。原因是 sessionStorage 与标签页绑定，关闭标签页后释放；项目里多标签同步状态要用 localStorage 的 storage 事件或服务端状态，而不是指望 sessionStorage 共享。
  • 为什么不建议把大量数据写进 localStorage？ 结论是它是同步 API，容量和性能都不适合大数据。原因是序列化和读写会占用主线程，还只能存字符串；项目里的离线列表、图片、文件和复杂对象应改用 IndexedDB 或 Cache Storage。
  • Cookie 的 Secure、HttpOnly、SameSite 分别解决什么问题？ 结论是 Secure 限制 HTTPS 发送，HttpOnly 阻止脚本读取，SameSite 降低跨站请求携带 Cookie 的风险。原因是 Cookie 会自动随请求发送；认证场景必须按威胁模型配置这些属性。
  • 浏览器存储写入失败或被清理时如何兜底？ 结论是把存储视为可失败缓存。原因是配额、隐私模式、用户清理和浏览器驱逐都可能发生；项目里写入要 try/catch，读取要能从服务端恢复或重新初始化。
