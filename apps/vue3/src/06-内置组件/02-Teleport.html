Teleport

在浏览器中操作下方交互演示；复习要点与面试答法见本复习区。

知识点要点

  • Teleport 可以把组件模板的一部分移动到指定 DOM 容器渲染；本页弹窗逻辑写在 #app 内，但实际 DOM 被挂到 body 下。
  • 它只改变真实 DOM 插入位置，不改变 Vue 组件逻辑层级，所以弹窗仍能读取当前组件的 open 状态并触发原组件中的关闭逻辑。
  • 弹窗、抽屉、通知、全局浮层常用 Teleport，因为它们容易受父级 overflow 、 transform 、层叠上下文和定位规则影响。
  • to 指向的目标容器必须存在；多个 Teleport 指向同一容器时会按挂载顺序追加，需要额外管理 z-index、焦点、滚动锁定和关闭交互。
  • 本页是最小交互示例，真实弹窗还应处理 Esc 关闭、焦点陷阱、背景不可滚动、ARIA 角色以及点击遮罩的边界；Vue2→Vue3 能力对比见 08-Vue2与Vue3差异 。

面试考点

  • Teleport 改变的是 DOM 位置还是组件父子关系？ 结论是只改变 DOM 位置，组件父子关系和响应式上下文不变。原因是 Vue 仍在原组件树中管理这段模板，只是在渲染阶段把节点插入目标容器。边界是 CSS 选择器、布局和事件冒泡会按真实 DOM 位置表现。
  • 为什么弹窗常 Teleport 到 body？ 因为弹窗需要脱离局部容器的 overflow、定位和层叠上下文限制。挂到 body 后更容易统一处理遮罩、z-index 和全局浮层栈。项目里仍要补充滚动锁定、焦点管理和可访问性，不能只靠 Teleport。
  • Teleport 的 to 目标不存在怎么办？ 结论是目标容器必须在挂载时可用，否则渲染会失败或产生警告。真实项目通常准备固定的 #modal-root ，或在目标未就绪时延迟渲染。SSR 场景还要保证服务端和客户端结构可匹配。
  • Teleport 和 CSS fixed / z-index 的区别是什么？ CSS 只能尝试在当前位置调整视觉层级，Teleport 直接改变节点所在容器。遇到父级 transform、overflow hidden 或复杂 stacking context 时，Teleport 更可靠。但它不替代浮层状态管理和交互细节。
