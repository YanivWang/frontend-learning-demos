PureComponent 与 shouldComponentUpdate 浅比较

打开控制台点击“切换无关 state”和“Alice +1”，比较普通 Component、PureComponent 与手写 shouldComponentUpdate 的 render 日志。

知识点要点

  • 基本机制：React.PureComponent 继承自 Component，但内置了基于浅比较的 shouldComponentUpdate；当新旧 props 和 state 的浅层值都相等时，React 会跳过该组件的 render。
  • 浅比较边界：原始值按值比较，对象、数组、函数按引用比较。示例中切换 theme 不改变每个 user 对象引用，所以 PureRow 和 ManualRow 可以跳过无关渲染。
  • 不可变更新前提：如果原地修改 user.score 再复用旧对象引用，浅比较会认为 props 没变，PureComponent 可能不重新渲染；因此配套写法是用 map、展开语法或不可变数据结构创建变化路径上的新引用。
  • 与手写 SCU 的区别：PureComponent 适合“渲染只依赖浅层 props/state”的组件；手写 shouldComponentUpdate 可以只比较特定字段，但更容易漏比较、写错比较逻辑或和新增 props 不同步。
  • 现代替代：函数组件里的相近能力是 React.memo；在 React 18 新代码中通常先保证组件纯渲染和状态局部化，再用 Profiler 或实际性能证据决定是否加 memo/PureComponent。

面试考点

  • PureComponent 到底比较了什么，为什么能跳过 render？ 结论是它浅比较新旧 props 和 state。原因是如果组件渲染结果只由这些浅层输入决定，输入没变就没有必要重新计算 UI。边界是它不做深比较，嵌套对象内部变化必须通过新引用表达出来。
  • PureComponent 为什么要求不可变更新？ 结论是浅比较依赖引用变化识别对象和数组是否变了。因为原地修改会保留旧引用，PureComponent 可能误判“没变化”并跳过 render；不可变更新让被修改对象和沿途容器都产生新引用。项目里列表项、表格行、树节点这类优化尤其依赖这一点。
  • PureComponent 和手写 shouldComponentUpdate 怎么选？ 结论是简单浅层依赖用 PureComponent，复杂比较才手写。原因是 PureComponent 减少样板代码，但无法表达“只比较某几个字段”或特殊业务判断；手写逻辑更灵活，也更容易遗漏字段。维护时如果 props 结构变化频繁，手写比较要配套测试或谨慎复核。
  • PureComponent 与 React.memo 的关系是什么？ 结论是二者都是基于输入相等性跳过渲染的性能优化，PureComponent 用于 class，React.memo 用于函数组件。React 18 新代码通常优先函数组件加 memo，旧 class 维护可继续用 PureComponent。二者都不应替代正确的数据流设计。
