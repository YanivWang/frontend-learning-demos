CSS 重排与重绘

打开 DevTools Performance 或 Rendering 面板观察动画：示例用 transform/opacity 做运动，并对比容易触发布局或绘制的属性。

知识点要点

  • 浏览器渲染通常会经历样式计算、布局、绘制和合成等阶段。修改 width、height、margin、padding、top、left、font-size 等影响盒子几何信息的属性，往往需要重新布局，也就是常说的 reflow/layout。
  • 修改 color、background、box-shadow 这类视觉属性通常不改变元素占位，但仍可能让浏览器重新绘制像素。它比重排少了布局计算，但在大面积阴影、滤镜或复杂背景上仍可能很贵。
  • 高频动画优先使用 transform 和 opacity，因为它们通常可以在合成阶段更新，不必每帧重新计算文档流。示例中的蓝色圆点用 translateX() 和透明度变化演示了这种写法。
  • will-change 只是提前告诉浏览器某个属性即将变化，不是“性能开关”。长期给大量元素设置会提前分配资源、增加内存或显存压力，实践中应只在动画即将发生前添加，并在结束后移除。
  • contain: layout paint 可以把组件内部的布局和绘制影响限制在局部，适合复杂卡片、列表项或独立小组件；边界是它会建立隔离上下文，可能影响尺寸依赖、定位、溢出和可访问性关联，需要按组件验证。

面试考点

  • 重排和重绘的区别是什么，为什么重排通常更贵？ 结论是重排会重新计算元素几何和文档流位置，重绘主要更新像素外观。原因是布局变化可能影响祖先、后代和后续兄弟节点，传播范围比单纯绘制更大。项目里滚动、拖拽、动画等高频场景要避免每帧读写布局属性交错，必要时批量读写或使用 requestAnimationFrame。
  • 为什么动画优先使用 transform 和 opacity？ 结论是这两个属性通常更容易交给合成阶段处理，帧率更稳定。原因是位移、缩放和透明度变化不一定需要重新排版其他元素。边界是“通常”不等于绝对，滤镜、巨大图层、复杂混合或设备资源不足仍可能掉帧，所以要用 DevTools 实测。
  • will-change 为什么不能长期写在大量元素上？ 结论是它会让浏览器提前做优化准备，滥用反而增加资源占用。原因是浏览器可能提前创建图层或缓存绘制结果，元素越多成本越高。项目里应在交互前短期添加，比如 hover、进入视口或动画开始前，动画结束后清理。
  • contain: layout paint 适合解决什么性能边界？ 结论是它适合把复杂组件的布局和绘制影响限制在组件内部。原因是 containment 告诉浏览器外部不依赖内部布局细节，从而减少无关区域失效。边界是组件如果依赖内容撑开外部、跨容器定位或被外部查询尺寸，就要谨慎测试。

面试回答框架

容易重排：width、height、margin、padding、top、left、font-size
容易重绘：color、background、box-shadow
常用于高性能动画：transform、opacity
