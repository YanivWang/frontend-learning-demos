原生 HTML5 拖拽 API（drag & drop）

本页聚焦 Drag and Drop 事件全景（dragstart → dragover → drop），与下一页「带 UI 反馈的复制拖放」形成递进。侧栏自由拖拽见 javascript/06-浏览器API/拖曳手柄/（Pointer Events）。

知识点要点

  • 三个角色：HTML 拖放围绕拖动源、拖拽数据和放置目标工作。普通元素默认不可拖，必须设置 draggable="true" 并在 dragstart 中准备数据。
  • 允许放置：大多数元素默认不是 drop target，必须在 dragover 中调用 preventDefault()，浏览器才会继续派发 drop。
  • 数据边界：dataTransfer 适合传字符串或文件；本页用 text/plain 传源节点 id。对象数据要先序列化，且不要信任跨页面拖入的数据。
  • 事件目标陷阱：ev.target 可能是放置区里的子节点，追加副本时用 ev.currentTarget 才能稳定落到绑定监听的容器；高亮状态同样应操作 currentTarget。
  • 复制而非移动：本页克隆源节点，并移除 id、draggable 属性，避免重复 id 或让副本继续参与拖拽。

面试考点

  • HTML5 拖放的拖动源、传输数据和放置目标分别是什么？ 结论是拖动源负责发起拖拽，dataTransfer 保存本次拖拽携带的数据，放置目标决定是否接收。原因是浏览器需要一个统一模型支持页面内拖拽、拖出页面和从系统拖入页面。项目里要区分“移动 DOM”“复制数据”和“接收文件”三种需求。
  • 为什么 dragover 里必须 preventDefault 才能 drop？ 因为浏览器默认不允许任意元素成为放置目标。取消 dragover 默认行为等于声明当前元素可接收拖放；否则释放鼠标时会失败，也不会进入 drop 逻辑。
  • dataTransfer 什么时候能写入和读取？ 结论是通常在 dragstart 写入，在 drop 读取。MDN 明确说明拖拽数据存储在其他 drag 事件中不可随意修改或读取；项目里要提前约定 MIME type，例如 text/plain 或 text/uri-list。
  • drop 处理里为什么要优先使用 currentTarget？ 因为 target 是事件实际命中的内部节点，currentTarget 才是绑定处理函数的放置区。边界是如果用事件委托做多个放置区，还要结合 closest() 找到真实业务容器。
  • 原生拖放和 Pointer Events 手写拖拽如何取舍？ 原生拖放适合需要浏览器拖放语义、文件拖入或跨上下文数据传递的场景；Pointer Events 更适合移动端手势、自由拖动和自定义动画。HTML Drag and Drop 在触屏体验上常需要额外适配。
