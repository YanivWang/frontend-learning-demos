函数组件事件处理与参数传递

请打开 DevTools Console 查看事件日志；下方 #root 为 React 挂载/交互演示区，可点击按钮与删除项。

知识点要点

  • 函数组件没有实例 this，事件处理函数可以直接定义在组件内部或组件外部，不需要像 class 组件那样在 constructor 中 bind。
  • onClick={handleClick} 传入的是函数引用，点击时由 React 调用；onClick={handleClick()} 会在渲染阶段立即执行，通常会造成意外更新或无限渲染。
  • 需要传业务参数时，用箭头函数延迟调用：onClick={() => onRemove(id)}。需要同时拿事件对象时，可以写 onClick={(event) => handle(event, id)}，让参数含义清楚。
  • 子组件不直接修改父组件列表，而是在事件里调用父级通过 props 传入的回调。本 demo 的 TodoItem 删除按钮把 id 交回父组件，由父组件更新数组 state。
  • React 事件处理器接收的是 React 合成事件对象，常用字段与 DOM 事件一致。涉及阻止默认行为、异步使用事件对象时，应按 react.dev 当前事件文档处理（详见 24-合成事件机制）。

面试考点

  • 函数组件事件处理为什么不需要 bind this？ 结论是函数组件没有实例 this。原因是组件函数每次渲染被调用，事件处理函数通过闭包访问当前 props/state；维护 class 组件时才需要关注方法 this 绑定。
  • onClick={handle(id)} 和 onClick={() => handle(id)} 有什么区别？ 结论是前者渲染时立即执行，后者点击时执行。原因是 JSX attribute 需要函数引用；项目里传参、埋点、删除项等场景都应使用箭头包装或提前返回函数。
  • 同时需要事件对象和业务参数怎么写？ 结论是显式写 (event) => handle(event, id) 或 (event) => handle(id, event)，团队保持一致即可。原因是这样不会丢失事件对象，也不会在 render 阶段执行业务逻辑。
  • 为什么子组件通过回调通知父组件？ 结论是状态归谁所有，就由谁更新。原因是本例列表 state 在父组件，子组件只有 id 和回调；边界是深层组件传递回调太多时，可考虑 Context 或状态管理。
  • React 事件和 DOM 原生事件有哪些边界？ 结论是 React 使用合成事件统一跨浏览器行为，事件在 root 委托处理。原因是性能与一致性；边界是 stopPropagation 只影响 React 树、异步访问 event 需 event.persist()（legacy）或提前读取字段，详见 react.dev Responding to Events。
