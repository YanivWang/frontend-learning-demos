路由 props 解耦组件与 $route

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • props: true 会把 route.params 按字段名传给组件 props，组件可以像普通业务组件一样声明 props: ['id']，减少对 $route 的硬依赖。
  • props: { ... } 是静态对象模式，适合传固定开关或常量；它不会自动注入 params，所以示例中的 newsletterPopup 和 $route.params.id 是两个来源。
  • props: route => ({ ... }) 是函数模式，适合把 params、query 做类型转换、重命名或组合；函数应尽量保持无状态，因为它只在路由变化时求值。
  • 命名视图要写成 props: { default: true, sideBar: { ... } }，每个 router-view 独立接收 props，避免默认视图和侧栏组件拿错字段。
  • props 只解决“参数如何传进组件”；同一个路由记录只变 params 时组件实例会复用，详情页仍应 watch $route、watch prop，或用 beforeRouteUpdate 重新拉取数据。

面试考点

  • 为什么路由组件推荐用 props 接收参数？ 结论是用 props 能把组件从 $route 上解耦出来。原因是组件只关心 id、postid 这类业务输入，复用和单测时不必伪造完整路由对象。边界是需要读取导航来源、完整 query 或 meta 时，仍可以在页面容器层使用 $route。
  • props: true、对象模式、函数模式怎么选？ 结论是 params 同名直传用 true，固定配置用对象，query/params 转换用函数。原因是三种模式对应不同数据来源。项目里常用函数模式把字符串 id 转成数字，或把 query.q 重命名为组件更清晰的 keyword。
  • 命名视图的 props 为什么不能只写一个 true？ 结论是多视图应按 default、sideBar 等 view 名称分别配置。原因是一个路由记录可能同时渲染多个组件，每个组件的入参不同。边界是只有默认视图时才可以直接写布尔、对象或函数。
  • props 解耦后还要处理 params 变化吗？ 结论是要处理。原因是 /user/1 到 /user/2 可能复用同一个组件实例，不会重新走 created。项目里详情页应 watch prop 或 $route，也可以用 beforeRouteUpdate 发起新请求和处理失败态。
