RSC 与 Server Components 面试边界

本页为概念向速记，下方 #root 展示 Server / Client 对照与代码示例；静态 HTML + UMD 无法真正运行 RSC 或 Server Functions，无需 Console 验证运行时。

知识点要点

  • Server Components 在服务端或构建阶段运行，渲染结果通过 RSC payload 传给客户端，默认不会把组件代码打进浏览器 bundle；适合数据读取、Markdown/模板拼装和减少客户端 JS。
  • Client Components 用文件顶部 'use client' 标记客户端入口，里面可以使用状态、Effects、事件处理和浏览器 API；该边界以下被纳入客户端 bundle。
  • RSC 边界要求传给客户端的 props 可序列化，不能直接传普通函数、数据库连接、DOM 节点等运行时对象；需要交互时通常把交互组件下沉到客户端边界。
  • React 19 稳定了 RSC 相关 API 给框架使用，但本仓库是静态 HTML + 本地 React 运行时，只能演示概念清单，不能真正执行 RSC、Server Functions 或流式 payload。
  • Server Actions 是服务端函数调用通道，客户端表单 Action 是浏览器中运行的函数提交模型；二者语法相近，但部署、序列化、安全校验和权限边界完全不同。

面试考点

  • Server Component 和 Client Component 的职责边界是什么？ 结论是数据读取和不需要交互的 UI 优先放 Server Component，状态、事件、Effects 和浏览器 API 放 Client Component。原因是 Server Component 默认不发送组件 JS，Client Component 才能在浏览器响应交互。项目里常把页面壳和数据列表放服务端，把按钮、弹窗、表单控件下沉到客户端。
  • 为什么 RSC 不能直接使用 useState、useEffect 或 window？ 结论是 Server Component 不在浏览器交互生命周期里运行。原因是状态和 Effects 依赖客户端持久实例与浏览器环境，而 RSC 生成的是可序列化 UI 描述。需要这些能力时应添加 'use client' 边界。
  • RSC 与传统 SSR 的区别是什么？ 结论是 SSR 主要生成初始 HTML，RSC 还定义了服务端组件与客户端组件的模块边界和 payload 协议。原因是 RSC 可以让部分组件永远不进客户端 bundle，并在导航中复用服务端渲染结果。边界是它需要框架和打包器支持，不能靠单个静态 HTML 复现。
  • Server Actions 和客户端 Action 的边界如何区分？ 结论是 Server Actions 在服务端执行，客户端 Action 在浏览器执行。原因是 Server Actions 需要编译和服务端入口，必须重新校验权限与输入；客户端 Action 只能处理本地状态或调用公开 API。面试中要特别说明不要把服务端私有能力暴露给客户端。
