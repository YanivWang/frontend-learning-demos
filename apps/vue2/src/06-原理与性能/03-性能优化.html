Vue2 常见性能优化

先点「逐个 push 1000 条」与「临时数组处理后整体赋值」对比两种生成方式，再点「触发一次重新渲染」 并打开 Console 观察 computed visibleTotal 与 methods visibleTotalByMethod 的执行次数差异；列表只展示前 5 条以控制 DOM 数量。

知识点要点

  • Vue 2 性能优化应先定位瓶颈：是首屏包体积、响应式数据过大、频繁重渲染、真实 DOM 太多，还是缓存策略不当；不要一上来泛泛加优化。
  • 大列表优先分页、虚拟滚动或只渲染可视区域；示例中的整体赋值减少了循环期间对响应式数组的逐项变更，但真实项目仍要控制 DOM 节点数量。
  • computed 基于响应式依赖缓存，依赖没变时复用结果；methods 每次渲染都会重新执行，适合便宜计算或需要每次执行的逻辑。
  • 稳定 key 能减少错误复用和不必要销毁，尤其是带输入框、组件状态、过渡动画的列表；随机 key 或 index key 都可能制造额外成本。
  • keep-alive 适合缓存频繁切换且创建成本高的组件，但缓存过多会占内存，需要 include、exclude、max 控制范围。
  • 路由懒加载减少首屏下载和解析压力，但首次进入懒加载页面可能变慢；弱网、chunk 404、版本缓存不一致时要有 loading 和失败兜底。Vue 2 已处于维护/历史版本语境，新项目通常优先 Vue 3 生态。

面试考点

  • Vue 2 页面卡顿时应该按什么顺序排查？ 结论是先用性能工具和日志定位瓶颈，再分别处理数据量、渲染次数、DOM 数量和资源体积。原因是不同瓶颈对应不同方案。项目里可先看首屏 bundle、接口返回量、列表节点数、computed 是否重复计算。
  • 大列表为什么不能只靠 v-for 硬渲染？ 结论是分页或虚拟滚动优先，批量数据先在普通数组里处理后整体赋值只是辅助。原因是真正贵的通常是大量 DOM 创建和更新。边界是 1000 条简单文本可能还能跑，复杂行组件、输入框、图片列表就必须控制可见范围。
  • computed 和 methods 的性能差别是什么？ 结论是 computed 有依赖缓存，methods 每次 render 都执行。原因是 computed 由响应式依赖驱动重新求值。项目里昂贵派生数据用 computed，事件处理或无缓存需求的即时逻辑用 methods。
  • keep-alive、路由懒加载和稳定 key 分别解决什么性能问题？ 结论是 keep-alive 减少重复创建，懒加载减少初始包体积，稳定 key 帮助正确复用节点。原因是它们作用在缓存、网络加载、diff 身份三个不同阶段。边界是缓存过多会占内存，懒加载失败要兜底，错误 key 会造成状态错乱。
