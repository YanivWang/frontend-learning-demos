onWatcherCleanup（Vue 3.5+）

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • onWatcherCleanup() 是 Vue 3.5+ 提供的 watcher 清理 API；当前 vendored Vue 为 3.5.34，因此示例可以直接从全局 Vue 解构并运行。
  • 它必须在 watch 或 watchEffect 的同步执行阶段调用，用来注册“本轮 watcher 失效时”的清理函数；下一次触发 watcher 或 watcher 被停止前会执行清理。
  • 示例中切换 id 会先清理上一个 id 的 setInterval，再开始新 id 的定时器。这类模式也常用于 AbortController 取消旧请求，避免慢响应覆盖新状态。
  • 如果清理逻辑写在 await 之后，当前活跃 watcher 已经丢失，onWatcherCleanup() 不能再关联到本轮 watcher；异步请求应先创建控制器并同步注册清理，再发起请求。
  • Vue 3.5 之前可使用 watch(source, (value, oldValue, onCleanup) => {}) 的第三个参数，watchEffect 也有 onCleanup 参数。跨版本库代码优先保留参数写法。

面试考点

  • onWatcherCleanup 解决什么问题？ 结论是它解决 watcher 每次重新运行前的旧副作用清理问题。原因是 watch 常用于请求、订阅和定时器，源变化后旧任务可能继续回调。项目里最典型场景是取消旧请求，避免旧响应覆盖最新筛选条件。
  • 为什么必须同步调用？ 结论是必须在 watcher 回调同步阶段注册。原因是 Vue 依赖当前活跃 watcher 记录清理函数，await 之后已经不再处于同一个同步追踪上下文。异步场景应先注册 AbortController 的清理，再执行请求。
  • Vue 3.5 前怎么写？ 结论是使用 watch 回调的第三个 onCleanup 参数，或 watchEffect 的参数。原因是这些参数在旧版本已存在且语义一致。组件库要兼容 Vue 3.4 或更早版本时，不应直接依赖 onWatcherCleanup。
  • 它和 onUnmounted 的职责有什么区别？ 结论是 watcher cleanup 管“本轮监听失效”，onUnmounted 管“组件实例销毁”。原因是同一组件内 watcher 可能运行很多轮。请求竞态、搜索联想、路由参数变化等场景应优先按 watcher 轮次清理。
