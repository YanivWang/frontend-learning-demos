getDerivedStateFromProps 与 getSnapshotBeforeUpdate

下方示例分别演示 props 派生 state 和滚动快照：输入框展示派生 state 的维护成本，消息列表展示追加内容前后如何保持滚动位置。

知识点要点

  • getDerivedStateFromProps：在初次挂载和后续更新的 render 前调用，接收最新 props 和当前 state，必须返回 state 补丁或 null；它是静态方法，不能访问 this、不能做请求、订阅或 DOM 操作。
  • 派生 state 风险：把 props 复制到 state 容易形成双数据源，出现用户输入被覆盖、父子状态不同步、边界条件难维护等问题。多数场景更推荐直接从 props 渲染、把组件做成受控组件，或通过改变 key 重置局部 state。
  • getSnapshotBeforeUpdate：在 React 把更新提交到 DOM 之前调用，适合读取更新前的滚动高度、选择区等 DOM 快照；返回值会作为 componentDidUpdate 的第三个参数，真正的 DOM 写操作应放到 componentDidUpdate。
  • 滚动边界：聊天/日志追加消息时，只应在用户原本接近底部时自动保持底部；如果用户已经向上阅读，强行滚动会破坏体验。还要处理 ref 为空、列表高度不足、连续追加等情况。
  • 现代写法：React 18 新代码通常用函数组件、受控状态、useEffect/useLayoutEffect 和 ref 处理类似需求；class 生命周期更多用于维护旧组件或面试理解渲染提交时机。

面试考点

  • getDerivedStateFromProps 为什么必须是静态纯函数？ 结论是它只能根据 props/state 计算下一段 state。原因是它运行在 render 前，没有实例上下文，也不保证适合副作用；如果在这里做请求或依赖实例字段，会破坏渲染可预测性。维护旧代码时应把副作用移到 componentDidMount/componentDidUpdate。
  • 为什么 props 派生 state 是高风险写法？ 结论是它经常制造两个状态来源。原因是父 props 更新和用户本地编辑都可能写同一份数据，稍不注意就覆盖用户输入或产生不同步。项目里能直接从 props 渲染就不要复制；需要“重置初始值”时常用 key 或受控组件模式。
  • getSnapshotBeforeUpdate 和 componentDidUpdate 如何配合？ 结论是前者读更新前快照，后者在 DOM 更新后应用快照。原因是某些值必须在 DOM 改变前读取，例如旧滚动高度和旧滚动位置；等 DOM 更新后再读就丢失了参照。示例中快照返回“底部距离”，更新后用新高度减去该距离恢复位置。
  • 聊天列表追加消息时如何避免打断用户阅读？ 结论是只有用户原本接近底部才自动跟随到底部。原因是用户向上滚动查看历史消息时，新增消息不应强行抢走视口。项目里还要处理消息批量追加、图片加载导致高度二次变化、虚拟列表和异步数据失败的边界。
