useState 基础与函数式更新

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • useState 返回当前渲染的状态快照和一个 set 函数；调用 set 函数不会改写本次 render 里的变量，而是让 React 安排下一次渲染。
  • 同一次事件处理中，count 始终是这次渲染捕获到的值，所以连续三次 setCount(count + 1) 通常只把同一个下一状态入队三次。
  • 当下一次状态依赖上一次状态时，使用函数式更新 setCount(prev => prev + 1)，React 会按队列顺序把每个 updater 应用到最新中间值。
  • state 适合驱动 UI；不会影响渲染的可变值放到 useRef。如果状态字段多且转移规则集中，再考虑 useReducer。
  • 对象和数组状态要创建新引用，不能原地修改后再传回同一个对象；否则 React 可能认为状态没有变化，UI 也难以追踪。

面试考点

  • 为什么连续三次 setCount(count + 1) 通常只加 1？ 结论是三次读取的都是同一次渲染中的 count 快照。原因是 React 在事件处理结束后再处理状态队列，普通写法会把相同的下一值入队。边界是如果下一状态只和事件参数有关，这样写可以；如果依赖旧状态，就要改成函数式更新。
  • 什么时候必须使用函数式更新？ 结论是“下一值依赖上一值”时优先使用 updater 函数。原因是 updater 会在 React 处理队列时拿到最新中间状态，能正确表达连续加减、切换、追加列表等逻辑。项目里常见于计数器、分页游标、队列式追加和定时器回调。
  • setState 后为什么立刻读不到新状态？ 结论是本次函数调用里的 state 不会被即时替换。原因是函数组件渲染产生的是快照，set 函数请求下一次渲染，新值会在下一次 render 中体现。需要基于新值做副作用时，通常放到事件里计算好，或用 useEffect 观察状态变化。
  • useState 和 useReducer 怎么选？ 结论是简单独立状态用 useState，复杂状态转移用 useReducer。原因是 reducer 能把 action 和转移规则集中管理。边界是不要为了“看起来高级”过早引入 reducer，小组件里的布尔值和输入框状态直接用 useState 更清晰。
