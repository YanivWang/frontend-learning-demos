Vuex actions 详解（载荷 / 异步 / Promise / 嵌套 dispatch）

点击下方按钮触发 commit / dispatch ：观察 count 与 status 行变化；「await checkoutPlus」与「模拟失败 action」演示 Promise 成功/失败路径。

知识点要点

  • mutation 应保持同步，专注记录一次可追踪的 state 变更； action 负责接口请求、定时器、组合多个 mutation 等异步或副作用流程。
  • dispatch 会返回 action handler 的返回值；当 action 返回 Promise 时，组件可以 await dispatch(...) 后更新按钮状态、跳转页面或展示错误。
  • 载荷可以写成 dispatch('incrementAsync', { amount: 3 }) ，也可以对象风格 dispatch({ type: 'incrementAsync', amount: 5 }) ；团队项目应统一一种形态，避免 action 内部解析混乱。
  • 异步 action 要考虑失败路径、重复点击和竞态。示例新增 checkoutFail ，组件用 try/catch 接住失败，真实项目还会配合 loading、禁用按钮和请求取消。
  • state 字段仍应提前声明，例如 count: 0 ；否则 mutation 中做自增可能遇到 undefined + 1 得到 NaN ，也不利于 Vue 2 响应式追踪。

面试考点

  • 为什么 mutation 同步、action 异步？ 结论是 mutation 用来记录确定的状态变更，action 用来承接异步过程。原因是 devtools 需要同步记录 mutation 前后的状态快照。项目里请求接口、组合多个提交、失败重试都放 action，最终再 commit。
  • 组件如何等待 Vuex action 完成？ 结论是让 action 返回 Promise，然后组件 await this.$store.dispatch(...) 。原因是 dispatch 会透传 handler 的返回值。项目里常用它控制提交按钮 loading、成功后跳转、失败后展示错误提示。
  • 载荷写法怎么选？ 结论是简单参数可用第二个参数，对象风格适合统一事件形态。原因是对象风格把 type 和 payload 放在同一个对象里，更接近事件描述。边界是 action 内部要按团队约定解析，别同时混用多个字段名。
  • 异步 action 的风险点是什么？ 结论是要处理接口失败、重复点击和后返回请求覆盖先返回请求。原因是 action 只是组织流程，不会自动保证幂等或取消。项目里通常加 loading 锁、请求 id、AbortController 或最后一次请求 wins 的策略。
