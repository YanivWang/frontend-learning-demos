Proxy 响应式原理

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • reactive() 返回的是原对象的 Proxy 代理。组件渲染、watchEffect() 或计算属性读取代理属性时会被收集为依赖，后续写入同一属性时再触发对应更新。
  • Proxy 能拦截属性读取、写入、删除、in、枚举和数组索引等操作，因此 Vue3 不需要像 Vue2 那样依赖 Vue.set 才能让新增属性进入响应式。
  • 常见坑是“脱离代理访问”：直接解构 reactive 对象的基础类型属性后，后续读写不再经过代理；需要保留 state.count 访问，或用 toRef() / toRefs() 暴露属性。
  • 代理对象与原始对象不是同一个引用，做严格相等比较、Map key、第三方实例缓存时要明确存的是代理还是原对象；不希望被深度代理的大对象可用 markRaw() 或 shallowRef() 管边界。
  • 示例中的新增 extra、修改数组索引和 push 都会让 watchEffect() 重新执行，说明这些操作都能被 Vue3 的响应式系统追踪。

面试考点

  • Vue3 的 reactive 为什么能监听新增属性和数组索引？ 结论是因为 reactive() 基于 Proxy 代理对象级别的操作，而不是逐个属性改写 getter/setter。新增属性、删除属性和数组索引写入都会经过代理陷阱，所以能触发依赖更新。边界是只有通过代理对象操作才有效，绕过代理直接改原始对象不会通知视图。
  • track 与 trigger 分别发生在什么时候？ 结论是读取响应式属性时 track，写入或删除相关属性时 trigger。Vue 用这种“读时收集、写时通知”的模型把组件渲染和状态字段关联起来。项目里排查“为什么没更新”时，先看读取是否发生在响应式 effect 中，再看修改是否经过代理。
  • reactive 对象为什么不建议直接解构后使用？ 结论是解构出的基础类型值不再经过代理访问，后续变化无法继续被追踪。原因是响应式依赖建立在属性访问上，而不是变量名本身。组合式函数返回多个字段时，通常用 toRefs()、单独的 ref()，或让调用方保持对象访问。
  • Vue3 Proxy 响应式相比 Vue2 defineProperty 改进了什么，仍有哪些边界？ 结论是它改进了新增/删除属性、数组索引和集合类型的追踪，降低了很多 Vue2 时代的显式补丁成本。原因是 Proxy 能拦截对象操作本身。边界是代理身份与原对象不同，且不是所有外部实例都适合深度代理，图表实例、复杂类实例常用 markRaw() 隔离。
