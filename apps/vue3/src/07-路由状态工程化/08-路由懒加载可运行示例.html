Vue Router 4 路由懒加载

本页为 可运行示例 ；点击「About Lazy」首次进入约延迟 800ms。守卫 + Pinia 见 06-Router与Pinia可运行示例 。

知识点要点

  • Vue Router 4 的 component 可以是返回 Promise 的异步组件工厂。Vite 工程常见写法是 component: () => import('@/views/About.vue') ，构建工具会把动态 import 识别为代码分割点。
  • 本仓库 HTML demo 没有打包步骤，因此用 new Promise + setTimeout 模拟 chunk 下载与解析延迟；行为上仍会在导航解析阶段等待 Promise resolved。
  • 点「About Lazy」首次进入时，下方会先显示 pending 文案，约 800ms 后显示异步组件内容；控制台会打印 [lazy-route] chunk resolved 。再次切换 Home ↔ About 可对比同步路由与懒路由。
  • 懒加载减少 首屏未访问页面 的初始 JS 体积，但用户 首次进入 懒路由会多一次异步等待；弱网还要考虑 loading、error、timeout 和重试。
  • keep-alive 缓存已创建组件实例，懒加载延后下载未访问页面；二者可配合，但分别解决「重复渲染」和「首屏体积」问题。概念见 06-KeepAlive 。

面试考点

  • Vue Router 4 路由懒加载在工程里怎么写？ 结论是路由 component 写返回 Promise 的工厂，常用 () => import('./views/X.vue') 。原因是 Router 在导航解析时会等待异步组件 resolved。边界是无构建 HTML 只能用 Promise 模拟，不能真正拆 chunk。
  • 路由懒加载解决什么性能问题？ 结论是把非首屏页面延后加载，降低初始 bundle。原因是大型 SPA 一次加载所有页面会拖慢首屏。边界是首次进入懒路由会等待 chunk，不能把全部性能问题都归因于懒加载。
  • 懒加载失败要怎么兜底？ 结论是要提供 loading、失败提示、超时和重试。原因是 chunk 可能 404、网络超时或发版后缓存不一致。项目里常配合全局错误监控与刷新提示。
  • 懒加载、预加载、keep-alive 怎么区分？ 结论是懒加载延后下载，预加载提前下载可能需要的资源， keep-alive 缓存已创建组件。三者优化阶段不同，应按路由热度取舍。
