setState 函数式更新（解决对象式合并陷阱）

请打开 DevTools Console 查看 prevState 递增序列与 2s 后的最终值；下方 #root 展示页面最终 count（预期为 10，对比 03 号 demo 的 1）。

知识点要点

  • 函数式 setState 接收 updater：(prevState, props) => partialState。React 会按队列顺序执行 updater，因此每一次都能拿到上一条更新合并后的最新 state。
  • 示例在 componentDidMount 中连续调用 10 次函数式更新，prevState.count 依次是 0 到 9，最终页面展示 10，和 03 号 demo 对象式更新最终只加 1 形成对比。
  • 只要新值依赖旧值，就用函数式写法，例如计数器递增、购物车数量、分页游标、批量选择等；如果只是把 loading 设成 true，对象式更新更简洁。
  • updater 应该保持纯函数风格：不要直接修改 prevState，不要在 updater 里发请求、读写 DOM 或触发额外状态更新；副作用应放在生命周期或事件流程中。
  • React 18 的自动批处理让"同一批次内多次更新"更常见，函数式更新是跨 class 和 Hooks 都适用的防旧快照写法。

面试考点

  • 函数式 setState 为什么能把连续 10 次更新累加到 10？ 结论是：因为每个 updater 都基于队列中前一次更新后的 prevState 计算。原因是 React 不会提前用同一个旧 this.state 算好所有对象；计数器这类累加逻辑必须依赖 updater，而不是对象式快照。
  • updater 函数的 prevState 和 props 分别是什么？ 结论是：prevState 是当前队列执行到这一步时的最新 state，props 是本次更新对应的当前 props。原因是 state 和 props 都可能影响下一状态；例如根据父组件传入的步长递增时，应使用第二个参数而不是闭包旧值。
  • 什么时候必须用函数式 setState？ 结论是：新 state 依赖旧 state 或 props 时必须用。原因是对象式更新可能被批处理并浅合并，无法表达"在前一次结果上继续计算"；项目中常见于连续点击、批量更新、异步回调和循环更新。
  • updater 函数里有哪些陷阱？ 结论是：不要直接改 prevState，也不要在 updater 中做副作用。原因是 React 可能为了调试或并发能力重复调用渲染相关逻辑；保持 updater 纯净，才能让批处理和重渲染行为可预测。
  • 函数式 setState 与 Hooks 的函数式更新有什么共同点？ 结论是：两者都是用"从上一个状态计算下一个状态"来避免旧快照。原因是 React 状态更新都有队列语义；差异是 class 的 setState 会浅合并对象，Hooks 的 setState 默认替换整个状态值。
