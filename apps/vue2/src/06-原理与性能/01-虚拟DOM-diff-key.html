虚拟 DOM / diff / key

在各列表项输入框中分别输入内容，再点「头部插入」或「反转列表」，观察稳定 id 作为 :key 时输入框状态是否跟随对应数据项移动；下方 <pre> 有面试答法摘要。

知识点要点

  • Vue 2 模板最终会进入 render 流程，render 产生 vnode tree；响应式数据变化后 watcher 触发重新 render，再由 patch 比较新旧 vnode 并更新真实 DOM。
  • 虚拟 DOM 的价值不是“比 DOM 一定快”，而是用 JavaScript 对象描述 UI，让 Vue 可以跨平台组织更新、批量调度，并避免开发者手写大量命令式 DOM 操作。
  • Vue 2 的 children diff 主要在同一层级内比较节点，不把跨层移动当作同一个节点优化；跨层结构变化通常会销毁并重建相关子树。
  • key 的核心是节点身份。稳定唯一的 key 能帮助 Vue 判断哪个 vnode 对应哪个真实 DOM 或组件实例，从而在插入、删除、排序时正确复用和移动。
  • 可变列表不推荐用数组 index 做 key，因为头部插入或排序后 index 会变化，输入框值、组件局部状态可能跟错数据；静态、永不重排的展示列表才勉强可接受。

面试考点

  • Vue 2 更新时 template、render、vnode、patch 的关系是什么？ 结论是 template 编译成 render（或直接使用 render），render 执行返回 vnode；数据变化触发 watcher 重新 render，新 vnode 与旧 vnode 经 patch 更新真实 DOM。原因是 Vue 不直接用模板字符串操作 DOM，而是把 UI 表达成 vnode 再 diff。项目里性能分析应关注数据变更规模、render 成本和 patch 后真实 DOM 成本。
  • key 在 diff 中解决的是性能问题还是节点身份问题？ 结论是先解决节点身份，性能收益是副作用。原因是 key 让 Vue 能确认两个 vnode 是否代表同一项数据，再决定复用、移动或销毁。边界是乱用随机 key 会让每次更新都像全量重建，反而更慢。
  • 为什么可变列表不能用 index 作为 key？ 结论是 index 会随插入、删除、排序改变，不能稳定代表数据项。原因是 Vue 会把同一个位置的 DOM 当作同一个节点复用。项目里带输入框、动画、组件状态的列表必须用数据库 id 或前端生成的稳定 id。
  • 什么时候可以不写 key，什么时候必须写稳定 key？ 结论是简单静态列表或只追加且没有局部状态的列表风险较低，但动态组件、表单项、可重排列表应写稳定 key。原因是无 key 时 Vue 会尽量就地复用。边界是为了强制重建组件，也可以主动改变 key。
