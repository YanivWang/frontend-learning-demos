Vitest 单元测试面试边界

本页为概念说明 + 迷你 mock 演示；Router/Pinia 集成测见 06-Router与Pinia可运行示例（可 mock router.push）。

知识点要点

  • 本页点「运行 vi.fn 风格 mock 演示」，下方 #log 会追加调用记录；点「清空日志」重置。Vitest 是面向 Vite 生态的测试框架，复用 Vite 的模块转换、插件和路径别名，提供 describe、it、expect、vi 等常见测试 API；需在 Vite 工程中运行，本 HTML 只做概念演示。
  • Vue 组件测试通常搭配 Vue Test Utils，通过 mount 或 shallowMount 渲染组件；是否 shallow 取决于是否要覆盖子组件交互，而不是为了让测试“更容易过”。
  • vi.fn() 记录函数调用，vi.mock() 替换模块依赖，vi.useFakeTimers() 控制定时器；使用后要恢复 mock/timer，避免测试之间互相污染。
  • 测试边界应按风险划分：纯函数、composable、Pinia store 适合单测；组件事件和 DOM 输出适合组件测试；路由、登录、网络失败和浏览器兼容流程更适合 Playwright E2E。
  • 控制台可配合观察；mock 演示仅模拟 vi.fn 参数记录，真实断言在 Vite 工程内运行。常见坑包括断言内部变量名、过度 snapshot、mock 掩盖真实集成问题、异步未等待、未清理 DOM 或 timer，以及把所有场景都塞进单元测试导致维护成本过高。

面试考点

  • Vitest 和 Jest 怎么取舍？ 结论是 Vite/Vue 3 项目通常优先 Vitest，已有 Jest 体系的大项目可以渐进迁移。原因是 Vitest 能复用 Vite 配置和转换链，启动与 watch 体验更贴近工程；边界是 Jest 生态成熟，遗留 mock、快照和 CI 配置不必一次性重写。
  • Vue 组件测试为什么强调测行为不测实现？ 结论是断言用户可见输出、事件和副作用比断言内部变量更稳定。原因是组件内部可能从 Options API 改到 Composition API，但外部行为不变；项目里应优先通过 DOM、props、emits 和可观察状态验证需求。
  • mock、fake timers 和真实依赖如何选择？ 结论是不可控、慢、昂贵或有副作用的依赖适合 mock，核心集成路径要保留真实依赖或更高层测试。原因是 mock 能稳定边界但也可能制造假信心；异步、定时器和网络失败路径要明确等待、恢复和错误断言。
  • 单元测试、组件测试和 E2E 测试边界是什么？ 结论是单测覆盖小逻辑，组件测试覆盖 UI 单元行为，E2E 覆盖真实浏览器流程。原因是测试越接近用户越可信但越慢、越脆；项目里常用金字塔或奖杯模型组合，而不是只追求某一种测试数量。
