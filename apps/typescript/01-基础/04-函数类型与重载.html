函数类型与函数重载

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • 函数类型是 API 契约：参数决定调用者能传什么，返回值决定调用者能拿到什么；本页展示了 type 别名、interface 调用签名和内联函数类型三种常用写法。
  • this 参数是 TypeScript 里的“假参数”，用于约束函数被调用时的 this 类型，编译后不会生成真实参数；适合对象方法、事件回调和需要禁止错误绑定的函数。
  • 可选参数 unit?: string 在类型上等价于可能为 undefined，函数体必须处理缺省；默认参数 unit = "CNY" 是运行时兜底，通常放在参数列表末尾，调用者才容易省略。
  • 剩余参数用数组类型表达，如 ...nums: number[]；如果每个位置含义不同，可以用元组 rest，例如 ...args: [level: "info" | "warn", message: string] 保留位置语义。
  • 函数重载用于一个函数名对应多组外部调用方式，调用者只能按重载签名调用；实现签名必须覆盖所有分支，并在函数体内通过 typeof、Array.isArray 等方式运行时收窄。
  • 常见坑是重载过度：如果返回类型不随参数形态明显变化，联合类型或泛型往往更简单；如果参数形态会决定返回类型，例如 string 返回 string[]、number 返回 number[]，重载可读性更好。

面试考点

  • 函数类型为什么既要看参数也要看返回值？ 结论是函数类型描述的是完整调用契约。原因是参数约束调用者输入，返回值约束调用者后续使用；只写一边都会让类型信息断裂。项目中工具函数、回调 props、事件处理器都依赖这个契约。
  • type、interface 调用签名和内联函数类型有什么区别？ 结论是三者都能描述函数，区别主要在复用方式和扩展能力。type 简洁，适合函数别名和联合类型组合；interface 适合可扩展的对象/函数形态；内联注解适合只用一次的短回调。项目里公共 API 优先抽成名字清楚的类型，局部回调再内联。
  • 可选参数和默认参数有什么区别？ 结论是可选参数是类型层面的“可能缺失”，默认参数是运行时给缺失值兜底。原因是 unit?: string 需要处理 undefined，unit = "CNY" 会在未传时赋默认值。实际项目中经常二者配合使用，但不能把默认参数当作外部校验。
  • 剩余参数为什么通常写成数组或元组类型？ 结论是 rest 参数在运行时就是数组，类型上也应表达每个元素或每个位置的约束。原因是 number[] 适合任意数量同类参数，元组 rest 适合固定前缀或位置含义。日志、事件总线和命令式 API 常用这类签名。
  • 什么时候用函数重载，什么时候用联合类型？ 结论是参数形态决定返回类型时用重载更清楚，简单分支用联合类型即可。原因是重载能给调用者暴露多组精确签名，但实现更复杂。项目中解析函数、DOM 包装函数可用重载；普通入参兼容通常用联合加收窄。
  • 重载实现签名能不能被外部直接调用？ 结论是不能，外部调用只匹配重载签名。原因是实现签名只是函数体内部为了覆盖所有分支而写的兼容类型，不是公开 API。面试里常用这个点考察“重载签名”和“实现签名”的边界。
