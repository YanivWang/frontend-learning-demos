固定宽度三栏布局的响应式基线

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见本复习区，脚本会输出视口与容器实测值。缩窄窗口并与下一页对比。

知识点要点

  • style.css 把 #pagewrap 写死为 980px，内容区 #content 与侧栏 #sidebar 通过 Grid 两列（600px + 280px）组成桌面布局；这能稳定复现桌面排版，但不会随视口主动重排。
  • <meta name="viewport" content="width=device-width, initial-scale=1.0"> 只是在移动浏览器里把布局视口对齐设备宽度，不会自动把固定宽度 CSS 变成弹性布局；如果 CSS 仍是 980px，窄屏会横向滚动。
  • 这页是响应式改造的“基线页”：先保留桌面样式，下一页再通过 @media (max-width: ...) 覆盖 Grid 列宽和间距，才能清楚看出每个断点解决了什么问题。
  • 固定宽度 Grid 布局的常见坑是列宽溢出、内容撑破网格和图片或第三方嵌入宽度超出容器；窄屏下应配合断点改成单列或流式列宽。
  • 示例中的 HTTP 第三方资源来自历史教学页面；生产环境应避免依赖不受控的 HTTP 脚本，改用本地资源、HTTPS 或静态占位，防止加载失败和混合内容风险。

面试考点

  • 只有 viewport meta 没有媒体查询时，页面为什么仍然不算响应式？ 结论是 viewport meta 只是设定移动端布局视口和缩放初始值，不会重写 CSS 布局。原因是响应式的核心变化仍来自 CSS：容器宽度、列布局、图片和嵌入内容必须能按视口调整。边界是简单单列流式页面可能只靠百分比宽度就够，但固定 980px 页面必须补断点或改布局模型。
  • 固定宽度 Grid 布局在移动端最常见的问题是什么？ 结论是横向滚动和列挤压最常见。原因是固定列宽不会自动按视口改成单列。项目里做旧站改造时，通常先用断点把 Grid 改为 1fr 单列，再逐步调整间距和字号。
  • 为什么响应式改造通常要先保留桌面基线，再逐步添加断点？ 结论是这样更容易控制回归范围。原因是桌面样式作为默认层，断点只覆盖窄屏真正需要改变的属性，便于定位是哪条规则影响了布局。边界是新项目可以采用移动优先，从小屏基础样式开始，再用 min-width 增强大屏。
  • HTTP 第三方脚本在响应式 demo 里要注意什么？ 结论是不应依赖不受控的远程脚本。原因是 HTTP 资源会带来混合内容风险、加载失败和隐私问题。项目里如果必须嵌入第三方内容，要准备失败占位、HTTPS 链接、懒加载和尺寸兜底。
