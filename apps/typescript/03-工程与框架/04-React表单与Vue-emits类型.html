React 表单事件与 Vue defineEmits 类型

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • React 的合成事件类型要绑定到真实元素：输入框变更常写 React.ChangeEvent<HTMLInputElement>，表单提交常写 React.FormEvent<HTMLFormElement>。这样 currentTarget、target 和表单控件属性才不会退化成宽泛的 EventTarget。
  • 读取表单值时要理解 DOM 类型：input.value 永远是字符串，type="number" 也需要 valueAsNumber 或手动转换；复选框读 checked，文件读 files。类型标注不能替代这些运行时分支。
  • Vue 3 的 defineEmits 可以用类型参数声明事件契约；Vue 3.3+ 的具名元组写法如 { submit: [email: string]; cancel: [] } 同时约束事件名、参数个数和 payload 类型，比字符串数组更适合组件库和表单组件。
  • 推荐在 React 表单中优先使用 currentTarget 读取绑定了泛型的元素，避免从冒泡来源的 target 上直接假设类型；Vue 中则把校验后的值作为 emit payload，不把 DOM 事件对象泄漏给父组件。
  • 涉及用户输入时，类型安全只覆盖编译期。邮箱格式、空字符串、重复提交、后端校验失败和 XSS 风险仍要在运行时处理；本页用 checkValidity() 拦截非法邮箱，并用 textContent 输出，避免把输入当 HTML 解析。

面试考点

  • React 表单事件为什么要写具体元素泛型？ 结论是让事件对象携带准确的 DOM 元素能力，例如 HTMLInputElement.value 和 HTMLFormElement.checkValidity()。原因是基础的 EventTarget 不保证有 value，直接访问会丢失类型安全。项目里复杂表单通常把事件处理器类型写在函数签名上，并优先用 currentTarget 读取当前绑定元素。
  • Vue defineEmits 的类型声明解决什么问题？ 结论是把子组件对外发出的事件名和参数类型变成组件公共契约。原因是父组件监听事件时依赖这些 payload，字符串数组只能限制事件名，不能限制参数形状。边界是运行时仍可配合对象校验器做值校验，尤其是表单提交、上传和跨团队组件库。
  • 表单类型安全和浏览器校验如何分工？ 结论是 TypeScript 负责“代码是否按约定读写”，浏览器约束验证和业务逻辑负责“用户输入是否有效”。原因是用户可以输入空值、非法邮箱或绕过前端，类型系统不会在运行时阻止。真实项目要同时处理前端提示、后端错误、重复提交和可访问性反馈。
  • 为什么不建议把事件参数写成 any？ 结论是 any 会绕过表单控件差异，容易把 checked、files、valueAsNumber 混成一个字符串读取路径。原因是不同输入类型的 DOM API 不同，精确类型能让编辑器提前暴露错误。边界是封装通用表单库时可以用泛型或受控字段模型，而不是把每个 handler 都退回 any。

上方为 React/Vue 类型片段；下方原生表单演示等价逻辑，提交后观察 #log。

// React
function onChange(e: React.ChangeEvent<HTMLInputElement>) {
  const value: string = e.currentTarget.value;
}
function onSubmit(e: React.FormEvent<HTMLFormElement>) {
  e.preventDefault();
  if (!e.currentTarget.checkValidity()) return;
}

// Vue 3.3+ defineEmits
const emit = defineEmits<{ submit: [email: string]; cancel: [] }>();
emit("submit", email.value);
