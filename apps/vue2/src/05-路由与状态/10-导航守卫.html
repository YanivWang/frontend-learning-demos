vue-router 全套导航守卫 + 受保护路由（meta.requiresAuth）

未登录时点击 About（需登录） 应跳转 Login；点「模拟登录」后进入 About；离开 About 时会弹出确认框。打开 Console 观察全局/路由独享/组件内守卫的打印顺序。

知识点要点

  • Vue Router 3 的守卫用于控制一次导航：可以放行、重定向到登录页，或用 next(false) 取消离开；示例中的 About 离开确认就是取消导航的边界。
  • 典型顺序是离开组件的 beforeRouteLeave 、全局 beforeEach 、复用组件的 beforeRouteUpdate 、路由独享 beforeEnter 、进入组件 beforeRouteEnter 、全局 beforeResolve 、确认导航、 afterEach 。
  • meta.requiresAuth 本身只是路由记录上的自定义字段，真正的拦截逻辑要在全局守卫中读取 to.matched ；嵌套路由时用 matched.some 比只看 to.meta 更稳。
  • beforeRouteEnter 发生在组件实例创建前，不能访问 this ，只有它支持 next(vm => ...) ； beforeRouteUpdate 和 beforeRouteLeave 已有实例，不支持这种回调。
  • 本 demo 用 sessionStorage 模拟登录态，刷新同标签页仍在，关闭标签页会丢失；真实项目还要处理 token 过期、后端 401、redirect 白名单和重复跳转失败。

面试考点

  • 导航守卫的完整流程怎么讲？ 结论是先处理离开和全局前置，再处理复用、路由独享、进入组件、resolve，最后确认导航并触发 afterEach。原因是 Router 要先确认旧组件能否离开，再解析新路由和异步组件。项目里排查“为什么页面没进来”时，应按这个顺序看日志。
  • 全局守卫、路由独享守卫、组件内守卫怎么分工？ 结论是全局守卫放登录、权限、埋点前置， beforeEnter 放单路由策略，组件内守卫放未保存表单、同组件参数更新。原因是它们的触发范围不同。边界是不要把业务请求都堆进全局守卫，否则路由层会过重。
  • beforeRouteEnter 为什么不能直接用 this ？ 结论是此时组件实例还没创建。原因是进入守卫运行在导航确认前，Vue 还没有挂载新组件。项目里如果要访问实例，应使用 next(vm => ...) ，或者把数据请求放在组件 created / watch 中。
  • 登录拦截有哪些坑？ 结论是要处理刷新、过期、重定向回跳和重复导航。原因是前端存储只代表本地状态，后端 401 才是最终认证结果。项目里通常用 meta.requiresAuth 做第一层拦截，请求拦截器统一处理 token 失效。
