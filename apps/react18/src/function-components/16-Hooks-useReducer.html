useReducer 管理复杂状态

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • useReducer 把“状态如何变化”从组件事件处理函数中抽离出来，适合多个字段有关联、更新分支多或需要记录 action 语义的局部状态。
  • reducer 必须是纯函数：相同的 state 和 action 得到相同的新 state，不直接修改原对象，也不在 reducer 中请求网络、写存储或操作 DOM。
  • dispatch(action) 描述“发生了什么”，示例中的 increment、decrement、reset 让事件处理更像发出意图，转移规则集中在 reducer。
  • 返回新对象时要保持不可变更新，数组用 concat、展开或 map 生成新引用；原地 push 再返回旧 state 会破坏 React 的变化判断。
  • useReducer 管的是组件局部状态；跨页面或远端缓存仍要结合 Context、路由状态、数据请求库或外部状态管理工具评估。

面试考点

  • useReducer 相比 useState 解决了什么问题？ 结论是它让复杂状态转移更集中、更可读。原因是多个事件都修改同一组字段时，把规则散在事件处理里容易重复和出错。项目里表单向导、编辑器、购物车局部状态和带历史记录的交互更适合 reducer。
  • reducer 为什么必须是纯函数？ 结论是为了让状态更新可预测。原因是 React 可能重新调用 reducer 来确认结果，副作用或原地修改会让结果依赖调用次数和外部环境。边界是副作用应放在事件处理、effect 或请求层，reducer 只返回新状态。
  • action 应该怎么设计？ 结论是 action 描述业务事件，而不是暴露任意字段赋值。原因是 { type, payload } 能让状态变化有语义，便于调试和复用。项目里 action 类型要覆盖真实用户行为，未知 action 可以抛错帮助尽早发现拼写问题。
  • useReducer 适不适合替代全局状态管理？ 结论是不适合作为默认替代。原因是它只解决单个组件树内的状态转移，跨页面共享、缓存失效、服务端数据同步还需要其他方案。边界是配合 Context 可做小范围全局状态，但要注意 Provider 值变化导致的渲染范围。
