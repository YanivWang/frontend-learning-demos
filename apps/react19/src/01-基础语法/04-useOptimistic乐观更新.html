useOptimistic 乐观更新

下方 #root 为发消息演示区：输入普通文本可观察乐观项立即出现；输入含「失败」可观察回滚与错误提示。请打开 DevTools Console 查看乐观层与真实 state 对齐日志。

知识点要点

  • useOptimistic(state, updateFn) 用于异步 Action 完成前先展示临时 UI，让用户马上看到「发送中」「已点赞」等反馈；返回 [optimisticState, addOptimistic] 。
  • updateFn 必须是纯函数，根据当前权威 state 和乐观输入计算下一屏状态；不要在里面发请求、改外部变量或生成不可控副作用。
  • 真实 state 更新后，乐观层会和最终结果对齐；如果请求失败且权威 state 没变，临时项应消失，并显示可理解的错误。
  • 与直接 setState 临时追加不同，乐观层由 React 在 Action / transition 生命周期内管理，提交完成或失败后会自动与权威 state 对齐，无需手动删除临时项。
  • 本页用 onSubmit + startTransition 触发发送，而非上一页的 <form action> ：原因是发消息常从按钮/快捷键触发，不一定走标准表单提交；乐观更新只要处在 transition / Action 生命周期内即可，与是否绑定 form action 无强绑定。
  • 适合发消息、点赞、列表追加等成功概率高、失败可回滚的交互；支付、库存扣减、权限变更这类强一致场景通常不应乐观确认。

面试考点

  • useOptimistic 解决的核心体验问题是什么？ 结论是提升异步提交的即时反馈。原因是用户发消息或点赞后不希望等待网络返回才看到变化。边界是它只处理临时 UI 层，服务端结果、失败文案和重试仍由业务负责。
  • 乐观 UI 为什么必须设计失败回滚？ 结论是失败时要回滚或标记失败，不能静默保留假状态。原因是乐观结果不是权威数据，和服务端冲突时必须以真实 state 为准。本页输入「失败」会触发回滚并展示错误。
  • updateFn 为什么要求纯函数？ 结论是它只应该计算乐观状态。原因是 React 可能根据渲染和 Action 生命周期重新计算乐观层，副作用会造成重复请求或状态污染。边界是 ID、时间戳等输入应在调用 addOptimistic 前准备好。
  • useOptimistic 与直接 setState 临时追加有什么区别？ 结论是乐观层由 React 管理，提交完成后自动与权威 state 对齐；手写临时 state 需自己维护添加、删除和对齐逻辑。原因是 useOptimistic 与 transition / Action 集成，失败时权威 state 不变则临时项自动消失。
  • 哪些业务不适合做乐观更新？ 结论是强一致、失败代价高或权限不确定的操作不适合直接乐观确认。原因是用户看到的结果如果随后回滚，会带来信任和业务风险。支付、审批、库存扣减通常应等待服务端确认再展示成功态。
