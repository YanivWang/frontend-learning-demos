类型收窄 · 可辨识联合

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • 类型收窄来自 TypeScript 的控制流分析：经过 typeof、instanceof、in、相等性判断或早返回后，编译器会在当前分支里排除不可能的联合成员。
  • 可辨识联合要求每个成员都有共同的字面量字段，例如 kind 或 type；本页 Shape 在 switch (s.kind) 后分别获得圆和矩形的专属字段。
  • 类型守卫函数 isXxx(value): value is Xxx 适合封装可复用的运行时判断；as 断言只是告诉编译器“相信我”，不会检查数据真假。
  • 真实项目里常用可辨识联合表达接口结果、加载状态、表单提交状态和 reducer action，比一堆可选字段更容易获得穷尽检查。
  • 涉及外部输入时，类型收窄只能基于当前 JS 判断；接口响应、表单值或本地存储仍要先做运行时校验，再让类型守卫把结果带回类型系统。

面试考点

  • TypeScript 为什么能在分支里推断出更窄类型？ 结论是编译器会做控制流分析。原因是条件判断、赋值、返回和抛错都会改变某个变量在当前位置可能具有的类型集合。边界是 TS 只相信能从代码静态看出的判断，外部数据仍要运行时校验。
  • 可辨识联合比可选字段建模好在哪里？ 结论是它让不同状态互斥且分支精确。原因是共同的字面量字段能告诉编译器当前是哪一个成员，避免在每个字段前都写非空判断。项目里适合 API 结果、UI 状态机和 reducer action。
  • 类型守卫函数和 as 断言怎么取舍？ 结论是能写真实判断就写类型守卫，只有编译器暂时无法表达但开发者确定安全时才用 as。原因是类型守卫同时执行运行时检查并反馈给类型系统，而断言不会生成校验逻辑。用户输入、网络响应和存储数据不能只靠断言。
  • 新增联合成员时如何避免漏处理？ 结论是配合 switch 和 never 穷尽检查。原因是当所有成员都被处理后，默认分支里的变量应被收窄为 never；新增成员但漏写分支时编译器会报错。复杂状态流可以把这个模式放进 reducer 或状态机。

type Shape =
      | { kind: "circle"; radius: number }
      | { kind: "rect"; width: number; height: number };
      function area(s: Shape): number {
      switch (s.kind) {
      case "circle": return Math.PI * s.radius ** 2;
      case "rect": return s.width * s.height;
      }
      }
