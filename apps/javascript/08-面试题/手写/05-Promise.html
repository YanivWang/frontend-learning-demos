知识点要点

  • Promise 有 pending 、 fulfilled 、 rejected 三种状态；一旦从等待态落定，就不能再被后续 resolve 或 reject 改写。
  • then 返回一个新的 Promise ，新状态由回调返回值决定：返回普通值则 fulfilled，抛错则 rejected，返回另一个 Promise 则跟随它的最终状态。
  • Promise 回调应异步执行；本 demo 用 queueMicrotask() 演示微任务调度，让 then 回调晚于当前同步代码执行。
  • catch 本质上是只传失败回调的 then ， finally 适合释放资源或收尾，但不应吞掉原本的成功值或失败原因。
  • 真实 Promise 还要处理 thenable 同化、自解析循环、宿主未处理拒绝追踪等细节；面试版可以先讲清状态机、回调队列、链式解析，再说明边界。
  • 项目中优先使用原生 Promise 和 async/await ；手写题用于理解异步模型，不应用自定义实现替换标准能力。

面试考点

  • Promise 为什么状态一旦改变就不能再变？ 结论：Promise 只能从 pending 变为 fulfilled 或 rejected，落定后不可逆。原因是异步结果必须有稳定的最终状态，否则多个回调链会看到互相矛盾的结果。边界是重复调用 resolve / reject 只应第一次生效，后续调用应被忽略。
  • then 为什么必须返回新的 Promise ？ 结论：为了让链式调用能表达“上一步处理后的下一个异步结果”。原因是每个 then 都可能返回值、抛异常或返回另一个 Promise，新 Promise 负责承接这些结果。项目里用 return 把异步结果传下去，否则后续链会拿到 undefined 。
  • 回调为什么要异步执行而不是 resolve 后同步调用？ 结论：Promise 反应任务要在当前同步调用栈结束后执行。原因是这能保证 then 注册顺序、同步代码顺序和异常传播语义稳定。边界是本 demo 用 queueMicrotask 近似演示，真实规范由宿主的 Promise Jobs / microtask 队列调度。
  • 手写 Promise 和原生 Promise 有哪些边界差异？ 结论：面试手写版通常只覆盖状态机、回调队列和链式解析。原因是完整实现还要处理 thenable、循环引用、静态组合方法、未处理拒绝事件和跨 realm 行为。实际项目必须使用原生 Promise 或成熟运行时，不要引入自制 Promise 替换标准实现。
