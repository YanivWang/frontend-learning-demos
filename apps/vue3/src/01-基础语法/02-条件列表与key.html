v-if / v-show / v-for / key

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • v-if 是真正的条件渲染，条件为假时分支不会保留在 DOM 中，重新变真时会重新创建节点和子组件状态。
  • v-show 会始终渲染元素，只通过 display 控制可见性；它初始渲染成本更高，但频繁切换时通常比反复创建销毁更稳定。
  • v-for 渲染列表时应提供稳定、唯一、可预测的 key，让 Vue 在插入、删除、反转时正确移动已有节点，而不是错误复用表单值或组件内部状态。
  • 不建议把 v-if 和 v-for 写在同一个元素上；Vue3 中 v-if 优先级更高，可能读不到 v-for 的局部变量。需要过滤时先用计算属性得到列表。
  • 本页通过“新增一项”和“反转列表”演示稳定 item.id 的意义；如果改成数组下标作为 key，列表顺序变化时更容易出现状态和 DOM 复用错位。

面试考点

  • v-if 和 v-show 如何按切换频率选型？ 结论是低频切换、条件不满足时不需要渲染的内容用 v-if，高频显示隐藏用 v-show。原因是 v-if 有创建销毁成本，v-show 有初始渲染成本但切换只是 CSS。项目里权限区块、异步加载内容常用 v-if，下拉面板或标签页可考虑 v-show。
  • v-for 的 key 为什么不能随便用 index？ 先结论：会增删、排序、反转的列表不要用下标 key。原因是下标会跟着位置变化，Vue 可能把旧节点和组件状态复用到错误数据上。只有纯静态、不会重排的展示列表，使用 index 的风险才相对可控。
  • v-if 和 v-for 为什么不建议写在同一个元素上？ 结论是优先级和作用域容易造成误解，Vue3 中 v-if 优先于 v-for。原因是条件表达式可能访问不到循环变量，逻辑也难以维护。项目里应先用计算属性过滤列表，或把 v-if 放到包裹层上。
  • 列表反转或插入时稳定 key 解决了什么问题？ 结论是稳定 key 帮 Vue 识别“同一条数据”而不是“同一个位置”。原因是 diff 时可以移动已有节点并保留正确状态。表单行、可编辑列表、动画列表都必须特别注意这一点。
