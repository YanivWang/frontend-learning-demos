var + function 同名提升经典题

先看本复习区，再查看下方输出是否与标准答法一致；细节可打开 DevTools Console。

知识点要点

  • 这题运行在普通非模块脚本中。代码执行前会先处理 var 声明和函数声明，函数声明会创建函数对象并绑定到名字 a。
  • 同一作用域里有多个同名函数声明时，后面的函数声明会覆盖前面的函数声明，所以第一处读取 a 时看到的是后一个 function a() { return 4; }。
  • var a 的声明部分不会覆盖已有函数绑定；但 var a = 1 的赋值部分会在执行到该行时把 a 改成数字 1。
  • 随后 var a = 3 再次执行赋值，把 a 改成数字 3；函数声明语句本身不会在运行到那一行时重新执行赋值。
  • 因此本例控制台依次输出：函数引用、1、3、3。面试答题要把“声明实例化阶段”和“运行期赋值阶段”分开讲。
  • 现代代码应避免同一作用域内 var 与函数声明重名；用 let / const、函数表达式或模块作用域能让命名冲突更早暴露。

面试考点

  • 为什么第一行输出的是后一个 function a，不是 undefined？ 结论：因为函数声明在执行前已经完成绑定，并且后面的同名函数声明覆盖前面的。原因是 var a 只声明变量，不会把已经由函数声明创建的绑定重置成 undefined。边界是不同环境中函数源码字符串格式可能不同，但它仍是函数引用。
  • var a = 1 为什么会把函数绑定改成数字 1？ 结论：因为赋值发生在运行期，会直接更新同一个名字绑定。原因是 var a = 1 的声明部分已在前置阶段处理，执行到这行时只剩赋值动作。项目里看到“提升”题时不能把声明提升误解为赋值也提前执行。
  • 多个同名函数声明和 var 声明的处理顺序怎么理解？ 结论：先建立作用域内声明绑定，再按代码顺序执行赋值和表达式。原因是函数声明会提供初始函数值，同名 var 声明不会覆盖它，而后续赋值会覆盖。边界是块级函数声明在严格模式和不同历史浏览器中曾有兼容差异，现代代码应避免写这种歧义代码。
  • 为什么现代代码要避免 var 和 function 同名？ 结论：它依赖提升细节，阅读成本高且容易被打包、拼接或维护修改放大风险。原因是同名绑定会让“函数还是值”随执行位置变化。实际项目用 const fn = function...、唯一命名和模块作用域更清晰，Lint 也通常会禁止重复声明。
