hash 路由的基本演示

请打开 DevTools Console 查看输出。

知识点要点

  • location.hash 对应 URL 中 # 后面的片段标识符；浏览器不会把片段发送给服务器，因此片段变化通常不会触发整页重新请求。
  • hash 路由的核心是监听 hashchange：当 location.hash 变化时，根据新片段匹配页面组件或渲染逻辑。
  • 直接给 location.hash 赋值会增加一条可后退的历史记录；location.replace("#/...") 会替换当前记录，更适合重定向或不希望用户回到中间态的场景。
  • popstate 主要服务于 History API 导航，不能替代 hashchange 作为 hash 路由的核心监听。
  • hash 路由部署简单，刷新不会要求服务端把所有路径 fallback 到入口 HTML；代价是 URL 带 #，语义、滚动锚点和部分 SEO/统计处理需要额外设计。

面试考点

  • hash 路由为什么切换页面时不会刷新整页？ 结论是 hash 只改变 URL 片段，浏览器不会为片段变化重新向服务器请求文档。原因是片段标识符用于客户端定位文档内资源，不属于 HTTP 请求路径。项目里可以据此在前端监听 hashchange 并局部更新视图。
  • hashchange 和 popstate 分别适合监听哪类导航？ 结论是 hash 路由监听 hashchange，History 路由监听 popstate。原因是二者对应不同的浏览器导航机制。边界是浏览器前进后退如果导致 hash 改变，也会触发 hashchange，但单纯调用 pushState 不会触发 popstate。
  • location.hash 赋值和 location.replace("#/...") 对历史栈有什么区别？ 结论是前者通常新增历史记录，后者替换当前记录。原因是 replace 的语义就是不保留当前条目。登录后跳转、非法路由纠正等场景常用 replace，普通页面导航更常用可后退的 hash 赋值。
  • hash 路由和 History 路由在部署与用户体验上如何取舍？ 结论是 hash 路由部署成本低，History 路由 URL 更自然。原因是 hash 路由刷新时服务器只看到入口路径，而 History 路由刷新深层路径需要服务端 fallback。项目里静态托管、旧系统嵌入常选 hash，对 URL 语义要求高的 SPA 通常选 History。
