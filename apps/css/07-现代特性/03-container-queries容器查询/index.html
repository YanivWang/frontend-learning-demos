Container Queries — 按父容器宽度自适应

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见本复习区，脚本会输出容器宽度与卡片 flex-direction。拖动虚线框改变容器宽度。

知识点要点

  • 媒体查询根据视口或设备特征切换样式，容器查询根据最近的查询容器尺寸切换样式。它解决的是“同一个组件放在主栏、侧栏、弹窗里宽度不同，布局也应不同”的组件级响应式问题。
  • 要使用尺寸容器查询，祖先元素需要声明 container-type 。本页的 .card-container 设置 container-type: inline-size ，表示可按容器的行向尺寸（通常是宽度）查询，同时用 container-name: card 命名。
  • @container card (min-width: 360px) 只在名为 card 的查询容器达到 360px 时生效。示例中两张卡片 HTML/CSS 相同，但因所在虚线容器宽度不同，可能呈现纵向或横向布局。
  • 容器查询不是媒体查询的替代品：页面整体导航、栅格和断点仍适合用媒体查询；可复用组件内部布局、卡片密度和按钮排列更适合容器查询。
  • container-type 会引入 containment，浏览器需要避免“组件样式改变自身尺寸、自身尺寸又改变查询结果”的循环依赖。实践中通常把容器设在组件外壳，查询内部子元素，而不是让被查询的元素直接查询自己。

面试考点

  • 容器查询和媒体查询的根本区别是什么？ 结论是媒体查询看全局视口或设备特征，容器查询看组件所在容器的尺寸。原因是组件复用时，视口相同不代表组件可用宽度相同。项目里页面级布局用媒体查询，组件级自适应用容器查询更准确。
  • 为什么要先设置 container-type 才能写 @container？ 结论是浏览器需要明确哪个祖先可以作为查询容器。原因是尺寸查询会引入 containment，避免子元素样式反过来无限影响被查询尺寸。项目里常把 container-type: inline-size 放在卡片外壳、列表项外壳或侧栏内容容器上。
  • container-name 有什么作用，什么时候需要命名容器？ 结论是命名容器让 @container card (...) 精确匹配目标容器，而不是只找最近的可查询祖先。原因是组件可能嵌套在多个容器里，未命名查询容易被更近的外壳截获。项目中简单组件可不命名，复杂嵌套或设计系统组件建议命名。
  • 容器查询有哪些布局、性能和兼容边界？ 结论是它适合组件自适应，但要注意 containment 副作用、循环依赖和浏览器兼容。原因是容器查询依赖浏览器对容器尺寸的独立计算，不能随意让被查询元素决定查询容器尺寸。项目里应提供基础布局作为降级，并在目标浏览器矩阵中确认支持情况。
