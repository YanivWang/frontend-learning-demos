effectScope / 副作用清理

在浏览器中操作下方交互演示；复习要点与面试答法见本复习区。

知识点要点

  • watch() 与 watchEffect() 都返回 stop 函数。本页“开启独立 watch”后改 count 会出现 独立 watch 日志；点“调用 watch 停止函数”后再改 count 不应再追加该前缀日志。
  • effectScope() 在 scope.run() 内创建的 watcher 会被收集，scope.stop() 一次性停止整个作用域，无需逐个保存 stop。本页 scope 内同时有 watch 与 watchEffect，停止后两者都不再触发。
  • 组件 setup() 同步创建的 watcher 会随组件卸载自动停止；组件外、插件内或“可开关模块”更适合显式 effectScope 或保存 stop 函数。
  • onScopeDispose() 在 scope 停止或所属组件卸载时执行，适合清理 setInterval 等非响应式资源；本页 scope 内注册了定时器，停止 scope 后 tick 日志应消失。watcher 每轮清理见 06-onWatcherCleanup与watch清理（Vue 3.5+）。
  • 重复点“开启”而不先停止会造成重复注册：本页在已开启时再次开启会 push 提示“已在运行”，避免演示日志爆炸。生产里应把启动/停止做成成对 API。
  • watch / watchEffect 选型与默认执行时机见 02-computed-watch-watchEffect；本页不重复讲解 lazy vs eager。

面试考点

  • watch stop 和 effectScope.stop 有什么区别？ 结论是 stop 只停一个 watcher，scope.stop() 停整个作用域内收集的响应式 effect。项目里单个监听用 stop，一组可开关逻辑用 scope。
  • effectScope 适合解决什么问题？ 结论是管理组件外、插件内或 composable 内部创建的一组响应式副作用。边界是不是通用资源管理器，定时器和请求仍要 onScopeDispose 或取消函数。
  • 组件卸载时哪些不会自动清理？ 结论是 DOM 监听、定时器、WebSocket、未取消请求等外部副作用。Vue 会处理绑定到实例的同步 watcher，但不会替你 clearInterval。
  • composable 如何设计释放？ 结论是创建时同步设计释放路径：返回 stop、暴露 scope，或在 scope 内用 onScopeDispose。调用方应清楚何时结束。
  • onScopeDispose 与 onUnmounted 如何配合？ 结论是 scope 停止或组件卸载都会触发 scope dispose；组件级资源仍可用 onUnmounted。异步 watcher 每轮清理优先 06 页。
