Router 与状态管理常见面试点

请打开 DevTools Console 查看模拟路由与 store 日志；下方 #root 为 React 挂载/交互演示区（本页用 path 状态模拟路由，非真实 React Router）。

知识点要点

  • 真实 React Router 负责 URL 与页面组件的映射，并提供 BrowserRouter 、 Routes 、 Route 、 Link 、 useNavigate 、 useParams 等能力。
  • 本静态 HTML 没有引入第三方库，只用 path 状态模拟页面切换；它不会改浏览器地址栏，也不能覆盖刷新、前进后退和深链接这些真实路由问题。
  • 状态管理先分层：组件内 UI 状态放本地，少量跨层客户端状态可用 Context 或外部 store，服务端数据缓存更适合 TanStack Query、SWR 等查询库。
  • Context + useReducer 可表达“小型 store”思想，但它缺少 Redux DevTools、规范化约束、中间件生态和大型团队协作约定。
  • 涉及登录态、路由守卫、刷新恢复、未知路径 fallback 时，要明确数据来源、持久化策略和失败路径；本 demo 增加未知路由提示来演示边界。

面试考点

  • React Router 解决什么问题，本 demo 为什么只是模拟？ 结论是 Router 让 URL 成为页面位置的来源。原因是刷新、分享链接、前进后退和参数解析都依赖真实地址。本 demo 只用状态切换组件，能说明映射思想，但不能替代真实 React Router 行为。
  • 本地状态、全局客户端状态、服务端缓存状态如何区分？ 结论是按所有权和生命周期区分。组件展开收起、输入草稿放本地；登录用户、主题这类跨组件状态放 Context 或 store；接口列表、分页缓存、失效重试属于服务端状态，通常交给查询缓存库。
  • Context + useReducer 与 Redux/Zustand 的边界是什么？ 小型页面或教学 demo 可以用 Context + useReducer 表达状态和 action。规模变大后，Redux/Zustand 在选择器、调试、模块拆分和团队约定上更有优势。边界是不要把所有状态塞进一个 Context 导致大范围重渲染。
  • 路由刷新、未知路径和登录态恢复有哪些风险？ 结论是要设计 fallback 和恢复流程。原因是用户可能直接打开深链接、刷新页面、token 过期或访问未知路径。项目里需要 404 页面、鉴权重定向、加载态和错误提示，本 demo 的“未知页”按钮只演示最小 fallback。
