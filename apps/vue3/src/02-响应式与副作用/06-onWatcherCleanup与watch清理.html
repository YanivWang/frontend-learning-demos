onWatcherCleanup（Vue 3.5+）

在浏览器中操作下方交互演示；复习要点与面试答法见本复习区。本页依赖 Vue 3.5+，低于 3.5 请用回调 onCleanup 参数。

知识点要点

  • onWatcherCleanup() 是 Vue 3.5+ 提供的 watcher 清理 API；本仓库 vue.global.js 为 3.5.34，可从全局 Vue 解构并直接运行。3.4 及以下项目应使用 watch 回调第三参数 onCleanup。
  • 必须在 watch 或 watchEffect 的同步执行阶段调用，注册“本轮 watcher 失效时”的清理函数；切换 activeId 时会先看到 cleanup：停止 id=…，再开始新 id 的定时器 tick。
  • 本页每个 id 启动 setInterval 模拟异步副作用；快速切换 A/B/C 时，旧 id 的 tick 应被清理，日志里不应混入多个 id 的并发 tick（开启“无清理对照”可对比脏数据）。
  • 若把 onWatcherCleanup 写在 await 之后，当前活跃 watcher 上下文已丢失，清理无法绑定到本轮；异步请求应先创建 AbortController 并同步注册清理，再 fetch。
  • Vue 3.5 之前写法：watch(src, (v, o, onCleanup) => { onCleanup(() => ...) })，语义与 onWatcherCleanup 一致。组件销毁级清理用 onUnmounted；整组 watcher 停止见 05-effectScope与副作用清理。
  • 异步竞态与 watch 选型背景见 02-computed-watch-watchEffect，本页专注“每轮 watcher 清理”而非 computed 缓存。

面试考点

  • onWatcherCleanup 解决哪类竞态？ 结论是 watcher 每次重新运行前旧副作用未取消，慢任务回调覆盖新状态。典型场景是搜索、路由参数变化、轮询定时器。本页切换 id 即验证。
  • 为什么必须同步调用？ 结论是 Vue 依赖当前活跃 watcher 记录清理函数，await 后已脱离同步追踪。应先注册 AbortController 清理再请求。
  • Vue 3.5 前怎么写？ 结论是 watch 第三参数 onCleanup 或 watchEffect 参数。库要兼容 3.4 时不应直接依赖 onWatcherCleanup。
  • 与 onUnmounted 的区别？ 结论是 watcher cleanup 管“本轮监听失效”，onUnmounted 管“组件实例销毁”。同一组件内 watcher 可能运行很多轮，应按轮次清理请求与定时器。
  • 与 effectScope.stop 的区别？ 结论是 scope.stop 停整个作用域内所有 effect；onWatcherCleanup 管单次 watcher 运行之间的清理。两者可组合：scope 停止时也会触发 watcher cleanup。
