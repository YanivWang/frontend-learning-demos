class 组件事件 this 绑定的三种方式

请打开 DevTools Console 观察各按钮点击时的 this 输出；依次点击三种绑定方式与反例按钮；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • React 不会自动把 class 方法绑定到组件实例。把 this.handle 作为回调传给事件系统时，它只是一个函数引用，调用时会丢失原来的 class 实例上下文。
  • constructor 中 bind 会生成稳定函数引用，适合老项目和需要向下传递 handler 的场景；缺点是样板代码较多。
  • JSX 内箭头函数写法直观，尤其适合传参：onClick={(e) => this.handle(id, e)}。它每次 render 都创建新函数，通常不是首要瓶颈，但传给大量 memo 子组件时可能触发额外渲染。
  • 类字段箭头函数把方法定义成实例属性，箭头函数词法绑定 this，在现代 Babel/构建配置中是 class 维护代码最常见写法。
  • 不要写 onClick={this.handle()}，这会在渲染阶段立即执行。函数组件没有 class 实例 this，同类问题通常改由闭包、useCallback 或直接内联函数处理。

面试考点

  • class 事件处理函数为什么需要绑定 this？ 结论是 class 方法默认不会绑定实例。原因是传给 onClick 的是函数引用，React 调用它时不会自动补上组件实例；边界是类字段箭头函数天然闭包住实例。
  • constructor bind、JSX 箭头函数和类字段箭头函数怎么取舍？ 结论是老代码常见 bind，传参常用 JSX 箭头函数，现代 class 维护代码常用类字段箭头函数。原因是它们分别优化稳定引用、写法便利和样板代码；大量列表中要关注函数引用变化。
  • 在 render 中创建箭头函数一定是性能问题吗？ 结论是不一定。原因是创建函数本身通常很小，真正影响常来自把新引用传给 memo/PureComponent 子组件；项目里应先用 DevTools 定位，再决定是否提取稳定 handler。
  • 事件传参时常见错误是什么？ 最常见是写成 onClick={this.handle(id)} 导致渲染时立即执行。正确做法是传函数，例如 onClick={() => this.handle(id)}，或预先 bind 参数。
  • class 维护代码迁移到函数组件后 this 问题如何消失？ 结论是函数组件没有实例 this，事件 handler 直接闭包捕获 props/state。原因是箭头函数和普通函数在函数组件里都能访问当前 render 的变量；迁移时用 useCallback 稳定引用即可，不再需要 bind。
