props / emit

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • props 是父组件传给子组件的输入契约。Vue 3 中 props 在子组件内是只读的，子组件应该把它当作外部输入，而不是本地可随意修改的状态。本页父组件 count 经 :count="count" 传入子组件。
  • emit 是子组件通知父组件的事件契约。点子组件「通知父组件 +1」只触发 increase 事件，真正的 count++ 发生在父组件 @increase 处理器中；pre 区会追加事件日志。
  • 这种 props down, events up 的单向数据流让状态来源可追踪：父组件拥有状态，子组件描述意图，父组件决定如何更新。点「父组件直接 +1」与通过子组件 emit 效果相同，但数据流路径不同。
  • Vue 3 推荐声明 emits: ["increase"]，这样组件的自定义事件会更清晰，也能避免未声明事件被当作透传监听器落到根元素上；透传细节见 02-attrs与透传。
  • 常见坑是子组件直接改 props、把本地临时状态和父级状态混在一起，或用层层 emit 穿透太深的组件树。跨层共享状态可考虑 03-provide-inject 或 Pinia；双向绑定语义见 04-组件v-model、05-defineModel（3.4+）。
  • 若子组件需要基于 props 初始化本地状态（如编辑表单草稿），应在 props 变化时用 watch 同步或明确「仅初始化一次」的语义，避免父级更新后子级仍显示旧值。

面试考点

  • 为什么强调 props down、events up？ 结论是为了保持单向数据流。原因是父组件拥有状态，子组件只接收输入并发出事件，状态变化来源更容易追踪。本页 pre 日志可区分「父直接改」与「子 emit 后父改」两条路径。
  • 子组件想改 props 怎么办？ 结论是不要直接修改 props，而是 emit 事件让父组件更新，或在子组件内基于 props 初始化本地状态。双向绑定语义可封装为 v-model，见 04-组件v-model。
  • emits 选项有什么价值？ 结论是它声明组件会发出的自定义事件。原因是 Vue 3 会据此区分组件事件和透传监听器，也便于校验 payload 和生成类型提示。组件库中应把公开事件写成明确契约。
  • props / emit 与其他通信方式怎么选？ 结论是父子近距离通信用 props / emit，表单双向语义用 v-model，跨层依赖用 provide-inject，全局业务状态用 store。边界是不要为了避免一两层传递就过早引入全局状态。
  • 子组件基于 props 初始化本地状态要注意什么？ 结论是明确 props 变化时是否同步本地副本。只初始化一次适合「打开弹窗时复制一份可编辑草稿」；props 持续驱动展示则应直接读 props 或用 computed。Vue 3.5+ props 解构见 06-props解构（3.5+）。
