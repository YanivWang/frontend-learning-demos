interface 与 type

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • interface 主要用来声明对象形状，支持 extends 继承和同名声明合并；适合描述可被第三方扩展的对象契约。
  • type 是类型别名，能给任意类型表达式命名，包括联合、交叉、元组、函数签名、条件类型和映射类型；本页的 Role 与 ApiResult 必须用 type 更自然。
  • 常见项目约定是“对象模型和组件 props 用 interface，联合状态、工具组合和类型计算用 type”；这不是语法强制，而是为了可读性和团队一致性。
  • 两者都能表达很多对象类型：interface extends 和 type A = B & C 都能组合形状，但冲突属性、错误提示和合并行为不同，要按可维护性选择。
  • 声明合并既是能力也是风险：写库声明和全局扩展时有用，业务代码中同名接口散落多处容易让真实结构难追踪。

面试考点

  • interface 和 type 的本质区别是什么？ 结论是 interface 声明对象形状，type 给类型表达式起别名。原因是 type 能表达联合、元组和条件类型，而 interface 有声明合并和面向对象式扩展。项目里二者都能写对象时，应遵循团队约定。
  • 为什么对象 API 经常优先使用 interface？ 结论是对象契约使用 interface 更便于扩展和阅读。原因是它的声明形式天然表示“形状”，并支持 extends 与库声明合并。边界是如果对象类型依赖联合、映射或条件计算，type 反而更合适。
  • 联合类型为什么只能用 type 表达？ 结论是联合是类型表达式，不是单个可声明的对象形状。原因是 interface 不能直接声明 "admin" | "guest" 或 { ok: true } | { ok: false }。项目中请求结果、状态机、事件名通常用 type。
  • 声明合并什么时候是优势，什么时候是风险？ 结论是库扩展和环境声明中是优势，普通业务模型中要谨慎。原因是同名 interface 会自动合并，方便扩展第三方类型，但也可能让字段来源分散。团队可通过命名和 lint 约束避免误合并。

TypeScript

// —— interface 声明对象形状与声明合并 ——
interface User {
  id: number;
  name: string;
}
interface User {
  email?: string; // 同名 interface 会声明合并
}

// —— type 表达联合与交叉 ——
type Role = "admin" | "guest";
type UserWithRole = User & { role: Role };

// —— type 可辨识联合 ApiResult ——
type ApiResult<T> =
  | { ok: true; data: T }
  | { ok: false; error: string };
