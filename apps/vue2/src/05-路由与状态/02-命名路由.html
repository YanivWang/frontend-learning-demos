命名路由（router-link :to="{ name: 'xxx' }"）

点击 User (foo/bar) 、 About 链接，或点「编程式 push」按钮，观察下方 params.id 变化；Cookie 行展示 js-cookie 读取结果（可在 Console 执行 Cookies.set('name','xxx') 后刷新验证）。

知识点要点

  • 命名路由是在路由记录上配置 name ，之后 <router-link> 和 router.push 都可以使用 { name: 'user', params: { id: 'foo' } } 。
  • 命名路由的主要价值是降低路径硬编码：当 /user/:id 改成 /members/:id 时，调用方只要仍然使用同一个 name ，改动面就更小。
  • 传动态参数时，推荐使用 name + params 。如果写 path ，Vue Router 不会自动把 params 拼进 path，应自己拼完整路径或改用命名路由。
  • 本页读取 Cookie 只是演示浏览器状态可以和路由同时展示。Cookie 属于客户端存储/请求状态，不应被当作路由匹配规则的一部分。
  • Vue 2 对应 vue-router 3，仍适合维护旧项目；新项目通常选择 Vue 3 + vue-router 4，但命名路由的选型思想基本一致。

面试考点

  • 命名路由相比 path 字符串有什么优势？ 结论是命名路由更抗重构。原因是调用方依赖稳定的语义名称，而不是散落各处的路径字符串。项目里详情页、编辑页、权限跳转等常用 name ，避免路径调整时漏改。
  • router-link 的对象写法如何传 params？ 结论是写成 :to="{ name: 'user', params: { id: 'foo' } }" 。原因是 Vue Router 根据 name 找到带动态片段的路由记录，再把 params 填入路径。边界是 params 字段名必须和 path 中的 :id 对上。
  • params 和 path 同时写时为什么容易失效？ 结论是不要期待 { path: '/user', params: { id: 1 } } 自动生成 /user/1 。原因是 path 已经是完整路径入口，params 不会再参与动态片段填充。需要动态片段时用命名路由，或自己拼出完整 path。
  • Cookie 这类浏览器状态和路由状态要如何区分？ 结论是 Cookie 表示客户端持久化/请求携带状态，路由表示当前页面位置。原因是 Cookie 会参与跨请求存储，路由只描述 UI 导航。项目里认证 token、偏好设置可在 Cookie 或存储中，当前页签、资源 ID 更适合路由。
  • 命名路由在重构路由路径时有什么价值？ 结论是调用方只依赖稳定的 name ，path 从 /user/:id 改为 /members/:id 时不必逐处改链接。原因是命名路由把语义与 URL 字符串解耦。项目里详情、编辑、权限跳转等高频入口尤其受益。
