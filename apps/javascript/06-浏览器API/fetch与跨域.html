fetch 基础与 CORS 跨域

请阅读本复习区，并在下方演示区点击按钮发起请求；需人工验证：公共 API 请求依赖外网，CORS 拦截细节请结合 DevTools Network 面板观察。

知识点要点

  • fetch() 返回 Promise，但只有网络层失败、请求被中止或浏览器安全策略阻止这类场景会 reject；HTTP 404/500 仍会 resolve 成 Response，所以业务代码必须检查 response.ok 或 status。
  • CORS 是浏览器在同源策略上的受控放行机制。是否允许跨域读取响应由目标服务端通过 Access-Control-Allow-Origin、Access-Control-Allow-Methods、Access-Control-Allow-Headers 等响应头决定，前端单独改请求参数不能替服务端授权。
  • 简单请求会直接发送；使用非简单方法、自定义请求头，或 Content-Type 不属于表单三类时，浏览器会先发 OPTIONS 预检请求。预检失败时 JS 通常只能看到泛化的 TypeError，具体原因要结合 Network 面板和服务端日志排查。
  • mode: "no-cors" 不能解决跨域读取，它只允许发出受限请求并返回不可读的 opaque response，前端拿不到状态码和响应体。真实项目要用后端 CORS 配置、同源代理或后端转发解决。
  • 本页用 textContent 展示请求结果，避免把远程响应拼成 HTML；网络 demo 还要准备失败路径、加载中状态、超时或 AbortController 取消，避免用户面对静默失败。

面试考点

  • fetch 与 axios 在错误处理上有何不同？ 结论是原生 fetch 对 HTTP 4xx/5xx 不会自动 reject，axios 默认会把非 2xx 状态当作 rejected Promise。原因是 Fetch 标准把“请求成功拿到响应”和“业务状态成功”分开。项目里封装请求层时应统一检查 response.ok、解析错误体，并把网络错误、HTTP 错误和业务错误区分开。
  • 什么是 CORS 预检请求？何时触发？ 结论是预检是浏览器在正式跨域请求前发送的 OPTIONS 请求，用来询问服务端是否允许方法、请求头和来源。原因是非简单请求可能产生更强副作用，浏览器需要先确认授权。项目里如果新增 Authorization 或自定义头，要同步配置 Access-Control-Allow-Headers。
  • mode: "no-cors" 能解决跨域吗？ 结论是不能解决“读取跨域响应”的问题，只会得到不可读 opaque 响应。原因是它不会绕过同源策略，浏览器仍不允许 JS 读取未授权响应体。项目里正确做法是服务端 CORS、同源反向代理或后端转发，而不是让前端强行 no-cors。
  • 前端请求失败时如何区分网络错误、HTTP 错误和 CORS 拦截？ 结论是先看 Promise 是否 reject，再看 response.ok/status，同时结合 DevTools Network 判断是否被 CORS 或预检拦截。原因是浏览器出于安全考虑不会把所有跨域失败细节暴露给 JS。项目中要给用户展示可理解的失败提示，并在日志里记录 URL、状态码、请求 ID 和失败阶段。
