展开折叠 · 三种方案对比

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • transition 描述两个状态之间的过渡，但只有可插值的值才能平滑补间；height: auto 依赖内容布局计算，不能直接过渡到具体高度。
  • 固定 height 方案最直观，适合内容高度可控的菜单；内容行数变化时容易截断或留空，需要 JS 测量高度才能精确。
  • max-height 可绕过未知高度问题，但必须设置一个足够大的上限；真实内容高度小于上限时，视觉展开时间会短于声明时长，速度看起来不均匀。
  • grid-template-rows: 0fr → 1fr 能让内容高度自适应，内部容器配 overflow: hidden 和 min-height: 0，避免收起时内容溢出。
  • 展开折叠是交互组件，真实项目还要同步按钮 aria-expanded、键盘焦点、动态内容高度和 prefers-reduced-motion。

面试考点

  • 为什么 height: auto 不能直接做过渡？ 结论是 CSS 不能在 0 和 auto 之间算出连续中间值。原因是 auto 是布局结果，不是一个固定数值。项目里如果必须精确高度动画，可以用 JS 读取 scrollHeight，或用 grid / max-height 方案替代。
  • 三种展开折叠方案怎么选？ 结论是固定高度适合内容稳定，max-height 适合简单自适应，grid 方案适合纯 CSS 且内容高度不固定的面板。原因是它们对内容变化、动画速度和维护成本的约束不同。边界是复杂手风琴、虚拟列表或频繁动态内容通常需要 JS 管理状态和高度。
  • max-height 方案为什么速度不均匀？ 结论是动画跑的是声明的上限，不是内容真实高度。原因是内容达到实际高度后视觉上已经展开完成，但 max-height 仍在继续补间到更大的值。项目里上限越夸张，视觉时长偏差越明显。
  • 展开折叠组件有哪些工程边界？ 结论是动画之外还要处理语义和状态。原因是按钮、面板、焦点和屏幕阅读器都需要知道当前是否展开。项目里应维护 aria-expanded、必要时设置 aria-controls，并在 reduced motion 下关闭过渡。
