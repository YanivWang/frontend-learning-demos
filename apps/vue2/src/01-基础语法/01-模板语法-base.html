Vue 2 基础指令与响应式绑定合集

在浏览器中操作下方交互演示：切换 app_3 对比 v-if / v-show，在 app_4 反转列表观察 :key；复习要点与面试答法见本区块。

知识点要点

  • Vue 2 模板把声明式 DOM 和实例数据连接起来：{{ msg }} 渲染文本（会转义）； v-bind / : 绑定 HTML 属性；v-on / @ 绑定事件；v-model 在表单控件与状态之间做双向同步。
  • v-if 为真时才创建对应 DOM，为假时销毁（惰性：初值为假则不渲染）； v-show 始终保留节点，只通过 display 切换显隐。频繁切换常用 v-show，权限或昂贵子树常用 v-if。
  • v-for 根据数组渲染列表。官方建议提供唯一的 string/number key，便于列表更新时识别节点身份；本页用 item.id，与官方示例一致。
  • 每个实例的 el 指向一个挂载容器（本页六个 #app_* 即六个独立实例）。 data 字段会代理到实例并转为响应式；在 methods 里改 this.msg 后，依赖它的模板会自动更新。
  • v-model 本质是语法糖：对 text/textarea 等价于 :value + @input。它只负责值同步，不等于表单校验、提交或业务规则。
  • 常见坑：Mustache 不能写在 HTML 属性里；列表勿省略稳定 key；同节点上慎用 v-if 与 v-for（v-for 优先级更高），通常先用 computed 过滤再渲染。
  • 本页为教学把知识点拆成多个根实例，避免演示互相影响。真实项目更常见「一个根实例 + 组件树」。

面试考点

  • Vue 2 模板里的插值、指令和响应式数据是什么关系？ 结论是模板声明数据如何映射到 DOM，指令声明特殊绑定行为，响应式系统在数据变化后更新视图。 原因是 Vue 在初始化时把 data 转为 getter/setter 并追踪模板依赖。项目里应改状态，而不是手写 DOM。
  • v-bind、v-on、v-model 分别解决哪类绑定问题？v-model 底层是什么？ 结论是 v-bind 绑属性，v-on 监听事件，v-model 封装表单双向同步。 对 text input，官方说明其等价于 value 属性 + input 事件。项目里属性用 :，事件用 @，表单控件再考虑 v-model。
  • v-if 和 v-show 的根本区别？ 结论是 v-if 控制是否创建/销毁 DOM，v-show 控制已存在节点的显示隐藏。 原因是 v-if 为假时不渲染节点且切换会重建子树；v-show 只改 CSS display。频繁切换用 v-show，不必渲染的块用 v-if。
  • v-if 和 v-for 各自承担什么职责？同元素混用要注意什么？ 结论是 v-if 做条件渲染，v-for 做列表渲染。同节点时 v-for 优先级高于 v-if，官方不推荐混写。项目里通常先过滤数据再 v-for。
  • 为什么 v-for 建议提供稳定 key？ 结论是 key 让 Vue 在 diff 时识别节点身份，便于复用、移动而非盲目就地补丁。 缺少合适 key 时，含子组件状态或表单输入的列表可能错位。优先业务唯一 id；索引仅在静态、无状态列表中可接受。
  • 多个 Vue 根实例演示和真实组件化项目有什么区别？ 结论是多个根实例适合隔离教学，真实项目通常一个根实例加组件树。原因是组件树更利于状态流、复用和调试。 同页多实例多见于迁移老页面或局部增强静态页。
