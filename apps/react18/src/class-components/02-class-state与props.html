class 组件的 state 与 props

请打开 DevTools Console 查看 props/state 输出；点击下方按钮验证 setState、子调父回调与 this 绑定；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • props 是父组件传给子组件的数据入口，子组件应把它当作只读值；示例中 Root 持有 appId ，并把它作为 id 传给 App 。
  • state 是组件自己的可变状态。class 组件可以在 constructor 或类字段中初始化 state，但事件、生命周期等位置必须调用 setState ，否则 React 不知道需要重新渲染。
  • setState 进入 React 更新队列，调用后立刻读取 this.state 通常仍是本轮 render 的旧快照；UI 变化要等下一次 render 提交后才可见。
  • 子组件不能直接修改 props；如果要影响父组件数据，应调用父组件传入的回调。示例中 App 点击按钮后调用 changeAppidFn ，由 Root 更新自己的 state，再把新 props 传回来。
  • class 普通方法作为事件处理器传给 React 后，调用时不会自动绑定组件实例；示例在 constructor 中 bind ，也可以用类字段箭头函数或在 JSX 中包一层箭头函数（详见 13 号 demo）。
  • react.dev 推荐新代码用函数组件 + Hooks 管理 state 与 props；class 组件主要用于维护旧代码和理解单向数据流。

面试考点

  • class 组件中 state 和 props 的职责怎么区分？ 结论是：props 负责父到子的输入，state 负责组件内部随交互变化的数据。原因是 React 的数据流是单向的，父组件重新渲染会把新 props 传给子组件；项目中可复用组件通常用 props 配置外部行为，用 state 管理局部 UI 状态。
  • 为什么不能直接修改 props 或 this.state？ 结论是：props 应视为只读，constructor 之后更新 state 必须用 setState 。原因是直接修改不会进入 React 的更新队列，也会破坏父子数据边界；边界是 constructor 初始化时可以直接给 this.state 赋初值。
  • class 事件处理函数为什么经常需要绑定 this？ 结论是：普通 class 方法不会自动绑定实例。原因是把 this.handleBtnClick 当函数引用传出后，调用方不会保留原来的接收者；维护老代码常见做法是在 constructor 中 bind，现代写法更多用函数组件事件闭包或类字段箭头函数。
  • JSX 里 onClick={fn()} 和 onClick={fn} 有什么区别？ 结论是：前者会在 render 时立即执行，后者才是在点击时执行。原因是 JSX 花括号里是 JavaScript 表达式；项目里需要传参时使用 onClick={() => fn(id)} ，注意避免在大量列表中无意义创建复杂闭包。
  • 子组件如何通知父组件更新传入的 props？ 结论是：调用父组件通过 props 传下来的回调。原因是 props 本身不属于子组件所有；示例中 App 调用 changeAppidFn ，父组件 Root 再通过 setState 改变传给子组件的 id 。
