React.memo 与不可变更新

请打开 DevTools Console 查看 UserCard render 输出；下方 #root 为 React 挂载演示区，切换无关状态时 memo 应跳过子组件渲染。

知识点要点

  • React.memo 包裹函数组件后，React 会用 Object.is 对新旧 props 做浅比较；相等时可以复用上次渲染结果。
  • 浅比较只看第一层引用，对象、数组、函数每次新建都会被视为变化；直接原地修改又会让 React 难以判断哪一项变了。
  • 本 demo 中点击“切换无关状态”时 users 引用不变，UserCard 可跳过；点击 Alice +1 时只为 Alice 返回新对象，Bob 的 props 仍稳定。
  • memo 优化组件重渲染，useMemo 缓存计算结果，useCallback 缓存函数引用；三者都应服务于真实性能瓶颈。
  • 常见坑是把 memo 当默认装饰器。组件很轻、props 总是新引用、或渲染问题来自状态设计时，memo 会增加比较成本和理解成本。

面试考点

  • React.memo 什么时候能减少渲染？ 结论是父组件重渲染但子组件 props 浅比较不变时最有效。原因是 React 可以跳过子组件函数执行。本 demo 切换主题不会改变每个 user 对象引用，所以未变化的卡片可避免重复渲染。
  • 为什么 memo 依赖不可变更新？ 因为 React 的比较依据主要是引用。不可变更新让变化项拿到新引用，未变化项保留旧引用，既能触发正确 UI 更新，也能让 memo 精确跳过未变化项。原地修改数组或对象会破坏这个判断。
  • memo、useMemo、useCallback 分别优化什么？ memo 优化组件是否重渲染，useMemo 优化昂贵计算结果，useCallback 优化函数 props 的引用稳定。边界是不要为了“少一次 render”盲目叠加，先用 React DevTools Profiler 证明瓶颈。
  • 自定义比较函数要注意什么？ 结论是只在浅比较不足且数据结构可控时使用。原因是比较函数本身也有成本，且漏比较函数 props 可能留下旧闭包。表格大行、图表配置这类明确重组件可以考虑，普通展示组件通常不需要。
