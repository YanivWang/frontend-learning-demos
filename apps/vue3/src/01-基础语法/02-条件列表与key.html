v-if / v-show / v-for / key

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • v-if 为真时才创建分支 DOM，为假时节点从文档中移除；本页点「切换显示」为 false 后再变 true，「v-if 挂载次数」会增加，说明分支被重新创建。打开 DevTools → Elements：visible=false 时带 v-if 的段落不在 DOM 树中。
  • v-show 始终渲染元素，只切换 display；同一按钮隐藏后，v-show 段落仍在 Elements 里且带 display: none，切换成本通常低于反复创建销毁。高频显隐实践中常用 v-show，低频或未授权内容常用 v-if。
  • v-for 应绑定稳定、唯一的 key（本页默认 item.id）。在每行 input 输入不同文字后点「反转列表」，内容会跟随任务名移动，因为 Vue 按 key 识别「同一条数据」。
  • 点「切换为 index key」后再反转：input 内容容易留在原位置而非跟随数据项，这就是下标 key 在重排时的典型错位。diff 与 vnode 机制见 08-虚拟DOM-diff-key。
  • 点「新增一项」在列表头部插入带新 id 的行；稳定 key 下旧行 input 状态不会串到新数据。遍历下标 index 可用于展示序号，但不应作为会重排列表的 key。
  • 不要把 v-if 与 v-for 写在同一元素上——Vue3 中 v-if 优先级更高，条件表达式可能访问不到循环变量。本页勾选「只显示新增任务」时，列表来自 filteredList 计算属性，而非在 v-for 标签上叠 v-if。
  • 条件分支与循环语法承接上一页 01-应用实例与模板语法；事件绑定与表单双向同步见下一页 03-事件与表单v-model。

面试考点

  • v-if 和 v-show 如何按切换频率选型？ 结论是低频切换、条件不满足时不必渲染的内容用 v-if，高频显示隐藏用 v-show。原因是 v-if 有创建销毁成本，v-show 有初始渲染成本但切换只是 CSS。本页「切换显示」配合 Elements 面板可对比 DOM 是否存在。
  • v-for 的 key 为什么不能随便用 index？ 先结论：会增删、排序、反转的列表不要用下标 key。原因是下标随位置变化，Vue 可能把旧 DOM/组件状态复用到错误数据上。本页「切换为 index key」+ 行内 input +「反转列表」即可复现错位。
  • v-if 和 v-for 为什么不建议写在同一个元素上？ 结论是 Vue3 中 v-if 优先于 v-for，易造成作用域误解。原因是条件可能读不到循环变量。项目里应先用计算属性过滤，或把 v-if 放到包裹层；本页 filteredList 即正例。
  • 列表反转或插入时稳定 key 解决了什么问题？ 结论是稳定 key 让 Vue 识别「同一条数据」而非「同一个位置」，从而移动节点并保留 input 等内容。原因是 patch 按 key 做同层比对。表单行、可编辑表格、动画列表都必须用业务 id。
  • v-if 重新显示时，分支内 DOM 状态为何会重置？ 结论是 v-if=false 时分支被销毁，再次为真会新建节点，内部非响应式 DOM 状态不会保留。原因是条件渲染不是 CSS 隐藏。本页「v-if 挂载次数」在反复切换后会递增；深层 diff 见 08-虚拟DOM-diff-key。

v-if vs v-show

v-show：节点始终存在，只切换 display。
