useLayoutEffect 与 useEffect 对比

请打开 DevTools Console 查看两种 Effect 的执行顺序；下方 #root 为 React 挂载/交互演示区，点击按钮切换 tooltip 文案。

知识点要点

  • useEffect 会在浏览器完成绘制后运行，适合网络请求、订阅、日志等不需要卡住首帧的副作用。
  • useLayoutEffect 会在 React 把 DOM 更新提交后、浏览器绘制前同步运行；如果在里面读布局并立刻写 state，用户通常看不到中间错误位置。
  • 本页 tooltip 先渲染到 DOM，再用 getBoundingClientRect() 判断是否需要上移，正是 useLayoutEffect 的典型场景。
  • 常见坑是把请求、计时器或昂贵计算放进 useLayoutEffect，它会延迟绘制并让页面显得卡顿；能用 useEffect 就不要升级为布局副作用。
  • 同构渲染中服务端没有布局信息，useLayoutEffect 只在客户端有意义；需要 SSR 的组件应避免依赖首屏同步测量，或延后到客户端交互后再显示。

面试考点

  • useLayoutEffect 和 useEffect 的核心区别是什么？ 结论是执行时机不同：前者在绘制前同步执行，后者在绘制后异步执行。原因是 React 需要给布局副作用一个修正 DOM 的窗口。边界是只有测量尺寸、滚动位置、避免闪烁这类视觉同步问题才用 useLayoutEffect。
  • 为什么数据请求不要默认放进 useLayoutEffect？ 结论是请求、订阅、日志应优先放进 useEffect。原因是这些工作不需要阻塞首帧，放进布局副作用会拖慢绘制。项目里只有请求结果会立即决定布局且无法接受闪烁时，才考虑重构交互或拆出占位 UI。
  • tooltip 测量为什么适合 useLayoutEffect？ 结论是它要在用户看到页面前完成位置修正。原因是先提交 DOM 才能读取真实尺寸，再同步更新 top 可避免 tooltip 从错误位置跳动。边界是测量逻辑要短小，并把依赖数组限制在真正影响尺寸的值上。
  • SSR 中使用 useLayoutEffect 要注意什么？ 结论是服务端没有浏览器布局，布局副作用只能在客户端运行。原因是 DOM 尺寸、滚动和视口都依赖真实浏览器环境。需要 SSR 的项目通常用 CSS 预留空间、客户端再增强，避免首屏强依赖同步测量。
