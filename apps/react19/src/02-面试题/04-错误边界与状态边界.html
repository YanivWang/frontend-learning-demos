错误边界与 Action 状态边界

下方 #root 为三类错误处理演示区：点击「触发渲染错误」观察 Error Boundary 兜底；点击「业务失败」观察 Action 返回 state 而非进入边界。请打开 DevTools Console 查看边界捕获日志。

知识点要点

  • Error Boundary 捕获子组件渲染、生命周期方法和构造函数中的错误；React 19 仍使用 class 组件的 static getDerivedStateFromError 与 componentDidCatch 实现边界。
  • 边界不会自动捕获事件处理器、定时器、Promise 回调、服务端渲染阶段或边界自身抛出的错误；这些位置需要 try/catch、状态建模或框架级错误处理。
  • useActionState 适合把表单 Action 的可预期业务失败返回为状态，例如本页的「库存不足」；它不会代替错误边界，也不应该把用户可修正错误都变成兜底异常页。
  • 状态边界处理「还能继续交互」的失败，错误边界隔离「子树无法继续渲染」的未知错误；项目里常按路由、卡片或高风险组件设置边界，避免整页白屏。
  • 重置错误边界时要同步修复触发错误的状态；本页按钮同时清掉边界内部错误和外层 throwRender，真实项目也可通过重置 key 重新挂载子树。

面试考点

  • Error Boundary 能捕获哪些错误，不能捕获哪些错误？ 结论是它捕获子树渲染、生命周期和构造阶段错误，不捕获事件处理器、异步回调、服务端渲染和自身错误。原因是边界参与的是 React 渲染提交链路，事件和异步任务已经离开该链路。项目里事件错误要本地处理，渲染风险点才用边界隔离。
  • React 19 中函数组件能不能直接写错误边界？ 结论是不能，官方错误边界仍要求 class 组件生命周期。原因是 getDerivedStateFromError 和 componentDidCatch 目前没有等价的内置函数组件 Hook。实际项目可封装 class 边界供函数组件树使用，或使用框架提供的路由错误边界。
  • 表单 Action 的业务失败为什么应该返回 state？ 结论是可预期失败应返回页面可展示状态。原因是用户需要看到错误并继续修正输入，进入错误边界会丢失上下文且体验过重。比如库存不足、校验失败、权限不足通常显示在表单旁；未知异常才交给边界或监控。
  • 错误边界和状态边界在项目里如何分层？ 结论是业务状态靠组件局部或 Action state，渲染崩溃靠 Error Boundary。原因是二者恢复方式不同：业务失败可重试，渲染错误需要隔离和重置子树。大型页面可按路由、微前端、数据面板设置边界，避免单点崩溃拖垮整页。
