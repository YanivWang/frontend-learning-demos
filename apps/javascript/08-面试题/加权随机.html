加权随机 / 抽奖概率

请打开 DevTools Console 查看输出。

知识点要点

  • 加权随机的核心是把权重转成连续区间：例如 A:0.5, B:0.3, C:0.2 对应 [0,0.5) 、 [0.5,0.8) 、 [0.8,1) ，随机数落在哪段就返回哪项。
  • 扩容数组法把每个候选项重复放入查找表，再用均匀随机取下标，写法直观但空间复杂度和权重放大倍数相关，权重很大或有小数时不适合。
  • 累加权重法先计算总权重，再生成 [0,total) 的随机阈值，遍历时不断累加，第一次超过阈值的项就是结果；单次抽取时间 O(n) ，额外空间可做到 O(1) 。
  • 权重输入要校验：空对象、非数字、负数、全 0、 NaN 都应拒绝或明确兜底，否则可能返回 undefined 或造成概率偏差。
  • Math.random() 适合前端演示、动画和非安全抽样，不提供密码学安全性；真实抽奖、红包、风控等涉及利益的场景应由服务端使用可信随机源并记录审计。
  • 验证概率只能看统计近似，样本越多越接近期望比例；单次或少量抽样不可能保证严格按权重分布。

面试考点

  • 加权随机如何用累加区间实现？ 结论：先求总权重，再生成一个小于总权重的随机阈值，遍历累加到超过阈值时返回当前项。原因是每个项占据的区间长度等于它的权重，随机数落入该区间的概率自然与权重成正比。边界是权重必须为有限非负数，并且总权重要大于 0。
  • 扩容数组法和前缀权重法怎么比较？ 结论：扩容数组适合讲解，累加权重更适合通用实现。原因是扩容数组需要把权重放大成整数份，空间成本和精度取舍都不可控；累加权重只依赖数值区间，空间可做到常量级。项目里如果需要高频抽取，可以预先存前缀和再用二分查找，把单次抽取降到 O(log n) 。
  • 为什么前端 Math.random() 不适合真实抽奖开奖？ 结论：它不是安全随机源，也不应由客户端决定有利益的开奖结果。原因是浏览器随机算法不保证密码学安全，客户端代码和请求都可被观察或篡改。真实抽奖应放在服务端，用可信随机源、签名日志和幂等校验保证可审计。
  • 如何验证加权随机实现是否大致符合权重？ 结论：用大量采样统计命中率，并和期望比例比较。原因是随机结果有波动，样本量越大越接近期望分布。边界是采样验证只能发现明显偏差，不能证明随机源安全，也不能保证每个短窗口都严格满足比例。
