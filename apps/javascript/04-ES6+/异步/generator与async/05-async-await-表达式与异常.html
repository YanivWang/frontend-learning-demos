await 表达式与异常处理

请打开 DevTools Console 查看输出。

知识点要点

  • await 后面可以是 Promise、thenable 或普通值；普通值会按已 fulfilled 的值处理，但 async 函数后续代码仍会让出到微任务恢复。
  • async 函数 return 的普通值会成为外层 Promise 的 fulfilled value；未捕获异常或 rejected await 会让外层 Promise rejected。
  • 任一 await 的 Promise rejected，当前 async 函数会像同步 throw 一样中断，后续语句不再执行，除非用 try...catch 捕获。
  • 本页用 localforage 演示存储 Promise：浏览器隐私模式、配额不足、序列化失败或底层存储不可用都可能 reject，不能只写成功路径。
  • 如果多个存储写入互不依赖，提前创建多个 Promise 会并发执行；如果需要严格顺序，应传入函数并在 async 函数内部逐个调用。

面试考点

  • await 后面不是 Promise 时会发生什么？ 它会按 Promise resolution 规则处理表达式，普通值会成为 fulfilled value，thenable 会被吸收。边界是即使值已经可用， await 后续代码仍会异步恢复，不会继续留在当前同步调用栈。
  • async 函数 return 和 throw 如何影响外层 Promise？ return value 会让返回的 Promise fulfilled 为 value；抛错或未捕获的 await rejection 会让返回的 Promise rejected。调用方必须用 await try...catch 或 .catch 处理失败。
  • await 的 Promise reject 后后续代码是否继续？ 默认不会继续，控制流会跳到最近的 catch 或让 async 函数返回 rejected Promise。项目里如果要失败后继续执行，应在当前步骤局部捕获并提供默认值。
  • 使用浏览器存储 API 时需要补哪些失败路径？ 要处理配额不足、权限限制、隐私模式、序列化失败和库未加载等情况。存储成功后再更新 UI，失败时给用户提示并避免把未确认的数据当作已持久化状态。
