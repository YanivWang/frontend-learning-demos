CommonJS 与 ES Module 对比

请打开 DevTools Console 查看输出。

知识点要点

  • CommonJS 使用 require / module.exports，依赖在运行时解析，同步加载语义更适合 Node.js 早期本地文件模块。
  • ES Module 是 ECMAScript 语言标准，使用静态 import / export，浏览器通过 <script type="module"> 原生加载。
  • ESM 的静态结构让构建工具能做依赖图分析、tree shaking 和预加载；需要条件加载时使用动态 import()。
  • ESM 导入的是 live binding，导出方变量变化时导入方能观察到；CommonJS 常见写法是导出对象，属性读取时机和重新赋值边界更容易混淆。
  • Node.js 中 CJS 和 ESM 可共存，但受 package.json 的 type、.cjs/.mjs 扩展名和互操作规则影响，不能简单等同于浏览器 ESM。

面试考点

  • CJS 和 ESM 的核心区别是什么？ 结论是 CJS 偏运行时加载，ESM 偏静态链接。原因是 require 是函数调用，可放在分支中；静态 import 必须在模块顶层，便于浏览器和构建工具提前建立依赖图。
  • 导出值和绑定有什么差异？ ESM 导入 live binding，导出方更新变量后导入方可观察到；CJS 常见导出是对象属性，重新给 module.exports 或属性赋值的观察时机取决于调用方何时读取。项目里迁移模块时要特别小心默认导出、命名导出和循环依赖。
  • 浏览器如何原生使用 ESM？ 通过 <script type="module">，模块默认严格模式、延迟执行，并按 URL 解析依赖。边界是浏览器不能直接识别裸模块名，通常需要 import maps 或构建工具。
  • 现代前端为什么优先 ESM？ ESM 是语言和浏览器标准，能支持静态分析、tree shaking、动态 import() 和更好的工具链协作。维护 Node 遗留包时仍会遇到 CJS，需要按运行时规则做互操作。
