HTML5 拖放复制（带 UI 反馈）

这个 demo 在放置区创建源元素副本，重点观察 dataTransfer、复制语义、拖拽高亮和副本删除这些项目里常见的拖放细节。

知识点要点

  • 复制闭环： dragstart 写入源元素的 data-item-id ， dragover 调用 preventDefault() 允许放置， drop 再读取 id 找回源节点并克隆。
  • 操作语义： effectAllowed = "copy" 表示拖动源允许复制， dropEffect = "copy" 表示当前放置目标按复制处理；两者配合能让光标反馈和真实行为一致。
  • 克隆边界： cloneNode(true) 会复制元素和后代，但不会复制通过 addEventListener() 绑定的监听器。示例给副本重新创建删除按钮监听器，而不是依赖源节点事件。
  • 副本清理： 副本不再保留 draggable ，并用 data-cloned 标记，避免被继续拖拽、误当成源数据或产生重复 id。
  • 交互边界： 放置区高亮用 dragenter / dragleave 计数器稳定状态，避免拖过内部子元素时高亮闪烁；复杂嵌套放置区还可结合 relatedTarget 判断。

面试考点

  • 如何用 HTML Drag and Drop 实现复制而不是移动？ 结论是在 drop 中根据拖拽数据找到源节点并创建副本，而不是把源节点 append 到新容器。原因是 append 原节点会移动 DOM；项目里还要清理副本的 id、拖拽属性和业务状态。
  • effectAllowed 和 dropEffect 分别解决什么问题？ effectAllowed 是拖动源声明允许哪些操作， dropEffect 是放置目标选择当前操作。两者不匹配时浏览器反馈可能和代码意图不同；复制、移动、链接场景要明确设置。
  • cloneNode(true) 复制节点时有哪些边界？ 它会深拷贝子节点和属性，但不会复制 addEventListener() 绑定的监听器，也可能复制重复 id。项目里通常会重新绑定交互、生成新 id，并只复制必要数据而不是整段 DOM。
  • 拖放 UI 反馈为什么容易在 dragleave 时闪烁？ 因为拖拽经过放置区子元素时，事件目标会变化，浏览器可能触发 dragleave 。复杂放置区需要判断是否真正离开放置区，否则高亮会反复闪烁。
  • 原生拖放做可视化编辑器时还缺哪些能力？ 原生 API 提供数据传递和基础事件，但触屏手势、自定义预览、吸附、排序动画和复杂命中检测通常还要额外实现。此时可结合 Pointer Events 或专门的拖拽库。
