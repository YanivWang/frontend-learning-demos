useOptimistic 乐观更新

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • useOptimistic(state, updateFn) 用于异步 Action 完成前先展示临时 UI，让用户马上看到“发送中”“已点赞”等反馈。
  • updateFn 必须是纯函数，根据当前权威 state 和乐观输入计算下一屏状态；不要在里面发请求、改外部变量或生成不可控副作用。
  • 真实 state 更新后，乐观层会和最终结果对齐；如果请求失败且权威 state 没变，临时项应消失，并显示可理解的错误。
  • 适合发消息、点赞、列表追加等成功概率高、失败可回滚的交互；支付、库存扣减、权限变更这类强一致场景通常不应乐观确认。
  • 本页输入“失败”会模拟服务端拒绝，消息会从乐观列表中回滚，并展示错误提示，避免用户误以为已经发送成功。

面试考点

  • useOptimistic 解决什么问题？ 结论是提升异步提交的即时反馈。原因是用户发消息或点赞后不希望等待网络返回才看到变化。边界是它只处理临时 UI 层，服务端结果、失败文案和重试仍由业务负责。
  • 乐观 UI 为什么必须处理失败？ 结论是失败时要回滚或标记失败，不能静默保留假状态。原因是乐观结果不是权威数据，和服务端冲突时必须以真实 state 为准。项目里常见做法是显示错误、提供重试或把条目标成“发送失败”。
  • updateFn 为什么要求纯函数？ 结论是它只应该计算乐观状态。原因是 React 可能根据渲染和 Action 生命周期重新计算乐观层，副作用会造成重复请求或状态污染。边界是 ID、时间戳等输入应在调用 addOptimistic 前准备好。
  • 什么时候不适合乐观更新？ 结论是强一致、失败代价高或权限不确定的操作不适合直接乐观确认。原因是用户看到的结果如果随后回滚，会带来信任和业务风险。支付、审批、库存扣减通常应等待服务端确认再展示成功态。
