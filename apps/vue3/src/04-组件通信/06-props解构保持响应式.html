props 解构保持响应式（Vue 3.5+）

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • Vue 3.5+ 在同一个 <script setup> 代码块中，会把从 defineProps 解构出来的变量访问编译为 props.xxx，因此读取时仍能追踪 prop 更新。
  • 普通 setup(props) 是运行时代码，const { label } = props 会把基本类型取成当前快照；本页点击按钮后可看到直接解构面板不再更新。
  • toRefs(props) 会把每个 prop 包成 ref，适合普通 setup、组合式函数传参或 Vue 3.4 及以下项目；模板直接使用 props.label 也是稳定写法。
  • Vue 3.5 的编译期解构不是通用 JavaScript 能力，离开 SFC 编译上下文、传给普通函数或在老版本中都不能假定它保持响应式。
  • watch 或组合式函数需要追踪 prop 时，优先传 getter，例如 () => props.label，或传 toRef(props, 'label')；不要把解构快照当作响应式源。

面试考点

  • Vue 3.5+ 的响应式 props 解构改变了什么？ 结论是只改变 <script setup> 中 defineProps 解构变量的编译结果。编译器会把变量访问改写成 props.xxx，所以父组件更新 prop 后子组件读取仍是最新值。边界是这不是普通 JS 解构语义，也不是全局构建能直接演示的运行时能力。
  • 为什么普通 setup 直接解构仍会丢响应？ 因为普通解构发生在运行时，只把当前 prop 值赋给局部常量。基本类型后续不会自动跟随父组件变化，对象虽然引用还在，但替换整个对象也会出现快照问题。普通 setup 中应保留 props.xxx 或使用 toRefs。
  • toRefs(props) 和 defineProps 解构怎么选？ Vue 3.5+ 的 SFC 内部本地读取可以用响应式解构，代码简洁。需要传入组合式函数、支持 Vue 3.4 及以下，或在普通 setup 里工作时用 toRef / toRefs 更明确。团队要按项目 Vue 版本统一约定。
  • watch props 时最稳的写法是什么？ 结论是传 getter 或 ref 源，例如 watch(() => props.label, ...)。原因是 watch 需要可追踪的响应式源，直接传解构后的普通值只会拿到一次快照。封装组合式函数时也应明确参数是 getter、ref 还是普通值。
