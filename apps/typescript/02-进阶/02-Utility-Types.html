Utility Types

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • Utility Types 是 TypeScript 标准库提供的内置泛型类型工具，用来基于已有类型派生新类型，避免手写重复的 DTO、表单补丁和展示模型。
  • Partial<T> 把属性变可选，Required<T> 把属性变必填，Readonly<T> 把属性变只读；它们只影响类型检查，不会冻结运行时对象。
  • Pick<T, K> 从对象类型中挑字段，Omit<T, K> 排除字段，本页示例用它们表达“更新入参”和“对外展示用户”的差异。
  • Record<K, V> 适合把联合键映射到同一种值，例如角色文案、权限配置、状态到渲染组件的查表。
  • 常见坑是过度叠加工具类型导致类型难读，或把 Partial 用在深层对象后误以为嵌套字段也会递归可选；需要深层变换时要单独定义递归工具或显式建模。

面试考点

  • Partial、Required、Readonly 分别解决什么问题？ 结论是它们分别把属性改成可选、必填和只读。原因是底层通过映射类型遍历 keyof T 并调整属性修饰符。边界是这些都是浅层类型变换，不会在运行时补字段或冻结对象。
  • Pick、Omit 和 Record 在业务里怎么用？ 结论是 Pick 表达白名单字段，Omit 表达排除敏感字段，Record 表达完整映射表。原因是它们能把业务模型和派生模型绑定起来，字段改名时让编译器提醒。项目里常用于 API DTO、列表展示和权限配置。
  • Utility Types 为什么不会改变运行时数据？ 结论是它们只存在于类型系统。原因是 TypeScript 类型会在编译后擦除，真正的数据拷贝、过滤或冻结仍要写 JavaScript 逻辑。示例里的 publicUser 是手动创建的运行时对象，不是 Omit 自动生成的。
  • 什么时候不该继续堆工具类型？ 结论是当类型表达已经影响阅读或包含业务语义时，应显式命名新类型。原因是 Partial<Pick<...>> 适合短派生，复杂组合会隐藏意图。团队项目里可以把常用派生命名为 UserPatch、PublicUser 这类稳定边界。

interface User {
      id: number;
      name: string;
      email: string;
      role: "admin" | "guest";
      }
      type UserPatch = Partial<Pick<User, "name" | "email">>;
      type PublicUser = Omit<User, "email">;
      type CompleteUser = Required<User>;
      type FrozenUser = Readonly<User>;
      type RoleLabels = Record<User["role"], string>;
      const roleLabels: RoleLabels = { admin: "管理员", guest: "访客" };
