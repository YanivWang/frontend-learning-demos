requestAnimationFrame 与递归调用对比

请阅读本复习区，并在下方演示区启动/停止动画；同时打开 DevTools Console 查看帧时间戳输出。

知识点要点

  • 直接同步递归会在同一轮调用栈里不断调用自己，缺少退出条件时会造成栈溢出； requestAnimationFrame() 则把下一次回调交给浏览器，在下一次重绘前执行。
  • rAF 与浏览器渲染节奏协同，适合驱动视觉动画、滚动跟随、拖拽中的合帧更新。它不是“固定 60Hz 定时器”，高刷新率屏幕、系统负载和后台标签页都会影响回调频率。
  • rAF 回调会收到 DOMHighResTimeStamp ，动画应根据时间差计算位移或进度，而不是每帧固定加一个数。这样在 60Hz、120Hz 或掉帧时速度才更稳定。
  • 想停止动画必须保存 requestAnimationFrame() 返回的 id，并用 cancelAnimationFrame(id) 取消下一帧；组件卸载、路由切换、弹窗关闭时都应清理，避免后台循环继续跑。
  • 与 setTimeout / setInterval 相比，rAF 更适合“下一次绘制前更新 UI”；定时器更适合非视觉任务，但可能和渲染不同步并产生掉帧或无效绘制。

面试考点

  • requestAnimationFrame 为什么比同步递归更适合动画循环？ 结论是同步递归会连续占用当前调用栈，而 rAF 会把下一次更新排到下一帧绘制前。原因是浏览器可以在帧边界统一处理样式、布局、绘制和回调。项目里做动画循环时应在回调尾部再次申请 rAF，而不是直接递归调用函数。
  • rAF 和 setTimeout/setInterval 做动画有什么差异？ 结论是 rAF 面向视觉更新并和渲染节奏同步，定时器只按任务队列延迟执行。原因是定时器可能在浏览器不准备绘制时触发，导致无效更新或帧不稳定。实际动画、拖拽和滚动联动优先 rAF，非视觉轮询或超时逻辑才用定时器。
  • 为什么动画逻辑应该使用回调时间戳计算进度？ 结论是应按时间差推进动画，而不是假设每秒 60 帧。原因是用户设备刷新率可能是 60Hz、120Hz，也可能因为负载掉帧或后台降频。项目里常用 delta = timestamp - lastTimestamp 计算位移，保证不同设备上的速度一致。
  • 组件卸载或页面隐藏时如何停止 rAF 循环？ 结论是保存 rAF id，在卸载、路由切换或显式停止时调用 cancelAnimationFrame() 。原因是只要下一帧已被申请，回调就可能继续运行并持有闭包引用。项目里还可监听 visibilitychange 暂停动画，回到前台后重新同步时间戳。
