defaultProps、参数默认值与 displayName

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • 函数组件默认值优先写在参数解构里，例如 function Greeting({ name = "访客" })，这与普通 JavaScript 默认参数一致。
  • defaultProps 在旧函数组件和 class 组件代码中常见；新函数组件更推荐参数默认值，类型工具、代码阅读和迁移都更直接。
  • 默认值只在 prop 为 undefined 或未传入时生效；如果父组件显式传 null，React 会把 null 作为真实值交给子组件。
  • displayName 不影响渲染结果，主要影响 React DevTools、错误堆栈和调试可读性；HOC、memo、forwardRef 包装后尤其需要保留可读名称。
  • 从 class 组件迁移时，static defaultProps 可改为函数参数默认值；如果旧代码仍使用 Component.defaultProps，应避免和参数默认值重复表达同一规则。

面试考点

  • 函数组件为什么更推荐参数默认值？ 结论是它是现代函数组件的首选写法。原因是默认值逻辑直接出现在函数签名里，和 JavaScript、TypeScript 推导更一致。维护旧代码时仍会看到 defaultProps，但新代码不必为了兼容旧风格而继续使用。
  • 默认值遇到 null 会不会生效？ 不会。默认值只处理未传入或 undefined。原因是 null 表示父组件有意传入“空值”。项目里如果 null 也要兜底，应在组件内部显式用空值合并或条件判断表达业务含义。
  • displayName 有什么作用？ 结论是它服务调试而不是业务逻辑。原因是 React DevTools 和错误堆栈需要可读组件名。HOC、memo、forwardRef 包装后函数名可能变得不直观，设置 displayName 能保留组件来源。
  • class 组件 defaultProps 如何迁移？ 结论是 class 的 static defaultProps 可迁移为函数组件参数默认值。原因是函数组件没有实例生命周期依赖默认 props 的必要。边界是大型旧代码库可分步迁移，但同一组件内不要同时维护两套默认值。
