useState 异步合并与连续更新

请打开 DevTools Console 查看点击瞬间 count 快照；下方 #root 为 React 挂载/交互演示区，分别点击两个按钮对比结果。

知识点要点

  • setCount 不会修改当前这次 render 里的 count 变量，而是把更新加入队列，请求 React 在下一次渲染中使用新 state。
  • 同一个事件回调中， count 是当前 render 的快照。连续三次 setCount(count + 1) 都计算出同一个新值，所以本 demo 的“快照式连续 +3”通常只增加 1。
  • 函数式更新 setCount(prev => prev + 1) 会让 React 按队列顺序把上一次更新结果传给下一次 updater，因此连续三次可以稳定累加 3。
  • React 18 会批处理同一事件中的多次 state 更新，减少重复渲染。批处理不等于丢更新；是否能累加取决于你传的是固定值还是 updater 函数。
  • 当新 state 依赖旧 state 时，优先使用函数式更新；当新值完全来自事件输入或接口结果时，可以直接传新值。

面试考点

  • setCount 后为什么同步读取仍是旧值？ 结论是当前 render 的变量不会被 setter 原地改写。原因是 React 把更新排队，等当前事件处理结束后再渲染；项目里需要观察更新后的结果应看下一次 render 或 effect。
  • 连续三次 setCount(count + 1) 为什么只加 1？ 结论是三次都基于同一个 count 快照计算。原因是 count + 1 在调用 setter 前已经算好了；边界是如果三次发生在不同 render 中，就会基于不同快照。
  • 函数式更新解决什么问题？ 结论是它解决“新值依赖旧值”的连续更新问题。原因是 React 会按队列依次调用 updater，把前一个 updater 的结果传给下一个；计数、追加数组、切换布尔值都适合这种写法。
  • React 18 批处理会不会丢更新？ 结论是批处理不会主动丢更新，但固定值更新可能覆盖成同一个结果。原因是批处理合并提交，更新队列仍会执行；项目里依赖旧状态时用 updater 就能表达正确意图。
  • 什么时候必须使用函数式更新？ 结论是新 state 依赖旧 state 时必须用 updater。原因是同一事件内闭包里的 state 是快照；典型场景：计数器连点、列表 append、toggle、基于 prev 的条件分支。
