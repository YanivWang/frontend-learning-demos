统一的图片 / 音频预加载器

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • new Image() / new Audio() 创建离屏资源对象后设置 src，浏览器会像普通资源一样发起请求；后续使用同 URL 时能复用 HTTP 缓存或内存缓存。
  • 本 demo 的 onProgress(current, total) 是“已完成资源数 / 总资源数”，适合驱动加载页进度条；它不是字节级进度，单个大图和小图权重相同。
  • 真实项目必须处理失败路径：图片 error、音频 error 也要计入完成并记录失败 URL，否则任一资源 404、跨域失败或网络中断都会让 onComplete 永远不触发。
  • 预加载不等于强制缓存。是否复用取决于 URL 完全一致、HTTP 缓存响应头、浏览器缓存策略和资源优先级；带随机查询参数的 URL 通常会绕开缓存。
  • <link rel="preload"> 更适合首屏关键资源的声明式优先加载；new Image() 更适合游戏、轮播、相册等运行时才能确定的图片列表。
  • 图片如果后续绘制到 canvas，还要考虑 CORS 与画布污染；仅作为 <img> 展示时，跨域限制相对少，但网络失败仍需要占位图或重试策略。

面试考点

  • new Image() 预加载为什么能让后续渲染更快？ 结论：因为设置 src 后浏览器会提前请求图片，后续相同 URL 的 <img> 通常能命中缓存。原因是资源请求和缓存由浏览器网络层管理，不要求元素已经插入 DOM。边界是缓存复用取决于 URL、响应头和浏览器策略，跨页面或弱网场景不能保证必然瞬时完成。
  • 资源预加载进度条为什么通常只能做数量进度？ 结论：用离屏图片对象只能可靠知道每个资源 load/error 完成，无法知道每个图片的实时字节进度。原因是 HTMLImageElement 不提供下载字节进度事件。项目里若需要精确字节进度，要改用 fetch 流式读取并自行解码，但会增加兼容性、缓存和 CORS 复杂度。
  • 预加载失败时应该怎么设计？ 结论：失败要进入进度统计，并单独暴露失败 URL 给业务兜底。原因是加载页的目标是进入可用状态，而不是因为某个非关键资源失败永久阻塞。项目中常见做法是记录失败、展示占位图、允许重试，并区分关键资源和非关键资源。
  • new Image() 和 <link rel="preload"> 如何选型？ 结论：首屏关键资源优先用 preload 声明意图，运行时列表优先用 JS 预加载。原因是 preload 能参与浏览器优先级调度且更早发现资源，JS 预加载则灵活可控。边界是 preload 使用错误会抢占带宽，JS 预加载过多也会影响首屏和移动端流量。
