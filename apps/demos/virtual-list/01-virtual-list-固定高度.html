虚拟列表（固定行高 40px）

这个 demo 渲染 10000 条固定高度数据，但 DOM 中只保留视口附近的行，重点观察滚动高度、可见范围和 translateY 偏移。

知识点要点

  • 核心思想： 虚拟列表不是真的渲染 10000 个 DOM 节点，而是用 .phantom 撑开总高度，只渲染视口可见行和少量 buffer。
  • 固定行高公式： start = floor(scrollTop / ROW_H) - BUFFER ， end = ceil((scrollTop + viewportHeight) / ROW_H) + BUFFER ，再夹到 0..TOTAL 范围内。
  • 位置还原： 真实行容器用 transform: translateY(start * ROW_H) 移动到它在完整列表中的位置，视觉上仍像完整列表在滚动。
  • 性能边界： scroll 事件可能高频触发，示例用 requestAnimationFrame 合并同一帧内的多次滚动回调；行数少时普通列表更简单；行内容复杂时还要考虑 DOM 复用和避免同步布局读取。
  • 动态高度差异： 固定高度可以直接用除法定位，动态高度需要缓存每行高度、前缀和或二分查找，否则滚动位置和占位高度会不准。

面试考点

  • 虚拟列表为什么能提升大列表性能？ 结论是它把 DOM 数量从总数据量降低到视口附近数量。原因是浏览器布局、绘制和事件处理都和 DOM 规模有关；项目里适合几千到几万行、每行结构相近的大列表。
  • 固定行高虚拟列表如何根据 scrollTop 计算渲染范围？ 先用 scrollTop / ROW_H 算出第一个可见索引，再根据视口高度算结束索引，并补上下 buffer。边界是结果必须 clamp，避免负索引或超过总数。
  • buffer 的作用是什么，设置过大或过小有什么问题？ buffer 用来提前渲染视口外少量行，减少快速滚动时的空白。过小容易闪烁，过大会增加 DOM 数量；项目里通常根据行高、滚动速度和设备性能调参。
  • 固定高度和动态高度虚拟列表的难点有什么不同？ 固定高度可用公式直接定位，动态高度必须测量和缓存每项高度。动态高度还要处理图片加载、内容展开导致高度变化的回算问题。
  • 高频滚动场景如何避免渲染抖动？ 结论是减少每次滚动做的工作。可以用轻量计算、复用 DOM、必要时用 requestAnimationFrame 合并更新，并避免在滚动处理中反复触发布局测量。
