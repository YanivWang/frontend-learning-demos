Promise.all 并发 与 then 串行

请打开 DevTools Console 查看输出。

知识点要点

  • Promise.all(iterable) 接收一组 Promise 或普通值，返回一个新的 Promise；全部 fulfilled 后，结果数组按输入顺序排列，而不是按完成先后排列。
  • Promise.all 本身不“创建任务”，并发来自你已经调用了多个异步函数并拿到多个 Promise；总耗时通常接近最慢的那个任务。
  • Promise.all 是 fail-fast：任一输入 reject，外层 Promise 就以首个拒因 reject；如果需要收集每个任务的成功/失败结果，应选择 Promise.allSettled 或给单个任务包 catch 。
  • 串行执行要在 then 回调里返回下一个任务的 Promise，例如 .then(() => taskP1()) ；直接写 .then(taskP1()) 会立刻启动任务并把 Promise 实例当作非函数参数传入。
  • new Promise(executor) 的 executor 会同步执行；真正异步的是 resolve 后排队执行的 then / catch 回调。

面试考点

  • Promise.all 为什么是并发等待而不是自动创建并发？ 结论是 Promise.all 只聚合已经传入的 Promise，不负责启动异步任务。原因是任务在调用 taskP1() 时就开始了；项目里要并发请求就先调用多个请求函数，要串行就把下一次调用放进前一个 then 或 await 之后。
  • Promise.all 的结果顺序如何确定？ 结果数组按输入 iterable 的顺序排列，和各任务完成时间无关。边界是任一任务 reject 会让整体 reject，成功数组不会返回；需要部分成功结果时不要直接用裸 Promise.all 。
  • Promise.all 、 allSettled 和串行 then 链如何选？ 多个互不依赖且任一失败就整体失败时用 all ；需要拿到每个任务状态时用 allSettled ；后一个任务依赖前一个结果或要限流时用串行链。项目里请求详情页多个独立接口常用 all ，批量上传需要逐项结果常用 allSettled 。
  • 为什么 then 里要传函数而不是直接传 Promise 实例？ then 需要的是回调函数，回调返回的 Promise 会决定下一步等待什么。直接传 Promise 实例既不会被当作回调执行，也容易让任务提前启动，导致串行代码实际变成并发。
