props 解构保持响应式（Vue 3.5+）

在浏览器中操作下方交互演示；复习要点与面试答法见本复习区。

知识点要点

  • Vue 3.5+ 在同一个 <script setup> 代码块中，会把从 defineProps 解构出来的变量访问 编译 为 props.xxx ，因此读取时仍能追踪 prop 更新。本页标题旁 Vue 3.5+ 标记编译宏能力。
  • 普通 setup(props) 是运行时代码， const { label } = props 会把基本类型取成当前快照；点「父组件改 label」后，红色「直接解构」面板 不更新 ，绿色 toRefs / props.label 面板同步更新。
  • toRefs(props) 会把每个 prop 包成 ref，适合普通 setup、组合式函数传参或 Vue 3.4 及以下项目；模板直接使用 props.label 也是稳定写法。toRefs 细节见 02-toRef-toRefs 。
  • Vue 3.5 的编译期解构不是通用 JavaScript 能力，离开 SFC 编译上下文、传给普通函数或在老版本中都不能假定它保持响应式。本页 pre 区展示 SFC 写法，下方三面板对比运行时行为。
  • watch 需要追踪 prop 时，优先传 getter（ () => props.label ）或 toRef(props, 'label') ；本页 WatchPanel 的 watch 日志仅在 toRef 写法下随父改 label 追加。不要把解构快照当作响应式源。
  • props 只读契约不变，见 01-props与emit 。defineModel（3.4+）是另一编译宏，见 05-defineModel 。

面试考点

  • Vue 3.5+ 的响应式 props 解构改变了什么？ 结论是只改变 <script setup> 中 defineProps 解构变量的编译结果。编译器会把变量访问改写成 props.xxx 。边界是这不是普通 JS 解构语义，全局构建 demo 只能用 toRefs 对照。
  • 为什么普通 setup 直接解构仍会丢响应？ 因为普通解构发生在运行时，只把当前 prop 值赋给局部常量。本页红色面板在父改 label 后冻结即证据。普通 setup 中应保留 props.xxx 或使用 toRefs 。
  • toRefs(props) 和 defineProps 解构怎么选？ Vue 3.5+ 的 SFC 内部本地读取可以用响应式解构，代码简洁。需要传入组合式函数、支持 Vue 3.4 及以下，或在普通 setup 里工作时用 toRef / toRefs 更明确。
  • watch props 时最稳的写法是什么？ 结论是传 getter 或 ref 源，例如 watch(() => props.label, ...) 。WatchPanel 的 toRef watch 日志随父更新而追加，直接解构值则不会。
  • Vue 3.4 及以下如何避免陷阱？ 结论是不使用 defineProps 响应式解构（尚未提供），统一 toRefs(props) 或 props.xxx 。升级 3.5 后 SFC 可简化，但 composable 传参仍建议 toRef 保持版本无关。
