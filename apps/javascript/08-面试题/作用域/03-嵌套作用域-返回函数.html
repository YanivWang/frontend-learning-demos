嵌套函数 - 返回内层函数引用

先看本复习区，再查看下方输出是否与标准答法一致；细节可打开 DevTools Console。

知识点要点

  • checkscope 返回的是内层函数 f 的引用，外部通过 checkscope()() 再执行它，因此这是典型闭包题。
  • f 引用了外层局部变量 scope ，函数被返回后仍保留对定义时词法环境的引用，所以最终输出 "local" 。
  • 词法作用域由函数定义位置决定，不由调用位置决定；即使 f 在全局位置被调用，它查找 scope 时仍先回到 checkscope 的局部环境。
  • 与上一题 return f() 对比：两题输出都为 "local" ，但上一题返回字符串结果，本题返回函数引用并让外层环境继续可达。
  • 闭包常用于函数工厂、私有状态、事件回调和缓存，但如果闭包长期持有大型对象、DOM 节点或定时器引用，可能导致内存无法及时释放。
  • 现代项目仍推荐优先用 let / const 明确变量边界；面试老题里的 var 主要用于突出函数作用域和提升行为。

面试考点

  • 为什么 checkscope()() 仍然输出 local ？ 结论：因为第一次调用返回了 f ，第二次调用执行的是带着外层词法环境的闭包。原因是 f 定义在 checkscope 内部，自由变量 scope 会优先解析到局部 "local" 。边界是如果返回前没有任何函数或对象引用该环境，局部变量才会在调用结束后变得不可达。
  • return f 和 return f() 有什么区别？ 结论： return f 返回函数本身， return f() 返回函数执行结果。原因是前者把函数引用带出外层作用域，后者在外层函数返回前就计算出结果。项目里返回函数通常用于延迟执行、参数预置或保存私有状态，返回结果则只是一次性计算。
  • 闭包为什么会让外层局部变量不被立即回收？ 结论：因为外部仍然可以通过返回的函数访问这些变量。原因是垃圾回收基于可达性， f 还活着，它引用的词法环境也必须保留。边界是闭包不会“复制”变量值，而是保留对变量环境的访问，多个闭包可能共享同一份状态。
  • 项目中使用闭包保存状态要注意什么？ 结论：闭包适合封装私有状态，但要控制生命周期。原因是长生命周期回调、缓存函数、事件监听器如果持有大对象或 DOM，会让这些对象持续可达。实际项目里要在组件卸载、路由切换或任务取消时移除监听、清理定时器和断开无用引用。
