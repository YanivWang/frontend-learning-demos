class 组件生命周期与常用实例 API

打开控制台观察 constructor、render、componentDidMount、shouldComponentUpdate、componentDidUpdate 的调用顺序，并操作下方按钮验证 state 与 forceUpdate 的区别。

知识点要点

  • 挂载阶段：constructor 用来初始化 state 和绑定实例字段，render 必须保持纯渲染，访问 DOM、启动 timer、订阅事件或发起请求应放到 componentDidMount。
  • 更新阶段：setState 或 props 变化会进入更新流程；shouldComponentUpdate 可返回 false 跳过当前组件渲染，componentDidUpdate 适合根据上一次 props/state 做 DOM 同步或补充请求，但必须避免无条件再次 setState。
  • 卸载阶段：componentWillUnmount 是清理副作用的位置，timer、订阅、手写 DOM 监听和未取消的异步回调都要在这里处理，否则容易出现内存泄漏或卸载后更新状态的警告。
  • 旧生命周期边界：componentWillMount、componentWillReceiveProps、componentWillUpdate 在现代 React 中被标记为不安全；维护旧 class 代码时只能以 UNSAFE_ 前缀保留，并优先迁移到 componentDidMount、componentDidUpdate、getDerivedStateFromProps 或函数组件 Hooks。
  • 实例 API：forceUpdate 会强制当前组件重新 render，并跳过当前组件的 shouldComponentUpdate；它只适合桥接非 React 数据源，常规业务状态应放进 state/props，事件处理器也要用箭头函数或显式绑定保证 this 指向组件实例。

面试考点

  • class 组件挂载、更新、卸载阶段各自适合放什么逻辑？ 结论是 render 只负责返回 UI，副作用按时机放到提交后的生命周期。因为 constructor 和 render 可能在渲染流程中被调用，不能依赖真实 DOM；DOM 测量、订阅和 timer 应在 componentDidMount 建立，在 componentWillUnmount 清理。项目里如果更新后要同步滚动或请求数据，放在 componentDidUpdate 并比较前后 props/state。
  • 为什么 constructor 里不能做网络请求或订阅？ 结论是 constructor 只做初始化。原因是此时组件尚未挂载，DOM 不存在，副作用也没有对应的清理时机；并发渲染和严格检查语境下，把副作用放进渲染前阶段更容易产生重复执行和难以回收的问题。真实项目里请求通常放到 componentDidMount，函数组件则放到 useEffect。
  • shouldComponentUpdate、forceUpdate 和 setState 的边界是什么？ 结论是业务更新优先用 setState，性能裁剪才考虑 shouldComponentUpdate，forceUpdate 是少用的命令式兜底。因为 setState 会进入 React 的调度和合并流程，shouldComponentUpdate 可以阻止不必要渲染，而 forceUpdate 会绕过当前组件的该判断。维护旧代码时只有 render 依赖了非 state/props 的外部可变值，才考虑临时使用 forceUpdate。
  • React 18 项目里遇到 UNSAFE_ 生命周期应该怎么迁移？ 结论是先确认它原来承载的是初始化、副作用、派生状态还是 DOM 快照，再迁移到对应现代 API。因为这些旧生命周期发生在提交前，容易和异步渲染模型冲突。常见做法是把请求和订阅迁到 componentDidMount/componentDidUpdate，把 props 派生状态收敛到受控数据流或 getDerivedStateFromProps，新代码则优先写函数组件和 Hooks。
