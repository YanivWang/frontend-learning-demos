Vuex 全家桶（state / mutations / actions / getters / mapState / mapGetters）

点「commit increment」「dispatch increment」「dispatch incrementAsync（约 3 秒）」观察 count 、 getCount 、 doneTodosCount 变化；下方 User / About 子组件展示 mapState / mapGetters 映射结果。

知识点要点

  • Vuex 是 Vue 2 时代常见的集中式状态管理库，适合多个组件共享登录用户、权限、购物车等状态；新项目选型通常把 Vue 3 + Pinia 作为主线，Vuex 多用于 Vue 2 或存量项目维护。
  • state 是单一状态树，字段最好在初始化时声明；Vue 2 响应式基于 Object.defineProperty ，后加根级字段不会自动具备完整响应式能力。
  • mutation 是同步修改 state 的唯一推荐入口，便于调试工具记录每一次状态变更； action 可以做异步，最终通过 commit 提交 mutation。
  • getter 类似 store 级 computed。普通 getter 返回派生值；返回函数的 getter 可接外部参数，例如 doneTodos(2) ，但 doneTodos.length 取到的是函数参数个数，不是结果数组长度。
  • mapState 、 mapGetters 会生成 computed 配置；如果映射函数里要访问组件局部状态 this.localCount ，应使用普通函数，箭头函数没有组件实例的 this 。

面试考点

  • Vuex 的核心角色怎么分工？ 结论是 state 存共享数据，mutation 同步改数据，action 处理异步流程，getter 做派生读取。原因是 Vuex 希望状态变化可预测、可追踪。项目里组件读取 state/getter，用户操作 dispatch action 或 commit mutation。
  • 为什么不能随意直接改 Vuex state？ 结论是应该通过 mutation 修改。原因是 mutation 是同步、可记录的状态变更入口，调试工具才能追踪变更来源。边界是技术上能直接改对象属性，但会破坏团队约定和调试体验，不适合作为项目写法。
  • getter 返回函数有什么坑？ 结论是返回函数的 getter 是“先取 getter，再调用函数”，不是一个数组结果。原因是它用于按参数查询，例如 doneTodos(2) 。项目里如果要显示已完成数量，应另写返回数组或数量的 getter，不能读函数的 length 当数组长度。
  • mapState 里什么时候不能用箭头函数？ 结论是需要访问组件实例时不能用箭头函数。原因是箭头函数不会绑定 Vue 组件的 this 。项目里像 countPlusLocalState(state) { return state.count + this.localCount } 应写普通函数。
