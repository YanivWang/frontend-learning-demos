Clock 演示 setState 批处理（对象式 vs 函数式）

请打开 DevTools Console 查看批处理前后 count 快照；点击下方按钮立即触发对比，或等待 6s 定时器自动触发；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • counterOld 连续三次对象式 setState 都先用旧 this.state.count1 算出同样的下一个值，React 批处理并浅合并后，count1 每轮只增加 1。
  • counterNew 连续三次函数式 setState 会按队列顺序拿到最新 prevState.count2，因此每轮增加 3；页面同时展示两个计数值，方便观察差异。
  • setState 后立即读取 this.state 仍可能是旧快照。需要观察提交后的结果时，可以使用 setState 第二个参数回调、componentDidUpdate，或把 UI 直接交给下一次 render 展示。
  • 定时器、订阅、手动 DOM 监听都属于副作用；class 组件应在 componentDidMount 建立，在 componentWillUnmount 清理，拼写错误会导致 React 无法调用清理逻辑。
  • 新项目通常用函数组件表达 Clock：用 useEffect 建立和清理 interval，用 useState 保存计数；维护老 class 时仍需理解本页的批处理和生命周期边界。

面试考点

  • 对象式 setState 和函数式 setState 在批处理中为什么结果不同？ 结论是：对象式会把旧 state 先算成对象，函数式会在队列执行时逐步读取最新 prevState。原因是 React 批处理时会合并多个更新；项目中只要依赖前一次结果，就应写函数式更新。
  • setState 后立即读取 this.state 为什么不可靠？ 结论是：因为 setState 安排更新，不等于同步改写实例字段。原因是 React 可能把多次更新合并到一次 render；如果要做提交后的日志、埋点或 DOM 测量，要放到回调或更新后的生命周期中。
  • 定时器驱动的 class 组件为什么必须清理 componentWillUnmount？ 结论是：否则组件卸载后定时器仍会运行，继续调用 setState 或持有资源。原因是浏览器定时器不受 React 自动管理；项目中还要同样清理订阅、WebSocket、DOM 事件监听等外部资源。
  • setState 第二个参数回调适合解决什么问题？ 结论是：适合在 state 提交并完成重新渲染后执行一次性逻辑。原因是它比立即读取 this.state 更可靠；边界是复杂派生逻辑不要塞进回调，优先通过 render 或 componentDidUpdate 表达。
  • 维护 Clock 这类 class 组件时如何迁移到函数组件？ 结论是：用 useState 存储计数和时间，用 useEffect 负责 interval 的建立与清理。原因是函数组件是现代 React 的主线；但迁移时仍要保留"依赖旧值用函数式更新、卸载时清理副作用"的规则。
