@layer — 级联层管理样式优先级

观察三个按钮：同一个 .demo-btn 会被后声明的 utilities 层覆盖，而未分层 override 又能覆盖所有普通分层规则。

知识点要点

  • @layer 创建的是 cascade layers（级联层），用于在 CSS 级联中新增一层“先比较层，再比较特异性”的管理维度。它适合把 reset、基础样式、组件样式、工具类和业务覆盖分清楚。
  • 本页先声明 @layer reset, components, utilities;，因此正常声明中 utilities 层优先于 components，components 又优先于 reset。按钮默认显示黄色，是因为 utilities 层最后胜出。
  • 未分层的普通样式在正常声明中优先于所有分层样式，所以 .demo-btn.override 即使没有使用 !important，也能覆盖 @layer 中的同类按钮规则。
  • 同一层内部仍遵守普通级联规则：来源、重要性、选择器特异性和源码顺序都会继续生效。@layer 不是取消特异性，而是把层顺序放到特异性比较之前。
  • @layer 和视觉层叠上下文无关，不控制元素谁压在谁上面；z-index、定位、transform 等影响的是绘制层叠，@layer 管的是 CSS 规则谁覆盖谁。

面试考点

  • @layer 在 CSS 级联顺序里解决了什么问题？ 结论是它让团队可以显式声明样式层级，减少靠选择器加重或 !important 抢优先级。原因是层顺序会在特异性之前参与比较，后面的层可以稳定覆盖前面的层。项目里常把第三方 reset 放低层，把组件和工具类放高层。
  • 为什么未分层样式能覆盖 @layer 里的规则？ 结论是在正常声明中，未分层规则优先级高于任意分层规则。原因是 CSS Cascade 规范把未分层正常声明排在分层正常声明之后。边界是 !important 的比较方向和普通声明不同，使用时要单独核对，不建议用它作为常规覆盖手段。
  • @layer、选择器特异性、源码顺序三者怎么比较？ 结论是先比较来源和重要性，再比较级联层，然后才在同层内比较特异性和源码顺序。原因是 @layer 的目的就是让“这类样式应该在哪个层级”比“选择器写得多重”更早决定结果。项目中应先设计层顺序，再控制选择器复杂度。
  • @layer 和 z-index 的层叠上下文有什么区别？ 结论是 @layer 管 CSS 规则覆盖，z-index 和 stacking context 管视觉绘制顺序。原因是二者属于不同模型：一个是级联算法，一个是布局绘制。项目里按钮颜色被谁覆盖看 @layer，弹窗是否压住遮罩看 stacking context。
