provide / inject

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • provide 在祖先组件登记依赖，inject 在任意深度后代读取依赖，适合主题、表单上下文、插件能力这类跨层共享，不适合替代清晰的父子 props。
  • 本页 provider 提供 theme 这个 ref，深层子组件注入后仍指向同一个响应式对象；Vue 不会自动解包注入的 ref，模板渲染时才会按模板规则解包。
  • 如果 provider 提供普通值，后代拿到的是当前值而不是响应式连接；需要共享变化时应提供 ref、reactive、computed 或显式更新函数。
  • 修改共享状态时推荐像本页一样同时提供 toggleTheme，由数据提供方集中控制写入逻辑，避免多个后代直接改同一份状态导致来源混乱。
  • 项目中应给注入项准备默认值，公共库或大型应用中优先用 Symbol 作为 key，避免字符串 key 冲突；缺失注入会得到 undefined，调用函数前要确认 provider 存在。

面试考点

  • provide / inject 解决什么问题？ 结论是解决祖先到深层后代的依赖传递，避免中间组件只为转发数据而声明 props。原因是注入按组件祖先链查找最近 provider，适合主题、表单上下文、i18n、插件服务等横切依赖。边界是它会弱化显式数据流，普通父子通信仍优先用 props / emit。
  • 注入的值一定是响应式的吗？ 不一定，响应式取决于 provider 提供的值本身。提供 ref 或 reactive 时后代能保持连接，提供普通字符串或对象快照时不会自动变成响应式。项目里要共享变化，应提供响应式值或由 provider 暴露只读值加写入函数。
  • 为什么不建议后代直接改注入状态？ 结论是写操作最好留在 provider 或由 provider 暴露命令函数。这样状态来源、校验、埋点和失败处理都集中，排查问题更容易。只有非常局部、约定明确的组合式上下文，才考虑让后代直接持有可写引用。
  • provide / inject 和 Pinia 怎么选？ 局部组件树依赖选 provide / inject，全应用共享、需要 DevTools、持久化或跨路由状态选 Pinia。provide / inject 没有全局状态管理的约束和调试能力，适合封装组件库内部上下文。复杂业务状态不要为了少建 store 而层层注入。
