类型声明与模块声明

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • .d.ts 是声明文件，只描述已有 JavaScript 或全局对象的类型形状，不应该放业务实现；它会参与类型检查，但不会生成运行时代码。
  • declare module "legacy-chart" 可为没有类型的第三方包补充模块声明，也可用于声明 *.svg、*.css 这类非 TS 资源模块。
  • declare global 可扩展 Window、ImportMetaEnv 等全局类型；在模块文件里通常需要配合 export {}，避免声明被当成脚本级全局污染。
  • 声明文件只让编译器知道“这里应该有什么”，不会创建真实包、函数或属性；本页脚本里仍手动提供 renderChart 和 window.appVersion 的运行时实现。
  • 常见坑是用宽泛的 any 声明掩盖第三方库风险，或把声明文件纳入错误的 include/typeRoots 范围导致 IDE 找不到类型。

面试考点

  • .d.ts 和普通 .ts 文件有什么区别？ 结论是 .d.ts 只声明类型契约，普通 .ts 可以包含实现。原因是声明文件描述已经存在的运行时能力，不负责创建它。项目里给 JS 库、全局变量和资源模块补类型时才写声明文件。
  • declare module 能解决什么问题？ 结论是它能让 TS 认识缺少类型的模块。原因是编译器需要知道模块导出的函数、对象和类型结构，才能检查调用方。边界是声明越粗糙，类型保护越弱；能安装官方类型或 @types 时优先使用维护好的声明。
  • declare global 扩展 Window 要注意什么？ 结论是要确保声明在正确的模块上下文中，并且运行时真的写入了对应属性。原因是全局扩展会影响整个项目，声明和实现不一致会让代码通过编译但运行时报错。团队项目里应把全局扩展集中放在专门的 env.d.ts 或 global.d.ts。
  • 声明文件为什么不能替代真实实现？ 结论是声明文件只影响编译器，不影响浏览器或 Node 的运行时。原因是类型会被擦除，模块加载仍需要真实 JS 文件或包。接入遗留库时既要补声明，也要确认依赖安装、打包解析和全局脚本加载路径都正确。
