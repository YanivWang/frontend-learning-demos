Node.js vs 浏览器事件循环

请打开 DevTools Console 查看输出。

知识点要点

  • 浏览器事件循环由 HTML 标准定义：一次任务执行完后会执行微任务检查点，清空 Promise.then 、 queueMicrotask 等微任务，然后浏览器才有机会渲染或处理下一任务。
  • Node.js 事件循环运行在 libuv 阶段模型上，常见阶段包括 timers、poll、check； setTimeout 进入 timers， setImmediate 进入 check，I/O 回调通常和 poll 阶段相关。
  • process.nextTick 是 Node 专有队列，不属于浏览器微任务 API；Node 会在继续事件循环前先清空 nextTick 队列，再处理 Promise 微任务，所以它通常早于 Promise.then 。
  • setImmediate 与 setTimeout(fn, 0) 的先后不能脱离上下文硬背：在主模块中顺序可能受调度和版本影响，在 I/O 回调内部通常 setImmediate 更早执行。
  • Node 20 起随 libuv 1.45 调整，timers 只在 poll 之后运行；涉及精确顺序的面试题要说明 Node 版本、代码位置和是否在 I/O 回调内。
  • 递归追加 process.nextTick 或微任务会让事件循环迟迟无法进入 timers/poll/check，实际服务端项目中可能造成 I/O 饥饿、请求延迟或监控误判。
  • 运行环境： 浏览器部分在 file:// 与 http(s):// 均可运行； process.nextTick 、 setImmediate 仅 Node 环境可用， 需人工在终端验证 ，不能在本 HTML 页面内直接执行。

面试考点

  • 浏览器与 Node 事件循环有何异同？ 结论是二者都有“同步先执行、微任务优先于后续任务”的共同点，但调度模型不同。原因是浏览器受 HTML 事件循环和渲染管线约束，Node 受 libuv 阶段、I/O 和专有队列约束；项目里分析顺序必须先确认运行环境。
  • process.nextTick 与 Promise.then 谁先执行？ 结论是在 Node 中 process.nextTick 通常先于 Promise 微任务。原因是 nextTick 队列会在事件循环继续前被优先清空；边界是浏览器没有 process.nextTick ，跨端代码不要依赖它。
  • setImmediate 与 setTimeout(0) 在 Node 中顺序如何？ 结论是不能只背固定顺序。原因是 setTimeout 属于 timers， setImmediate 属于 check；主模块中可能受计时器阈值和版本影响，在 I/O 回调内通常 setImmediate 先运行。
  • 为什么不能用浏览器事件循环规则硬套 Node？ 结论是会误判 I/O、 setImmediate 和 nextTick 。原因是浏览器需要协调渲染、用户输入和任务队列，Node 更关注服务端 I/O 阶段；排查线上延迟时要结合 Node 官方事件循环阶段和实际版本。
  • nextTick 递归调用有什么风险？ 结论是可能饿死事件循环。原因是 Node 会优先清空 nextTick 队列，如果回调不断追加 nextTick，timers、poll 和 I/O 回调都得不到执行；项目里应改用任务拆分、流式处理或背压机制。

Node.js 对照代码

console.log("sync start");
          setTimeout(() => console.log("timer"), 0);
          setImmediate(() => console.log("immediate"));
          Promise.resolve().then(() => console.log("promise"));
          process.nextTick(() => console.log("nextTick"));
          console.log("sync end");

在 Node 中通常先看到同步日志，再看到 nextTick 、Promise 微任务； timer 与 immediate 的相对顺序要结合代码位置和 Node 版本判断。
