Worker 基础与子脚本

请打开 DevTools Console 查看输出。

知识点要点

  • new Worker("worker.js") 会创建一个独立的 worker 执行环境，适合把计算、解析、压缩等耗时任务移出主线程，避免阻塞点击、滚动和渲染。
  • Worker 没有主线程的 window 和 DOM 访问能力，脚本里的全局对象通常通过 self 使用；需要更新页面时只能把结果发回主线程。
  • 主线程和 worker 通过 postMessage() 与 message 事件通信，消息默认按结构化克隆算法复制；函数、DOM 节点等不可克隆对象会失败，大型 ArrayBuffer 可考虑 transfer 避免复制成本。
  • worker.js 里用 importScripts("childWorker.js") 同步加载子脚本，这是 classic worker 的写法；现代模块化代码可用 new Worker(url, { type: "module" }) 后在 worker 中使用静态 import。
  • 生命周期要显式收尾：主线程不再需要任务时调用 worker.terminate()，worker 内部完成或收到停止命令时可调用 self.close()；同时要监听 error，避免失败后资源悬挂。

面试考点

  • Web Worker 能解决什么问题，不能直接操作哪些浏览器对象？ 结论是它能把耗时 JS 放到独立线程执行，降低主线程卡顿，但不能直接操作 DOM。原因是 worker 有独立的全局作用域和事件循环，只暴露一部分 Web API。项目里常把图片处理、JSON 解析、加解密和大量计算放进 worker，再把结果发回主线程渲染。
  • postMessage 为什么不能传函数或 DOM 节点，大数据传输如何优化？ 结论是普通消息依赖结构化克隆，函数、DOM 节点这类带执行环境或页面引用的对象不可克隆。原因是浏览器需要在线程边界复制或转移数据，不能共享任意对象引用。传大块二进制时优先使用 transferable objects，例如把 ArrayBuffer 所有权转给 worker，减少复制成本。
  • importScripts() 与 module worker 的 import 有什么区别？ 结论是 importScripts() 属于 classic worker，同步加载并执行脚本；module worker 使用标准 ESM import，更适合现代工程。原因是两者的加载、作用域和依赖解析模型不同。老 demo 或简单脚本可用 importScripts()，新项目更推荐 { type: "module" } 配合构建工具。
  • terminate() 和 self.close() 分别适合在哪一侧调用？ 结论是主线程调用 worker.terminate() 强制结束 worker，worker 内部调用 self.close() 主动结束自己。原因是控制权所在不同：主线程负责管理任务生命周期，worker 负责在任务完成或收到停止命令时自我收尾。实际项目还要处理错误、取消和重复创建，避免僵尸 worker 或重复计算。
