知识点要点

  • localforage 的价值是用接近 localStorage 的 key-value 接口封装 IndexedDB，并在不支持时降级到 WebSQL 或 localStorage，让业务代码统一使用 Promise。
  • 所有读写方法都是异步的：setItem 完成后再 getItem 才能稳定读到最新值；示例用 async / await 串行展示，避免多个 Promise 并发导致日志顺序误导。
  • createInstance({ name, storeName }) 可以把 demo 数据隔离到独立仓库；业务代码不应随意调用 clear()，除非确认要清空整个 store，删除单个 key 优先用 removeItem()。
  • iterate、keys、length 适合调试和管理小规模 key 集合；大量数据仍应设计索引、分页或直接使用 IndexedDB 游标，避免一次性遍历造成性能问题。
  • 失败路径必须处理：配额不足、用户清理、隐私模式、降级驱动差异、Blob 支持差异都可能让 Promise reject；缓存类数据要能从网络或默认值恢复。

面试考点

  • localforage 相比直接用 localStorage 解决了什么问题？ 结论是它把存储改成异步 Promise 模型，并优先使用 IndexedDB。原因是 localStorage 同步、只能存字符串且不适合大数据；项目里需要离线缓存对象或文件时，localforage 比手写 IndexedDB 更轻量。
  • localforage 的异步 API 为什么要按顺序 await？ 结论是有依赖关系的读写必须串行等待。原因是 setItem、getItem、removeItem 都返回 Promise，不等待就会产生竞态；例如刚写入就读取、刚清理就遍历，日志顺序可能和真实存储状态不一致。
  • clear 和 removeItem 在业务数据安全上有什么区别？ 结论是 removeItem 删除单个 key，clear 清空整个 store。原因是同一个 store 里可能有多个业务模块数据；真实项目要用独立实例隔离命名空间，并谨慎暴露清空能力。
  • localforage 与直接使用 IndexedDB 如何选？ 结论是简单 key-value 缓存选 localforage，复杂查询、索引、事务控制选 IndexedDB。原因是 localforage 屏蔽了大量样板代码，但也隐藏了底层能力；离线数据库、搜索索引和多对象仓库通常需要直接建模 IndexedDB。
  • localforage 写入失败时应该处理哪些边界？ 结论是把写入失败当成正常异常处理。原因是配额、浏览器策略、隐私模式、驱动降级和数据类型都可能失败；项目里应捕获错误、提示用户、清理旧缓存或回退到网络数据。
