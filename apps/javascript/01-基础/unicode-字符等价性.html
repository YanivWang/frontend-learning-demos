Unicode 字符等价性（NFC / NFD）

请阅读本复习区，并在下方演示区输入两段文本比较；同时打开 DevTools Console 查看 normalize 示例输出。

知识点要点

  • Unicode 允许多种编码表示同一「字符」（如 é 可以是单码点或 e+组合重音）。
  • NFC 会尽量组合为预组合字符，NFD 会拆成基字符 + 组合标记；str.normalize('NFC') 后再比较可减少误判。
  • NFKC / NFKD 是兼容规范化，可能把某些兼容字符折叠成普通形式；适合搜索或宽松匹配，不一定适合保留用户原文。
  • 全角/半角、不同空格（如 NBSP）、大小写和语言特定排序不完全由 normalize 解决，必要时配合 Intl.Collator 或业务规则。
  • 用户输入清洗流程常见为 trim → 折叠连续空白 → normalize → 再比较/入库，但应保留原始输入用于展示或审计。
  • normalize 不能消除所有同形字符风险，例如拉丁 a 与西里尔 а 是不同字符；账号、域名等安全场景需要白名单或混合脚本检测。

面试考点

  • 相同外观为何不相等？ 结论是它们可能由不同 Unicode 码点序列组成。比如 é 可以是单个预组合码点，也可以是 e 加组合重音；直接 === 比较的是序列，不是视觉外观。
  • NFC、NFD、NFKC、NFKD 有什么区别？ 结论是 NFC/NFD 处理规范等价，NFKC/NFKD 还处理兼容等价。NFC 适合大多数存储与比较，NFKC 会改变更多字符形态，适合搜索归一化但不一定适合展示原文。
  • 用户名、搜索和去重要注意什么？ 结论是先定义业务等价规则，再做规范化。常见做法是 trim、折叠空白、normalize('NFC')，再按场景决定是否忽略大小写或全半角；数据库唯一约束要和前端规则一致。
  • normalize 能解决同形字符攻击吗？ 结论是不能完全解决。它处理的是 Unicode 规范化形式，不会把所有视觉相似字符合并；安全敏感标识需要限制字符集、检测混合脚本，或使用成熟的 IDN / 账号策略。
