Error 类型、try/catch/finally 与 error.cause

请打开 DevTools Console 查看输出。

知识点要点

  • try 包裹可能抛错的代码；catch (err) 接收任意 thrown 值（不一定是 Error 实例）；finally 在 try/catch 之后总会执行，常用于释放资源。
  • 内置 Error 子类表达语义：TypeError、ReferenceError、SyntaxError、RangeError 等；err.message 是人类可读描述，err.stack 含调用栈（非标准但普遍实现）。
  • new Error("msg", { cause: rootErr }) 或 err.cause = rootErr 建立错误链，便于日志与监控还原根因（ES2022）。
  • Promise 的 rejection 是异步的，外层同步 try/catch 接不住未 await 的 Promise；须 .catch()、try { await p } 或全局 unhandledrejection。
  • 自定义错误：class AppError extends Error { constructor(msg) { super(msg); this.name = "AppError"; } }，配合 instanceof 区分业务错误与系统错误。

面试考点

  • try/catch/finally 执行顺序？ 结论：try 抛错则进 catch，finally 总在 try/catch 之后执行；finally 里的 return 可能覆盖 try/catch 的 return。原因是 finally 设计用于清理。项目中避免在 finally 里 return，以免掩盖真实结果。
  • error.cause 解决什么问题？ 结论：包装错误时保留原始异常，便于排查。原因是高层 API 常把底层错误转成统一格式，若只留 message 会丢栈。监控上报应递归展开 cause 链。
  • async throw 外层 try/catch 为什么接不住？ 结论：async 函数返回 Promise，throw 变成 rejection。原因是同步 try 只包当前调用栈，Promise 在微任务阶段才 reject。必须 await 或在 Promise 链上 .catch。
  • 如何设计错误类型？ 结论：用 Error 子类区分可恢复/不可恢复、用户输入/系统故障。原因是统一 catch (e) 后可根据 instanceof 或 code 字段分支。避免 throw 字符串，优先 Error 对象保留 stack。
