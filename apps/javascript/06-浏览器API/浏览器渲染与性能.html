浏览器渲染与性能

请打开 DevTools Console 查看输出。

知识点要点

  • 关键渲染路径通常包括解析 HTML 得到 DOM、解析 CSS 得到 CSSOM，再结合可见节点生成渲染树；浏览器随后计算布局、绘制像素并把图层合成到屏幕。
  • Layout 负责计算元素几何信息，Paint 负责把背景、文字、边框等绘制成像素，Composite 负责合成图层；不同属性变化会触发不同阶段，成本也不同。
  • 重排（layout/reflow）来自尺寸、位置、字体等几何信息变化；重绘（paint）来自颜色、阴影等视觉变化；只改 transform / opacity 更可能停留在合成阶段。
  • 示例里的“逐个插入”故意在每次写 DOM 后读取 offsetHeight ，会迫使浏览器尽快计算布局；这类读写交错就是常见的 layout thrashing。
  • DocumentFragment 可以先在内存中组装多个节点，再一次性插入文档，减少中间状态对布局和绘制的影响；真实项目还应使用虚拟列表、分页和按需渲染控制节点数量。
  • requestAnimationFrame 适合把视觉更新安排到下一帧执行，但它不能自动修复低效 DOM 结构；优化仍要从减少节点、读写分离、避免强制同步布局入手。

面试考点

  • 重排、重绘、合成分别是什么？如何减少？ 结论是重排改几何，重绘改像素，合成组合图层。原因是布局变化会影响后续位置尺寸计算，通常成本最高；项目里应减少同步布局、批量 DOM 操作，动画优先使用 transform / opacity 。
  • DocumentFragment 批量插入为何更快？ 结论是它能把多个节点先离线组装，再一次插入文档。原因是中间节点不在当前文档树里，不会每添加一个都形成可见中间状态；边界是最终插入大量节点仍然会有一次布局和绘制成本。
  • 为什么 transform / opacity 动画性能更好？ 结论是它们更容易在合成阶段完成。原因是改变平移、缩放、透明度通常不需要重新计算其他元素布局；但如果元素很多、图层过多或有复杂滤镜，仍可能占用 GPU/内存并掉帧。
  • 什么是 layout thrashing，如何用读写分离避免？ 结论是 layout thrashing 是反复写样式/DOM 后马上读布局信息，导致浏览器多次强制同步布局。原因是浏览器本可批量计算布局，但读 offsetHeight / getBoundingClientRect() 会要求立即给出最新值；项目里应先批量读，再批量写。
  • requestAnimationFrame 在渲染性能优化中适合做什么？ 结论是适合调度下一帧的视觉更新和动画循环。原因是回调会在浏览器下一次重绘前执行，能和刷新节奏对齐；边界是它不是节省计算的魔法，重任务仍需拆分、降采样或移到 Worker。
