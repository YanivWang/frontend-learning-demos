provide / inject

在浏览器中操作下方交互演示；复习要点与面试答法见本复习区。

知识点要点

  • provide 在祖先组件登记依赖， inject 在任意深度后代读取依赖，适合主题、表单上下文、插件能力这类跨层共享，不适合替代清晰的父子 props 。本页中间层 MiddlePanel 不声明 theme 相关 props，深层子组件仍能注入。
  • 本页 provider 提供 theme 这个 ref ，深层子组件注入后仍指向同一个响应式对象；Vue 不会自动解包注入的 ref ，模板渲染时才会按模板规则解包。点「切换主题」父/深子任一处点击，背景 class 同步变化。
  • 如果 provider 提供普通字符串，后代拿到的是当前值快照而不是响应式连接；需要共享变化时应提供 ref 、 reactive 、 computed 或显式更新函数。本页 pre 区对比「注入 ref」与「注入普通值」差异说明。
  • 修改共享状态时推荐同时提供 toggleTheme ，由数据提供方集中控制写入逻辑，避免多个后代直接改同一份状态导致来源混乱。事件日志区记录每次切换来源（父或深子）。
  • 项目中应给注入项准备默认值（ inject('key', defaultValue) ），公共库或大型应用中优先用 Symbol 作为 key，避免字符串 key 冲突。props 逐级传递对比见 01-props与emit 。
  • composable 也可内部使用 provide/inject 封装上下文，见 03-composable复用 。全应用共享状态仍优先 Pinia。

面试考点

  • provide / inject 解决什么问题？ 结论是解决祖先到深层后代的依赖传递，避免中间组件只为转发数据而声明 props 。本页 MiddlePanel 零 props 转发即例证。边界是普通父子通信仍优先 props / emit。
  • 注入的值一定是响应式的吗？ 不一定，响应式取决于 provider 提供的值本身。提供 ref 或 reactive 时后代能保持连接，提供普通字符串时不会自动变成响应式。本页 theme ref 切换时 UI 同步即 ref 连接证明。
  • 为什么不建议后代直接改注入状态？ 结论是写操作最好留在 provider 或由 provider 暴露命令函数。这样状态来源、校验、埋点和失败处理都集中。本页 toggleTheme 由 provider 定义，深子只调用函数。
  • provide / inject 和 Pinia 怎么选？ 局部组件树依赖选 provide / inject，全应用共享、需要 DevTools、持久化或跨路由状态选 Pinia。复杂业务状态不要为了少建 store 而层层注入。
  • 注入 key 缺失时如何兜底？ 结论是 inject('key', defaultValue) 或工厂函数 inject('key', () => ref('light')) 。缺失时无 provider 会得到默认值而非 undefined 报错。公共库应用 Symbol key 避免与业务字符串冲突。
