拖拽调整侧栏宽度（Vue3 + requestAnimationFrame 版）

阅读本复习区后，在下方拖拽侧栏右侧手柄，观察宽度变化与主内容 margin 联动。

知识点要点

  • 拖拽调整宽度的核心流程是：按下时记录 clientX 和初始宽度，移动时计算偏移量，松开时解绑全局监听并恢复状态。
  • 把 mousemove/mouseup 监听临时绑定到 document，可以避免鼠标移出手柄或侧栏后拖拽丢失；结束时必须解绑，避免内存泄漏和重复响应。
  • 拖拽中临时添加 .no-transition，并在 mousedown 里 preventDefault()，可以减少宽度跟手滞后和文本被选中的问题。
  • requestAnimationFrame 不是让宽度变化“更少计算”，而是把高频 mousemove 合并到浏览器下一帧提交，避免一帧内多次写响应式状态。
  • 侧栏宽度要设置最小/最大边界，保证菜单仍可用、主内容不被完全挤压；真实项目还应支持键盘操作或可访问的替代设置项。
  • 运行环境： 鼠标拖拽在 file:// 与 http(s):// 均可；触屏需 Pointer Events。本页通过 CDN 同级路径加载 Vue3，不依赖 HTTPS 特权 API。

面试考点

  • 拖拽调整宽度为什么要把 move/up 监听绑到 document？ 结论是为了让拖拽在鼠标离开手柄后仍能继续并正确结束。原因是 mousemove 只绑在手柄上很容易断流；边界是拖拽结束必须解绑，否则会造成重复监听和异常更新。
  • requestAnimationFrame 在拖拽节流里解决什么问题？ 结论是把高频事件合并到浏览器绘制节奏中提交。原因是鼠标移动事件可能远高于屏幕刷新率，一帧内多次改宽度只会制造额外布局压力；项目里通常记录最新值，在下一帧一次性写入。
  • 拖拽过程中为什么要禁用 transition 并阻止默认行为？ 结论是为了保证跟手和避免误选文本。原因是宽度过渡会让视觉变化滞后于指针，默认拖选文本会干扰交互；拖拽结束后再恢复过渡效果即可。
  • 为什么要限制侧栏最小和最大宽度？ 结论是拖拽必须有业务边界。原因是过窄会隐藏菜单，过宽会挤压内容；后台系统常把边界和用户偏好一起持久化，下次打开恢复。
  • Mouse Events 和 Pointer Events 在拖拽交互里如何选？ 结论是现代项目优先 Pointer Events，简单桌面 demo 可用 Mouse Events。原因是 Pointer Events 统一鼠标、触摸和触控笔，还支持 pointer capture；如果只支持鼠标，移动端拖拽要额外写 touch 逻辑。
