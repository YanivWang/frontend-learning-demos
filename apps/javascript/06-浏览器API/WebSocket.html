WebSocket 连接与消息

请阅读本复习区，并在下方演示区连接公共 echo 服务；需人工验证：WebSocket 依赖外网，部分网络环境可能拦截 wss://。

知识点要点

  • new WebSocket(url) 会发起握手，服务端同意 Upgrade 后形成持久的全双工连接，客户端和服务端都可以主动发送消息。
  • readyState 包含 CONNECTING、OPEN、CLOSING、CLOSED；发送前必须确认是 OPEN，否则会失败或抛错。
  • 本 demo 使用公共 echo 服务演示连接、发送、接收与关闭，依赖外网和服务可用性；失败时会显示错误或关闭信息。
  • WebSocket 只提供传输通道，不自动保证业务消息不丢、不重、不乱序处理；项目里需要设计心跳、重连、消息 id、幂等、补偿拉取和退避策略。
  • 安全上应优先使用 wss://，鉴权通常通过握手时的 Cookie、短期 token 或子协议约定完成；不要把长期敏感 token 明文拼在可泄漏的 URL 中。

面试考点

  • WebSocket 与 HTTP 轮询、SSE 的核心区别是什么？ 结论是 WebSocket 是持久全双工连接，轮询是客户端反复请求，SSE 是服务端到客户端的单向事件流。原因是 WebSocket 握手后双方都能主动发消息，适合聊天、协同、行情等高实时双向场景。只需要服务端推送文本事件时，SSE 往往更简单。
  • 为什么 WebSocket 发送前要检查 readyState？ 结论是只有 OPEN 状态才能可靠发送。原因是连接建立、关闭中或已关闭时底层通道不可用，直接 send() 可能抛错或丢失业务意图。项目里通常维护发送队列，在 open 后 flush，关闭后进入重连或降级。
  • WebSocket 断线重连和心跳应该怎么设计？ 结论是用应用层 ping/pong 或业务心跳检测活性，断开后按指数退避重连。原因是浏览器的连接关闭不一定及时暴露，移动网络和代理也可能静默断开。重连后要补拉关键状态，避免只依赖未送达的实时消息。
  • wss、鉴权和消息可靠性有哪些安全边界？ 结论是生产环境应使用 wss://，并设计短期鉴权、权限校验和消息确认。原因是 WebSocket 连接建立后仍可能被滥用，服务端必须对每条业务消息做授权和限流。敏感 token 不宜长期放在 URL 中，消息处理要考虑重复、乱序和重放。
