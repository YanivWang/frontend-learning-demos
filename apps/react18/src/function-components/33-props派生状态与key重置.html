props 派生状态与 key 重置

请打开 DevTools Console 查看切换用户时的 key 重置日志；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • props 是父组件传入的数据源，state 是组件自己的记忆；把 props 无条件复制到 state 会制造两个来源，容易出现旧值覆盖用户输入。
  • 能由当前 props 或 state 直接计算出的值，应在渲染阶段派生，例如本页的 displayLabel，不需要再用 useEffect 同步。
  • key 不只是列表 diff 标识，也决定 React 是否复用同一个组件实例；key 变化会卸载旧实例并重新创建 state。
  • 本页切换 userId 时给 Editor 设置 key，表示草稿属于某个用户，换用户就应该丢弃旧草稿并回到新的 initialName。
  • 如果业务要求跨用户保留未提交草稿，就不应使用 key 重置，而应把草稿提升到父组件或按 userId 存成映射。

面试考点

  • 为什么不推荐 mirror props 到 state？ 结论是不要无条件复制，因为会形成两个数据来源。原因是父组件重新渲染或 props 变化时，本地 state 可能覆盖用户正在编辑的草稿。项目里只有「初始值变成本地可编辑草稿」这类明确需求，才把 props 作为 state 初始值。
  • 什么场景用 render 派生，什么场景用 key 重置？ 结论是可计算值直接派生，需要丢弃整块内部状态时用 key。原因是派生计算不会额外触发渲染，而 key 会让 React 创建新实例。切换用户/记录时，要丢弃草稿就加 key={id}；要保留草稿则把草稿提升到父组件或按 id 存映射。
  • key 为什么能重置组件状态？ 结论是 key 参与 React 对组件身份的判断。原因是类型和位置相同但 key 不同，React 会卸载旧组件、挂载新组件，Hook state 也随之重建。边界是不要为了刷新数据滥用 key，否则会丢失焦点、动画和未保存状态。
  • 函数组件如何替代 getDerivedStateFromProps？ 结论是优先用直接派生、受控组件或 key 重置。原因是 getDerivedStateFromProps 本来就容易引入派生状态反模式。维护旧 class 组件时可以保留它，但新函数组件通常不需要写等价 Hook。
