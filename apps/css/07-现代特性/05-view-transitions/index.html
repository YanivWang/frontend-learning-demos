View Transitions

点击下方色块切换状态。支持 View Transitions 的浏览器会对旧状态和新状态做快照过渡，不支持时直接切换颜色。

知识点要点

  • View Transitions API 的核心是把一次 DOM 状态变更包装起来：浏览器捕获旧状态快照，执行更新回调，再捕获新状态快照，随后用 CSS 动画在两张快照之间过渡。
  • document.startViewTransition(updateCallback) 是脚本入口。本页点击色块时先计算下一个颜色，再把背景色与文本更新放进回调；支持的浏览器执行快照过渡，不支持时直接调用同一个更新函数。
  • ::view-transition-old(root) 和 ::view-transition-new(root) 是默认根过渡的伪元素，可设置动画时长、动画曲线或自定义关键帧。更细粒度的元素过渡需要给元素设置 view-transition-name。
  • 它适合主题切换、SPA 路由切换、卡片展开和列表重排这类“DOM 结构或状态已经变了，但希望视觉连续”的场景；普通 hover、按钮反馈、单个属性变化仍用 transition 或 animation 更轻量。
  • 失败路径必须显式处理：CSS 的 @supports (view-transition-name: none) 只能保护样式，脚本仍要判断 document.startViewTransition 是否存在；更新回调不要做慢网络请求，用户开启减少动态效果时也应避免过度动画。

面试考点

  • View Transitions API 解决的不是哪类普通 CSS transition 问题？ 结论是它解决的是 DOM 更新前后两个视觉状态之间的连续过渡，不是单个 CSS 属性从 A 到 B 的基础动画。原因是普通 transition 需要同一个元素的可插值属性连续变化，而路由切换或结构重排常常会替换节点。项目里页面切换、主题切换可考虑它，按钮 hover 仍用普通过渡。
  • startViewTransition 的 DOM 更新回调为什么必须可快速完成？ 结论是回调应该只做同步或很短的状态提交。原因是浏览器需要在旧快照和新快照之间建立动画，回调太慢会让交互卡顿或让过渡时机不可控。项目里网络请求应在过渡前完成，或先展示 loading 状态再触发过渡。
  • ::view-transition-old/new 和 view-transition-name 分别控制什么？ 结论是前者控制过渡快照伪元素的动画，后者给具体元素命名以建立更细粒度的共享元素过渡。原因是默认 root 过渡只能处理整页快照，命名后浏览器才能把旧元素和新元素对应起来。项目里只做淡入淡出可用 root，复杂卡片迁移再加命名。
  • View Transitions 在兼容性、可访问性和失败路径上要怎么降级？ 结论是页面功能不能依赖动画存在。原因是并非所有浏览器都支持该 API，用户也可能偏好减少动态效果。项目里脚本要检测 document.startViewTransition，CSS 用 @supports 包住伪元素样式，不支持时直接完成 DOM 更新。
