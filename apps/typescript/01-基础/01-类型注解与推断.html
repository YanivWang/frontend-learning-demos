类型注解与推断

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • TypeScript 是 JavaScript 的静态类型层：IDE 和 tsc 在编译期发现类型问题，但浏览器执行的仍是擦除类型后的 JavaScript；本页故意展示 add("1", 2) 在 JS 中会拼接。
  • 现代写法是让简单局部变量依赖推断，给函数参数、公共返回值、对象边界和难推断泛型补显式注解；过度注解会增加噪声，缺少边界注解会让 API 变得不稳定。
  • 基本类型包括 string、number、boolean、null、undefined、bigint、symbol；启用 strictNullChecks 后，null 和 undefined 不再默认可赋给所有类型。
  • any 会关闭后续类型检查，适合极少数迁移逃生口；unknown 表示“目前不知道”，使用前必须通过 typeof、判空、用户自定义类型守卫等方式收窄。
  • 常见坑是把 TypeScript 当运行时校验：外部接口、表单输入和本地存储读出的数据仍需要运行时验证，否则类型声明只能让调用处“看起来安全”。

面试考点

  • 类型注解和类型推断应该如何取舍？ 结论是局部简单值依赖推断，函数参数、公共 API 和复杂对象边界显式注解。原因是推断能减少重复，边界注解能稳定契约。项目中导出的函数、组件 props、接口返回值通常要写清，临时变量不必处处标注。
  • TypeScript 的类型检查为什么不能改变 JavaScript 运行时？ 结论是 TypeScript 类型在编译后会被擦除。原因是 TS 设计目标是给 JS 加静态检查，而不是改变 JS 语义；所以本页的 add("1", 2) 运行时仍按 JS 拼接。边界是外部数据必须做运行时校验。
  • any 和 unknown 的核心区别是什么？ 结论是 any 放弃检查，unknown 保留检查。原因是 any 可以任意访问属性和调用方法，unknown 必须先收窄。迁移旧代码可短期用 any，新边界更推荐 unknown 加解析逻辑。
  • strictNullChecks 打开后 null 和 undefined 有什么影响？ 结论是它们只属于各自类型，不能随意赋给普通 string、number 等类型。原因是这能把空值风险提前到编译期。项目里接口可空字段要写成联合类型，并在使用前判空。
