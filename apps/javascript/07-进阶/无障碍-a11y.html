无障碍 a11y：语义化、焦点与 ARIA

用 Tab 聚焦按钮，再按 Enter 或 Space 触发展开；观察焦点样式、aria-expanded 和动态状态文本。

知识点要点

  • 无障碍不是只给元素加 ARIA，而是让页面结构、名称、角色、状态和交互都能被键盘用户与辅助技术理解；原生 button 自带按钮角色、键盘触发和禁用语义。
  • ARIA 用来补充语义，不改变真实行为。把 div 写成 role="button" 后，开发者仍要自己处理 Tab 可聚焦、Enter/Space 触发、禁用态和焦点样式，所以优先使用原生控件。
  • 本页按钮用 aria-expanded 表达折叠状态，用 aria-controls 指向被控制面板；面板的 hidden 决定真实可见性，避免只改 ARIA 但视觉和 DOM 状态不同步。
  • aria-live="polite" 适合“已展开/已折叠”这类非紧急动态消息，读屏软件会在合适时机播报；错误、支付失败等紧急消息才考虑 assertive，否则容易打断用户。
  • 验证交互组件至少要测试键盘 Tab 顺序、焦点是否可见、Enter/Space 是否可操作、状态是否同步，以及在屏幕阅读器或浏览器无障碍树中是否有正确名称与角色。

面试考点

  • 为什么优先使用原生 button？ 结论是原生控件默认具备正确角色、键盘行为和可访问状态，成本最低也最不容易漏。原因是 ARIA 只改变无障碍树，不会自动实现交互逻辑。项目中只有样式或组件结构确实无法使用原生控件时，才考虑自定义角色并补齐键盘和焦点行为。
  • aria-expanded、aria-controls 和 hidden 分别表达什么？ 结论是 aria-expanded 告诉辅助技术当前是否展开，aria-controls 建立控制关系，hidden 控制元素实际是否显示和暴露。原因是状态、关系和可见性是不同层面的信息。边界是不能只改 ARIA 不改 UI，也不能只隐藏 UI 不更新状态。
  • aria-live 适合哪些动态内容？ 结论是适合不移动焦点也需要告知用户的状态变化，例如保存成功、筛选结果数量、展开折叠状态。原因是读屏用户不一定能看到视觉变化。边界是频繁变化或非关键消息不要用 assertive，否则会造成播报噪音和用户打断。
  • 如何验证一个组件键盘可用？ 结论是不用鼠标也能发现、聚焦、操作、退出，并且焦点始终可见。原因是键盘访问是 WCAG 的基础要求，也影响读屏和开关设备用户。项目中应把 Tab 顺序、Enter/Space 行为、Esc/方向键等约定纳入测试用例。
