Vue CLI 知识点速查（笔记型）

这是笔记型 demo，没有交互示例；重点复习 Vue CLI 在 Vue 2 历史项目中的工程职责。

知识点要点

  • @vue/cli 是全局命令行脚手架，负责创建项目、调用插件和提供图形化管理入口；安装后会暴露 vue 命令。
  • @vue/cli-service 是项目内运行时依赖，封装 webpack、webpack-dev-server 与默认配置，常见命令是 serve 、 build 、 inspect 。
  • Vue CLI 插件通常包含 generator 与 runtime plugin：前者生成或修改项目文件，后者调整 webpack 配置、注入命令或扩展项目能力。
  • Vue CLI 适合维护历史 Vue 2 / webpack 项目；官方文档已经标注 Vue CLI 维护模式，新 Vue 项目推荐使用 create-vue 搭配 Vite。
  • 排查构建问题时不要只看表层配置，先用 vue-cli-service inspect 输出最终 webpack 配置，再判断 loader、plugin、alias、publicPath 是否符合预期。

面试考点

  • @vue/cli 和 @vue/cli-service 的区别是什么？ 结论是前者偏全局脚手架，后者偏项目内构建服务。原因是 @vue/cli 负责创建和管理项目，而 @vue/cli-service 才真正参与本项目的开发服务器、构建和配置解析。项目里排查命令失败时，要先看失败发生在全局 vue 命令阶段还是项目内 service 阶段。
  • Vue CLI 插件为什么能扩展项目？ 结论是插件通过 generator 改文件、通过 runtime plugin 改服务配置或命令。这样能把 Babel、Router、Vuex、ESLint 等能力拆成可组合插件。边界是插件越多，最终 webpack 配置越不直观，所以要用 inspect 看合并结果。
  • vue-cli-service inspect 有什么排查价值？ 结论是可导出 webpack 最终合并配置，定位 loader/plugin/alias 是否符合预期。原因是 CLI 把配置封装在 service 和插件中，源码里不一定能直接看到。项目里构建异常时先用 inspect 确认实际生效的配置。
  • 现在新项目还应该用 Vue CLI 吗？ 结论是通常不建议新项目首选 Vue CLI，官方推荐 create-vue 和 Vite。原因是 Vue CLI 已处于维护模式，现代 Vue 3 工程链路围绕 Vite 优化。例外是存量 Vue 2 / webpack 项目、企业内部模板和历史插件依赖暂时无法迁移时，可以继续维护。
  • 维护 Vue 2 老项目时 Vue CLI 和 Vite 怎么取舍？ 结论是存量 Vue 2 + webpack 项目继续用 CLI 维护成本低；新功能或迁移阶段可评估 Vite。原因是 CLI 插件生态与 webpack 配置深度绑定，强行切 Vite 需改构建链。边界是仅维护 bugfix 不必强行迁移，大版本升级时再规划 Vite 迁移。

常用命令

  • vue create my-project — 交互式创建项目
  • npm run serve — 开发服务器（vue-cli-service serve）
  • npm run build — 生产构建
  • vue-cli-service inspect > webpack.config.js — 导出最终 webpack 配置
