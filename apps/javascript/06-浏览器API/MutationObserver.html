MutationObserver 监听 DOM 变化

请打开 DevTools Console 查看输出。

知识点要点

  • MutationObserver 用来异步观察 DOM 树变化，回调收到一组 MutationRecord，可以区分子节点、属性和文本内容变化。
  • observe(target, options) 必须至少启用 childList、attributes 或 characterData 之一；subtree: true 会扩大到所有后代节点，适合容器级监听但成本更高。
  • attributeOldValue 与 characterDataOldValue 能记录旧值，便于 diff 和调试，但也会保留更多快照信息，不适合无边界地监听大 DOM。
  • MutationObserver 替代了同步且性能差的旧 Mutation Events；它的回调是异步批量投递，不能假设每次 DOM 改动都会立即单独触发。
  • 常见坑是在回调里继续修改被观察区域，导致回调再次入队形成循环；组件卸载、一次性扫描结束或切换观察目标时要调用 disconnect()。

面试考点

  • MutationObserver 与 DOM Mutation Events 有什么区别？ 结论是 MutationObserver 是现代推荐方案，旧 Mutation Events 已不适合作为监听 DOM 变化的实现。原因是 MutationObserver 异步批量投递记录，能减少同步事件带来的性能和递归问题。项目里应使用它做第三方 DOM 接入、富文本插件适配或调试辅助。
  • observe 配置里 childList、attributes、subtree 分别影响什么？ 结论是 childList 看子节点增删，attributes 看属性变化，subtree 把观察范围扩展到后代。原因是 MutationObserver 不会默认监听所有变化，必须明确声明关心的类型。实际项目应缩小 target 和 options，避免在整个 document 上长期监听。
  • MutationObserver 回调里继续修改 DOM 有什么风险？ 结论是可能触发新的 mutation 并反复进入回调。原因是回调本身的 DOM 写入也会被同一个 observer 观察到。安全做法是加条件过滤、临时 disconnect()、或把修正操作限定到不被观察的节点。
  • MutationObserver 和 ResizeObserver、IntersectionObserver 如何选型？ 结论是 DOM 结构或属性变化用 MutationObserver，元素尺寸变化用 ResizeObserver，可见性或曝光用 IntersectionObserver。原因是三者观察的信号不同。不要用 MutationObserver 推断布局尺寸或滚动曝光，否则容易误判且成本更高。
