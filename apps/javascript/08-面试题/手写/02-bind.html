手写 bind

先看本复习区，再查看下方输出是否与标准答法一致；细节可打开 DevTools Console。

知识点要点

  • Function.prototype.bind() 返回一个新的绑定函数，不会立即执行；这是它和 call / apply 最核心的区别。
  • 绑定函数普通调用时，内部会用原函数配合绑定的 thisArg 执行，并把预置参数与调用时传入的参数拼接。
  • 绑定函数也可以作为构造函数使用；当它被 new 调用时， this 必须指向新创建的实例，绑定的 thisArg 会被忽略。
  • 为了让 new BoundPerson() 创建的实例仍能访问 Person.prototype 上的方法，手写实现需要让绑定函数的 prototype 继承原函数 prototype 。
  • 规范里的绑定函数还涉及 length 、 name 、不可作为普通函数时的异常等细节；面试手写一般重点答 this 、参数、 new 和原型链。
  • 真实项目不建议重写 Function.prototype.bind ；需要固定回调上下文时直接用原生 bind ，或在现代代码中用箭头函数捕获外层 this 。

面试考点

  • bind 和 call/apply 最大区别是什么？ 结论： bind 返回新函数并延迟执行， call/apply 会立即执行。原因是 bind 的用途是预先固定 this 和部分参数，等待之后作为回调或构造函数调用。项目里事件回调、定时器、类方法传参常用 bind 或箭头函数解决上下文丢失。
  • 手写 bind 如何同时支持预置参数和后续参数？ 结论：在外层保存 presetArgs ，绑定函数执行时再和 laterArgs 拼接。原因是 bind 支持偏函数能力，先传的参数会排在后续实参之前。边界是参数拼接不改变原函数的形参数量，规范里的 length 调整面试实现通常不强求。
  • new 调用绑定函数时为什么要忽略绑定的 thisArg ？ 结论：构造调用的 this 必须是新创建的实例。原因是 new 的语义是创建对象、连接原型、以新对象作为 this 执行构造函数；如果仍使用绑定对象，实例属性会写错位置。边界是绑定参数仍然有效，只是绑定的 thisArg 被忽略。
  • 手写 bind 为什么要处理 prototype ？ 结论：为了让绑定函数作为构造函数时，实例仍然能继承原构造函数的原型方法。原因是 instanceof 和方法查找都依赖原型链。本 demo 用 Object.create(fn.prototype) 连接原型，避免直接共享并修改原函数的 prototype 对象。
