嵌套路由 + 命名视图组合（settings 综合页）

页面加载后自动进入 /settings ；点击 emails / profile 链接，观察主内容区变化；切到 profile 时同时出现编辑区与右侧 helper 预览区（命名视图）。

知识点要点

  • 本页把 /settings 作为父路由，先渲染 UserSetting 布局组件，再由布局内部的子 <router-view> 渲染设置页内容。
  • 命名视图允许一个路由记录同时渲染多个组件：无名出口对应 components.default ， <router-view name="helper"> 对应 components.helper 。
  • /settings/profile 同时渲染编辑区和预览区，说明嵌套路由负责页面层级，命名视图负责同一层级内的多区域布局。
  • 本仓库 HTML demo 需要支持直接打开，使用默认 hash 模式更稳定；如果项目改用 history 模式，刷新深层路径时服务器必须把请求回退到入口 HTML。
  • 复杂路由配置要按布局层、业务页、辅助区拆清楚，并给不存在的子路径准备 fallback；否则页面会出现布局存在但局部内容空白的问题。

面试考点

  • 嵌套路由和命名视图组合能解决什么布局问题？ 结论是它能同时表达页面层级和同级多区域布局。原因是 children 决定子页面嵌在哪里，命名视图决定同一路由往哪些出口渲染组件。项目里设置页、用户中心、工作台右侧预览都适合这种组合。
  • components.default 和 router-view name 如何对应？ 结论是无名 router-view 对应 default ，有名出口按 name 找同名组件。原因是 Vue Router 需要用对象键把多个 outlet 和多个组件一一映射。边界是写成单数 component 时只能填充默认出口。
  • 为什么 settings/profile 可以同时渲染内容和预览区？ 结论是它的路由记录使用了 components 对象。原因是一个匹配结果可以给默认出口放编辑组件，同时给 helper 出口放预览组件。项目里这比在组件内部写大量条件渲染更容易表达路由驱动的布局。
  • hash 模式和 history 模式在部署上有什么差异？ 结论是 hash 模式对静态文件更友好，history 模式 URL 更干净但需要服务端兜底。原因是 hash 后面的片段不会发给服务器，而 history 深层路径刷新会请求真实路径。静态 demo 和 GitHub Pages 常用 hash，生产 history 要配置 fallback。
  • 复杂路由页面如何避免路由配置失控？ 结论是按布局层、业务页、辅助区拆分 children，命名视图 key 与 outlet name 一一对应，并给未知子路径配 fallback。原因是路由表膨胀后难维护且易出现空白出口。项目里可配合路由模块化与常量命名。
