构造函数最基本用法

请打开 DevTools Console 查看输出。

知识点要点

  • 构造调用： 函数被 new 调用时，引擎创建空对象、将其 [[Prototype]] 链到构造函数的 prototype ，并以该对象为 this 执行函数体。
  • 实例属性： 构造函数内 this.xxx = … 写入的是每个实例独立的自有属性；共享行为应放在 Person.prototype 上。
  • 原型方法共享： 多个实例的 sayHello 引用同一份 Person.prototype.sayHello ，节省内存且行为一致。
  • 类型判断： instanceof 沿 [[Prototype]] 链查找 Person.prototype ； constructor 来自原型，替换整个 prototype 后可能不准。
  • 漏写 new： 普通调用时 this 在非严格模式下可能指向 globalThis ，污染全局；严格模式或 ESLint 规则可防此类错误。
  • class 关系： class 是构造函数和原型模式的语法糖，实例方法仍然定义在 prototype 上（见 04-ES6+/class详解 ）。

面试考点

  • 构造函数和普通函数的区别是什么？ 函数本身没有硬性区别，关键在调用方式；用 new 调用时才会创建实例并绑定 this 。项目里通常用首字母大写或 class 表示「需要 new」。
  • 为什么实例方法通常放在 prototype 上？ 方法是共享行为，不需要每个实例都保存一份函数对象。放到原型上能节省内存，并让实例通过原型链访问同一份方法。
  • instanceof 和 constructor 判断类型有什么边界？ instanceof 看原型链，适合判断某个实例是否链接到某构造函数原型； constructor 可被改写，不能单独作为可靠依据。跨 realm 场景还要更谨慎。
  • 构造函数与 class 是什么关系？ class 让构造函数、原型方法和继承写法更清晰，但底层仍依赖原型链。维护旧代码时理解构造函数，有助于理解 class 的运行机制。
  • 漏写 new 会发生什么？ 非严格模式下 this 可能变成 globalThis ，属性挂到全局；严格模式抛 TypeError 。可用 new.target 在构造函数内检测是否被 new 调用。
