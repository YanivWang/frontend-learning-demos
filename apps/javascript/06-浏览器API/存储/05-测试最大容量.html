测试 localforage 最大可写容量

请打开 DevTools Console 查看输出。

知识点要点

  • 浏览器存储配额不是 Web API 规范里给定的固定值，不同浏览器、设备空间、隐私模式、持久化策略和 localforage 后端都会影响可写容量。
  • 一次构造巨大数组再写入会同时压垮 JS 堆内存和存储配额，不能区分“内存不够”和“存储写满”；更安全的压测方式是分块写入并记录第几个 chunk 失败。
  • QuotaExceededError 表示本次写入超过当前可用配额或被策略拒绝，项目里要捕获并提示用户清理缓存、减少离线数据或改为服务端保存。
  • navigator.storage.estimate() 适合快速看总体配额压力，真实写入压测适合验证当前数据形态能否写入；前者成本低但不精确，后者更接近结果但有副作用。
  • 容量压测会写入真实本地数据，demo 和生产诊断工具都必须由用户主动触发，并提供停止、清理和错误提示，避免页面打开就污染用户存储。

面试考点

  • 为什么不能通过一次大对象写入准确测出最大容量？ 结论是一次大对象只能测到“这次写入是否成功”，不能精确代表最大配额。原因是失败可能来自 JS 内存、序列化成本、单条记录大小限制或浏览器配额；项目里应分块写入并结合 StorageManager.estimate() 观察。
  • localforage 写满时如何处理 QuotaExceededError？ 结论是必须捕获错误并走降级流程。原因是离线缓存、草稿和文件都可能超过配额；业务上可以清理旧缓存、压缩数据、限制文件大小，或提示用户改用云端保存。
  • 为什么容量压测要分块写入并提供清理按钮？ 结论是分块能定位失败边界，清理按钮能撤销副作用。原因是压测数据真实落盘，占用同源配额；demo 若自动写入几十 MB，会影响后续学习页面和用户浏览器状态。
  • StorageManager.estimate() 和真实写入压测各自适合什么场景？ 结论是 estimate() 适合低成本监控，写入压测适合开发诊断。原因是前者给的是同源总体近似值，后者会验证当前数据结构和后端，但不能在生产页面频繁运行。
  • 生产项目如何避免离线缓存把用户磁盘打满？ 结论是设置业务上限、LRU 清理、按需下载和写入失败兜底。原因是浏览器可能驱逐非持久化数据，用户设备空间也有限；PWA、图片编辑器和文档离线包尤其需要可见的缓存管理页。
