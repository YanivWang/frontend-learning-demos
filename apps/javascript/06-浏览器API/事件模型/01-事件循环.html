事件循环综合（同步 + Promise + async/await + setTimeout）

阅读本复习区后，在下方点击按钮观察输出顺序；也可打开 DevTools Console 对照。

知识点要点

  • 浏览器事件循环以“任务”为入口执行 JavaScript：当前脚本本身就是一个任务，调用栈里的同步代码会先从上到下执行完。
  • 当前任务结束后，浏览器会清空微任务队列，再进入渲染机会或下一个任务；Promise.then、queueMicrotask 和 await 后续代码都属于微任务。
  • async 函数在遇到 await 前仍是同步执行；await 会把后面的 continuation 包装成 Promise 反应，因此本例的 async1 end 会进入微任务队列。
  • setTimeout(fn, 0) 不是“立刻执行”，而是把回调排入后续任务；它必须等当前脚本任务和由它产生的所有微任务执行完，才有机会运行。
  • 微任务适合在当前任务末尾补齐状态一致性，但不能无限递归追加；过长的微任务链会推迟渲染和输入响应，实际项目中应避免把耗时工作塞进微任务。
  • 运行环境： 本页仅依赖同步脚本、Promise 与 setTimeout，file:// 与 http(s):// 均可运行；输出顺序以当前浏览器 HTML 事件循环为准，不能套用 Node 的 process.nextTick 规则（见 Node 对比 demo）。

面试考点

  • 这段同步、Promise、async/await、setTimeout 混合代码输出顺序是什么？ 结论是先同步输出 script start、async1 start、async2、promise1、script end，再执行微任务 async1 end、promise2、queueMicrotask，最后执行 setTimeout。原因是脚本任务先跑完，Promise 与 await 后续进入微任务，同轮微任务按入队顺序清空，定时器属于后续任务。
  • await 后面的代码为什么晚于当前同步代码？ 结论是 await 后续会异步恢复。原因是 async 函数会把 await 之后的代码转换成 Promise 反应任务；边界是 await 前的表达式仍会立即求值，不能把整个 async 函数都理解成异步。
  • Promise.then 为什么通常先于 setTimeout(0)？ 结论是同一轮脚本里 Promise 微任务先执行。原因是 HTML 事件循环在一个任务结束后执行微任务检查点，然后才进入下一个任务；但定时器还有最小延迟、后台节流等浏览器策略，不能用它做精确计时。
  • queueMicrotask 与 Promise.then 如何选？ 结论是只想显式排微任务时优先 queueMicrotask。原因是它语义更直接，也不会把异常包装成 rejected Promise；项目里常用于让同步/异步分支都在同一微任务时机派发状态变更。
  • 微任务过多有什么风险？ 结论是会阻塞后续任务、渲染和用户输入。原因是浏览器会持续清空微任务队列，微任务不断追加微任务可能造成饥饿；耗时计算应拆分到任务、Worker 或可中断调度中。
