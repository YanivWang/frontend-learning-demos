class 组件 refs 访问 DOM 与实例

点击下方按钮验证 createRef 聚焦 DOM、父组件调用子 class 实例方法，以及“只改实例字段不会触发 render”的差异。

知识点要点

  • createRef：在 class 构造函数中创建 ref 容器，React 提交挂载后把 DOM 节点或 class 组件实例写入 ref.current，卸载后再置空；因此事件和生命周期里可读，render 中不应依赖它计算 UI。
  • 回调 ref：ref={(node) => ...} 允许自定义保存节点；挂载时收到节点，卸载时收到 null。如果用内联函数，每次 render 都会得到新的回调，React 可能先清空旧 ref 再设置新 ref。
  • 适用场景：ref 是命令式逃生口，适合输入框聚焦、文本选择、滚动定位、测量尺寸、播放媒体、集成非 React DOM 库；业务数据同步应优先用 props、state、Context 或回调函数。
  • 访问实例：ref 指向 class 组件时拿到的是组件实例，可以调用公开方法；这会增加父子耦合，通常只在表单校验、动画控制、兼容旧组件时使用，不能当作常规状态管理。
  • 版本边界：字符串 ref 已废弃；React 18 中函数组件没有实例，想让父组件传入 ref 需要 forwardRef 暴露 DOM 或配合 useImperativeHandle 暴露受控命令式方法。

面试考点

  • class 组件里 createRef 和回调 ref 有什么区别？ 结论是 createRef 适合大多数稳定节点，回调 ref 适合自定义保存或处理动态节点。原因是 createRef 提供固定容器，React 自动维护 current；回调 ref 直接接收节点和 null，控制力更强。项目里输入框聚焦通常用 createRef，复杂列表节点管理才考虑回调 ref。
  • ref.current 什么时候可用，为什么 render 里不能依赖它？ 结论是提交挂载后才可靠可用。原因是 render 阶段只是描述 UI，真实 DOM 尚未提交，ref 写入发生在提交阶段；ref 变化也不会触发 render。需要读取尺寸或操作 DOM 时应放在事件、componentDidMount 或 componentDidUpdate。
  • 什么时候应该用 ref，什么时候应该改用 props/state？ 结论是命令式 DOM 操作用 ref，声明式业务数据用 props/state。因为 React 的渲染一致性依赖单向数据流，绕过数据流会让状态来源不清。项目里主题、表单值、是否展开这类 UI 状态不要放 ref；聚焦、滚动、媒体播放可以用 ref。
  • 父组件通过 ref 调子 class 方法有什么风险？ 结论是可以做，但会增强耦合。原因是父组件直接依赖子组件实例方法名和实现细节，子组件重构为函数组件或替换实现时容易破坏调用方。维护旧组件时可用于少量命令式能力，新设计更推荐通过 props 回调或受控状态表达交互。
