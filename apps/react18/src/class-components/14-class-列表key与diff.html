class 组件列表 key 与 diff

请先勾选某个任务，再点击"从头部插入"，对比稳定 id key 与 index key 两组列表的勾选状态是否错位；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • key 是 React 在同级 children 中识别元素身份的提示。它帮助 React 判断哪一个旧 fiber 可以复用，哪一个需要挂载或卸载，但不会作为普通 props 传给子组件。
  • 稳定业务 id 优先于数组下标。列表只追加且不排序、不删除时 index 勉强可用；一旦从头部插入、删除或排序，用 index 会把旧组件实例复用到错误数据上，造成输入值、勾选状态等内部 state 错位。
  • 本 demo 同时渲染"稳定 id key"和"index key"两组列表：先勾选某个任务，再从头部插入，可以看到 index key 列表中的勾选状态跟随位置而不是任务。
  • key 只需要在相邻兄弟节点之间唯一，不要求全局唯一。如果子组件需要业务 id，应显式传 todoId={todo.id}。
  • 改变 key 会让 React 把组件视为新实例，可用于重置表单或局部状态；但把 key 当作强制刷新工具滥用，会丢失用户输入并增加挂载成本。

面试考点

  • key 在 React diff 中解决的是什么身份问题？ 结论是它告诉 React 同级列表中哪些元素代表同一个组件实例。原因是 React 需要在重新渲染时复用或销毁 fiber；项目里稳定 key 能保留正确的输入值、动画状态和组件 state。
  • 为什么会变动的列表不推荐用 index 作为 key？ 结论是 index 表示位置，不表示数据身份。原因是插入、删除、排序会让同一个位置对应不同数据，React 复用实例后就可能出现 state 错位；只有静态列表或纯追加列表风险较低。
  • key 会作为 props 传进子组件吗？ 不会。原因是 key 是 React 的特殊字段，供协调算法使用；需要业务 id 时要额外传一个显式 prop。
  • 为什么可以用 key 重置组件状态？ 结论是 key 变化会让 React 卸载旧实例并挂载新实例。原因是身份变了，原来的 state 不再复用；常见场景是切换用户时重置表单草稿，但要避免误删未保存输入。
  • class 组件和函数组件的 key 规则有什么区别？ 结论是没有区别。原因是 key 是 React 协调算法的规则，与组件是 class 还是函数无关；两者都应在同级列表中使用稳定唯一标识。
