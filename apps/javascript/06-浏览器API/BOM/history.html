history pushState / replaceState 与 popstate

请打开 DevTools Console 查看输出。

知识点要点

  • history.pushState(state, unused, url) 会向会话历史追加一条同源记录；history.replaceState 会替换当前记录，不增加后退层级。
  • state 会和这条历史记录绑定，用户前进/后退回到该记录时可从 popstate 事件的 event.state 中取回。
  • 调用 pushState 或 replaceState 本身不会触发 popstate；只有用户触发前进/后退或代码调用 history.back/forward/go 切换历史记录时才触发。
  • url 必须与当前页面同源，否则会抛出异常；state 需要可结构化克隆，过大或不可序列化的数据不应塞进历史记录。
  • History 路由的 URL 没有 #，用户体验更自然；但刷新深层路径时服务器会收到真实路径，所以生产部署必须把 SPA 路径 fallback 到入口 HTML。

面试考点

  • pushState 和 replaceState 对历史栈的影响有什么区别？ 结论是 pushState 新增一条可后退记录，replaceState 覆盖当前记录。原因是它们分别对应“进入新页面状态”和“修正当前页面状态”。列表页进入详情通常用 push，登录重定向、规范化 URL、替换临时参数常用 replace。
  • 为什么调用 pushState 后不会立即触发 popstate？ 结论是不会触发，因为 popstate 表示激活历史记录发生变化，而不是创建历史记录。原因是 pushState 只是写入一条新记录并更新地址栏。SPA 需要在调用 pushState 后主动渲染当前路由，同时监听 popstate 处理前进后退。
  • history state 有哪些同源、序列化和容量边界？ 结论是 URL 必须同源，state 应是可结构化克隆的小对象。原因是浏览器要持久化会话历史，不能保存函数、DOM 节点这类运行时对象。项目里只放路由恢复需要的轻量数据，业务大状态仍放 store、URL 参数或服务端。
  • History 路由为什么需要服务端 fallback？ 结论是刷新深层路径时浏览器会请求真实路径，服务器必须返回 SPA 入口 HTML。原因是 History 路由没有 hash 把路径留在客户端。部署到 Nginx、静态托管或 CDN 时，如果没有 fallback，刷新 `/users/1` 这类路径会变成 404。
