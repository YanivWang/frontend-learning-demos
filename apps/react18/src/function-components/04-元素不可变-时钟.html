React 元素的「不可变」特性

请打开 DevTools Console 查看每秒 tick 日志；下方 #root 为 React 挂载演示区，观察时间文本如何更新。

知识点要点

  • React element 是描述某一刻 UI 的不可变对象。创建后不要修改它的属性或子节点；要表达新界面，就创建新的 element 并交给 React 渲染。
  • 本 demo 的时钟每秒重新创建包含当前时间的 element，再调用同一个 root.render(element)。这演示的是“用新描述替换旧描述”，不是直接改 DOM 文本。
  • ReactDOM 会比较前后两次 element 描述，只把必要变化提交到真实 DOM。时钟示例里主要变化是时间文本，而不是整页 DOM 都被粗暴重建。
  • createRoot 应在容器上创建一次并复用；在 setInterval 里反复 createRoot 会重复创建根，违背 React 18 root 的使用方式。
  • 真实项目中通常不会手动反复调用 root.render 来更新局部 UI，而是用组件 state 或外部状态触发渲染；本页是为了单独说明 element 不可变。

面试考点

  • React element 为什么说是不可变的？ 结论是 element 表示某一时刻 UI 的快照，创建后不应修改。原因是 React 依赖新旧描述比较来决定 DOM 更新；项目里要改变 UI，应改变 state/props 或创建新 element。
  • 时钟示例为什么每秒创建新 element？ 结论是因为当前时间变了，需要一份新的 UI 描述。原因是旧 element 不会被原地修改；边界是这个写法适合讲原理，实际组件里会用 useState 和 effect 管理定时器。
  • React 会不会每次 render 都重建整个 DOM？ 结论是不会简单粗暴重建整棵 DOM。原因是 React 会比较前后描述并提交必要变更；但这不代表 render 没成本，频繁定时更新仍应控制范围和清理定时器。
  • createRoot 为什么只创建一次？ 结论是一个容器对应一个 React root，后续更新复用 root.render。原因是 root 保存调度与渲染上下文；在循环里创建 root 会制造错误用法和潜在资源问题。
  • 真实项目为什么用 state 而不是手动 root.render？ 结论是 state 让更新声明式、可组合、可测试。原因是组件内部 useState + useEffect 能自动管理定时器清理；手动 root.render 只适合入门演示 element 不可变原理。
