Proxy 的局限性

请打开 DevTools Console 查看输出。

知识点要点

  • Proxy 可以代理普通属性访问，但不能让所有内建对象“透明工作”。Date、Map、Set 等方法依赖规范里的内部槽位，代理对象本身没有这些槽位。
  • 本 demo 中空 handler 的 proxy instanceof Date 仍为 true，但调用 proxy.getDate() 会报不兼容接收者错误，因为方法收到的 this 是代理对象而不是原始 Date 实例。
  • 常见修复是在 get trap 中遇到函数就 bind 到原对象，或只针对少量方法手动转发；这能让示例运行，但会让方法始终绑定原对象，影响继承、mock 和后续二次包装。
  • Proxy 还有其他边界：对象相等性不会被代理抹平，带私有字段的类方法也可能因为 this 不是原实例而失败，且旧环境无法完整 polyfill。
  • 实践中不要默认给所有内建对象、DOM 对象或第三方类实例套通用代理；更稳妥的方式是封装明确 API、只代理普通数据对象，或针对需要的操作做白名单转发。

面试考点

  • 为什么空 handler 的 Proxy 也不能透明代理 Date？ 结论是空 handler 只代表不自定义 trap，不代表代理对象拥有目标对象的内部槽位。原因是 Date.prototype.getDate 会检查 this 上的 Date 内部数据，而代理对象不是那个原始 Date 实例。项目里给内建对象做日志、权限或响应式包装时，要先验证其方法是否依赖内部槽位。
  • bind 原对象修复内建方法调用有什么副作用？ 结论是它能解决当前调用，但不是完全透明代理。原因是方法的 this 被固定为原对象，后续继承、二次代理、测试替身或希望 receiver 参与逻辑的场景都会受影响。工程上更推荐只对白名单方法转发，并把这种包装边界写清楚。
  • Proxy 在私有字段、=== 和旧浏览器兼容上有哪些边界？ 结论是代理不会改变对象身份，proxy !== target；私有字段方法可能因 this 不匹配而失败；旧环境也无法完整 polyfill。原因是这些能力依赖语言内部机制而不是普通属性访问。需要跨环境兼容或代理 class 实例时，要优先考虑显式适配层。
  • 遇到内建对象或类实例时是否应该直接套 Proxy？ 结论是不应默认直接套通用代理。原因是普通对象数据和带内部槽位/私有状态的对象行为差异很大。项目里更稳的做法是代理 plain object，或为 Date、Map、DOM 节点、第三方实例分别设计清晰的封装 API。
