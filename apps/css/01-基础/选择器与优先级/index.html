选择器种类与优先级（specificity）

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见本复习区，脚本会读取实际生效样式。

知识点要点

  • MDN 定义 specificity 为三元组 (a, b, c)：a 为 ID 选择器数量，b 为类、属性、伪类数量，c 为类型选择器和伪元素数量；通配符和组合符本身不增加权重。
  • 内联 style 属性通常高于普通作者样式，但本例 .box { background-color: yellow !important; } 能覆盖 style="background-color: red"，因为 !important 把声明提升到 important 层级。
  • !important 不是日常覆盖工具，它会让后续维护必须进入同样的 important 层级；在同一重要性、来源和层级下，才继续比较 specificity，完全相同则后写覆盖先写。
  • 属性选择器可以按属性存在、精确值、包含子串、前缀、后缀、空格分隔词列表和语言前缀匹配；[class~="important"] 匹配独立 class 词，[href*="baidu"] 只是子串匹配。
  • 工程推荐让选择器短而可预测，例如 BEM、CSS Modules、组件作用域或级联层（@layer）；覆盖第三方样式时优先调整加载顺序、作用域和公开变量，最后才提高权重或使用 !important。

面试考点

  • CSS 优先级冲突的比较顺序是什么？ 结论是先看级联来源、重要性和级联层，再在同一层级内比较 specificity，最后才看声明顺序。原因是 CSS Cascade 不只是选择器权重算法，!important、内联样式、用户样式、作者样式和层级都会参与。项目里排查样式不生效时，应先在 DevTools 里看被哪条规则覆盖，再判断是顺序、权重还是 important 问题。
  • 为什么不建议滥用 !important？ 结论是 !important 只适合少数强约束或覆盖不可控来源的场景，不适合常规布局。原因是它提升声明优先层级，后续覆盖往往只能继续加 !important 或提高来源层级，维护成本很高。本例用它是为了演示可以覆盖普通内联背景色，真实项目更推荐通过作用域、加载顺序或设计 token 解决。
  • 属性选择器 ^=、$=、*=、~=、|= 分别适合什么场景？ 结论是 ^= 匹配前缀，$= 匹配后缀，*= 匹配任意子串，~= 匹配空格分隔词，|= 匹配语言代码或连字符前缀。原因是属性值可能是 URL、class 词列表或语言标签，匹配语义不同。项目里用子串匹配要谨慎，URL 和 class 命名变化都可能造成误命中。
  • 覆盖组件库或第三方样式时如何避免权重失控？ 结论是先控制样式加载顺序和作用域，再使用组件库暴露的 class、CSS 变量或主题 API，最后才提高选择器权重。原因是权重越高越难回收，深层选择器和 !important 会把局部问题扩散成全局维护成本。边界是修复浏览器扩展、第三方 iframe 外壳或紧急线上样式时可以临时用 important，但应标注原因并尽快移除。
