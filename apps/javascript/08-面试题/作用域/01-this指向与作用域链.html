经典题 - this 指向与作用域链

先看本复习区，再查看下方输出是否与标准答法一致；细节可打开 DevTools Console。

知识点要点

  • 这题运行在普通 <script> 且未开启严格模式： bar() 是独立函数调用，普通函数的 this 会默认绑定到全局对象 window 。
  • 顶层 var val = 1 在浏览器普通脚本中会创建 window.val ，所以 console.log(window.val) 输出 1 ；如果改成 let / const ，不会生成 window 属性。
  • bar 内部的 var val = 2 是局部变量，只影响标识符 val 的词法查找，不会改变 this.val 。
  • test() 在全局作用域定义，它的外层词法环境是全局环境；即使在 bar() 里调用，变量 val 仍沿定义时作用域链找到全局 val 。
  • this 绑定和作用域链是两套规则：变量名由词法环境决定， this 由调用方式决定；箭头函数没有自己的 this ，会捕获外层 this 。
  • 如果开启严格模式，独立调用普通函数时 this 是 undefined ，访问 this.val 会抛错；如果放到 type="module" 中，顶层 var 也不会挂到 window 。

面试考点

  • 为什么 bar() 中 this.val 输出 1 而不是局部变量 2 ？ 结论：因为 this.val 读的是 window.val 。原因是 bar() 作为普通函数独立调用，非严格模式下 this 默认指向 window ，而顶层 var val = 1 又创建了 window.val 。边界是严格模式下 this 为 undefined ，这行会抛错。
  • test() 为什么读取全局 val ，不是 bar() 里的 val ？ 结论：普通变量查找按函数定义时的词法作用域链，不按调用栈动态查找。原因是 test 在全局定义，外层环境就是全局环境， bar 的局部环境不是它的外层词法环境。项目里排查闭包和回调变量时，要先看函数在哪里定义，再看它在哪里调用。
  • 严格模式、 let / const 或模块化会怎样改变结果？ 结论：这些都会改变全局绑定或 this 默认值。原因是严格模式下独立调用的 this 不再装箱到全局对象， let / const 不创建 window 属性，ES module 顶层作用域也不是传统脚本全局对象。面试答题要先说明运行环境，否则同一段代码可能结果不同。
  • 如何区分作用域链和 this 绑定？ 结论：标识符变量看“定义在哪里”， this 看“怎么调用”。原因是词法作用域在函数创建时确定， this 是调用表达式的一部分。边界是箭头函数没有自己的 this ， call/apply/bind 可以显式指定普通函数的 this 。
