字符串中反斜杠转义与 JSON 序列化

请打开 DevTools Console 查看输出。

知识点要点

  • 这页真实主题不是现代 escape() 用法，而是反斜杠在 JS 字符串字面量、JSON 文本和二次嵌套中的表现。
  • JSON 字符串必须转义双引号、反斜杠以及 U+0000 到 U+001F 控制字符；/ 允许转义但通常不是必须。
  • JSON.stringify 会生成合法 JSON 文本，JSON.parse 会把转义序列还原为实际字符，业务代码不要手写拼接复杂 JSON。
  • 把 JSON 文本再作为另一段 JS 字符串、HTML 属性或外层 JSON 的值时，反斜杠本身也会被转义，所以日志里可能看到 \\\"。
  • escape() / unescape() 是已废弃的全局 API，和 JSON 转义不是一回事；URL 参数用 encodeURIComponent，结构化数据用 JSON.stringify。
  • 不要扩展 String.prototype 实现项目转义工具，原因是会污染所有字符串对象，增加枚举、命名冲突和第三方代码兼容风险。

面试考点

  • JSON 哪些字符必须转义？ 结论是双引号、反斜杠和控制字符必须转义。原因是它们会破坏 JSON 字符串边界或不可直接出现在 JSON 文本中；边界是 \uXXXX 只是表示字符的另一种写法，不等于 URL 百分号编码。
  • 为何会看到双重反斜杠？ 结论是因为同一段内容经过了两层字符串语法。第一层为了 JSON 合法性转义引号，第二层为了把 JSON 文本放进另一个字符串而转义反斜杠；排查时要区分「源码/日志展示」和「解析后的实际值」。
  • escape、JSON 转义和 encodeURIComponent 有什么区别？ 结论是三者服务于不同语境。JSON 转义维护数据格式，encodeURIComponent 维护 URI 组件，escape 是历史 API 且已废弃；项目里不要混用。
  • 为什么不要改 String.prototype？ 结论是除非 polyfill 且遵循规范，否则不应修改内建原型。原因是所有字符串都会继承该方法，可能与未来标准或库冲突；局部工具函数更可控。
