LRU Cache（Map 简化版）

请打开 DevTools Console 查看输出。

知识点要点

  • LRU（Least Recently Used）按“最近使用”维护缓存顺序：越久没有被访问的项越靠前，容量超限时优先淘汰最旧项。
  • ES Map 会按插入顺序迭代键；命中 get 后先 delete 再 set ，可以把该键移动到顺序末尾，表示最近使用。
  • put 已存在的键时也要刷新顺序；插入后如果 map.size > capacity ，通过 map.keys().next().value 取到第一个键并删除。
  • 严格算法题常要求 get / put 都是 O(1) ，典型实现是哈希表定位节点、双向链表维护新旧顺序；本页 Map 版更短，适合解释思路。
  • 容量必须是正整数，非法容量要明确抛错；真实缓存还要考虑过期时间、内存上限、并发读写、缓存击穿和数据一致性。
  • 本 demo 用 textContent 输出操作日志，不把日志拼成 HTML；带用户输入或服务端数据的缓存可视化也应避免把内容直接写入 innerHTML 。

面试考点

  • LRU 为什么命中 get 后也要刷新顺序？ 结论：因为“使用”不仅包括写入，也包括读取。原因是 LRU 淘汰的是最近最少使用项，如果读取命中后不刷新，该热点数据仍可能被误判为旧数据而淘汰。项目里接口响应缓存、图片缓存、计算结果缓存都需要在命中时刷新热度。
  • Map 版 LRU 为什么能删除最旧 key ？ 结论：因为 Map 按插入顺序迭代键，第一个键就是当前最旧项。原因是每次命中或更新都会删除再插入，把最近使用项移动到末尾。边界是这依赖 ECMAScript 对 Map 迭代顺序的定义，不适用于普通对象属性顺序来做严谨 LRU。
  • Map 简化版和哈希表 + 双向链表版有什么差异？ 结论：Map 版代码短，双向链表版更贴近严格算法题。原因是双向链表能显式把任意节点移到头部或尾部，哈希表负责 O(1) 定位；Map 版利用语言内置顺序，适合前端面试讲解和小型缓存。边界是大规模缓存还要考虑内存、淘汰策略和过期策略。
  • LRU 缓存适合哪些项目场景，容量边界怎么处理？ 结论：适合缓存最近访问更可能再次访问的数据，比如接口结果、编译产物、图片缩略图。原因是它用有限容量换取热点数据命中率。边界是容量应按内存和业务命中率评估，非法容量要拒绝，缓存数据过期或权限变化时还要主动失效。
