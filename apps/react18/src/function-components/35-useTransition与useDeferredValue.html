useTransition 与 useDeferredValue

请打开 DevTools Console 查看 tab 切换与 deferred 过滤日志；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • createRoot 启用 React 18 新根行为（如更广范围的自动批处理），但 不会 因此全面开启并发渲染；需要并发调度时，应通过 startTransition 等 API 按需标记非紧急更新。
  • useTransition 返回 [isPending, startTransition] 。把 state 更新包进 startTransition(() => ...) 后，React 可以中断或延后这次更新，让更紧急的交互（如当前输入、点击反馈）先完成。
  • useDeferredValue(value) 会返回一个“滞后版”的值：当 value 快速变化时，UI 可以先继续显示旧值，再在后台追上最新值。搜索框 + 大列表过滤是典型场景。
  • 当 query !== deferredQuery 时，说明列表仍基于旧 deferred 值渲染，可配合 opacity 提示“结果尚未跟上输入”。这与 isPending 类似，但语义上一个是“Transition 进行中”，一个是“展示值仍落后”。
  • Transition 解决的是 更新优先级 ，不能替代函数式 setState 、依赖数组或数据请求边界；昂贵计算仍应配合 useMemo 、虚拟列表等手段。

面试考点

  • createRoot 与 startTransition 分别解决什么问题？ 结论是 createRoot 提供 React 18 新客户端根与更广批处理前提； startTransition 把某次更新标记为非紧急。原因是并发调度需要新根能力，但只有你显式标记 Transition 时，React 才会按较低优先级处理该更新。
  • useTransition 和 useDeferredValue 分别适合什么场景？ 结论是 useTransition 适合你在事件处理里主动发起 state 更新（如切 tab、打开面板）； useDeferredValue 适合“已有快速变化的值，希望下游渲染慢半拍”（如搜索词驱动列表）。原因是前者控制更新入口，后者延迟消费某个值。
  • isPending 与 query !== deferredQuery 各表示什么？ 结论是 isPending 表示 Transition 更新尚未完成； query !== deferredQuery 表示 deferred 值还没追上最新 query。原因是一个跟踪 Transition 生命周期，一个跟踪值滞后；项目里都可用来展示 loading 或 stale 样式。
  • Transition 更新被中断时会怎样？ 结论是 React 可以丢弃未完成的 Transition 工作并重新开始。原因是更高优先级更新到来时，React 优先保证紧急 UI；边界是 Transition 内更新不应依赖“一定完整执行一次”的副作用顺序。
  • Transition 能否替代函数式 setState？ 结论是不能。原因是 Transition 只改变调度优先级，不改变 state 快照语义；连续计数、闭包、依赖数组等规则仍然成立，项目里仍要写 updater 和正确依赖。
