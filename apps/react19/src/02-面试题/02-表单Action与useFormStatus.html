表单 Action 与 useFormStatus

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • React 19 扩展了表单的 action：函数 Action 会收到 FormData，可直接由提交触发异步逻辑；React 18 项目通常需要自己写 onSubmit、阻止默认行为并维护 pending。
  • useFormStatus 从 react-dom 导出，读取最近父级 <form> 的 pending、data、method、action；本页的 SubmitButton 不需要 props 下钻就能禁用按钮。
  • 常见坑是调用位置：useFormStatus 只能在表单内部的子组件里读到状态，在渲染 <form> 的同一个组件中调用只会读到外层表单或空状态。
  • useActionState 更适合保存提交后的业务结果，useFormStatus 更适合做局部提交反馈；两者组合时要同时处理空输入、重复点击、异步失败和用户可读错误。
  • 本页演示的是静态 HTML 中的客户端 Action；真正的 Server Action 属于 React 19 生态中的框架能力，需要构建器、服务端环境和可序列化参数约束。

面试考点

  • React 19 表单 Action 相比 React 18 表单提交解决了什么问题？ 结论是把“提交触发异步逻辑、pending、结果状态”收进 React 的表单模型里。原因是 action 函数天然接收 FormData，并能和 useActionState、useFormStatus 联动。边界是它不是替代表单校验和错误建模，服务端 Action 还需要框架运行时支持。
  • useFormStatus 为什么通常要封装成 SubmitButton 子组件？ 结论是它读取的是最近父级表单状态，子组件位置正好在表单上下文内部。原因是 Hook 的状态来自 DOM 表单提交上下文，而不是当前组件自己创建的某个变量。项目里常把按钮、loading 文案和重复提交禁用放到独立组件中。
  • useFormStatus 与 useActionState 的 pending 分别适合放在哪里？ 结论是 useFormStatus 适合表单内部局部控件，useActionState 适合拥有 Action 的组件统一展示结果。原因是前者跟随最近表单，后者返回 action 的最新 state 和 pending。复杂表单可用前者禁用按钮，用后者渲染成功或失败提示。
  • 表单 Action 失败时为什么要返回可展示状态而不是只抛异常？ 结论是业务失败应进入页面状态，异常边界更适合兜底未知错误。原因是“评论不能为空”“库存不足”这类失败需要展示在表单旁并允许用户修正。项目中还要考虑网络失败、重复提交、服务端校验和可访问性提示。
