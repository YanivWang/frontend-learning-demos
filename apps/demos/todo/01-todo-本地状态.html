Todo 本地状态管理

这个 demo 用原生 DOM 写一个小 Todo，重点观察：状态数组如何驱动视图、动态列表如何用事件委托、用户输入如何安全写入页面。

知识点要点

  • 状态唯一来源：todos 数组是页面的业务状态，DOM 只是状态的展示结果。新增、切换、删除都先更新数组，再调用 render() 重新生成列表。
  • 不可变更新思路：新增用 [...todos, newTodo]，切换用 map() 返回新对象，删除用 filter() 返回新数组。这和 React 官方推荐的数组状态更新方式一致，也更容易追踪状态变化。
  • 事件委托：列表项由 render() 动态创建，不需要给每个 checkbox / 删除按钮单独绑定事件；把监听器放在父级 ul 上，通过冒泡上来的 event.target 判断用户点了谁。
  • 安全渲染文本：任务内容来自输入框，属于不可信字符串；示例使用 textContent 写入文本，避免把用户输入交给 innerHTML 当 HTML 解析，从源头降低 XSS 风险。
  • 和框架的关系：React / Vue 帮你做了更复杂的状态追踪和 DOM 更新，但最基础的心智模型仍然是「状态变化 -> 视图重新计算」。先理解这个原生版本，再学框架会更稳。

面试考点

  • 原生 Todo 如何组织状态、事件和渲染？ 可以回答为：用一个数组保存任务列表，把新增、切换、删除都收敛成状态更新，然后统一调用渲染函数把状态同步到 DOM。这样业务状态不会散落在多个 DOM 节点里，后续迁移到 React / Vue 也容易。
  • 为什么列表更新推荐生成新数组，而不是直接 push / splice 原数组？ 原生 JS 里直接改数组也能跑，但生成新数组能让“这次状态发生了变化”更明确。React 官方也要求把 state 中的数组当作只读数据，新增用展开或 concat()，删除用 filter()，替换用 map()。
  • 动态列表为什么适合用事件委托？ 因为列表项会频繁新增、删除和重建，给父级容器绑定一次事件更稳定。事件从子元素冒泡到父元素后，通过 event.target 和 dataset 判断动作类型，可以避免重复绑定和遗漏新节点。
  • 用户输入为什么不要直接拼到 innerHTML？ innerHTML 会把字符串按 HTML 解析，不可信输入可能变成可执行或危险的 DOM。展示纯文本时优先用 textContent；如果确实要渲染 HTML，必须先做严格的可信来源控制或净化。
  • 这个 demo 和 React / Vue 的状态驱动视图有什么关系？ 它是框架心智模型的最小版：状态是数据源，视图由状态推导，事件只负责触发状态变化。区别是框架会自动调度更新、做局部 diff 或响应式追踪，而这里手动调用 render()。
