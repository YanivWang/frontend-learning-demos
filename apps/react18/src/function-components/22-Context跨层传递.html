Context 跨层传递

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • Context 用来把数据提供给某棵组件树下的所有后代，典型场景是主题、语言、当前用户、权限配置等跨层但相对稳定的数据。
  • useContext(ThemeContext) 会沿组件树向上查找最近的 ThemeContext.Provider；如果没有匹配 Provider，才会使用 createContext 的默认值。
  • Provider 的 value 按引用传递，父组件每次创建新对象都可能让消费者重渲染；本 demo 用 useMemo 稳定主题对象，只在 dark 改变时生成新值。
  • Context 不负责异步请求、缓存、时间旅行调试或复杂派生状态；大型业务状态通常要拆分 Provider，或引入 Redux、Zustand、TanStack Query 等更专门的工具。
  • 现代写法优先用函数组件与 useContext；维护旧 class 代码时才会遇到 Context.Consumer 或 contextType。

面试考点

  • Context 适合传什么数据，不适合传什么数据？ 结论是适合跨层、低频、读多写少的数据。原因是它避免层层透传 props，但 Provider value 变化会影响所有消费者。项目里主题、登录用户可放 Context，输入框草稿、列表缓存和高频动画状态不宜直接塞进一个大 Context。
  • useContext 如何决定读取哪个 Provider？ 它读取调用组件上方最近的同一个 Context Provider。原因是 React 按组件树而不是 DOM 层级查找上下文；没有 Provider 时使用默认值。嵌套主题、局部权限覆盖等场景可以用内层 Provider 覆盖外层值。
  • Provider value 为什么常配合 useMemo 或拆分 Context？ 结论是为了减少不必要的消费者重渲染。原因是对象、数组、函数每次重新创建都会变成新的引用。本 demo 的主题对象只依赖 dark，真实项目还会把读状态和写 dispatch 分到不同 Context。
  • Context 与 props 组合、状态库如何取舍？ 少量中间层优先 props 或 children 组合，跨很多层共享稳定配置再用 Context。需要复杂更新流程、调试工具、服务端缓存或跨页面协作时，Context 通常只是底层通道，不应当承担完整状态管理。
