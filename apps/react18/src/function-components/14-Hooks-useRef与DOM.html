useRef 保存 DOM 与可变值

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • useRef 返回同一个 ref 对象，React 会在提交 DOM 后把节点写入 current；示例中的输入框聚焦就是典型命令式 DOM 操作。
  • 修改 ref.current 不会触发重新渲染，所以它适合保存定时器 ID、上一次值、点击次数这类“不直接驱动 UI”的可变数据。
  • useState 用于让页面随数据变化重新渲染，useRef 用于跨渲染保存可变引用；把 UI 状态塞进 ref 会让界面和数据不同步。
  • 不要在渲染期间随意读写可变 ref，因为 React 可能重新渲染、丢弃渲染或并发调度；读写 DOM ref 通常放在事件处理或 effect 中。
  • DOM ref 需要考虑节点尚未挂载或已经卸载的边界，实际项目里调用 focus()、测量尺寸、滚动前应先确认 current 存在。

面试考点

  • useRef 和 useState 的本质区别是什么？ 结论是 state 参与渲染，ref 不参与渲染。原因是 setState 会通知 React 重新计算 UI，而修改 ref.current 只是改一个普通对象属性。项目里输入值、列表和错误信息用 state，定时器 ID、DOM 节点和上一次值用 ref。
  • 什么时候应该通过 ref 操作 DOM？ 结论是只在 React 声明式模型覆盖不了的命令式场景使用。原因是聚焦、滚动、测量尺寸、调用第三方组件实例方法都需要真实 DOM 或实例。边界是不要用 ref 手动改 React 正在管理的文本和样式，否则容易和下一次渲染冲突。
  • 为什么修改 ref.current 页面不会立刻更新？ 结论是 ref 的变化不会触发 render。原因是 React 不追踪 ref 对象内部属性的变化。示例里点击“只改 ref”会更新控制台值，页面要等其他状态触发渲染后才读到新值。
  • 在渲染期间读写 ref 有什么风险？ 结论是容易破坏组件纯渲染假设。原因是渲染可能被 React 重复调用或丢弃，写 ref 会产生不可预测的外部变化。项目中应把这类读写放到事件处理、effect 或 layout effect 中。
