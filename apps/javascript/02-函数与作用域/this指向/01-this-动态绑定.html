this 的动态绑定

请打开 DevTools Console 查看输出。

知识点要点

  • 普通函数的 this 绑定在调用时确定。示例中 window.a() 是属性引用调用，所以 a 内部的 this 是 window 。
  • window.a()() 分两步：先执行 window.a() 得到一个匿名函数，再把这个匿名函数作为普通函数独立调用；第二次调用没有接收者，非严格模式下 this 会被替换为 window 。
  • 「this 指向最后调用它的对象」只是记忆口诀，不是完整规则。更准确的说法是：根据调用表达式判断引用基值，或由 call / apply / bind 、 new 、箭头函数词法捕获等规则决定。
  • 严格模式下，独立调用普通函数的 this 保持为 undefined ；非严格模式下 null / undefined 会被替换为全局对象，原始值会装箱。
  • 箭头函数没有自己的 this ，会捕获外层词法环境的 this ，适合回调保留上下文，但不能作为构造函数使用。

面试考点

  • 普通函数的 this 到底由什么决定？ 先结论：由调用形式决定。原因是函数调用会根据引用的基值或显式传入的 thisArg 设置 this 。项目中判断 obj.fn() 、 const fn = obj.fn; fn() 、事件回调和 bind 时，都要先还原实际调用表达式。
  • window.a()() 为什么第二次调用拿不到第一次的 this ？ 先结论：第一次调用返回的是一个新函数，第二次是独立调用。原因是普通函数不会自动继承外层调用的 this 。如果要保留上下文，可返回箭头函数、使用 bind ，或在外层保存 const self = this 。
  • 严格模式、非严格模式下独立函数调用的 this 有什么不同？ 先结论：严格模式是 undefined ，非严格模式会替换为全局对象。原因是非严格调用会执行 this 替换规则。边界是 ES modules 和 class 方法默认严格，所以很多现代代码里不会退化到 window 。
  • 箭头函数和普通函数的 this 机制有什么区别？ 先结论：箭头函数没有自己的 this ，普通函数有调用时绑定。原因是箭头函数从外层词法环境读取 this 。项目中箭头函数适合作为回调保留外层上下文，但不适合定义需要动态接收者的对象原型方法。
