函数组件 + props 传值（计数器示例）

请打开 DevTools Console 查看每秒 props 输出；下方 #root 为 React 挂载演示区。

知识点要点

  • 函数组件本质上是接收 props 并返回 React element 的函数。父级写在 JSX attribute 上的值，例如 count={count} ，会组成一个 props 对象传给组件。
  • props 是组件的外部输入，应按只读值处理。子组件不能通过修改 props.count 改变父级数据；需要改变数据时应由父级重新传入 props，或让组件使用自己的 state。
  • 本 demo 用外部变量 count 加 root.render(<Counter count={count} />) 演示“父级重新传 props”。真实项目中更常见的是父组件用 useState 更新 count，然后自然触发子组件重渲染。
  • props 适合表达父级控制的输入和配置， state 适合表达组件自身随交互变化的数据。不要把 props 复制到 state 里再维护两份来源，除非有明确的派生状态需求。
  • createRoot 只负责在 DOM 容器上创建 React root；之后每次更新复用同一个 root.render 。旧 ReactDOM.render 已废弃。

面试考点

  • 函数组件接收 props 的规则是什么？ 结论是 React 调用函数组件时会传入一个 props 对象。原因是 JSX attribute 会被收集到这个对象中；边界是组件名必须大写，小写标签不会按自定义函数组件调用。
  • 为什么 props 必须只读？ 结论是 props 是父级传入的输入，子组件不能修改它。原因是 React 的数据流是从父到子，修改 props 不会通知父级也会破坏可预测性；项目里要改数据应调用父级回调或使用 state。
  • 本例为什么不用 useState ？ 结论是本例刻意用手动 root.render 突出 props 传值过程。原因是它处在入门阶段；真实业务计数器应放在组件 state 中，而不是维护全局变量和定时 render。
  • props 和 state 如何区分？ 结论是 props 表示外部输入，state 表示内部可变数据。原因是它们的所有权不同；边界是受控组件会把值提升到父级 state，再通过 props 下发。
  • createRoot 与 root.render 各负责什么？ 结论是 createRoot 在容器上创建 React 18 root（只做一次）， root.render 提交新的 element 树触发更新。原因是 root 保存渲染上下文；计数器每秒 tick 只需反复 render ，不要反复 createRoot 。
