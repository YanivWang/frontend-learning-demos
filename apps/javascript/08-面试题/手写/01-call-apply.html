手写 call / apply

先看本复习区，再查看下方输出是否与标准答法一致；细节可打开 DevTools Console。

知识点要点

  • Function.prototype.call() 和 apply() 都会立即调用目标函数，并把第一个参数作为函数执行时的 this 。
  • 两者主要区别是参数形态： call(thisArg, a, b) 逐个传参， apply(thisArg, [a, b]) 接收数组或类数组参数列表。
  • 手写实现常用“临时属性调用”模拟隐式绑定：把函数挂到 thisArg 包装对象上，再执行 context[key](...args) ，这样函数内部 this 就指向 context 。
  • 临时属性名要用 Symbol 避免覆盖业务属性，并在 finally 中删除；否则被调函数抛异常时，目标对象会残留临时方法。
  • 非严格普通函数语义下， null / undefined 会替换为 globalThis ，数字、字符串、布尔值等原始值会通过 Object() 装箱。
  • 生产代码一般不污染 Function.prototype ；这里挂原型只是为了演示面试题，真实项目优先使用原生 call / apply 。

面试考点

  • call 和 apply 的共同点与区别是什么？ 结论：它们都会立即执行函数并指定 this ，区别只在传参形式。原因是两者都调用同一个函数，只是 call 接收参数列表， apply 接收数组或类数组。项目里动态参数数组通常用 apply 或展开语法，固定参数用 call 更直观。
  • 手写 call/apply 为什么要临时挂载函数？ 结论：为了把显式绑定转成对象方法调用。原因是 obj.fn() 调用时，普通函数内部的 this 会指向 obj ；临时挂载后调用即可模拟这个规则。边界是必须用唯一键并及时删除，避免覆盖业务属性或留下副作用。
  • thisArg 为 null 、 undefined 或原始值时怎么处理？ 结论：按非严格普通函数语义，空值转为 globalThis ，原始值装箱为对象。原因是原生 call/apply 在非严格函数调用中也会做类似替换。边界是严格模式函数内部的 this 不会被强制替换，这种简化版手写实现不能完全复刻规范所有细节。
  • 手写 apply 为什么要校验第二个参数并清理临时属性？ 结论：第二个参数应能转成参数列表，临时属性无论调用成功或失败都要清理。原因是错误参数会导致展开或转换异常，被调函数本身也可能抛错。项目里写 polyfill 或工具函数时，清理逻辑要放在 finally 中，保证失败路径也不污染对象。
