v-on 事件三种形态（直接方法名 / 调用方法 / 表达式）

依次点「点击1/2/3」对比方法名绑定与内联调用的事件参数差异；点「计数」看表达式写法； 下方日志区显示每次触发的参数信息。

知识点要点

  • v-on 用来监听 DOM 事件， @click 是 v-on:click 的简写。Vue 只负责把事件绑定到模板声明的位置，真正的业务逻辑通常写在 methods 。
  • @click="clickHandle1" 属于方法事件处理器，Vue 会把原生 DOM 事件对象传给方法。本页点击“点击1”会把事件对象写入日志。
  • @click="clickHandle3('say', $event)" 属于内联调用。只要写了括号，Vue 不会自动补事件对象；需要原生事件时必须显式写 $event 。
  • @click="count = count + 1" 这类表达式适合非常小的状态更新。边界是表达式一旦包含分支、异步、校验、复用逻辑，就应该移到 methods ，否则模板会变难读。
  • 事件对象来自浏览器原生事件，所以涉及默认行为、冒泡、捕获时，要结合事件修饰符或 DOM 事件机制理解。Vue 2 已 EOL，新项目优先 Vue 3。

面试考点

  • Vue 2 中 @click="fn" 和 @click="fn()" 的事件参数有什么区别？ 结论是不带括号时 Vue 会把原生事件对象传入，带括号时需要显式写 $event 。原因是前者是方法事件处理器，后者是内联表达式调用。项目里如果既要业务参数又要事件对象，应写 @click="submit(row, $event)" 。
  • 什么场景可以直接在模板里写事件表达式？ 结论是只适合非常简单、一次性的状态更新。原因是模板表达式应该保持可读，复杂逻辑会让视图层承担业务流程。项目里 count++ 可以接受，表单校验、接口请求、权限判断应放到 methods 。
  • 为什么复杂事件逻辑应该放到 methods？ 结论是为了复用、测试和维护。原因是 methods 有清晰函数名，能拆分分支、处理异常和复用参数，而模板适合表达“哪个事件触发哪个动作”。边界是 methods 里仍不要混入大量 DOM 事件细节，可用修饰符处理冒泡和默认行为。
  • v-on 和事件修饰符分别解决什么问题？ 结论是 v-on 负责监听事件，修饰符负责声明事件机制细节。原因是监听点击和阻止提交刷新是两层关注点。项目里常写 @submit.prevent="save" ，让 save 专注保存逻辑。
  • 维护 Vue 2 项目时如何说明事件写法与 Vue 3 选型边界？ 结论是 Vue 2/3 的 v-on / @ 语法基本一致，差异在项目选型与 Composition API。原因是 Vue 2 已 EOL，新项目应优先 Vue 3；存量项目的事件绑定写法可直接迁移。边界是 Vue 3 移除了 .native 修饰符。
