TypeScript mini-project

本页对应一个真实 TypeScript 小工程，重点不是浏览器交互，而是用 tsc --noEmit 验证源码和配置。

知识点要点

  • 这个 mini-project 用真实文件演示 TypeScript 工程闭环：src/user.ts 定义领域类型和函数，src/index.ts 通过 import type 引入类型，通过普通 import 引入运行时代码，tsconfig.json 决定编译器如何检查。
  • tsc --noEmit -p tsconfig.json 会读取项目配置并执行完整类型检查，但不会写入 dist。这适合放进 CI 或本仓库的 npm run validate:tsc，用来证明示例源码仍能在 strict 规则下通过。
  • strict 是一组严格检查的总开关，本项目还显式打开 noImplicitAny、strictNullChecks 和 noUnusedLocals。它们分别帮助发现未标注的宽泛参数、可能为空的访问，以及重构后遗留的未使用变量。
  • moduleResolution: "bundler" 面向现代打包器解析模型；示例里 import type { User } 表示只在类型层使用 User，避免读者误以为该导入一定会变成运行时代码。
  • 常见坑是把类型检查当成运行时测试。formatUser 能保证传入值符合 User 的静态形状，但外部 JSON、接口返回、用户输入和构建产物仍需要运行时校验、单元测试或端到端验证补齐。

面试考点

  • tsc --noEmit 在工程里解决什么问题？ 结论是它把 TypeScript 当作类型检查器使用，不生成 JS 产物。原因是很多项目已经由 Vite、Webpack、Babel 或 swc 负责转译，仍需要独立命令保证类型没有退化。项目里通常把它放在 CI、提交前检查或发布前验证中。
  • 为什么推荐开启 strict？ 结论是 strict 比零散类型注解更能建立工程级安全基线。原因是它会系统性收紧隐式 any、空值、函数参数等规则，让错误在改代码时暴露。边界是迁移老项目可以分阶段开启，但新项目应默认 strict。
  • import type 和普通 import 有什么区别？ 结论是 import type 只服务类型系统，普通 import 表示运行时也需要这个模块值。原因是 TypeScript 会在 emit 阶段擦除类型导入，清晰区分后能减少循环依赖和副作用误判。项目里类型和函数来自同一文件时，也建议像本例一样分开写。
  • 类型检查通过是否等于程序没有 bug？ 结论是否定的，类型检查只证明代码满足静态契约。原因是接口返回、用户输入、时间、网络失败和权限状态都发生在运行时。真实项目还要配合运行时 schema 校验、单元测试、集成测试和错误监控。

npm run validate:tsc

npx --yes -p typescript@5.8.3 tsc --noEmit -p apps/typescript/mini-project/tsconfig.json

源码：src/user.ts、src/index.ts · 配置：tsconfig.json（strict: true）

// src/user.ts
export type User = { id: number; name: string; email?: string };

export function formatUser(u: User): string {
  return u.email ? `${u.name} <${u.email}>` : u.name;
}

// 取消下面这行注释，validate:tsc 应报错：
// const bad: User = { id: "x", name: 1 };
