class 组件中的受控与非受控表单

请填写下方表单并点击提交，观察受控字段实时回显与非受控字段通过 ref 读取的差异；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • 受控组件把输入值放在 this.state 中， value 或 checked 由 React 传回输入框， onChange 再把用户输入写回 state，因此校验、联动、禁用提交按钮和回显都可以由同一份数据驱动。
  • 非受控组件让 DOM 保存当前值，React 只用 defaultValue 或 defaultChecked 设置初始值，提交时通过 ref 读取。它适合简单一次性读取、迁移旧表单或文件输入这类 DOM 自有状态。
  • value 与 defaultValue 不是同一种控制权：前者每次渲染都以 React state 为准，后者只影响初始值。不要在同一个输入生命周期内从未定义 value 切到有值，避免受控/非受控切换警告。
  • class 组件里表单事件仍是合成事件，读取 event.target.value 后用 setState 更新；如果下一步依赖更新后的 state，应放到 setState 回调或 componentDidUpdate 。
  • 涉及用户输入时不要把输入内容拼进 innerHTML 。本 demo 提交时用 alert(JSON.stringify(...)) 展示数据，没有把用户输入作为 HTML 注入页面。

面试考点

  • 受控组件为什么说 React state 是唯一数据源？ 结论是输入显示值由 state 决定。原因是 value 绑定 state 后，用户输入必须经过 onChange 和 setState 才能反映到界面；边界是高频大表单要注意渲染成本，可拆分字段组件或局部状态。
  • defaultValue 和 value 的行为差异是什么？ value 是持续控制， defaultValue 只是初始值。原因是 React 后续渲染不会再用 defaultValue 覆盖 DOM 当前值；项目中如果要重置非受控输入，通常改 key 重新挂载或直接操作 ref。
  • 表单校验、联动和一次性提交时如何选择？ 结论是需要实时校验、联动、回显、禁用提交时优先受控，只在简单采集或接入旧 DOM 插件时考虑非受控。原因是受控值在 React 数据流里更容易组合；边界是文件输入不能被脚本设置文件路径，通常用非受控读取。
  • 为什么文件输入和部分第三方控件更适合非受控？ 结论是浏览器安全限制和 DOM 自有状态。原因是 <input type="file"> 的值只能由用户选择，React 不能通过 value 设置文件路径；集成 jQuery UI、富文本编辑器等命令式 DOM 库时，也常用 ref 读取而非受控 state。
  • class 表单里如何避免受控/非受控切换警告？ 结论是给 value 稳定默认值（如空字符串），不要在受控/非受控之间切换。原因是 React 检测到 value 从 undefined 变为字符串时会警告；项目中字符串字段初始化为 "" ，布尔字段用 checked + 明确布尔值。
