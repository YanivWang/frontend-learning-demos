Vue Router 4 + Pinia 可运行最小示例

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • 本页是真正可运行的 Router + Pinia 最小示例：createPinia() 和 createRouter() 都通过 app.use() 安装，组件里再通过 store 和 router 完成登录、跳转、退出。
  • 示例使用 createWebHashHistory()，因为 hash 路径不会交给服务器解析，适合本仓库可直接用 file:// 打开的 HTML；真实线上 history 模式需要服务端 fallback。
  • beforeEach 守卫读取 meta.requiresAuth 与 Pinia 登录态，未登录访问 /profile 时返回 "/login"；返回值写法比混用 next 更不容易重复放行。
  • 示例补了空用户名提示、退出后跳转登录页、未知路由 fallback，演示路由类 demo 的失败路径；用户名通过 Vue 文本插值显示，不会被当作 HTML 执行。
  • 真实项目里前端登录态只负责体验层拦截，不能替代服务端权限校验；token 持久化、刷新、过期、XSS 风险和接口 401 处理都需要单独设计。

面试考点

  • Router 守卫里读取 Pinia store 要注意什么？ 结论是要确保 Pinia 已创建并在应用生命周期内作为插件安装，守卫执行时才能拿到正确的 active pinia。原因是 useStore() 依赖当前 Pinia 实例；边界是 SSR 或多个 app 场景应显式传入 pinia，避免状态串用。
  • 为什么这个 demo 选择 hash history？ 结论是为了让文件协议和静态服务都能打开并刷新。原因是 hash 后面的路径不会发给服务器，避免刷新子路由时 404；项目上线如果能配置服务器回退，通常可以改用 history 模式获得更自然 URL。
  • 登录跳转如何避免无限重定向？ 结论是守卫只拦截带 requiresAuth 的目标页，并把登录页排除在受保护集合外。原因是如果 /login 本身也触发同一重定向条件，会形成循环；项目里还要保存来源页、处理空输入和接口登录失败。
  • 前端登录态和真实鉴权有什么区别？ 结论是前端状态只能改善导航体验，安全鉴权必须在后端接口完成。原因是用户可以修改本地状态或跳过前端路由；真实项目需要服务端校验 token、权限、过期时间，并在 401/403 时统一清理状态和重定向。
