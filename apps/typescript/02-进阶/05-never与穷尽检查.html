never 与穷尽检查

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • never 表示不可能正常出现的值，常见于总是抛错的函数、无限循环函数，以及联合类型被所有分支处理完后的剩余类型。
  • 可辨识联合配合 switch 能做穷尽检查：当每个 type 都有对应 case，default 分支里的 action 应被收窄成 never。
  • assertNever(x: never): never 的价值是让“新增 action 但漏写 case”从运行时隐藏 bug 变成编译期错误，本页 reducer 就演示了这个结构。
  • void 表示函数返回值不会被使用或没有有意义返回值；never 表示函数不会正常返回，两者不能混成“都没有返回值”。
  • 常见坑是在 default 分支直接返回原状态或空值，这会掩盖新增联合成员；业务 reducer、状态机和消息处理器更适合显式调用 assertNever。

面试考点

  • never 和 void 有什么区别？ 结论是 void 是没有有意义的返回值，never 是不会正常返回。原因是 void 函数可以执行完，never 函数只能抛错、死循环或进入不可能分支。项目里错误抛出函数和穷尽检查辅助函数通常返回 never。
  • assertNever 为什么能发现漏处理分支？ 结论是它要求参数类型必须已经被收窄为 never。原因是所有联合成员都处理完后，默认分支不应再有实际成员；新增成员但漏写 case 时，传入的就不再是 never，编译器会报错。这个错误比线上遇到未知 action 更早。
  • reducer 里如何落地穷尽检查？ 结论是用字面量 type 建模 action，并在 switch 的 default 调用 assertNever(action)。原因是 reducer 往往随着业务迭代新增 action，最容易漏改某个分支。边界是运行时仍可能收到非法外部数据，需要在进入 reducer 前校验。
  • default 分支直接 return 原状态可以吗？ 结论是公共 reducer 有时为了兼容外部未知 action 会这么做，但应用内部强类型 action 不推荐。原因是它会让漏处理的新 action 静默通过。项目里可在边界层兼容未知输入，在内部状态机里保持穷尽检查。

type Action =
      | { type: "increment"; payload: number }
      | { type: "reset" };
      function assertNever(x: never): never {
      throw new Error(`Unhandled action: ${JSON.stringify(x)}`);
      }
      function reducer(state: number, action: Action): number {
      switch (action.type) {
      case "increment": return state + action.payload;
      case "reset": return 0;
      default: return assertNever(action);
      }
      }
