结合 localforage 与 StorageManager 估算剩余容量

请打开 DevTools Console 查看输出。

知识点要点

  • localforage.iterate() 可以遍历当前 store 的 key/value，适合估算“业务写入了多少数据”；它不等于浏览器底层真实占用，也不会告诉你整体配额。
  • navigator.storage.estimate() 返回当前 origin 的 usage 和 quota ，可用 quota - usage 粗略判断总体剩余空间；这个值包含 IndexedDB、Cache Storage 等同源数据。
  • 剩余容量不是固定值。浏览器会根据设备剩余磁盘、站点参与度、是否持久化存储、隐私模式等因素调整配额，并可能在存储压力下清理非持久化数据。
  • 容量估算要按值类型处理：字符串可用 TextEncoder 估算 UTF-8 字节数， Blob 读 size ， ArrayBuffer 读 byteLength ，对象再考虑 JSON 序列化和异常兜底。
  • 项目里通常把业务估算用于缓存详情页，把 StorageManager.estimate() 用于判断是否接近 origin 配额；不要在首屏频繁全量遍历大 store。
  • 运行环境： 剩余配额依赖 navigator.storage.estimate() ； file:// 下数值可能不准确。不能假设固定 50MB 上限。

面试考点

  • 如何估算 localforage 当前 store 还剩多少可用空间？ 结论是只能粗略估算：先用 iterate() 统计当前 store 的业务数据大小，再用 navigator.storage.estimate() 查看同源总体 quota - usage 。原因是 localforage 没有暴露浏览器真实配额 API；项目里应把结果标成估算值，并在写入失败时处理 QuotaExceededError 。
  • 为什么不能用固定 50MB 当作浏览器真实剩余容量？ 结论是固定 50MB 只能是业务阈值，不是规范保证。原因是 IndexedDB/localStorage 等配额由浏览器按设备空间和策略决定；离线缓存、图片编辑、PWA 等场景必须用真实写入结果和 StorageManager.estimate() 共同判断。
  • StorageManager.estimate() 的 usage/quota 和 localforage.iterate() 有什么区别？ 结论是前者看整个 origin，后者看当前 localforage store。原因是 estimate() 可能包含 IndexedDB、Cache Storage 等全部同源数据，而 iterate() 只能遍历 localforage 管理的数据；排查配额压力时两者互补。
  • 容量估算在大数据量或多标签页场景有哪些误差？ 结论是误差来自序列化格式、并发写入和浏览器内部存储实现。原因是 JSON 字符数不等于字节数，其他标签页可能同时写入，浏览器也可能压缩或额外记录元数据；项目里应展示近似值并在关键写入前后重新检查。
  • 项目里什么时候应该提示用户清理离线缓存？ 结论是当总体剩余配额很低、写入失败或业务缓存超过阈值时提示。原因是主动清理比等浏览器驱逐更可控；离线文档、音视频草稿等场景还要告诉用户哪些数据会被删除。
