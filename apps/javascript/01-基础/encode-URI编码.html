URI 编码 API（encodeURI / encodeURIComponent / decode）

请打开 DevTools Console 查看输出。

知识点要点

  • encodeURI ：编码完整 URL，保留 : / ? # 等 URI 结构字符，只编码中文等特殊字符。
  • encodeURIComponent ：编码 URL 组件（query 值、path 段），会编码 / ? & = 等。
  • 解码用 decodeURI / decodeURIComponent ；输入里存在非法百分号编码或不完整 UTF-8 序列时会抛 URIError 。
  • 拼接查询字符串时可写 ?q=${encodeURIComponent(keyword)} ，更推荐用 URLSearchParams 处理空格、重复 key 与自动转义。
  • escape / unescape 已废弃，编码规则不是 WHATWG URL 使用的 UTF-8 百分号编码，不要用于新代码。

面试考点

  • encodeURI vs encodeURIComponent？ 结论是完整地址用 encodeURI ，单个组件用 encodeURIComponent 。原因是前者保留 URI 分隔符，后者会把 :// 、 & 、 = 等也编码；边界是已经由 URL / URLSearchParams 管理的地址不要重复编码。
  • 查询参数为什么优先 URLSearchParams？ 结论是它比手写拼接更稳。它会按标准序列化 key/value，减少漏编码 & 、空格、中文等字符的风险；项目里仍要注意服务端对 + 与 %20 的解析约定。
  • decodeURIComponent 什么时候会抛错？ 结论是遇到格式非法的百分号转义会抛 URIError 。例如孤立 % 或非法 UTF-8 字节无法还原；边界是解码用户输入前通常要捕获异常，避免整页脚本中断。
  • escape 为何废弃？ 结论是它不是现代 URL 编码方案。它会产生 %uXXXX 这类历史格式，和 UTF-8 百分号编码、URL 标准不一致；新代码用 encodeURIComponent 或 URLSearchParams 。
