模板 ref / nextTick

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • 模板 ref 是逃逸口，常用于访问 DOM、调用子组件暴露的方法、聚焦输入框或测量元素尺寸；它不应该替代响应式状态成为主要数据来源。
  • Vue 会把同一轮状态变更合并到异步 DOM 更新队列中，状态赋值后立即读取 DOM，可能读到更新前的文本、尺寸或滚动位置。
  • nextTick() 返回一个 Promise，等待当前刷新队列把 DOM 更新完成；本页按钮先同步读取旧文本，再在 await nextTick() 后读取新文本。
  • 模板 ref 在组件挂载前、条件分支未渲染、列表项卸载后都可能是 null，访问时应根据生命周期和条件渲染做判断。
  • nextTick 只保证 Vue 本轮 DOM patch 完成，不保证图片加载、CSS 动画结束或网络请求完成；这些场景要监听对应事件或 Promise。

面试考点

  • 数据变了为什么同步读取 DOM 还是旧值？ 结论是 Vue 的 DOM 更新是异步批量执行的。原因是同一轮多次状态修改会被合并，减少重复渲染。项目里修改状态后要测量高度、滚到底部或读取文本，应等待 nextTick()。
  • nextTick 适合解决什么问题，不适合解决什么问题？ 先结论：它适合等待 Vue 完成本轮 DOM 更新，不适合等待外部异步资源。原因是它只和 Vue 更新队列有关。图片加载、接口返回、动画结束要分别用 load、请求 Promise 或动画事件处理。
  • 模板 ref 什么时候可能是 null？ 结论是挂载前、分支未渲染或节点被卸载后都可能为空。原因是 ref 的值由真实渲染结果赋值，不是声明后立即可用。项目里访问 DOM ref 通常放在 onMounted 之后，或在事件中做空值判断。
  • 什么时候应该用响应式状态而不是直接操作 DOM？ 结论是凡是能由数据描述的 UI 都应优先用响应式状态。原因是直接改 DOM 会绕过 Vue 的渲染模型，下一次 patch 可能覆盖手动修改。只有聚焦、测量、滚动、第三方 DOM 库接入这类命令式需求才使用模板 ref。
