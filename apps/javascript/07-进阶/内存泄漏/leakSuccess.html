通过手动置 null 释放闭包引用

请打开 DevTools Console 与 Memory 面板，等待 15 秒后对比闭包引用被置空前后的快照。

知识点要点

  • 闭包是函数和其词法环境的组合。只要返回的函数仍可达，它就可以继续访问创建时捕获的变量，因此本页 inner 会保留 outer 中的 bigData。
  • 把 inner 置为 null 的作用是断开外部变量到闭包函数的强引用；如果没有其他引用链指向该闭包或 bigData，这些对象会在后续 GC 中变为可回收。
  • 常见误区是把 null 当成立即释放内存的命令。JavaScript 标准不提供同步触发 GC 的能力，DevTools 中看到的内存下降受引擎调度、快照时机和对象晋升策略影响。
  • 真实项目里更推荐按生命周期清理：组件卸载时移除事件监听、取消定时器和异步订阅，清空大型缓存或回调引用；只有在长生命周期对象确实持有短生命周期资源时，手动置空才有意义。
  • 排查闭包泄漏时，用 Heap snapshot 查看闭包、数组和 retainers 的引用路径，比单纯观察 Console 日志更可靠；本页的 15 秒定时器只是制造一个可观察的释放时点。

面试考点

  • 闭包为什么可能让大对象长期留在内存中？ 结论是闭包函数可达时，它捕获的词法环境也会随之可达。原因是函数需要保留外层变量以便后续调用。边界是现代引擎可能优化未使用变量，但被闭包实际读取的大数组、DOM 节点或业务对象仍可能被保留。
  • 把闭包变量置为 null 到底释放了什么？ 结论是它释放的是“引用”，不是直接释放内存。原因是当保存闭包的变量不再指向函数，闭包和它捕获的对象可能从可达图中消失。项目里常见场景是取消回调、清理订阅、把不再使用的大对象从长生命周期实例上移除。
  • 为什么置 null 后内存不一定立刻下降？ 结论是对象只是变成可回收，真正回收由垃圾回收器择机执行。原因是 JavaScript 没有标准的同步 GC API，浏览器还可能为了性能延迟回收或保留堆空间复用。排查时应通过多次快照和 retainers 验证引用是否断开，而不是只盯总内存数字。
  • 组件或事件场景如何避免闭包泄漏？ 结论是在生命周期结束时移除所有能继续触达组件状态的外部引用。原因是事件监听、定时器、Promise 回调和全局缓存都可能持有闭包。实际项目中要在卸载钩子里 remove listener、clear timer、abort 请求或取消订阅。
