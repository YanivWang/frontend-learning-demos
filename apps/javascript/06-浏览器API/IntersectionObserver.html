IntersectionObserver 图片懒加载

请阅读本复习区，向下滚动下方演示区观察懒加载；图片 URL 依赖外网，加载失败会显示错误占位。DevTools Console 可查看触底日志。

知识点要点

  • IntersectionObserver 让浏览器异步计算目标元素与视口或指定 root 的交叉状态，回调拿到的是本轮状态发生变化的 IntersectionObserverEntry 集合。
  • root 默认为视口；rootMargin 可以扩大观察区域，本页设置底部 100px 提前加载；threshold 控制可见比例达到多少时触发。
  • 图片懒加载的推荐流程是先渲染占位内容，把真实地址放到 data-src，进入观察区域后再赋给 img.src，加载成功后移除占位并停止观察。
  • 常见坑是只处理成功路径：图片可能跨域失败、网络失败或被重置按钮重复初始化，所以需要 onload/onerror、unobserve 和旧 observer 的 disconnect。
  • 它适合可见性驱动的图片懒加载、曝光统计和无限滚动哨兵；如果需要逐帧跟随滚动位置，仍应使用节流后的 scroll 或 requestAnimationFrame。

面试考点

  • IntersectionObserver 为什么比 scroll 事件更适合懒加载？ 结论是它把可见性判断交给浏览器异步完成，通常比每次滚动手动计算元素位置更稳。原因是浏览器可以批量调度交叉状态变化，减少主线程上的频繁布局读取。项目里仍要为不支持的旧环境准备降级，或直接使用原生 loading="lazy" 满足简单图片场景。
  • rootMargin 和 threshold 在图片预加载里分别控制什么？ 结论是 rootMargin 控制观察区域边界，threshold 控制目标可见比例。原因是提前加载通常希望图片还没真正进入视口就触发，而不是等完全可见。移动端可根据网络、图片尺寸和首屏策略调大底部 rootMargin。
  • 懒加载图片进入视口后为什么要 unobserve？ 结论是加载完成后应停止观察这个元素。原因是同一元素滚入滚出会继续产生回调，可能重复赋值、重复计数或浪费 observer 资源。列表重置、虚拟滚动销毁和组件卸载时还要对 observer 调用 disconnect。
  • 图片加载失败或重复重置时要补哪些边界处理？ 结论是至少要处理 onerror、保持可访问的 alt、避免重复计数并清理旧 observer。原因是懒加载触发只代表开始请求，不代表图片真的加载成功。真实项目还会加入占位图、重试、错误样式和监控埋点。
