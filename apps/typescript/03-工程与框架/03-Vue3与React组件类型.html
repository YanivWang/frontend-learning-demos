Vue3 / React 组件类型

下方展示 Vue/React Props 类型片段；完整组件类型须在 Vue/React 工程中由 IDE/tsc 验证，本页用等价 JS 演示运行时形态。

知识点要点

  • Vue 3 的 <script setup lang="ts"> 常用 defineProps<T>() 和 defineEmits<T>() 描述组件输入输出，编译器会把宏调用转换为组件选项和类型信息。
  • React 函数组件可以直接给参数标注 Props，例如 function Button(props: ButtonProps)；React.FC<Props> 可用，但不是必须，团队应统一 children、default props 和泛型组件写法。
  • 组件 Props 类型应表达业务约束，例如 variant: "primary" | "danger"、可选角色、事件参数类型，而不是把所有字段都写成宽泛的 string 或 any。
  • TypeScript 对组件边界的价值在调用方：父组件传错 props、事件回调参数用错、表单值类型不匹配时，IDE 和 tsc 能提前报错。
  • 边界是框架类型主要保护编译期调用；服务端返回、用户输入、URL 参数和本地存储仍是不可信数据，需要运行时校验或解析后再传给组件。

面试考点

  • Vue 3 defineProps<T> 和运行时 props 声明怎么选？ 结论是纯 TS 项目常用类型参数声明，确实需要运行时校验或默认值时结合运行时声明、withDefaults 或校验逻辑。原因是类型参数主要服务编译期推导，运行时 props 选项还能参与校验和默认值。项目里要根据组件是否暴露给不可信调用方决定强度。
  • React 组件为什么常直接标注函数参数 Props？ 结论是直接标注更直观，也避免把 React.FC 当成必需模板。原因是 React 组件本质是函数，参数类型已经能约束 props；是否使用 React.FC 更多是团队风格问题。泛型组件、children 约束和默认值场景要按项目约定处理。
  • Props 类型如何表达业务约束？ 结论是尽量用字面量联合、可选字段、事件参数和精确对象类型表达真实边界。原因是宽泛的 string 或 any 只能说明“有这个字段”，不能防止传错业务值。示例里的 variant 只能是 primary 或 danger，比任意字符串更安全。
  • 组件类型不能替代什么？ 结论是不能替代运行时数据校验。原因是接口响应、表单输入和 URL 参数进入应用时仍是普通 JS 值，类型标注不会自动验证。项目里应在边界层解析数据，再把可信类型传入 Vue 或 React 组件。

// Vue3 <script setup lang="ts">
type UserCardProps = { id: number; name: string; role?: "admin" | "guest" };
const props = defineProps<UserCardProps>();

// React
type ButtonProps = {
  variant: "primary" | "danger";
  onClick: (event: React.MouseEvent<HTMLButtonElement>) => void;
};
function Button(props: ButtonProps) { return <button onClick={props.onClick} />; }
