async 是 generator + 自动执行器的语法糖

请打开 DevTools Console 查看输出。

知识点要点

  • 历史上可以用 generator 暂停在 yield Promise 处，再由执行器在 Promise fulfilled 后调用 next(result)，从而模拟顺序异步流程。
  • async/await 可以理解为语言内置的 Promise 执行器：自动等待、自动恢复、自动把 return 包装为 fulfilled，把未捕获异常包装为 rejected。
  • await 和 yield Promise 相似但不是完全等价；await 会按 Promise.resolve 规则处理普通值、Promise 和 thenable。
  • 示例中的 runGenerator 只处理 fulfilled 路径，真实执行器还要把 Promise reject 通过 gen.throw(error) 送回 generator，才能在 generator 内部 try...catch。
  • 多个 await 写在函数体内默认串行；如果任务互不依赖，应先启动多个 Promise，再用 Promise.all 并发等待。

面试考点

  • 为什么说 async/await 类似 generator + Promise 执行器？ 因为 generator 可以在 yield 暂停，执行器等 Promise 决议后再 next 推进；async 函数把这套驱动流程做进语言里。边界是 async 还规定了返回 Promise、异常转 rejection、await thenable 等完整语义。
  • 手写执行器和 async 函数在错误处理上有什么差异？ async 函数会把 await 的 rejection 当作异常抛进函数体，可用 try...catch 捕获。手写执行器如果只写 fulfilled 路径，rejection 会丢到外层 Promise，不能被 generator 内部捕获。
  • 多个 await 为什么默认串行，如何改成并发？ 写成 await a(); await b(); 时，b 会在 a 完成后才启动。互不依赖时应先执行 const pa = a(); const pb = b();，再 await Promise.all([pa, pb])。
  • 现代项目为什么很少手写 generator 异步执行器？ 因为 async/await 已经覆盖常见异步控制流，并且错误传播、返回值包装和调试体验更标准。generator 仍适合迭代器、惰性序列和需要显式暂停/恢复的高级场景。
