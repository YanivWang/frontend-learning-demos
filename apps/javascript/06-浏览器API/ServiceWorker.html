Service Worker 注册

请阅读本复习区，并在下方演示区点击注册/注销；需人工验证：须 HTTPS 或 localhost 静态服务，file:// 无法注册。

知识点要点

  • ServiceWorker 是独立于页面生命周期运行的 worker，可拦截其作用域内的网络请求、管理缓存、配合 Push/Notification 做 PWA 能力。
  • 注册必须发生在安全上下文中，通常是 HTTPS 或 localhost；file:// 直接打开不会成功，本页会把失败原因写到日志里。
  • navigator.serviceWorker.register(scriptURL, { scope }) 的默认 scope 受脚本 URL 所在目录限制；本页使用 ./sw.js 和 scope: "./"，避免错误声明站点根作用域。
  • 生命周期分为注册、安装、激活和拦截请求。离线缓存不是自动发生的，必须在 worker 脚本里选择缓存清单、更新策略、失败回退和旧缓存清理。
  • 常见坑是缓存版本没有失效、接口请求被错误缓存、旧 Service Worker 长期控制页面。生产项目应提供版本化缓存、灰度发布、skipWaiting/clients.claim 的取舍和一键注销方案。

面试考点

  • Service Worker 为什么必须运行在 HTTPS 或 localhost？ 结论是它能拦截网络请求，浏览器必须限制在可信来源中。原因是如果不要求安全上下文，恶意脚本可能长期驻留并篡改资源响应。开发时可用 localhost，本地直接双击 HTML 的 file:// 不能注册。
  • register 的 scope 由什么决定，为什么本页使用相对 scope？ 结论是 scope 受 Service Worker 脚本所在路径限制，默认不能控制脚本目录以上的页面。原因是这避免低层级目录脚本接管整个站点。当前 demo 的 sw.js 和 HTML 同目录，所以使用 scope: "./" 才是最小权限。
  • Service Worker 生命周期里 install、activate、fetch 分别做什么？ 结论是 install 适合预缓存，activate 适合清理旧缓存，fetch 负责按策略响应请求。原因是 Service Worker 不是普通页面脚本，它会经历安装、等待、激活和控制客户端。项目里要明确缓存优先、网络优先或 stale-while-revalidate 等策略。
  • 离线缓存有哪些常见坑和回滚策略？ 结论是最大风险是缓存陈旧、接口误缓存和新旧 worker 版本并存。原因是被 Service Worker 控制的页面可能继续使用旧逻辑直到刷新或重新激活。实践中要做缓存版本号、白名单路由、错误 fallback、监控和必要时的注销/清缓存入口。
