HTTP 强缓存 vs 协商缓存

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • 强缓存由响应头里的 Cache-Control、兼容旧协议的 Expires 等信息决定；命中且未过期时，浏览器直接使用本地副本，不需要发网络请求。
  • 协商缓存发生在缓存过期或需要重新验证时：浏览器携带 If-None-Match 或 If-Modified-Since 请求服务器，资源未变化则返回 304 Not Modified。
  • no-cache 不是“不缓存”，而是可以存储但使用前必须向服务器验证；no-store 才表示不应存储请求或响应，常用于隐私和敏感数据。
  • ETag 通常比 Last-Modified 更精确，因为后者时间精度有限且受文件时间影响；两者同时存在时，现代服务器通常优先用 ETag 验证。
  • 生产实践常见组合是 HTML 入口短缓存或每次协商，带内容 hash 的 JS/CSS/图片使用 Cache-Control: public, max-age=31536000, immutable，通过文件名变化完成更新。
  • 缓存策略也是风险控制：错误地长缓存 HTML 会让用户停留在旧版本，缓存个性化响应可能泄露隐私，过度禁用缓存会导致带宽和首屏性能下降。

面试考点

  • 强缓存和协商缓存的请求链路有什么区别？ 结论是强缓存命中时不发请求，协商缓存会发条件请求并可能拿到 304。原因是强缓存依据本地 freshness 判断是否可直接复用，协商缓存需要服务器确认资源是否变化。排查线上问题时可在 DevTools Network 里看是否出现 memory/disk cache 或 304。
  • Cache-Control: no-cache 和 no-store 有什么区别？ 结论是 no-cache 表示用前必须重新验证，no-store 表示不应存储。原因是二者控制的是“是否可直接复用”和“是否允许保存”两个层面。登录态接口、隐私页面更偏 no-store，普通 HTML 入口常用 no-cache 或短缓存配合协商。
  • ETag 和 Last-Modified 同时存在时如何理解优先级？ 结论是 ETag 通常更精确，现代实践一般优先按 If-None-Match 验证。原因是 Last-Modified 只能表达修改时间，可能受秒级精度、部署时间和文件系统影响。边界是分布式服务要保证 ETag 生成稳定，否则会导致缓存命中率下降。
  • 前端静态资源为什么常用文件 hash 加长期缓存？ 结论是内容 hash 让 URL 和内容绑定，可以安全设置长期强缓存。原因是内容变化会生成新文件名，旧缓存不会污染新版本。项目里通常让 HTML 引用最新 hash 资源，所以 HTML 本身不能被错误地长期强缓存。
  • 缓存配置错误会带来哪些线上风险？ 结论是既可能影响正确性，也可能影响安全和性能。原因是缓存会改变资源更新、用户隔离和网络请求路径。典型风险包括 HTML 长缓存导致发布不生效，用户专属数据被共享缓存保存，以及所有资源 no-store 导致首屏和带宽成本变差。

常见 Cache-Control

  • no-store：完全不缓存（敏感数据）
  • no-cache：可存但必须协商验证后才能用
  • max-age=31536000, immutable：带 hash 的静态资源长缓存
