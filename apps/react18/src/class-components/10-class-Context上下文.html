class 组件消费 Context

请打开 DevTools Console 观察 contextType 与 Consumer 读取结果；点击下方"切换主题"验证 Provider value 变化；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • contextType：class 组件声明 static contextType = ThemeContext 后，实例方法和 render 中都可以通过 this.context 读取最近 Provider 的 value；一个 class 只能通过该属性绑定一个 Context。
  • Consumer：ThemeContext.Consumer 使用 render props 接收 value，适合在 class 组件中读取多个 Context、只包裹局部 UI，或避免把整个 class 绑定到一个 contextType。
  • 适用范围：Context 适合主题、语言、当前登录用户、权限配置这类跨层共享且多数后代都需要的数据；父子近距离传值仍优先 props，复杂频繁更新的业务状态通常需要更明确的状态管理方案或组件局部化。
  • 性能边界：Provider 的 value 变化会通知所有消费该 Context 的组件重新渲染；对象字面量每次 render 都是新引用，真实项目里要稳定 value、拆分读写 Context，或把高频变化下沉到更小的 Provider。
  • 版本与迁移：React 18 函数组件推荐 useContext；class 维护场景继续使用 contextType/Consumer。旧版 legacy context API 已不适合作为新代码方案。

面试考点

  • class 组件如何读取 Context，contextType 和 Consumer 有什么区别？ 结论是 contextType 让 class 通过 this.context 读取一个 Context，Consumer 用 render props 在局部读取 Context。原因是 contextType 更简洁但一次只能绑定一个，Consumer 更啰嗦但可组合多个 Context。项目里单一主题用 contextType，多数据源或局部读取用 Consumer。
  • Context 适合解决什么问题，不适合解决什么问题？ 结论是它解决跨层传递共享值，不是所有状态管理的替代品。原因是 Context 会让消费者隐式依赖 Provider，且 value 更新会影响所有消费者。主题、语言、当前用户适合放 Context；表单输入、局部展开状态和高频动画状态不应默认上 Context。
  • Provider value 为什么可能导致大范围重新渲染？ 结论是消费者会响应最近 Provider 的 value 变化。原因是 React 按引用判断对象 value 是否变化，对象字面量或新函数会在每次 render 产生新引用。真实项目里要把 value 稳定下来、拆分 Context，或让 Provider 范围尽量小。
  • 多个 Context 或动态主题在 class 组件里怎么组织？ 结论是一个 class 只能绑定一个 contextType，多 Context 用 Consumer 嵌套或拆分组件。原因是 static contextType 只能声明一次；项目中主题 + 语言 + 用户可拆成多个 Provider 嵌套，需要读多个 Context 的子组件用 Consumer 或拆成更小的消费组件。
  • React 18 新代码如何从 class Context 迁移到函数组件？ 结论是函数组件使用 useContext 读取同一个 Context。原因是 Hooks 是 React 18 新代码的主流组织方式，能和 useMemo、useReducer 等组合管理 Provider value。迁移时注意函数组件接收 ref 仍需 forwardRef。
