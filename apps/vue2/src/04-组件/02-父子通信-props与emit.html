父子组件通讯（props 下传 + $emit 上抛）

在 #my-comp1-demo 内点「放大/缩小/计数」观察字号与 count 变化；对比挂载点外无效的 <my-comp1>；打开 Console 查看 $emit 事件与 count 日志。

知识点要点

  • 父子通信的主线是单向数据流：父组件用 v-bind:article 把文章对象传给子组件，子组件只读取 prop，不直接改父组件状态。
  • 子组件需要影响父组件时，通过 this.$emit('extend') 或 this.$emit('small', payload) 发出自定义事件；父组件用 v-on 监听并决定如何修改 fontSize。
  • 组件的 data 必须是函数。每次创建组件实例都会调用该函数返回新对象，否则多个 my-comp1 会共享同一个 count，点击一个会影响全部。
  • 事件负载应保持清晰且可预测：本页 small 传出对象负载，父组件拿到后先处理边界，再更新字号，避免让子组件直接控制父组件内部数据。
  • Vue 只编译挂载点内部的模板；挂在 body 里的 <my-comp1> 不属于 #my-comp1-demo，因此不会获得 props、指令或事件能力。

面试考点

  • Vue 2 父子组件通信为什么强调单向数据流？ 结论：父传子用 props，子通知父用 $emit，状态拥有者负责更新。原因是 Vue 2 组件提倡单向数据流，子组件不应直接改父级状态。边界是兄弟、跨层或全局状态不适合层层转发，应考虑 EventBus、provide/inject 或 Vuex。
  • 组件 data 为什么必须写成函数？ 结论：让每个组件实例拿到独立状态。原因是组件会被复用多次，data 若为同一对象引用则状态会串在一起。项目里可复用组件都要把内部状态写成返回对象的函数。
  • $emit 适合传什么，什么时候该改用状态管理或 provide/inject？ 结论：payload 应表达「发生了什么」和必要数据，父组件决定如何更新状态。原因是 $emit 适合父子直接通信。边界是跨多层、兄弟或全局共享状态应改用 provide/inject、EventBus 或 Vuex，避免 prop 钻取和事件链过长。
  • 为什么根实例外的组件标签不会被 Vue 接管？ 结论：Vue 只编译 el 对应 DOM 树内部的模板。原因是编译和响应式绑定都从根实例挂载点开始。项目里不要把需要 Vue 管理的组件散落在根节点外。
