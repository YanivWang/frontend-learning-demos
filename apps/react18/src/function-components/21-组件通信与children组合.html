props 回调、状态提升与 children 组合

请打开 DevTools Console 查看子→父回调输出；下方 #root 为 React 挂载演示区。

知识点要点

  • React 数据流默认是自上而下的：父组件持有 name ，通过 props 传给 NameInput 和 Preview ，两个子组件都从同一状态源读取数据。
  • 子组件不能直接修改父组件 state，而是调用父组件传入的回调；示例中 NameInput 触发 onChangeName ，由父组件执行 setName 。
  • 兄弟组件通信的第一选择通常是状态提升到最近共同父组件。这样数据来源清楚，避免两个子组件各维护一份可能不同步的副本。
  • children 适合表达“把某段 UI 塞进组件壳子里”的组合关系；示例的 Panel 负责边框和标题，具体输入与预览由调用方传入。
  • 当通信跨越很多层或很多页面时，再考虑 Context 或状态管理库；不要把所有局部交互提前做成全局状态，也不要用大量布尔 props 拼复杂布局。

面试考点

  • 子组件如何把输入变化通知父组件？ 结论是父组件把回调作为 props 传下去，子组件在事件里调用。原因是 React 遵循单向数据流，真正修改状态的函数仍在状态拥有者那里。项目里回调命名通常写成 onChangeXxx 或 onSubmit ，表达“发生了什么”。
  • 兄弟组件通信为什么通常先状态提升？ 结论是把共享 state 放到最近共同父组件最直接。原因是两个兄弟组件都能通过 props 读取同一份数据，更新路径也清晰。边界是层级过深或消费范围过大时，再引入 Context 或状态管理工具。
  • children 组合相比一堆布尔 props 有什么优势？ 结论是组合能让组件壳子和具体内容解耦。原因是调用方可以直接传入任意 React 节点，而不是让组件内部根据多个布尔值拼分支。项目里卡片、弹窗、布局容器、表单区域都适合这种模式。
  • props 回调、Context 和状态管理库怎么选？ 结论是局部父子和兄弟通信优先 props 与状态提升，跨多层但变化不频繁用 Context，复杂全局状态再考虑 store。原因是每升一级抽象都会增加维护成本。边界是性能敏感的 Context 要拆分 Provider 或结合 memo 化策略。
