class 组件通信、状态提升与 children 组合

请在输入框修改名字，观察 Preview 实时回显；点击"清空"验证父组件通过回调控制 state；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • React 组件通信遵循单向数据流：父组件通过 props 把数据传给子组件，子组件不能直接改 props，只能调用父组件传下来的回调来请求更新。
  • 兄弟组件需要共享状态时，优先把状态提升到最近的共同父组件。本 demo 中 NameInput 只发出新值，App 保存 name，Preview 读取同一份 state。
  • children 适合表达"把一段 UI 放进容器"的组合关系；具名 props（如 footer）适合多个插槽式区域。它们能让容器组件更通用，减少大量布尔配置 props。
  • 当共享范围只是局部父子树，状态提升比 Context 或全局状态更简单；当跨层很多、多个远端组件都需要同一份数据时，再考虑 Context、外部 store 或路由状态。
  • class 与函数组件在通信原则上没有差异，差别只在写法：class 用实例方法和 this.setState，函数组件用闭包、useState 和回调 props。

面试考点

  • React 单向数据流下父子和兄弟组件如何通信？ 结论是父传子用 props，子传父用回调，兄弟共享状态提升到共同父组件。原因是 React 更新应从状态所有者向下传播；项目里这样更容易追踪数据来源。
  • 状态提升什么时候比 Context 或全局状态更合适？ 结论是共享范围小、只服务当前页面局部交互时优先状态提升。原因是 Context 和全局 store 会扩大依赖范围；跨很多层或多个远端区域共享时才值得引入。
  • children 和具名 props 组合能解决什么问题？ 结论是它们让容器组件接收 UI 片段，而不是通过大量配置项猜测所有布局。原因是 React 元素也是值；项目中卡片、弹窗、布局容器常用 children、header、footer 这类组合方式。
  • 子组件为什么不应该直接修改 props？ 结论是 props 只读。原因是直接修改会破坏单向数据流，父组件下一次渲染还会覆盖子组件修改；正确做法是调用回调让父组件更新状态。
  • class 组件通信迁移到函数组件时哪些原则不变？ 结论是单向数据流、状态提升、props 只读、回调通知父组件这些原则不变。变化只是写法：class 的 this.setState 换成 useState，实例方法换成普通函数和闭包。
