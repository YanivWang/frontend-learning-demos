:is() 与 :where()

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见本复习区，脚本会输出 :is() / :where() 命中元素的 computed color。

知识点要点

  • :is() 和 :where() 都是函数式伪类，可以把多个候选选择器写进同一个参数列表，减少 .demo article h2, .demo article h3, .demo article strong 这类重复父级。
  • :is() 的特异性不是固定值，而是取参数列表中最高特异性的选择器。本页 :is(h2, h3, strong) 只包含类型选择器，所以权重较低；如果写成 :is(#hero, .title) ，整条规则会按 ID 权重参与级联。
  • :where() 的匹配能力类似 :is() ，但自身特异性恒为 0。它适合写 reset、base、主题默认值和设计系统低优先级规则，让业务选择器更容易覆盖。
  • 两者使用 forgiving selector list：参数列表里某个选择器无效时，浏览器可以忽略无效分支，而不是让整条选择器列表全部失效。实践中仍应避免把实验性选择器混入核心样式，兼容性要单独核对。
  • 选择器写短不代表优先级更低。项目里需要“合并写法但保留真实权重”时用 :is() ；需要“命中很多元素但不抢覆盖权”时用 :where() 。

面试考点

  • :is() 和 :where() 的共同点与根本区别是什么？ 结论是二者都能把多个候选选择器合并到一个函数式伪类里，根本区别是特异性计算。原因是 :is() 取参数中最高权重， :where() 永远是 0。项目里组件状态选择器常用 :is() 简化书写，低优先级默认样式常用 :where() 。
  • 为什么 :where() 很适合写 reset、base 或设计系统默认样式？ 结论是它能扩大命中范围但不抬高覆盖成本。原因是 :where(p, li) 的特异性为 0，后续普通元素、类选择器更容易覆盖它。边界是源码顺序仍会在同等特异性时生效，不能把它当成完全“无影响”的样式。
  • :is() 参数里出现 ID 选择器为什么可能造成覆盖困难？ 结论是参数中最高特异性会代表整个 :is() 。原因是 :is(#hero, .card) 会让匹配 .card 的分支也按 ID 级别参与级联。项目里不要为了少写几行把 ID 和普通类混在同一个 :is() ，否则后续覆盖可能不得不加重选择器。
  • :is()、:where()、普通选择器列表在可维护性上怎么取舍？ 结论是普通列表最直观， :is() 适合消除重复上下文， :where() 适合低权重批量命中。原因是三者的可读性和级联影响不同。团队样式规范里应明确 reset/base 优先用 :where() ，组件强匹配再用 :is() 。
