迭代器协议、for...of 与可迭代对象

请打开 DevTools Console 查看输出。

知识点要点

  • 可迭代协议 ：对象实现 [Symbol.iterator]() 并返回迭代器；迭代器的 next() 返回 { value, done } 。
  • for...of 反复调用 next() 直到 done: true ，遍历的是 值 ； for...in 遍历可枚举 键 ，数组上会包含索引字符串且可能沿原型链向上。
  • 内置可迭代：Array、String、Map、Set、TypedArray、arguments、DOM collections 等；普通 plain object 默认不可 for...of 。
  • function* 返回的 Generator 既是迭代器也是可迭代对象，可 for...of 或手动 .next() 。
  • Array.from(iterable) 、 [...iterable] 、 new Map(iterable) 等都消费 Iterable； Array.fromAsync 用于异步 Iterable（需较新运行时）。

面试考点

  • for...of 和 for...in 有什么区别？ 结论： for...of 遍历迭代值， for...in 遍历可枚举键。原因是前者依赖 Symbol.iterator ，后者走属性枚举。数组上 for...in 可能扫到原型方法，业务遍历数组元素优先 for...of 或数组方法。
  • 如何手写可迭代对象？ 结论：实现 [Symbol.iterator]() { return { next() { … } } } 。原因是 ES 内置语法和 API 都统一消费这一协议。项目里分页、游标、树遍历都可封装成 Iterable，便于与 for...of 、展开语法组合。
  • Generator 与迭代器协议的关系？ 结论：Generator 是迭代器协议的语法糖。原因是 function* 自动生成带 next 的对象，且自身也可被再次迭代。async generator 则对应异步迭代，配合 for await...of 。
  • Array.from 消费 Iterable 的边界？ 结论：只拉平一层 Iterable，不会递归；对 Map 会得到 [key, value] 条目数组。原因是 Array.from 按迭代协议逐个取值。无限 Iterable 会导致死循环，需要业务侧限制步数。
