Performance API

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • performance.now() 返回相对 performance.timeOrigin 的高精度时间戳，并且是单调递增时钟；它适合测量一段代码耗时，不适合展示真实日期时间。
  • performance.mark() 与 performance.measure() 属于 User Timing API：先给关键阶段打标记，再用 measure 生成可查询的 PerformanceMeasure 记录。
  • performance.getEntriesByType("navigation") 返回 Navigation Timing 2 的导航记录，可读取 DNS、连接、请求响应、DOM 解析、load 事件等页面加载阶段数据。
  • 旧的 performance.timing 已不推荐用于新代码；现代项目优先读取 PerformanceNavigationTiming，并用 PerformanceObserver 监听后续性能条目。
  • 常见坑是重复打同名 mark/measure 导致条目堆积，或者把本地调试指标当作线上真实体验；上报前要清理无用条目，并结合采样、网络状态、设备分层和隐私限制分析。

面试考点

  • performance.now() 为什么比 Date.now() 更适合测耗时？ 结论是 performance.now() 更适合做耗时测量。原因是它基于单调时钟，精度更高，并且不受系统时间被用户或 NTP 调整的影响。项目里用它算 duration，用 Date 或服务端时间处理业务时间。
  • mark / measure 在业务性能埋点里如何使用？ 结论是用 mark 标记开始和结束节点，再用 measure 生成可查询的耗时记录。原因是 Performance API 会把这些记录纳入统一的性能条目模型，便于调试和上报。真实项目要统一命名、清理旧条目，并避免给高频路径打过多标记。
  • navigation entry 与旧 performance.timing 有什么区别？ 结论是新代码应优先使用 PerformanceNavigationTiming。原因是它以 PerformanceEntry 的形式暴露导航指标，和资源、mark、measure 等条目模型一致，而旧 performance.timing 已进入历史兼容语境。兼容旧浏览器时才考虑兜底读取旧字段。
  • Performance API 指标上报有哪些隐私和兼容边界？ 结论是指标会受浏览器精度限制、跨源资源 Timing-Allow-Origin、缓存和页面生命周期影响。原因是高精度时间可能被滥用于侧信道，跨源资源也不会默认暴露完整细节。线上监控要做采样、分桶和异常过滤，不能只看单次本地结果。
