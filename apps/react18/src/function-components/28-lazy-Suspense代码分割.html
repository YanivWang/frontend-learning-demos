React.lazy 与 Suspense

请打开 DevTools Console 查看 lazy 加载与错误边界日志；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • React.lazy 用来声明懒加载组件，加载函数必须返回 Promise，并解析为包含 default 组件的模块对象。
  • Suspense 负责在懒加载组件尚未就绪时展示 fallback；它处理“等待中”，不负责处理“加载失败”。
  • 真实项目里常写成 lazy(() => import('./Page.jsx'))，配合路由级分包、首屏之外的大模块、后台低频页面使用。
  • 加载失败、网络断开、chunk 404、部署后旧页面引用旧 chunk 都会让 lazy Promise reject；这类错误应交给 Error Boundary 展示失败 UI 和重试入口。
  • React 18 中客户端代码分割的 Suspense 已稳定；数据获取 Suspense 在不同框架里有不同约束，不能简单把 React 19 的 use() 能力混进 React 18 demo。

面试考点

  • React.lazy 和 Suspense 各自负责什么？ 结论是 lazy 定义如何加载组件，Suspense 定义等待期间显示什么。原因是懒加载组件渲染时会“挂起”，React 用最近的 Suspense fallback 填补 UI。项目里常把页面级路由组件放进 Suspense 边界。
  • lazy 组件加载失败为什么需要错误边界？ 结论是失败会作为错误抛出，Suspense fallback 只处理 pending。原因是 reject 代表加载已失败，不再是等待状态。本 demo 增加“模拟加载失败”，由 Error Boundary 给出错误提示和重试边界。
  • 路由级代码分割如何设计体验？ 结论是按页面或大模块分割，并提供稳定 loading、失败提示和重试。原因是 chunk 下载受网络、缓存和部署版本影响。项目里还要考虑预加载关键路由、部署后 chunk 失效兜底和监控上报。
  • Suspense for data fetching 与 React 18 lazy 有什么边界？ React 18 客户端稳定场景是代码分割等待组件加载。数据获取 Suspense 通常由框架或库约定支持；React 19 的 use() 属于新版本能力，讲 React 18 时应单独说明版本差异。
