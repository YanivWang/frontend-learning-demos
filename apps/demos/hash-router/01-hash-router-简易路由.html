Hash 路由简易 SPA 与 fallback

这个 demo 用 location.hash 和 hashchange 实现一个零依赖路由，重点观察首屏渲染、动态段匹配和未知路径 fallback。

知识点要点

  • 核心机制：# 后面的 fragment 改变会触发 hashchange，但页面首次加载不会自动触发一次，所以初始化时必须主动调用 render()。
  • 静态托管优势：fragment 不会作为请求路径发送给服务器，刷新 #/user/42 仍然请求当前 HTML，因此 hash 路由通常不需要额外后端 fallback。
  • 动态段处理：示例用正则匹配 /user/:id，并只接受数字 ID；路径经 decodeURIComponent 解码后再匹配，真实项目还需要校验和权限判断。
  • 安全渲染：路由参数来自地址栏，属于用户可控输入；示例用 DOM API 和 textContent 写入页面，不把参数拼进 innerHTML。
  • 模式取舍：hash 模式部署简单但 URL 带 #；History API URL 更自然，但刷新深层路径需要服务器把请求回退到入口 HTML。

面试考点

  • Hash 路由为什么刷新时通常不需要后端 fallback？ 结论是浏览器请求服务器时不会把 # 后面的 fragment 当成路径发送。原因是 hash 主要在客户端解析，所以静态站点刷新深层 hash 路由仍拿到同一个 HTML。边界是 SEO、分享链接美观度和服务端渲染场景通常更偏向 History 模式。
  • 首屏渲染为什么不能只依赖 hashchange？ 因为 hashchange 只在 fragment 发生变化时触发，首次加载当前 URL 不一定有变化。实践中会在绑定监听后主动执行一次渲染，保证直接打开 #/about 也能显示正确页面。
  • Hash 路由和 History API 路由怎么选？ 部署约束强、静态托管为主时 hash 更省事；希望 URL 无 #、配合服务端路由或 SSR 时用 History API。History 模式需要服务器把未知前端路由 fallback 到入口页，否则刷新会 404。
  • 动态路由参数为什么要匹配、解码和安全渲染？ 地址栏是用户可控输入，参数必须先确认形状再进入业务逻辑。展示时用 textContent，避免把恶意字符串拼成 HTML；项目里还要做权限校验和不存在资源的提示。
  • 未知路由应该如何处理？ 结论是显式提供 404 fallback，而不是静默显示空白。原因是用户可能手输、收藏过期链接或从旧版本跳转；项目里还可以记录埋点并提供返回首页入口。
