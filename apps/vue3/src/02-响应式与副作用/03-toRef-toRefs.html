toRef / toRefs

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • toRef(state, "count") 会创建一个与 state.count 双向联动的 ref；修改 countRef 会写回源对象，修改源对象也会更新 countRef。
  • toRefs(state) 会把响应式对象当前可枚举属性批量转换成 ref，适合在返回对象或解构时保留每个属性的响应性。
  • ref(state.count) 只是用当前值创建新的 ref，不会和 state.count 保持连接；这是它和 toRef 最容易混淆的地方。
  • toRefs 只处理调用时已经存在的属性，后续新增属性不会自动出现在返回结果里；需要为可能不存在的单个属性建立连接时，用 toRef(state, "maybeKey")。
  • 本页还演示了嵌套对象属性：name 来自当前 state.user 的 name。如果项目会整体替换 state.user，需要重新设计状态结构或改用计算属性，避免绑定到旧对象。

面试考点

  • toRef 和 ref(state.foo) 有什么区别？ 结论是 toRef 保持与源属性联动，ref(state.foo) 只复制当前值。原因是 toRef 的 getter/setter 仍读写源对象属性。项目里要把响应式对象的单个字段传给函数或组件时，应优先考虑 toRef。
  • toRefs 适合解决什么解构问题？ 先结论：它适合让调用方解构响应式对象后仍保留响应性。原因是直接解构会绕过代理 getter，而 toRefs 返回的是一组 ref。composable 返回多个状态字段时，常用 return { ...toRefs(state), action }。
  • toRefs 为什么不适合处理未来新增属性？ 结论是 toRefs 只为调用时已有的可枚举属性创建 ref。原因是它不是持续监听对象结构变化的工具。项目中如果某个 key 可能稍后才出现，应提前初始化字段，或对该字段单独使用 toRef。
  • props 解构时为什么常提到 toRef / toRefs？ 结论是直接解构 props 可能丢失响应性，而 toRef/toRefs 可以保留与 props 字段的连接。原因是 props 本身是响应式只读对象。边界是 props 不应在子组件里直接写回，派生值用 computed，修改请求通过 emit 交给父组件。
