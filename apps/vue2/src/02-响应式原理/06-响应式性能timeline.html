用 Performance 时间线观察大数组对响应式性能的影响

先打开 DevTools → Performance 并开始录制，再依次点击「慢：循环 push」与「快：整体赋值」对比 arr.length 与日志耗时；也可点错误 onclick 按钮观察报错；复习要点与面试答法见本区块。

知识点要点

  • Vue 模板中的 @click 会在组件实例上下文中解析 methods；原生 onclick="handleClickFn()" 在全局作用域查找函数，访问不到当前 Vue 实例的 methods。
  • 本页按钮对比 1 万次逐项 push 与一次整体赋值。逐项写入会多次经过 Vue2 包装的数组变更方法，制造更多依赖通知和调度压力。
  • 批量数据处理时，实践中通常先在普通数组 buffer 中完成计算、过滤、排序，再一次性赋给响应式状态，减少不必要的中间更新。
  • Chrome DevTools Performance 面板可观察 scripting 耗时、long task、渲染和布局成本。示例中的 performance.now() 只测局部 JS 耗时，不能替代完整 timeline 分析。
  • 性能优化边界：大列表不应只靠减少 push 次数，还要考虑分页、虚拟列表、稳定 key 、避免深层响应式大对象、减少 watcher 数量和不必要的 DOM 渲染。

面试考点

  • 为什么大数组逐项 push 可能比整体赋值慢？ 结论是逐项 push 会让响应式系统经历更多次数组变更拦截和更新调度。原因是 Vue2 对数组变更方法做了包装，每次变更都可能通知依赖。项目里导入、筛选或聚合大量数据时，先构造普通数组再整体替换通常更稳定。
  • 原生 onclick 和 Vue 的 @click 有什么作用域区别？ 结论是 @click 在 Vue 模板编译后的组件上下文中调用 methods，原生 onclick 是浏览器内联事件，默认找全局函数。原因是二者绑定和作用域模型不同。Vue 项目里应使用 v-on / @ ，并把错误路径作为调试信号。
  • Performance 面板应该观察哪些信号？ 结论是看长任务、scripting 耗时、渲染和布局是否被数据更新放大。原因是 performance.now() 只能给一段代码打点，无法展示浏览器主线程完整工作。项目里要结合录制结果、组件更新范围和 DOM 数量判断瓶颈。
  • Vue2 大列表性能优化有哪些边界方案？ 结论是减少响应式写入次数、减少渲染节点、减少 watcher 和稳定 diff。原因是性能压力可能来自数据观察、虚拟 DOM diff、真实 DOM 更新或布局。项目边界上，大量可见行优先虚拟列表，大量静态数据可考虑冻结或不放入深层响应式对象。
