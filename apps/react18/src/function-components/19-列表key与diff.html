列表 key 与 diff

请打开 DevTools Console 查看插入日志；下方 #root 为 React 挂载演示区，勾选 checkbox 后点击「从头部插入」对比两组列表。

知识点要点

  • key 是 React 在同级兄弟元素中识别身份的线索，帮助 diff 判断“哪一项还是原来的组件”。它只在当前数组层级内需要唯一。
  • key 不会作为普通 props 传给组件；如果子组件也需要业务 id，要显式传入 id={todo.id} 或类似字段。
  • 稳定业务 id 能让组件状态跟着数据项移动。示例中正确列表用 todo.id，从头插入新任务后，已勾选项仍对应原任务。
  • index key 只适合完全静态的列表；一旦从头插入、删除或排序，React 会按位置复用组件，导致输入框内容、勾选状态等局部状态错位。
  • 改变 key 会让 React 认为这是新组件，从而销毁旧状态并重新创建；项目里可用它重置表单或详情页局部状态，但不要用随机 key 造成每次渲染都重建。

面试考点

  • key 在 diff 中解决什么问题？ 结论是帮助 React 在同级列表中判断元素身份。原因是没有稳定身份时，React 只能按位置复用组件。项目里稳定 key 能减少错误复用，并让局部状态与真实数据项保持一致。
  • 为什么列表增删排序时不推荐 index key？ 结论是 index 表示位置，不表示数据身份。原因是从头插入后，同一个位置变成了另一条数据，React 却会复用旧组件状态。示例中反例列表的 checkbox 状态就可能跟错任务。
  • key 为什么不会出现在 props 里？ 结论是 key 是 React 自己使用的特殊字段。原因是它参与元素协调，不属于传给组件的业务数据。项目里子组件需要 id 时必须额外传 id 或完整对象。
  • 什么时候可以改变 key 来重置状态？ 结论是当你希望 React 把组件视为全新实例时可以这么做。原因是 key 变了，旧状态会被丢弃。场景包括切换用户后重置表单、切换文档后重置编辑器；边界是不要用 Date.now() 或随机数作为常规 key。
