useEffect 与 class 生命周期对照

请打开 DevTools Console 查看 mount/update/unmount 日志；下方 #root 为 React 挂载/交互演示区，可切换 userId 与挂载/卸载。

知识点要点

  • useEffect(fn, []) 常被类比为 componentDidMount，因为它在组件提交到 DOM 后运行一次；但它表达的是“这个副作用不依赖响应式值”，不是生命周期方法的逐字替代。
  • useEffect(fn, [deps]) 会在首次 mount 后运行，也会在依赖变化后的提交中运行。若要模拟“只在更新时执行”，本 demo 用 useRef 跳过首次 effect。
  • useEffect 返回的 cleanup 会在组件卸载前，以及下一次执行同一个 effect 之前运行；它适合清理订阅、定时器、事件监听和过期异步任务。
  • Hooks 推荐按副作用语义拆分 effect，而不是把所有逻辑塞进一个“生命周期对照表”。例如订阅、日志、请求、DOM 集成可以放在不同 effect 中，各自声明依赖。
  • React 18 StrictMode 在开发环境会额外执行一次 setup + cleanup，用来暴露副作用清理不完整的问题；生产环境不会因为 StrictMode 做这次额外检查。

面试考点

  • useEffect 能否和 class 生命周期一一对应？ 结论是不能一一对应，只能做近似类比。原因是 effect 描述“同步到外部系统”的副作用，而 class 生命周期按组件阶段组织；项目迁移时应按副作用拆分，而不是机械翻译。
  • useEffect(fn, []) 和 componentDidMount 有何差异？ 结论是二者都在首次提交后运行，但 effect 仍有 cleanup 语义和 StrictMode 开发检查。原因是 Hooks 关注依赖和清理；边界是如果 effect 读取了 props/state，就不能为了“只跑一次”省略依赖。
  • useEffect(fn, [deps]) 为什么不是纯粹的 componentDidUpdate？ 结论是带依赖的 effect 首次 mount 后也会执行。原因是 React 需要先运行 setup；如果只想处理更新，可以像本 demo 一样用 ref 跳过首次，或重新设计副作用语义。
  • cleanup 和卸载生命周期如何对应？ 结论是 cleanup 可以覆盖卸载清理，也会在依赖变化导致 effect 重跑前执行。原因是 React 要先清理旧订阅再建立新订阅；网络请求、定时器、事件监听都要在 cleanup 中处理边界。
  • StrictMode 下 effect 为什么会重复执行？ 结论是 React 18 开发环境会额外执行一次 setup + cleanup 来检查副作用是否可恢复。原因是这能提前暴露遗漏清理的问题；生产环境不会做这次额外执行，不能把它当线上行为。
