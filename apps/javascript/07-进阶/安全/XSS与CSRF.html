知识点要点

  • XSS 的本质是把不可信数据放进了 HTML、属性、URL、JavaScript 或 CSS 等可解释上下文，导致攻击者控制的内容被浏览器当代码或危险资源执行；防护核心是按输出上下文编码，并尽量使用安全 DOM API。
  • 本页演示 DOM XSS：innerHTML 会把输入当 HTML 解析，<img src=x onerror=alert(1)> 这类 payload 可能触发脚本；textContent / createTextNode 会把输入当纯文本，适合昵称、评论等纯文本场景。
  • Stored、Reflected、DOM XSS 的区别在于恶意数据来源和触发链路不同：存储型来自服务端持久化数据，反射型来自请求参数回显，DOM 型由前端脚本从 URL、存储或输入读取后写入危险 sink。
  • CSRF 的本质是攻击站点诱导浏览器向受害站点发送带 Cookie 的状态变更请求。服务端应使用 CSRF Token、校验 Origin / Referer 或 Fetch Metadata，并把 Cookie 设置为合适的 SameSite。
  • CSP 可以限制脚本来源，HttpOnly 可以降低 Cookie 被脚本读取的风险，SameSite 可以减少跨站自动带 Cookie；这些都是纵深防御，不能替代输出编码、输入校验和服务端权限校验。

面试考点

  • XSS 和 CSRF 的攻击目标有什么区别？ 结论是 XSS 让攻击者代码在受害页面上下文执行，CSRF 让受害者浏览器替攻击者发起已认证请求。原因是 XSS 利用输出和 DOM sink，CSRF 利用浏览器自动携带 Cookie 等凭据。边界是二者可以组合：一旦有 XSS，攻击者往往能读取页面 token 或直接发起站内请求，CSRF 防线会被削弱。
  • 为什么不可信输入不能直接写入 innerHTML？ 结论是 innerHTML 会触发 HTML 解析，攻击者可构造标签、事件属性或危险 URL 进入可执行上下文。原因是浏览器无法知道这段字符串是“用户文本”还是“可信模板”。项目中纯文本使用 textContent，富文本必须使用成熟 sanitizer，并按输出上下文处理属性和 URL。
  • SameSite、CSRF Token、Origin/Referer 分别解决什么问题？ 结论是 SameSite 减少跨站请求自动带 Cookie，CSRF Token 证明请求来自真实页面流程，Origin/Referer 验证请求来源。原因是 CSRF 发生在服务端状态变更接口，最终必须由服务端拒绝伪造请求。边界是 SameSite 受兼容性和业务跨站需求影响，Origin/Referer 可能缺失，Token 要防泄漏和重放。
  • CSP 和 HttpOnly 为什么不能单独解决 XSS？ 结论是它们是纵深防御，不是根因修复。原因是 CSP 配置可能被放宽或绕过，HttpOnly 只能保护 Cookie 不被 JS 读取，不能阻止攻击脚本操作页面或调用接口。项目里仍要优先消除危险 sink、做上下文编码，并在高风险页面启用严格 CSP 和 Trusted Types。

CSRF 防护要点（服务端为主）

  • Cookie 设置 SameSite=Lax/Strict，敏感操作用 SameSite=Strict
  • 表单/接口携带不可预测的 CSRF Token（服务端校验）
  • 校验 Origin / Referer 是否来自可信域
// 响应头示例
Content-Security-Policy: default-src 'self'; script-src 'self'
Set-Cookie: session=...; HttpOnly; Secure; SameSite=Lax
