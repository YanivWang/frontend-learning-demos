composable 组合式函数

在浏览器中操作下方交互演示；复习要点与面试答法见本复习区。

知识点要点

  • composable 是封装「有状态逻辑」的普通函数，通常以 useXxx 命名。它可以返回 ref、computed、方法和派生状态，让组件专注渲染。本页 useMousePosition() 即官方 Mouse Tracker 思路的简化版。
  • 示例同时组合了 ref、computed、watch、onMounted 和 onUnmounted，展示了组合式函数能封装完整行为而不依赖 mixin；生命周期注册仍须在调用 composable 的组件 setup 同步阶段执行，见 01-生命周期钩子。
  • 本页左右两个面板各调用一次 useMousePosition()，移动鼠标可见 x/y 独立更新，证明每次调用默认创建独立状态。若要共享，需把 ref 提升到模块作用域或使用 Pinia / provide-inject，见 04-provide-inject。
  • 涉及 DOM 事件时 composable 必须设计清理路径。本页在 onMounted 监听 mousemove，onUnmounted 移除；点「卸载左/右面板」后对应坐标冻结，控制台打印 [composable] 已移除 mousemove 监听。
  • isNearOrigin 由 computed 派生（x < 200 且 y < 200），watch 在布尔值翻转时追加日志；鼠标移入/移出左上角区域时 pre 区会出现新行。副作用清理进阶见 02-onWatcherCleanup。
  • 常见坑是返回 reactive 对象后被调用方解构导致响应丢失，或在 composable 中隐藏过多业务副作用。实践中通常返回 refs 或用 toRefs 保持可组合性，见 02-ref-reactive与解构。

面试考点

  • composable 和 mixin 相比优势是什么？ 结论是 composable 的来源、依赖和返回值更明确。原因是它是普通函数调用，可以通过参数和返回值表达契约，不会像 mixin 那样隐式合并选项并产生命名冲突。项目里可测试性和类型推导也更好；本页双面板独立状态即直观对比。
  • composable 的状态默认共享吗？ 结论是不共享，函数内部创建的状态每次调用都是新实例。本页左/右面板 x/y 可完全不同。若要共享，应显式把状态放在模块作用域、store 或 provide 中，避免调用方误判生命周期。
  • 为什么 DOM 事件必须清理？ 结论是因为浏览器事件监听不受 Vue 响应式系统自动管理。监听器持有回调引用，组件卸载后仍可能触发并更新已销毁组件的状态。本页卸载面板后坐标不再变化即证明清理生效。
  • 什么时候升级为 store？ 结论是当状态跨多组件共享、需要 DevTools、持久化或跨路由时，优先 Pinia。composable 更适合局部可复用逻辑；过多全局状态藏在 composable 中会让数据流不清晰。
  • composable 里注册 onUnmounted 与组件自己注册有何区别？ 结论是无本质区别，都会绑定到调用 composable 的当前组件实例。原因是生命周期钩子按「当前 setup 上下文」注册。composable 只是把注册逻辑封装成可复用函数，卸载时机仍跟随使用该 composable 的组件。
