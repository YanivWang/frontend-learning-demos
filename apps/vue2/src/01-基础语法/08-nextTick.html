nextTick 与异步 DOM 更新队列

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • Vue 2 修改响应式数据后不会立刻同步改 DOM，而是把 watcher 放入异步更新队列，并在本轮调用栈结束后批量 flush。这样可以把同一轮里的多次状态变化合并，减少重复渲染。
  • 本页点击“修改 message”后，同步读取 $refs.text.textContent 仍可能是旧文本；进入 this.$nextTick 回调后，Vue 已完成本轮 DOM 更新，才能读到新文本。
  • 连续执行 this.count++ 三次会让数据变成新值，但 DOM 更新会被排队合并。不要把“数据已经变了”和“页面已经渲染完”混为一谈。
  • this.$nextTick(callback) 适合组件实例内部使用，Vue.nextTick(callback) 是全局 API。它们等待的是 Vue 的 DOM 更新队列，不保证网络请求、CSS 动画、图片解码或第三方组件内部异步也完成。
  • Vue 2 已 EOL。维护旧项目时仍常用 nextTick 处理聚焦、测量尺寸、滚动到底部等 DOM 读取；新项目应优先按 Vue 3 当前文档选型。

面试考点

  • Vue 2 为什么更新数据后不能马上读到最新 DOM？ 结论是数据更新和 DOM patch 不是同步完成的。原因是 Vue 会把 watcher 放入异步队列，等待本轮调用栈结束后批量更新 DOM。项目里更新数据后立即测量高度、聚焦输入框或滚动容器，都应该放进 nextTick。
  • nextTick 适合解决哪些实际问题？ 结论是它适合“依赖最新 DOM”的动作。原因是回调执行时 Vue 已完成本轮响应式更新对应的 DOM patch。项目里常见场景是列表新增后滚动到底部、弹窗出现后聚焦输入框、条件渲染后读取元素尺寸。
  • nextTick 和 setTimeout 有什么区别？ 结论是 nextTick 等 Vue 自己的更新队列，setTimeout 只是把回调放到宏任务。原因是 Vue 会优先选择合适的异步策略来 flush 更新队列，语义更明确。边界是 nextTick 不负责等待网络、图片加载或 CSS 动画结束。
  • 连续修改同一个响应式数据时 DOM 为什么不会同步刷新三次？ 结论是 Vue 会去重并批量更新 watcher。原因是同一轮事件循环内多次状态变化可以合并成一次渲染，减少性能浪费。项目里不要依赖中间 DOM 状态，应以最终数据状态和 nextTick 后的 DOM 为准。
