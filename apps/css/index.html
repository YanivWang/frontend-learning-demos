知识点要点

  • 本页是 CSS 模块的静态导航入口，不负责讲解某一个 CSS API。它把 demo 按 01-基础、02-布局、03-动画、04-视觉效果、05-响应式、06-性能、07-现代特性分组，方便在不启动文档站时快速打开页面。
  • 导航项使用普通 <a href> 相对路径，而不是依赖 JavaScript 路由。这样页面在本地静态服务、VitePress 外部入口以及部分 file:// 兜底访问场景下都能保持基本可用。
  • 页面结构用标题和列表表达信息层级，链接文本直接写出 demo 主题。相比“点击这里”这类泛化文案，可读性、键盘导航和屏幕阅读器体验都更好。
  • manifest.json、VitePress 侧边栏和搜索索引才是仓库的权威索引；本页是手写的轻量入口。因此新增 CSS demo 后，除了同步生成索引，也要检查这里是否需要补充人工分组链接。
  • 常见坑是目录页链接过期、只覆盖部分 demo，或把导航脚本做成访问前提。项目中应让核心链接保持静态可访问，脚本只做增强，例如本页页脚在 file:// 下修正总目录路径。

面试考点

  • 静态 demo 目录为什么要用普通链接而不是只依赖脚本路由？ 结论是普通链接是最稳的基础能力，脚本路由只能作为增强。原因是 <a href> 天然支持复制链接、键盘访问、浏览器历史和无脚本环境。项目里文档站构建失败或 JS 加载失败时，静态目录仍能作为兜底入口。
  • 相对路径入口页在 file:// 和本地服务下分别要注意什么？ 结论是 demo 内部链接应优先使用相对路径，并避免依赖服务器重写。原因是 file:// 下没有 SPA fallback，本地服务下根路径和部署子路径也可能不同。项目里跨目录跳转要核对层级，页脚这类增强脚本也要保留无脚本时的可用链接。
  • 目录页和 manifest / 文档站索引的职责应该怎么划分？ 结论是 manifest 和生成索引负责权威完整性，目录页负责人工可读的学习分组。原因是完整索引适合脚本维护，手写目录适合表达学习顺序和主题聚合。项目里新增 demo 后先同步生成索引，再人工确认目录页是否需要补链接或调整分组。
  • 导航页如何兼顾可访问性、可维护性和构建失败时的兜底访问？ 结论是用语义化标题、列表和明确链接文本构建基础内容，再用少量脚本做渐进增强。原因是可访问性和兜底访问依赖 HTML 本身，而不是运行时框架。项目里目录页应避免空链接、模糊文案和只靠点击事件跳转。
