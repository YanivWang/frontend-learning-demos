柯里化与参数收集

知识点要点

  • 柯里化是把 fn(a, b, c) 变成 fn(a)(b)(c) 或分批传参的形式。示例中的 curried 每次返回新函数，直到收集的参数数达到 sum3.length 。
  • 闭包保存已传入的 args ，下一次调用再用展开语法拼接 rest 。这说明柯里化不是语法特性，而是函数、闭包和参数列表组合出的模式。
  • Array.prototype.slice.call(arguments, 1) 是把类数组 arguments 转成真数组并跳过第一个参数的旧写法；现代函数更推荐 function fn(first, ...rest) 。
  • 柯里化常用于固定上下文参数、构造可复用校验器、组合中间件或函数式工具。普通业务代码如果只是少传几个参数，用默认参数、对象参数或简单包装函数通常更直观。
  • 通用 curry 依赖 fn.length 判断参数是否收齐；默认参数、剩余参数和可选参数会影响 length ，实际库通常需要更完整的占位符和 arity 设计。

面试考点

  • 柯里化和偏函数应用有什么区别？ 先结论：柯里化强调把多参函数拆成连续的一元或分批调用，偏函数应用强调先固定一部分参数。原因是二者都依赖闭包保存参数，但抽象目标不同。项目中固定接口前缀、日志上下文这类场景更像偏函数，构建函数组合流水线更像柯里化。
  • 通用 curry 为什么依赖 fn.length ，边界是什么？ 先结论：示例用 fn.length 表示原函数声明的形参数量，参数够了就执行。原因是函数对象的 length 记录第一个带默认值参数前的形参数量。边界是默认参数、rest 参数和运行时可选参数会让这个判断不可靠。
  • arguments 为什么要借用 Array.prototype.slice ？ 先结论：因为 arguments 是类数组，不是真数组。原因是它有索引和 length ，但没有数组方法； slice.call 可按数组切片算法读取它。现代项目里优先用 rest 参数，类型更清楚，也避免箭头函数没有自己的 arguments 。
  • 项目里什么时候适合使用柯里化？ 先结论：当函数需要复用固定参数或参与组合时适合。原因是闭包能把配置前置，调用点只关心剩余业务参数。边界是过度柯里化会增加阅读成本，团队不熟悉函数式风格时应优先选择对象参数或明确命名的包装函数。
