JSX 表达式与 React 19 自动批处理

请打开 DevTools Console 查看 render 次数与批处理日志；下方 #root 中可点击按钮对比 setTimeout 批处理、flushSync 与函数式 setState，并观察 && 短路渲染数字 0 的陷阱。

知识点要点

  • JSX 花括号内放的是 JavaScript 表达式，例如变量、三元表达式、数组 map 或函数调用；if、for 这类语句要提前写在 JSX 外，因为 JSX 会被编译为函数调用的参数列表。
  • && 条件渲染要小心数字 0：React 会渲染数字和字符串，只有 false、null、undefined 等不会生成节点；安全写法是 items.length > 0 && ... 或三元表达式。
  • 自动批处理从 React 18 的 createRoot 开始成为默认行为，React 19 继续沿用；事件处理器、setTimeout、Promise 回调里的多次 state 更新通常合并成一次渲染。
  • 批处理会放大闭包旧值问题：同一事件或异步回调里连续写 setCount(count + 1) 可能只加 1；应使用函数式更新 setCount(c => c + 1)，让 React 基于最新 pending state 排队。
  • flushSync 可以强制 React 立即提交更新，适合少量需要同步读取 DOM 的集成场景；滥用会破坏批处理收益并影响性能。

面试考点

  • JSX 花括号里为什么只能放表达式不能放语句？ 结论是 JSX 花括号需要表达式，if、for 是语句。原因是 JSX 最终会被 Babel 编译为函数调用参数，参数位置只能放表达式。项目里复杂条件应先在组件体内算出变量，再嵌入 JSX。
  • && 条件渲染为什么可能把 0 渲染出来？ 结论是左侧为 0 时会把 0 渲染出来。原因是 0 && expr 的求值结果就是数字 0，React 会渲染数字。边界是布尔判断应写成 items.length > 0 && ... 或使用三元表达式。
  • React 18+ 自动批处理覆盖哪些异步边界？ 结论是在 createRoot 下，事件处理器、setTimeout、Promise 回调、原生事件等中的多次 setState 默认合并为一次渲染。原因是 React 18 扩展了批处理范围，React 19 延续。本页点击「setTimeout 内两次 setState」可在 Console 看到 render 只增 1。
  • flushSync 适合解决什么同步 DOM 读取问题？ 结论是只在必须立即读到更新后 DOM 时使用。原因是它会打断自动批处理并强制同步提交。典型场景是与第三方 DOM 库集成、马上测量尺寸或滚动位置；普通业务状态更新不需要它。
  • 自动批处理与函数式 setState 有什么关系？ 结论是批处理把多次更新合并后，基于旧闭包值的 setCount(count + 1) 可能只生效一次；函数式 setCount(c => c + 1) 让每次更新基于最新 pending 值排队。本页「闭包旧值 vs 函数式 +3」按钮可对比 Console 输出。
