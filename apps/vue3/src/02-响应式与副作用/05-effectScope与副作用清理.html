effectScope / 副作用清理

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • watch() 和 watchEffect() 都会返回停止函数。组件 setup() 同步创建的 watcher 会随组件卸载停止；组件外或异步创建的 watcher 更需要手动停止。
  • effectScope() 会创建一个响应式 effect 作用域，作用域内创建的 computed、watch、watchEffect 会被收集，调用 scope.stop() 后统一停止。
  • 示例中独立 watch 需要保存停止函数，而 scope 内的 watcher 不必逐个保存；这正是 composable、插件和非组件状态模块使用 effectScope 的价值。
  • effectScope 只管理 Vue 响应式 effect，不会自动清除 setInterval、DOM 事件、WebSocket、请求等外部资源；这些资源仍要配合 onScopeDispose()、组件卸载钩子或业务取消函数。
  • 常见坑是重复开启 watcher 或 scope 却没有停止，导致日志重复、内存泄漏和重复请求。实际项目应把“启动”和“停止”做成成对 API，并防止重复注册。

面试考点

  • watch 停止函数和 effectScope.stop() 有什么区别？ 结论是停止函数只停一个 watcher，effectScope.stop() 停整个作用域内收集到的响应式 effect。原因是 scope 在运行期间统一记录依赖 effect。项目里单个监听用停止函数即可，一组可开关的响应式逻辑更适合 scope。
  • effectScope 适合解决什么问题？ 结论是它适合管理组件外、插件内或 composable 内部创建的一组响应式副作用。原因是这些逻辑不一定天然绑定组件生命周期。边界是它不是通用资源管理器，网络、事件监听和定时器仍要单独清理。
  • 组件卸载时哪些资源不会自动清理？ 结论是 Vue 会处理同步绑定到组件实例的响应式 effect，但不会自动清理所有外部副作用。比如手动添加的 DOM 监听、定时器、订阅、WebSocket 和未取消请求都需要显式释放。否则页面切换后可能继续触发回调或占用内存。
  • composable 里如何设计副作用释放？ 结论是创建副作用时同步设计释放路径。组件内可用 onUnmounted 或 onScopeDispose，组件外可暴露 stop 方法或返回 scope。这样调用方能清楚知道状态何时结束。
