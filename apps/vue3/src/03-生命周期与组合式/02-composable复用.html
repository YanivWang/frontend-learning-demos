composable 组合式函数

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • composable 是封装“有状态逻辑”的普通函数，通常以 useXxx 命名。它可以返回 ref、computed、方法和派生状态，让组件专注渲染。
  • 示例中的 useMousePosition() 同时组合了 ref、computed、watch、onMounted 和 onUnmounted，展示了组合式函数能封装完整行为而不依赖 mixin。
  • 每次调用 composable 默认都会重新创建函数内部状态，因此多个组件调用通常彼此独立。若要共享状态，需要把响应式对象提升到函数外，或使用 Pinia / provide-inject 等明确的共享机制。
  • 涉及 DOM 事件、定时器、订阅和异步请求时，composable 必须同时设计清理路径。示例在挂载时监听 mousemove，卸载时移除监听，避免组件销毁后仍更新状态。
  • 常见坑是返回 reactive 对象后被调用方解构导致响应丢失，或在 composable 中隐藏过多业务副作用。实践中通常返回 refs 或用 toRefs 保持可组合性。

面试考点

  • composable 和 mixin 相比优势是什么？ 结论是 composable 的来源、依赖和返回值更明确。原因是它是普通函数调用，可以通过参数和返回值表达契约，不会像 mixin 那样隐式合并选项并产生命名冲突。项目里可测试性和类型推导也更好。
  • composable 的状态默认共享吗？ 结论是不共享，函数内部创建的状态每次调用都是新实例。原因是普通函数执行会重新创建局部变量。若要共享，应显式把状态放在模块作用域、store 或 provide 中，避免调用方误判生命周期。
  • 为什么 DOM 事件必须清理？ 结论是因为浏览器事件监听不受 Vue 响应式系统自动管理。原因是监听器持有回调引用，组件卸载后仍可能触发。示例用 onUnmounted 移除 mousemove，真实项目中的滚动、resize、媒体查询监听也一样。
  • 什么时候升级为 store？ 结论是当状态跨多组件共享、需要调试持久化或有明确业务领域时，优先使用 Pinia 等 store。原因是 composable 更适合局部可复用逻辑，过多全局状态藏在 composable 中会让数据流不清晰。
