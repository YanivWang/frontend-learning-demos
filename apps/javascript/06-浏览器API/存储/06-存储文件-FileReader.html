FileReader 读取文件并存入 localforage

请打开 DevTools Console 查看输出。

知识点要点

  • FileReader.readAsDataURL() 会把 File / Blob 读成 data:<mime>;base64,... 字符串，适合小图片预览，但不是最省空间的持久化格式。
  • base64 会带来额外体积和 JS 字符串内存开销；如果只是离线缓存文件，优先把原始 Blob 存进 localforage，或用 readAsArrayBuffer() 处理二进制数据。
  • localforage 常用 IndexedDB 后端时可以保存 Blob、ArrayBuffer 等值；如果浏览器只能回退到 localStorage，容量和二进制支持都会变差，生产代码要确认 driver 或做降级。
  • 读取文件要处理用户未选择文件、读取失败、用户重复选择同一文件、文件过大等情况；写入存储还要捕获配额不足和 IndexedDB 不可用。
  • 缓存用户文件涉及隐私和安全边界：不要默认长期保存敏感文件，提供删除入口，不要把 data URL 注入 innerHTML ，展示时应使用受控的 src 或对象 URL。
  • 运行环境： 文件选择需用户手势， file:// 与 http(s):// 均可；IndexedDB 在隐私模式可能不可用。不涉及需 HTTPS 的 File System Access API。

面试考点

  • readAsDataURL 和 readAsArrayBuffer 存文件有什么区别？ 结论是 data URL 适合小文件预览，ArrayBuffer/Blob 更适合二进制处理和离线缓存。原因是 data URL 是 base64 字符串并带 MIME 前缀，体积和内存开销更高；项目里头像预览可用 data URL，大文件缓存应保留 Blob。
  • 为什么大文件不建议先转 base64 再缓存？ 结论是会增加体积、占用 JS 堆内存并拖慢序列化。原因是 FileReader 会一次性把结果放到内存中，localforage 再持久化；图片编辑器、离线附件等场景应限制大小、分片或直接存 Blob。
  • localforage 存 Blob 时要注意哪些浏览器后端差异？ 结论是 IndexedDB 后端最适合，localStorage 回退不适合大二进制。原因是 IndexedDB 支持结构化克隆存储 Blob，而 localStorage 只有字符串且容量很小；生产里应确认 driver，并为老浏览器提供提示或上传替代方案。
  • 文件读取和写入 IndexedDB 分别有哪些失败路径？ 结论是读取可能因为未选文件、权限或 I/O 错误失败，写入可能因为配额、浏览器策略或数据库异常失败。原因是这两步是独立异步流程；代码应分别捕获并给出不同提示，不能只在最后打印一个笼统错误。
  • 前端离线缓存用户文件时有哪些安全和隐私边界？ 结论是必须让用户知道文件会保存在本机，并提供清理能力。原因是浏览器存储通常按同源长期保留，敏感文件可能被同源后续代码读取；项目里要最小化保存、避免 HTML 注入，并在退出登录或清理缓存时删除。
