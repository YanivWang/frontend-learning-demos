React 18 自动批处理对比

请打开 DevTools Console 查看 setTimeout 批处理日志；下方 #root 为 React 挂载/交互演示区，对比两种按钮的 effect 日志条数。

知识点要点

  • React 18 在 createRoot 根下启用自动批处理：不只 React 事件， setTimeout 、Promise、原生事件等异步回调中的多次 state 更新也会合并为更少提交。
  • 批处理的目标是减少不必要的 render 和 DOM commit。它不会改变“更新入队并按顺序计算”的语义；本 demo 用 effect 日志观察一次异步点击通常只追加一条。
  • 批处理并不保证固定值更新都累加成功。如果新值依赖旧 state，仍应写 setCount(prev => prev + 1) 这类函数式更新。
  • flushSync 可以退出批处理并强制同步提交，本 demo 连续两次 flushSync 用来对比更多 commit。真实业务只在必须同步读 DOM 或集成第三方 API 时使用。
  • React 17 的旧根在异步回调里不会覆盖所有自动批处理场景；React 18 讨论自动批处理时必须说明 createRoot 前提，不能把 legacy root 行为混成当前推荐写法。

面试考点

  • React 18 自动批处理扩展了哪些场景？ 结论是 createRoot 下 React 会自动批处理更多来源的更新，包括 timeout、Promise、原生事件等。原因是 React 18 root 统一了更新调度；边界是 legacy root 不应拿来代表 React 18 推荐行为。
  • createRoot 对自动批处理有什么前提意义？ 结论是自动批处理的更广覆盖依赖 React 18 root。原因是 legacy ReactDOM.render 在 React 18 中以 React 17 模式运行，异步批处理行为不完整；新项目必须用 createRoot 。
  • 批处理会改变 state 计算结果吗？ 结论是不会改变更新队列语义，但会减少提交次数。原因是更新仍按顺序入队计算；如果传固定值，结果可能看起来被覆盖，如果传 updater，则会基于队列逐步累加。
  • flushSync 如何退出批处理？ 结论是把更新放进 flushSync 回调，React 会同步提交。原因是它告诉 React 这个更新必须立刻刷新 DOM；代价是打断批处理和调度优化，因此不适合常规点击或输入更新。
  • 异步回调连续更新应怎么写？ 结论是依赖旧值就用函数式更新，不依赖旧值才直接传新值。原因是批处理场景下当前闭包里的 state 可能是旧快照；例如计数、追加日志、切换布尔值都应写 updater。
