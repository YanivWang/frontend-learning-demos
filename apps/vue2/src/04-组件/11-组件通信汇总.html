Vue2 组件通信方式汇总

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • Vue 2 组件通信的主线是单向数据流：父组件通过 props 把数据传给子组件，子组件通过 $emit 发出事件请求父组件修改数据，避免子组件直接改父级状态。
  • provide / inject 适合传主题、表单上下文、祖先组件能力等稳定依赖。Vue 2 官方也提醒它不是响应式状态管理方案；如果要跨页面共享并追踪变更，应优先用 Vuex。
  • EventBus 本质是一个 Vue 实例上的事件发布订阅。它适合小范围临时通信，但监听方销毁前必须 $off，否则旧回调可能继续存在，带来重复响应和内存泄漏风险。
  • $attrs 收集未被 props 接收的 attribute，$listeners 收集父组件监听器，常用于二次封装按钮、输入框、弹窗等基础组件时把能力透传到底层元素。
  • Vue 2 已处于维护/历史版本语境；新项目通常应选择 Vue 3。维护 Vue 2 项目时，先按父子、跨层、全局状态、组件封装四类场景选通信方式，再考虑迁移成本。

面试考点

  • Vue2 父子组件为什么优先用 props 和 $emit？ 结论是它最符合 Vue 的单向数据流。原因是父组件拥有状态，子组件只声明输入和事件，数据来源清楚、调试也容易。边界是兄弟或跨多层组件通信会变得绕，这时要考虑 EventBus、provide/inject 或 Vuex。
  • provide / inject 和 Vuex 的边界怎么划分？ 结论是 provide/inject 用来传稳定上下文，Vuex 用来管理可追踪的共享状态。原因是 Vue 2 的 provide/inject 默认不是为了响应式状态流设计的，调试链路也不如 Vuex 明确。项目里常把主题、表单校验上下文放 provide/inject，把登录用户、购物车、权限放 Vuex。
  • EventBus 为什么容易内存泄漏，如何清理？ 结论是 EventBus 注册的回调如果不在组件销毁前移除，就可能在组件销毁后继续触发。原因是事件总线持有回调引用，页面切换或组件反复创建会累积监听。Vue 2 中通常在 created 里 $on，在 beforeDestroy 里用同一个函数引用 $off。
  • $attrs / $listeners 适合解决什么组件封装问题？ 结论是它们适合透明封装基础组件。原因是包装组件不必显式声明每个原生属性和事件，也能把父层传入的能力转交给内部按钮或输入框。边界是关键业务 props 仍建议显式声明，避免组件接口过于隐式。
