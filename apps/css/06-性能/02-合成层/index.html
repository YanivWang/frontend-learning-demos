合成层与 GPU 加速

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见本复习区，脚本会输出合成友好属性的 computed 值。可配合 DevTools Layers 面板观察。

知识点要点

  • 浏览器会把页面拆成若干可绘制区域，再在合成阶段把这些内容合并成最终画面。某些元素被提升为独立合成层后，后续 transform 或 opacity 变化通常可以只更新合成，不重新布局整棵文档。
  • “GPU 加速”是实现层面的优化结果，不是 CSS 规范保证的固定行为。不同浏览器、设备、元素大小、滤镜和混合效果都会影响图层策略，所以面试里应说“通常更容易合成友好”，不要说“一定走 GPU”。
  • will-change: transform 用于提前告知浏览器元素即将变化，让浏览器有机会准备图层或缓存。它应靠近交互发生的时间点使用，例如 hover 前后、进入视口前后，而不是全站常驻。
  • 图层过多会增加内存/显存占用、纹理上传和合成开销，低端移动设备尤其明显。优化目标不是把所有元素提升为层，而是让真正高频变化的少量元素稳定渲染。
  • 排查时不要只看代码猜测，应结合 DevTools 的 Layers、Rendering、Performance 面板，观察是否出现长任务、频繁 Layout/Paint、图层爆炸或帧率下降。

面试考点

  • 合成层和 GPU 加速是什么关系，为什么不能简单等同？ 结论是合成层是浏览器渲染管线中的分层合成策略，GPU 加速是常见实现手段之一。原因是浏览器可能把某些层交给 GPU 处理，但是否提升、如何合成由实现和设备决定。项目里应通过 DevTools 验证，而不是用 translateZ(0) 这类技巧盲目强推。
  • transform / opacity 动画为什么通常比 left/top 更稳？ 结论是它们通常不改变文档流几何，能减少布局和绘制压力。原因是 left/top 会影响定位盒的位置计算，可能带来布局失效； transform 更像在合成阶段移动已绘制内容。边界是大面积元素、复杂阴影或滤镜仍可能导致合成成本上升。
  • will-change 应该何时添加和移除？ 结论是只在元素即将发生高成本变化前添加，变化结束后移除。原因是 will-change 会让浏览器提前准备资源，长期保留会浪费内存或显存。项目里可以在 hover、拖拽开始、轮播切换前短期设置，也可以用类名或 JS 在 transitionend 后清理。
  • 图层过多会带来哪些性能风险？ 结论是图层过多会增加内存/显存、纹理上传和合成管理成本。原因是每个层都需要浏览器维护绘制结果和合成顺序，低端设备更容易掉帧或白屏。项目里要优先定位真正的瓶颈，只提升少量高频动画元素。
