把文本复制到剪贴板

请打开 DevTools Console 查看输出。

知识点要点

  • navigator.clipboard.writeText() 是现代异步剪贴板写入 API，返回 Promise；成功只代表浏览器接受了写入请求，不代表业务上可以忽略失败提示。
  • Clipboard API 通常只能在安全上下文（HTTPS 或 localhost）和用户激活下使用。非安全页面、后台脚本、自动触发或权限策略限制都可能抛出 NotAllowedError 。
  • document.execCommand("copy") 已废弃，但一些旧浏览器仍可用；它依赖当前选区或临时输入框，容易影响焦点和选择状态，只适合作为兼容兜底。
  • 复制 DOM 内容时优先使用 textContent 得到纯文本；不要把用户生成的 innerHTML 直接复制或展示成可信 HTML，避免把 XSS 载荷在系统间传播。
  • 用户体验上必须处理失败路径：按钮应在点击回调里调用复制、捕获异常、显示可理解的失败原因，并可提供“手动选择复制”的 fallback。

面试考点

  • 复制文本为什么优先使用 navigator.clipboard.writeText ？ 结论是现代项目优先用 Clipboard API。原因是它是异步 Promise API，不需要人为操作选区，语义比 execCommand 明确；边界是它受安全上下文、用户激活和权限策略限制，失败时要兜底提示。
  • Clipboard API 为什么经常要求 HTTPS 和用户点击？ 结论是这是浏览器的安全约束。原因是剪贴板可能包含敏感信息，任意页面自动读写会造成隐私和钓鱼风险；项目里应把复制调用放在按钮点击等用户手势内，并在 HTTP 环境下提示手动复制。
  • execCommand("copy") 为什么只能作为兜底？ 结论是它已废弃且行为依赖选区。原因是浏览器保留它主要是兼容旧内容，未来行为和支持都不应作为新代码基础；兼容方案应创建临时输入框、恢复焦点，并捕获失败。
  • 剪贴板复制失败时产品上怎么处理？ 结论是失败是正常路径，不应只在控制台报错。原因是权限、协议、iframe 权限策略、浏览器设置都可能拒绝写入；项目里应展示“复制失败，请手动选择文本”的提示，并保留可选中文本。
  • 复制 DOM 内容时为什么优先取 textContent ？ 结论是复制纯文本更安全、更可预测。原因是 innerHTML 可能包含用户注入的标签和事件属性；分享码、命令、邀请链接这类场景通常只需要文本，应该避免把 HTML 当作可信载荷。
