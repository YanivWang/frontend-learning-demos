悬浮侧栏拖拽 V2（带圆角与外边距）

阅读本复习区后，拖拽带圆角与外边距的侧栏，注意主内容 margin-left = 侧栏宽度 + 16px。

知识点要点

  • 这个 V2 和贴边版的主要区别是侧栏有 left: 16px、圆角和页面内边距，因此主内容的 margin-left 要用“侧栏宽度 + 间距”一起计算。
  • 拖拽时同时修改 width 和 margin-left 会影响布局阶段，浏览器需要重新计算相关盒子的尺寸和位置；requestAnimationFrame 只能减少提交频率，不能把这类更新变成纯合成。
  • will-change 适合短时间提示浏览器某个属性即将变化，但它不是“GPU 加速开关”；对 width、margin-left 这类布局属性长期使用可能增加内存和图层管理成本。
  • 拖拽交互要处理用户体验边界：按下时阻止默认选中文本，移动时限制宽度范围，结束时解绑监听，避免页面后续交互被残留监听影响。
  • 现代实现可以考虑 Pointer Events 和 setPointerCapture()，统一鼠标、触摸、触控笔；本 demo 保留 Mouse Events，便于对照基础拖拽流程。
  • 运行环境： 与贴边版相同；file:// 可加载 Vue3 本地脚本。改 width/margin-left 会触发布局，rAF 仅合并提交频率。

面试考点

  • 带外边距的 fixed 侧栏为什么要额外补偿主内容位置？ 结论是主内容需要同时避开侧栏宽度和页面间距。原因是 fixed 侧栏脱离普通文档流，不会自动占位；项目里如果只用宽度计算，内容会和侧栏边距或阴影发生重叠。
  • 拖拽中改 width 和 margin-left 会触发哪些渲染阶段？ 结论是通常会触发布局，并可能继续触发绘制和合成。原因是几何尺寸和位置变化会影响布局树；如果动画可以改成 transform，才更可能降低到合成层更新。
  • will-change 应该如何谨慎使用？ 结论是只在变化前短时间使用，变化结束后移除或避免长期声明。原因是浏览器可能提前分配资源，滥用会增加内存占用；对布局属性也不能承诺一定走 GPU 合成。
  • requestAnimationFrame 能否避免重排？ 结论是不能消除由布局属性变化引起的重排，只能把更新安排到合适的帧。原因是 width 和 margin-left 的语义就是改变布局；它的价值在于合并高频事件、减少一帧内多次写入。
  • 可调整布局组件有哪些可访问性和移动端边界？ 结论是不能只依赖鼠标拖拽。原因是触屏、键盘和辅助技术用户可能无法操作手柄；真实项目应提供键盘调整、重置按钮、持久化设置，并用 Pointer Events 覆盖更多输入设备。
