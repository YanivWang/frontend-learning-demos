工程化与组合式组织

知识点要点

  • Vue 3 工程常见分层是 views 放路由页面、components 放可复用 UI、composables 放组合式逻辑、stores 放全局状态、router 放路由表与守卫、services 放请求封装。
  • 组合式函数应围绕业务能力或技术能力命名为 useXxx，例如 useUserSearch、usePolling；它解决的是逻辑复用和关注点聚合，不是把一个组件按生命周期硬切成多个文件。
  • store 适合跨页面共享、需要调试或持久化的状态；composable 适合复用一段响应式逻辑；纯格式化、计算、校验优先放 utils，避免把所有函数都包装成 Hook 风格。
  • 工程化组织要配合 TypeScript 类型、路径别名、ESLint/Prettier、单元测试和目录约束，否则目录命名只是形式，无法阻止状态泄漏和隐式依赖。
  • 常见坑包括循环依赖、composable 在组件外错误依赖当前实例、store 承载过多临时 UI 状态、请求层直接操作组件状态，以及路由页面和基础组件互相引用。

面试考点

  • Vue 3 项目目录为什么要按职责分层？ 结论是分层是为了降低协作成本和依赖混乱，不是为了追求目录数量。原因是路由页面、基础组件、业务逻辑、请求层和全局状态的变化频率不同；边界是小 demo 可以扁平一些，大项目要用约定和 lint/code review 保证单向依赖。
  • composable 和 store 的边界怎么划？ 结论是 composable 复用逻辑，store 保存跨组件共享状态。原因是 composable 每次调用可以创建独立状态，也可以组合生命周期和监听；store 是全局容器，适合登录态、购物车、权限等共享数据。项目里不要把一次性表单输入都放进 store。
  • 为什么不建议按生命周期拆分组合式函数？ 结论是组合式 API 推荐按关注点组织代码。原因是一个业务能力往往同时需要 state、computed、watch 和生命周期，按 mounted/watch 拆分会把同一能力打散；边界是底层工具型 composable 可以只暴露某个生命周期相关能力。
  • service、utils、components、views 如何分工？ 结论是 service 负责外部 I/O，utils 负责无副作用纯函数，components 负责 UI 复用，views 负责路由页面编排。原因是这样可测试、可替换、可定位问题；项目里若 service 直接改 Pinia 或组件状态，需要明确这是业务层设计而非通用工具。
