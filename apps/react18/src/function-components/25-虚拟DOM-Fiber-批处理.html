虚拟 DOM、Fiber 与 React 18 自动批处理

请打开 DevTools Console 查看 render 次数与批处理日志；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • React 元素是描述 UI 的普通 JavaScript 对象，渲染时 React 会根据前后两次描述计算最小必要更新，再提交到真实 DOM。
  • “虚拟 DOM 一定更快”是常见误解。它真正提供的是声明式编程模型、跨平台描述能力和可预测更新流程；手写小范围 DOM 可能更快，但复杂 UI 更难维护。
  • Fiber 是 React 内部的节点与工作单元结构，支撑把渲染拆成可调度任务，并为并发渲染、优先级、暂停和恢复提供基础。
  • React 18 在 createRoot 下启用更广泛的自动批处理，定时器、Promise、原生事件处理器中的多次 setState 也会合并成更少渲染。
  • 批处理通常减少重复渲染；如果必须在状态更新后立即读取已经提交的 DOM，可使用 flushSync ，但它会降低调度空间，应少用。

面试考点

  • 虚拟 DOM 解决什么问题？ 结论是它让 UI 可以用状态到视图的声明式描述表达。原因是 React 通过元素树比较和提交阶段统一管理 DOM 更新。边界是它不是性能神话，极小 DOM 操作手写可能更快，但复杂应用更看重可维护性和一致性。
  • Fiber 为什么重要？ 结论是 Fiber 让渲染工作能被拆分和调度。原因是每个 Fiber 节点携带组件、状态、子树和副作用等信息，React 可以按优先级推进工作。面试不必手写 Fiber，只要能说明它为并发渲染和避免长任务阻塞提供基础。
  • React 18 自动批处理有什么变化？ 结论是 createRoot 下批处理范围从 React 事件扩展到定时器、Promise、原生事件等异步回调。原因是 React 统一调度同一轮任务里的状态更新。本 demo 点击按钮后定时器里更新 a 和 b ，通常只产生一次合并渲染。
  • 什么时候需要 flushSync？ 结论是只有在状态更新后必须同步读取已更新 DOM 时才考虑。原因是 flushSync 会强制 React 立即提交，削弱自动批处理和并发调度收益。常见场景是测量 DOM 尺寸或对接必须同步读取 DOM 的第三方库。
