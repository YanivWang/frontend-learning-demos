虚拟 DOM / diff / key

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • 本页先在两侧 input 输入不同文字，再点「调换顺序」观察内容是否跟随数据项。Vue 模板会被编译成渲染函数，渲染函数返回 vnode。状态变化后，运行时对比新旧 vnode，再用最小必要 DOM 操作把真实页面更新到新状态。
  • key 不是给循环“消警告”的装饰，而是同层兄弟节点的身份标识。稳定唯一的 key 能让输入框焦点、组件状态和 DOM 复用跟随同一个数据项移动。
  • 常见坑是把数组下标、随机数或会变化的字段当 key。列表插入、删除、排序时，下标 key 会让 DOM 状态留在位置上而不是跟着数据项走；随机 key 则会让节点每轮都被销毁重建。
  • 现代 Vue3 不只是运行时全量 diff：编译器会做静态提升、缓存事件处理器，并用 patch flags 标记动态部分，实际更新时优先处理真正变化的节点。
  • 列表 key 基础用法见 01-条件列表与key；性能相关指令见 05-性能优化。

面试考点

  • 虚拟 DOM 在 Vue3 中解决的核心问题是什么？ 结论是它把“状态如何映射到 DOM”抽象成 vnode 对比和 patch 流程，让声明式渲染可以跨平台、可组合并可优化。原因是组件只描述目标 UI，运行时负责找出变化并更新真实 DOM。边界是虚拟 DOM 不等于天然最快，静态页面或超大列表仍需要编译优化、缓存或虚拟滚动配合。
  • key 为什么要稳定且唯一，不能随便用 index？ 结论是 key 决定节点身份，应该使用业务上稳定唯一的 id。原因是 diff 需要用 key 判断“同一个数据项是否还在”，从而移动或复用 DOM。列表只追加不重排时 index 风险较低，但一旦有插入、删除、排序、输入框或子组件状态，就容易出现状态串位。
  • 列表重排时有 key 和 index key 的 DOM 状态差异是什么？ 结论是稳定 key 会让 DOM/组件实例跟随数据项移动，index key 会让实例按位置复用。示例中先在 A/B/C 的输入框输入内容再调换顺序，左侧内容跟着数据项走，右侧内容更容易留在原位置。项目里表单行、可编辑表格、动画列表尤其要避免 index key。
  • Vue3 的编译优化如何降低 diff 成本？ 结论是 Vue3 会在编译阶段标记动态节点、提升静态节点，运行时不必每次把整棵树都当作未知结构比较。原因是模板语法提供了比手写 render 更明确的静态/动态信息。边界是动态插槽、大量条件分支或第三方渲染内容仍可能扩大更新范围，需要结合组件拆分和性能分析。
