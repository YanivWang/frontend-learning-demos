调度器 / nextTick

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • Vue 的状态变更不会每次都同步 patch DOM，而是交给调度器排队。连续多次修改同一个响应式状态时，组件更新会被去重并合并到同一轮刷新中。
  • 因此在同一个同步函数里，刚改完状态就读取模板对应 DOM，常会读到旧文本；等待 await nextTick() 后，才能读到本轮 Vue DOM 更新后的结果。
  • nextTick() 可以接收回调，也可以返回 Promise，组合式 API 中更推荐 await nextTick()，让“改状态 -> 等 DOM -> 读尺寸/聚焦/滚动”的流程更清晰。
  • 不要把 nextTick 当成通用延时器。它只等待 Vue 刷新队列完成，不保证网络请求完成、图片解码完成，也不等价于浏览器下一帧绘制；动画测量可能还需要 requestAnimationFrame。
  • 示例连续执行三次 count.value++，同步日志中的 DOM 仍是旧值，nextTick 后才显示合并后的新值，用来观察调度器批处理效果。

面试考点

  • Vue 为什么连续修改三次状态通常只触发一次 DOM 更新？ 结论是 Vue 会通过调度器批量合并同一轮同步更新，并对组件更新任务去重。原因是立即逐次 patch DOM 成本高，也会产生中间态 DOM。边界是不同异步任务之间会进入不同刷新轮次，不能简单理解为“所有更新永远合并一次”。
  • nextTick 解决的是哪个时序问题？ 结论是它用于等待 Vue 当前状态变更对应的 DOM 更新完成。原因是状态写入是同步的，但 DOM patch 被延后到刷新队列。项目里修改数据后要读取 DOM 尺寸、滚动到底部或聚焦新出现的输入框，才需要 await nextTick()。
  • nextTick 和 setTimeout、requestAnimationFrame 有什么边界差异？ 结论是 nextTick 等 Vue 的更新队列，setTimeout 等宏任务，requestAnimationFrame 更接近下一帧绘制前的回调。原因是它们服务的调度层不同。需要读 Vue 更新后的 DOM 用 nextTick，需要配合视觉帧或动画测量时再考虑 requestAnimationFrame。
  • 项目里哪些场景不应该滥用 nextTick？ 结论是普通业务状态流转不需要每次更新后都写 nextTick。原因是 Vue 模板会自动响应状态变化，过度等待会让逻辑变慢且掩盖状态设计问题。只有后续逻辑真的依赖“已 patch 的 DOM”时，才把它作为明确的时序边界。
