编程式导航（push / replace / go / 命名路由）

点击 push / replace / 命名路由 / query 等按钮，观察「当前路径」行与下方组件变化；用浏览器后退/前进或 back() 、 go(-1) 对比 push 与 replace 的历史栈行为。

知识点要点

  • $router.push 会导航到新地址并新增历史记录，适合用户主动进入详情、编辑、下一步等可回退流程。
  • $router.replace 会导航但替换当前历史记录，适合登录成功、无效地址修正、默认页跳转等不希望用户后退回来的场景。
  • $router.go(n) 、 back() 、 forward() 只是沿浏览器历史栈移动，不保证一定有可用记录；在移动端或新开页面中要准备无历史记录的兜底按钮。
  • 对象写法更适合复杂跳转：动态路由用 { name, params } ，可选条件用 { path, query } 或命名路由加 query。
  • 真实项目中导航可能被守卫取消、重定向，或因为跳转到当前地址产生冗余导航；Vue Router 3.1+ 的 push / replace 会返回 Promise，必要时应捕获失败。

面试考点

  • push、replace、go 的历史记录语义分别是什么？ 结论是 push 新增记录，replace 替换当前记录，go 按偏移量移动历史栈。原因是它们分别对应浏览器导航的三类行为。项目里列表到详情用 push，登录后跳首页用 replace，返回上一页可用 back 或 go(-1)。
  • 什么时候用命名路由对象而不是 path 字符串？ 结论是有动态参数、路径可能重构、跳转语义重要时用命名路由。原因是 name 比 path 更稳定，params 也能按路由记录自动填充。简单静态地址或外部拼好的路径用 path 字符串即可。
  • path + query 和 name + params 应该如何搭配？ 结论是 params 通常配 name，query 可以配 path 或 name。原因是 params 需要动态路径片段承接，query 是 URL 查询串。边界是写了 path 再传 params 通常不会自动拼接，应避免这种混搭。
  • 重复导航或导航守卫拦截时要怎么处理？ 结论是要把导航当成可能失败的异步动作处理。原因是守卫可以取消或重定向导航，重复跳当前地址也可能产生失败结果。项目里常在按钮点击后捕获 Promise，或统一封装安全跳转函数。
  • 编程式导航和 router-link 各适合什么场景？ 结论是声明式链接用 router-link ，业务逻辑触发的跳转用 $router.push/replace 。原因是 link 适合菜单/锚点，编程式适合表单提交后、权限判断后、异步完成后再导航。项目里两者常并存。
