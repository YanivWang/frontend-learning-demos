闭包的作用域链与变量提升

请打开 DevTools Console 查看输出。

知识点要点

  • 闭包是“函数 + 它引用的外层词法环境”。示例里 innerFn 和 innnnerFn 继续读取 outerVar、outerLet、innerVar、innerLet，说明相关环境没有在外层函数返回后立刻释放。
  • var 声明会被提升并初始化为 undefined，所以赋值前读取 outerVar 得到 undefined。let/const 也会创建绑定，但初始化前处于暂时性死区，读取会抛 ReferenceError。
  • 浏览器普通脚本顶层的 var 与函数声明会成为全局对象属性；顶层 let/const 属于全局词法环境，不会挂到 window。模块脚本还有独立模块作用域。
  • arguments 是函数调用时的类数组实参对象，示例中可观察实参；但 arguments.callee 已废弃，严格模式中会报错，递归或自引用应使用函数名。
  • 闭包常用于封装私有状态、工厂函数、回调记住上下文；边界是长期持有大对象或 DOM 引用时要主动释放无用引用。

面试考点

  • 什么是闭包，为什么外层函数返回后变量仍可访问？ 先结论：闭包让函数保留对外层词法环境的引用。原因是函数对象创建时记录了外层环境，内层函数还被外部变量持有时，垃圾回收不能释放这部分环境。项目中常见于事件回调、节流防抖、模块私有变量。
  • var 提升和 let/const 暂时性死区有什么区别？ 先结论：var 可在声明前读取到 undefined，let/const 在初始化前不能访问。原因是 var 创建时已初始化，词法声明创建了绑定但要等执行到声明语句才初始化。边界是“提升”不是把代码文本移动，而是声明绑定阶段的行为。
  • 浏览器脚本顶层 var 与 let/const 对 window 有什么影响？ 先结论：普通脚本顶层 var 会成为 window 属性，let/const 不会。原因是全局环境同时有对象环境记录和声明式环境记录。项目里要避免用顶层 var 暴露全局状态，模块化代码也不应依赖 window.xxx 读取声明。
  • 为什么不推荐 arguments.callee？ 先结论：它已废弃且严格模式禁用。原因是它会暴露当前执行函数，影响优化并和现代严格模式设计冲突。实际递归应使用具名函数、函数声明或把函数引用保存到明确变量中。
