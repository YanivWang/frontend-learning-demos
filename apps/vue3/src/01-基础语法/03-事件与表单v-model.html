事件绑定 / 表单 v-model

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • @click 是 v-on:click 的缩写，事件处理函数可以直接修改响应式状态；需要原生事件对象时可以传 $event 或在函数参数中接收。
  • v-model 在原生表单上会按控件类型绑定值并监听更新事件：文本框通常是 value + input，复选框会处理布尔值或数组，选择框会同步选中项。
  • .trim 会在同步时去掉首尾空白，.number 会尝试转成数字，.lazy 会从实时 input 改为 change 时同步；本页备注框需要失焦后才更新。
  • 用户输入必须有失败路径：本页提交时检查空用户名、非法年龄和未同意协议；真实项目还要做服务端校验，因为前端状态和 DOM 都可以被绕过。
  • 本页用插值和 <pre>{{ form }}</pre> 展示输入，Vue 会按文本转义；如果改用 v-html 渲染用户输入，就会引入 XSS 风险。

面试考点

  • v-model 在原生表单上本质是什么？ 结论是语法糖，本质是把表单值绑定到状态，并在用户输入时更新状态。原因是不同控件的值来源和事件不同，Vue 会替你处理文本框、复选框、单选框和选择框的差异。边界是自定义组件的 v-model 依赖 props 和 emits 约定，和原生表单展开规则不同。
  • .trim、.number、.lazy 分别改变了什么行为？ 先结论：.trim 清理首尾空白，.number 尝试类型转换，.lazy 改变同步时机。原因是表单输入默认都是字符串且通常实时同步。项目里搜索框常保持实时输入，备注或低频字段可以用 .lazy 减少更新频率。
  • 为什么表单不能只依赖前端 v-model 类型转换？ 结论是前端转换只改善交互，不是可信校验。原因是空输入、无法解析的数字、脚本篡改请求都可能绕过页面逻辑。项目中应同时做前端提示和服务端校验，并明确失败提示。
  • 用户输入展示时如何避免 XSS 风险？ 结论是默认使用插值或 v-text 展示文本，不把用户输入传给 v-html。原因是 Vue 的文本插值会转义 HTML，而 v-html 会按 HTML 解析。只有经过可信白名单消毒的富文本才考虑使用 v-html。
