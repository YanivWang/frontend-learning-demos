watch 与 computed 的取舍

在 app1 ～ app3 修改 firstName/lastName 观察 fullName 同步方式； 在 app4 输入含 ? 的问题（试 yes/no/fail）观察防抖与异步竞态处理。

知识点要点

  • computed 负责“由已有响应式数据得到一个新值”，例如 firstName + ' ' + lastName 。它有依赖缓存，模板多次读取同一个派生值时不会无意义重复计算。
  • watch 负责“某个值变化后要做一件事”。回调能拿到 newValue 和 oldValue ，适合做异步请求、防抖、表单校验、同步本地存储或通知外部系统。
  • 本页第一个示例用 watch 手动维护 fullName ，能运行但不是推荐写法；第二个示例用 computed 派生 fullName ，更少状态、更不容易漏同步。
  • computed 也可以写成 { get, set } ，适合把一个可读写派生值拆回多个源字段。边界是 setter 仍应保持同步、可预测，不要把请求、计时器这类副作用塞进去。
  • 异步 watch 必须考虑失败路径和竞态。本页使用防抖计时器和 requestVersion ，避免重复输入时旧异步结果覆盖新答案；真实项目还要处理取消请求、错误提示和组件销毁后的清理。

面试考点

  • watch 和 computed 最核心的职责区别是什么？ 结论是 computed 用来得到派生值，watch 用来响应变化后执行副作用。原因是 computed 有依赖缓存并作为属性读取，watch 是命令式回调，能做异步和外部同步。项目里全名、过滤列表、汇总价格用 computed；输入变化后请求接口或写缓存用 watch。
  • 为什么 fullName 用 computed 比用 watch 同步字段更好？ 结论是 computed 状态更少，数据来源更清楚。原因是 watch 方案需要额外维护 fullName ，任何一个源字段同步漏掉都会出错；computed 每次都从 firstName 和 lastName 派生，不存第二份真相。边界是如果变化后要触发请求或埋点，才应使用 watch。
  • watch 里发请求为什么要处理防抖和旧结果覆盖？ 结论是用户快速输入会触发多次变化，旧异步结果可能晚于新请求返回。原因是异步完成顺序不等于输入顺序，不处理会把界面改回过期答案。项目里通常配合防抖、取消请求、版本号或 AbortController，并明确展示错误状态。
  • computed setter 适合解决什么问题？ 结论是适合把一个可写的派生值拆分回多个源数据。原因是 getter 定义如何组合，setter 定义外部赋值时如何反向更新源字段。项目里可用于姓名拆分、区间输入等同步逻辑，但复杂副作用仍应放在 methods 或 watch。
  • 维护 Vue 2 项目时如何说明 watch 与 Vue 3 选型边界？ 结论是 Vue 2/3 的 watch 语义基本一致，Vue 3 新增 watchEffect 等组合式 API。原因是 Vue 2 已 EOL，新项目应优先 Vue 3 Composition API；存量项目的 watch 选项可直接迁移。边界是复杂副作用在 Vue 3 中也可用 composable 封装。
