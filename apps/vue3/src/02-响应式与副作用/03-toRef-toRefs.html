toRef / toRefs

在浏览器中操作下方交互演示；复习要点与面试答法见本复习区。

知识点要点

  • toRef(state, "count") 创建的 countRef 与 state.count 双向联动；本页“通过 countRef +1”与“改 state.count”都会同步两处显示，控制台打印 [toRef]。
  • ref(state.count) 只在创建时读取一次当前值，之后与源对象断开；本页 brokenCount 在源 count 变化后不会更新，点“对比 ref 快照”可验证。
  • toRefs(state) 只为调用时已存在的可枚举属性创建 ref；本页 label 来自 toRefs，会随 state.label 更新。点“新增 state.tag”后，refsSnapshot 里不会出现 tag，说明 toRefs 不追踪后续新增键。
  • 对可能稍后才有的单个属性，应使用 toRef(state, "maybeKey") 而不是指望 toRefs 自动补全；本页 tagRef 在首次访问前为 undefined，赋值后仍与源联动。
  • 嵌套字段可用 toRef(state.user, "name")；若业务会整体替换 state.user，旧 toRef 仍指向被替换前的对象，需要重新 toRef 或改用 computed。解构基础见 01-ref-reactive与解构。
  • composable 常见返回形态：return { ...toRefs(state), action }，让调用方解构后每个字段仍是 ref。props 在子组件中不应直接写回，派生用 computed，修改走 emit。

面试考点

  • toRef 和 ref(state.foo) 有什么区别？ 结论是 toRef 保持与源属性联动，ref(state.foo) 只复制当前值。原因是 toRef 的 getter/setter 仍读写源对象。本页 brokenCount 行就是对比入口。
  • toRefs 适合解决什么解构问题？ 先结论：让调用方解构响应式对象后仍保留响应性。原因是直接解构会绕过代理 getter。composable 返回多字段时常用 ...toRefs(state)。
  • toRefs 为什么不适合处理未来新增属性？ 结论是只为调用时已有属性创建 ref。本页“新增 state.tag”按钮后，refsSnapshot 不会自动出现 tag；应对单字段用 toRef。
  • props 解构时为什么常提到 toRef / toRefs？ 结论是直接解构 props 可能丢失响应性。原因是 props 是响应式只读对象。边界是子组件不应直接写 props，应 emit 给父组件。
  • 整体替换嵌套对象后 toRef 绑定会怎样？ 结论是仍绑定被替换前的子对象。原因是 toRef 记录的是创建时的属性引用。项目中应避免频繁整体替换 reactive 的嵌套对象，或替换后重新建立 toRef。
