浅拷贝 vs 深拷贝

请打开 DevTools Console 查看输出。

知识点要点

  • 浅拷贝复制的是第一层属性值：原始值会得到新值，引用值仍指向同一个对象，所以改 s1.b.c 会影响原对象。
  • Object.assign() 和对象展开都适合合并扁平配置或做不可变更新的第一层拷贝，但它们会读取 getter，并不会递归复制嵌套结构。
  • JSON.parse(JSON.stringify(value)) 只适合纯 JSON 数据；它会丢失 undefined、函数、Symbol，Date 变字符串，RegExp 变普通对象，遇到循环引用会抛错。
  • structuredClone() 是现代浏览器和 Node.js 可用的原生深拷贝能力，基于结构化克隆算法，支持 Date、RegExp、Map、Set、循环引用和部分可转移对象；函数、DOM 节点等不可克隆值会抛 DataCloneError。
  • 手写 deepClone 可按业务补齐特殊类型，并用 WeakMap 解决循环引用；但属性描述符、不可写属性、getter/setter、类实例私有状态等边界很容易漏，生产代码通常优先选原生 API 或成熟库。
  • 状态管理里不建议无脑深拷贝整棵对象树；现代推荐做结构共享的不可变更新，只复制发生变化的路径，避免性能浪费和引用稳定性被破坏。

面试考点

  • Object.assign 和展开运算符为什么不是深拷贝？ 结论是它们都是浅拷贝，只复制对象自身可枚举属性的第一层值。原因是嵌套对象的属性值本身就是引用，复制后两个对象仍指向同一份嵌套数据。项目里合并一层配置可以用它们，更新深层状态时要沿修改路径逐层复制或使用专门工具。
  • 为什么不推荐把 JSON 序列化当通用深拷贝？ 结论是它只适合纯 JSON 数据，不是通用对象克隆。原因是 JSON 格式本身不能表达函数、undefined、Symbol、循环引用、Map/Set 等对象语义，部分类型会被丢弃或降级。接口 DTO、缓存纯数据可以用，业务对象、表单状态、图结构不应依赖它。
  • structuredClone() 能替代所有手写深拷贝吗？ 结论是大多数结构化数据优先用它，但不是万能。原因是它按结构化克隆算法复制，能处理很多内置类型和循环引用，也会对函数、DOM 节点等不可克隆值抛错。需要自定义类实例规则、保留访问器或兼容旧环境时，仍要评估 polyfill、库或业务专用克隆函数。
  • 深拷贝在前端状态管理里什么时候反而有问题？ 结论是整树深拷贝通常不是首选。原因是它会带来不必要的 CPU 与内存开销，并破坏未变化对象的引用稳定性，影响 memo、diff 和缓存命中。实际项目更常用结构共享的不可变更新，只复制变化分支。

观察要点

浅拷贝只复制第一层，引用类型（数组/对象）仍指向同一份；深拷贝递归复制每一层。具体数值对比见 Console。
