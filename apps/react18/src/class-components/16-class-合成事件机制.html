class 组件中的 React 合成事件

请打开 DevTools Console；提交表单验证 preventDefault，点击子按钮验证 stopPropagation；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • React 事件处理函数收到的是 SyntheticEvent。它封装浏览器原生事件，提供接近原生事件的接口，让 class 与函数组件使用同一套事件模型。
  • 需要访问底层浏览器事件时可以读 event.nativeEvent，但不要依赖 React 没有承诺的内部映射关系；业务代码优先使用合成事件暴露的标准方法和字段。
  • 阻止默认行为用 event.preventDefault()，阻止继续冒泡用 event.stopPropagation()。React 中返回 false 不会自动阻止默认行为。
  • React 17 起事件委托从 document 调整到 root 容器，多个 React 根或渐进升级页面中，事件边界更贴近各自 root；React 18 沿用这一模型。
  • React 17 起 SyntheticEvent 不再池化，异步读取事件字段不需要 event.persist()。不过涉及表单输入时仍建议尽早取出需要的值，避免 DOM 状态后续变化带来误解。

面试考点

  • SyntheticEvent 和浏览器原生事件是什么关系？ 结论是 SyntheticEvent 是 React 对原生事件的封装。原因是 React 需要统一事件接口并接入自己的更新调度；边界是确实要底层事件时读 nativeEvent，但不依赖内部映射细节。
  • React 中为什么不能 return false 来阻止默认行为？ 结论是 React 事件不会把返回值当作阻止信号。原因是官方要求显式调用 preventDefault()；表单提交、链接跳转等场景都应这样写。
  • preventDefault 和 stopPropagation 分别解决什么问题？ preventDefault 阻止浏览器默认动作，stopPropagation 阻止事件继续冒泡。项目里提交表单通常用前者，嵌套点击区域避免父级响应通常用后者。
  • React 17 事件委托从 document 变到 root 有什么影响？ 结论是事件监听绑定在各自 React root 上，而非全局 document。原因是多 root 共存或 React 与非 React 代码混用时，事件边界更清晰；React 18 createRoot 沿用此模型。
  • React 18 中还需要 event.persist() 吗？ 结论是不需要。原因是 React 17 起事件不再池化；但旧 React 16 维护代码里曾需要 persist() 才能异步读取事件对象。
