tsconfig 与 strict

在浏览器中打开本页，结合下方演示区观察效果；要点与面试答法见上方复习区。

知识点要点

  • tsconfig.json 是项目级 TypeScript 配置入口，决定哪些文件参与检查、目标语法、模块系统、模块解析策略和严格程度。
  • strict 是一组严格检查开关的总开关，会启用 strictNullChecks、strictFunctionTypes 等能力；新项目建议默认开启，旧项目可分目录逐步治理。
  • noImplicitAny 禁止参数和变量悄悄退化成隐式 any，能尽早暴露回调、事件参数和第三方数据建模缺口。
  • noUncheckedIndexedAccess 会把索引读取结果标成可能为 undefined，适合暴露数组越界、字典缺 key 这类运行时风险，但可能增加必要的空值处理。
  • Vite、Webpack、esbuild 等前端项目通常让构建工具输出代码，TypeScript 只跑 tsc --noEmit 做类型检查；moduleResolution: "Bundler" 适合现代打包器解析 ESM、exports/imports 的场景。

面试考点

  • strict 为什么建议新项目默认开启？ 结论是它能把空值、函数参数兼容性、隐式 any 等高风险问题前移到编译期。原因是 TypeScript 的收益来自约束边界，而不是只写几个注解。旧项目迁移时可以先开启局部配置或逐项打开，避免一次性产生过多错误。
  • noImplicitAny 和 noUncheckedIndexedAccess 分别防什么？ 结论是前者防类型缺失，后者防索引读取不存在。原因是隐式 any 会让类型检查断链，索引读取在 JS 里天然可能得到 undefined。项目里常见于事件回调、字典表、数组查找和接口缓存。
  • 为什么前端项目常用 tsc --noEmit？ 结论是把类型检查和代码转译分工给不同工具。原因是 Vite/esbuild/SWC 更适合快速打包，而 tsc --noEmit 专注类型正确性。边界是如果项目需要生成声明文件，就不能只依赖 noEmit。
  • moduleResolution: Bundler 适合什么场景？ 结论是适合由现代打包器处理模块解析的前端应用。原因是它更贴近 bundler 对 package.json exports/imports 和扩展名的处理方式。库项目、Node 运行时代码或老构建链路仍要按目标环境选择 NodeNext、Node16 等策略。
