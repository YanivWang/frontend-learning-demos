React 19 新特性概览

本页运行在 React 18.3.1，仅概览 React 19 能力边界；请打开 DevTools Console 查看手写异步状态日志；下方 #root 为 React 18 模拟演示区，完整 React 19 API 见 apps/react19/ 。

知识点要点

  • 本页属于 React 18 模块，运行时仍是 React 18.3.1；它只做 React 19 能力边界概览，真实 API 演示请看 apps/react19/ 。
  • React 19 把异步提交流程抽象为 Actions，并提供 useActionState 处理表单提交状态、返回值和 pending 状态。
  • useOptimistic 用于在异步 Action 完成前展示乐观 UI，真实结果返回后再与权威 state 对齐。
  • use() 可以在渲染中读取 Promise 或 Context，读取 Promise 时依赖 Suspense 处理挂起；这不是 React 18 的数据请求写法。
  • React 19 支持函数组件直接接收 ref prop， forwardRef 进入逐步迁移语境；维护 React 18 项目时仍要继续使用 forwardRef 。

面试考点

  • React 18 项目里怎么介绍 React 19？ 结论是先讲版本边界：React 18 仍用现有 Hooks，React 19 新增 Actions、 useActionState 、 useOptimistic 、 use() 和 ref as prop。原因是这些能力改变了表单、乐观 UI、数据读取和 ref 传递的写法。项目中要等依赖链、测试和框架支持确认后再升级。
  • Actions 与 useActionState 主要解决什么？ 结论是把异步表单提交的状态管理标准化。原因是过去常用 useState 手写 pending、error、result，容易遗漏重复提交和失败路径。边界是复杂表单校验、缓存和路由刷新仍可能交给框架或表单库处理。
  • useOptimistic 和手写乐观 UI 有什么区别？ 结论是它把临时乐观层和真实 state 的对齐交给 React 模型。原因是发送消息、点赞等交互需要先反馈，再等待服务端确认。边界是失败回滚、冲突合并和重试文案仍要由业务明确设计。
  • use() 是否替代 useEffect 请求数据？ 结论是不是简单替代。原因是 use() 读取 Promise 会挂起并依赖 Suspense ，更适合与框架数据层、缓存和 Server Components 配合；客户端事件后的请求仍常用事件处理或 effect。
  • React 19 ref as prop 对 forwardRef 迁移有什么影响？ 结论是函数组件可直接声明 ref prop， forwardRef 仍可用但进入逐步废弃语境。原因是 React 19 把 ref 当作普通 prop 传递，减少包装样板。边界是 React 18 项目仍需 forwardRef ；混用库时要等依赖升级，并检查 TypeScript 类型与 ref 回调写法。
