DOM 事件流：捕获、冒泡、委托

阅读本复习区后，在下方点击嵌套方块、切换模式并点击列表项； once 选项见 DevTools Console。

知识点要点

  • DOM 事件从根节点向目标元素传播是捕获阶段，到达目标后再沿祖先链向外冒泡；默认 addEventListener(type, listener) 监听冒泡阶段，传 { capture: true } 或旧写法 true 才监听捕获阶段。
  • event.target 是实际触发事件的节点， event.currentTarget 是当前正在执行监听器的节点；做委托时通常从 target.closest(selector) 找业务元素，再确认它仍在委托容器内。
  • stopPropagation() 阻止事件继续进入后续祖先节点，但不会阻止当前元素上已经注册的其他监听器； stopImmediatePropagation() 还会阻止当前元素后续监听器执行。
  • 事件委托适合动态列表和大量子元素，能减少绑定和销毁成本；不适合所有事件，例如不冒泡的事件或需要独立生命周期的复杂交互通常应单独绑定。
  • addEventListener 现代推荐使用 options 对象： capture 控制阶段， once 自动解绑， passive 声明不会调用 preventDefault() ，滚动场景可减少浏览器等待。
  • 运行环境： DOM 事件在 file:// 与 http(s):// 下均可演示；本页不依赖 HTTPS 或用户手势权限。

面试考点

  • DOM 事件捕获和冒泡的执行顺序是什么？ 结论是先捕获、到目标、再冒泡。原因是浏览器沿祖先链分阶段派发事件，本例点击 inner 时会先看到 outer/middle/inner 的 capture，再看到 inner/middle/outer 的 bubble；边界是并非所有事件都冒泡，委托前要查事件类型。
  • stopPropagation 和 stopImmediatePropagation 有什么区别？ 结论是前者只阻止继续传播，后者还阻止同一元素上的后续监听器。原因是同一节点可以注册多个监听器；本例 middle 调用 stopPropagation 后仍会执行 middle 的第二个冒泡监听，而 inner 调用 stopImmediatePropagation 会截断后续监听器。
  • 事件委托为什么适合动态列表？ 结论是把监听器放在稳定父节点上，让冒泡事件统一处理。原因是新增或删除子项时不需要逐个绑定/解绑；项目里要用 closest 找目标项，并用 currentTarget.contains() 防止误命中容器外节点。
  • addEventListener 第三个参数如何选？ 结论是新代码优先用 options 对象而不是布尔值。原因是 { capture, once, passive } 可读性更强，能同时表达阶段、自动解绑和默认行为约束；边界是老代码里的 true 等价于捕获阶段。
  • 事件委托处理用户生成内容时要注意什么？ 结论是事件命中不等于数据可信。原因是 DOM 节点和 data-* 属性都可能来自动态内容；实际项目里应校验 id、权限和容器范围，渲染用户文本使用 textContent ，避免把事件委托和 HTML 拼接组合成 XSS 风险。
