高阶组件 HOC 与 render props

请打开 DevTools Console 查看 HOC 与 Hook 加载日志；下方 #root 为 React 挂载/交互演示区，移动鼠标可观察坐标更新。

知识点要点

  • 高阶组件 HOC 是“接收组件，返回新组件”的函数，常用于注入 loading、权限、埋点、主题等横切能力；它不应修改传入组件本身。
  • render props 是组件通过函数 prop 暴露状态或行为，由调用方决定如何渲染；本 demo 的 MouseTracker 把鼠标坐标交给 render 函数。
  • 自定义 Hook 把有状态逻辑提取到普通函数中，函数组件里复用更直接；本 demo 的 useMousePosition 与 render props 版本演示了同一逻辑的现代写法。
  • 常见坑：HOC 要透传无关 props、设置可读 displayName ，避免在 render 内动态创建 HOC；render props 每次传新函数可能影响基于浅比较的优化。
  • 鼠标移动是高频 DOM 事件，真实项目要按场景节流、限制监听范围，并确保 Effect cleanup 正确移除监听，避免性能问题和内存泄漏。

面试考点

  • HOC 为什么要返回新组件而不是修改传入组件？ 结论是 HOC 应保持纯函数式组合。原因是修改原组件会让多个增强器互相污染，也破坏组件可复用性。本 demo 的 withLoading 返回 WithLoading ，并把剩余 props 透传给原组件。
  • render props 的优缺点是什么？ 优点是调用方完全控制渲染结构，逻辑组件只负责提供状态。缺点是 JSX 嵌套可能变深，传入函数引用每次变化也可能影响浅比较优化。鼠标位置、表单状态、权限结果都可以用这种模式表达。
  • Hooks 时代为什么迁移到自定义 Hook？ 结论是 Hook 复用状态逻辑更轻，少一层组件包装。原因是它直接在函数组件里返回数据和方法，不需要 render props 嵌套或 HOC 命名转发。边界是 Hook 只能在 React 函数组件或其他 Hook 中调用，不能替代所有组件增强场景。
  • HOC、render props、Hook 如何取舍？ 新函数组件业务逻辑复用优先自定义 Hook；需要包裹组件、统一权限、埋点、主题注入时 HOC 仍有价值；需要调用方决定 UI 结构时 render props 仍清晰。维护旧 class 代码或第三方库时，要能读懂三种模式。
