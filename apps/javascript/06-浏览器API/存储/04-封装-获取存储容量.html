封装 localforage 容量估算工具

请打开 DevTools Console 查看输出。

知识点要点

  • 容量统计工具应返回清晰的数据结构，例如 storeUsageBytes 、 originUsageBytes 、 originQuotaBytes 和 originRemainingBytes ，避免调用方误把业务估算当真实配额。
  • localforage.iterate() 负责遍历当前 store，适合做局部缓存大小估算； navigator.storage.estimate() 负责读取同源总体用量和配额，适合判断写入风险。
  • 比起把累加值挂到外部对象，封装函数内部维护局部变量并返回结果更直观，也减少并发调用时互相污染的风险。
  • 估算函数要按类型处理：字符串用 TextEncoder ， Blob 用 size ， ArrayBuffer 用 byteLength ，普通对象再尝试 JSON 序列化。
  • 失败路径包括序列化异常、浏览器不支持 StorageManager.estimate() 、localforage 后端不可用或配额已满；工具应返回可解释的降级结果或抛出给调用方处理。
  • 运行环境： 与 IndexedDB / storage.estimate 相同； file:// 下 origin 字段可能为 null ，工具应做降级展示。

面试考点

  • 如何把 localforage 容量统计封装成可复用工具？ 结论是把“遍历 store 估算业务数据”和“读取 origin 配额”拆成函数，最后返回结构化对象。原因是两类数据来源不同、精度不同；项目里调用方可以分别展示缓存大小和剩余配额。
  • 为什么封装时不建议把统计值写进外部对象？ 结论是局部变量加返回值更稳定。原因是外部可变对象会让数据流不清晰，多次并发调用也可能共享状态；只有需要跨多个回调累加时，才在函数内部用闭包变量。
  • 工具返回的 storeUsage 和 originQuota 分别代表什么？ 结论是 storeUsage 是业务层估算， originQuota 是浏览器给整个同源站点的粗略配额。原因是 localforage 只管理自己的 store，而浏览器配额还包含 Cache Storage、其他 IndexedDB 数据等。
  • 浏览器不支持 StorageManager.estimate() 时如何降级？ 结论是仍可返回 store 估算值，并把 origin 级字段置为 null 。原因是 API 支持度和浏览器策略不同；用户界面应显示“无法获取浏览器配额”，而不是显示一个伪造数字。
  • 大数据量缓存页面多久统计一次更合理？ 结论是不要每次渲染或首屏都全量统计。原因是 iterate() 会读取和序列化每条数据；项目里适合在设置页用户主动打开时统计，或在写入/删除时维护增量元数据。
