手写 EventEmitter

先看本复习区，再查看下方输出是否与标准答法一致；细节可打开 DevTools Console。

知识点要点

  • EventEmitter 的基本结构是“事件名 → 监听函数列表”，on 负责追加监听，emit 按注册顺序同步调用监听。
  • off 要能移除同一个函数引用；once 通常通过包装函数实现一次性监听，因此需要保存原始 listener，让 off(type, listener) 也能删掉包装函数。
  • emit 时复制监听数组可以冻结本轮派发快照，避免某个监听在执行中新增或删除监听导致本轮遍历顺序不稳定。
  • 本 demo 的 emit 是同步派发：监听里的异常会沿调用栈抛出，耗时监听会阻塞后续监听；真实项目要按场景决定是否捕获异常或切到异步队列。
  • Node.js EventEmitter 对 error 事件、最大监听数、this 绑定等有更多约定；浏览器 DOM 事件则使用 EventTarget、捕获/冒泡和事件对象模型。
  • 常见风险是忘记解绑导致内存泄漏，尤其是组件卸载、路由切换、长连接消息订阅等场景；注册和清理应成对出现。

面试考点

  • EventEmitter 的 on、emit、off、once 分别要保证什么行为？ 结论：on 注册，emit 触发，off 删除，once 只触发一次。原因是发布订阅模型需要把事件生产者和消费者解耦，同时保留可清理能力。项目里消息总线、插件钩子、Node.js stream 都会用到类似模式。
  • once 为什么通常要保存原始 listener 引用？ 结论：因为 once 实际注册的是包装函数，但用户解绑时传入的是原始函数。原因是如果不保存映射关系，off(type, listener) 无法删除已包装的监听。边界是包装函数还要在执行前或执行后解绑，通常执行前解绑可避免监听内部再次触发同一事件造成重复调用。
  • emit 派发时为什么要复制监听列表？ 结论：复制后按快照派发，本轮执行顺序更稳定。原因是监听函数可能在执行中调用 on 或 off 修改同一个事件列表，直接遍历原数组容易漏调或重复调用。边界是复制有少量额外开销，但对面试实现和大多数业务事件总线更容易保证语义。
  • EventEmitter 和 DOM EventTarget 有什么区别？ 结论：EventEmitter 更偏 Node.js 风格的同步发布订阅，DOM EventTarget 面向浏览器事件流。原因是 DOM 事件有事件对象、捕获、目标、冒泡、取消默认行为等机制，而普通 EventEmitter 只按事件名派发参数。项目中 DOM 交互用 addEventListener，内部消息总线才考虑 EventEmitter 模式。
