手写 instanceof

先看本复习区，再查看下方输出是否与标准答法一致；细节可打开 DevTools Console。

知识点要点

  • value instanceof Constructor 的核心是原型链匹配：从左值对象的 [[Prototype]] 开始向上查找，能找到 Constructor.prototype 就返回 true 。
  • 左值如果是原始值、 null 或 undefined ，没有普通对象原型链可遍历，原生 instanceof 会直接得到 false 。
  • 右值不是函数时，原生 instanceof 会抛 TypeError ；面试手写版至少要区分“左值不匹配返回 false”和“右值非法抛错”。
  • 遍历原型链时推荐使用 Object.getPrototypeOf() ，它是标准 API； __proto__ 是历史访问器，不适合作为现代实现的首选。
  • instanceof 依赖构造函数原型，跨 iframe、手动改原型、 Symbol.hasInstance 自定义判断时都可能和直觉不同；判断数组优先用 Array.isArray() 。
  • 业务代码通常不需要手写 instanceof ，这个题主要考原型链、异常边界和类型判断 API 的取舍。

面试考点

  • instanceof 的判断过程是什么？ 结论：它判断右侧构造函数的 prototype 是否出现在左侧对象的原型链上。原因是实例方法查找和构造关系都依赖原型链，而不是只看构造函数名称。边界是自定义 Symbol.hasInstance 、跨 realm 对象和被改写的 prototype 会影响判断结果。
  • 为什么原始值和 null 做 instanceof 会返回 false ？ 结论：左侧没有可遍历的对象原型链时不会命中构造函数原型。原因是 instanceof 检查的是对象内部原型，而不是装箱后的临时包装对象。项目里判断基础类型应优先用 typeof ，不要用 instanceof String 判断字符串字面量。
  • 手写 instanceof 为什么要校验右侧构造函数？ 结论：右侧不是可调用对象时应抛出 TypeError ，不能简单返回 false 。原因是原生运算符把“类型错误”和“原型链不匹配”作为不同语义处理。面试实现可以覆盖常规函数构造器，但要说明未完整实现 Symbol.hasInstance 等规范细节。
  • instanceof 、 typeof 和 Array.isArray() 应该如何选？ 结论：基础类型用 typeof ，数组用 Array.isArray() ，自定义类实例关系才考虑 instanceof 。原因是 typeof 不看原型链， Array.isArray() 对数组品牌判断更可靠。边界是多窗口对象或可伪造原型时， instanceof 不适合当安全边界。
