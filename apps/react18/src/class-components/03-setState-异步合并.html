setState 异步合并（对象式连续更新只加 1）

请打开 DevTools Console 查看循环内 this.state.count 与 2s 后的最终值；下方 #root 展示页面最终 count（预期为 1）。

知识点要点

  • setState 不是同步赋值，而是把更新请求交给 React；调用后立刻读取 this.state ，通常仍会看到本轮渲染时的旧值。
  • 对象式 setState({ count: this.state.count + 1 }) 会先用当前 this.state.count 算出对象，再把对象放进更新队列；示例循环 10 次时，10 个对象都近似是 { count: 1 } 。
  • React 会批处理同一批次内的多个更新以减少渲染次数，并把传入对象浅合并到 state；更新同一个字段时，后面的对象会覆盖前面对象对该字段的结果。
  • 只要下一个 state 依赖上一个 state，就应使用函数式 setState((prevState, props) => nextState) ；这是计数器、分页、队列等场景最常见的正确写法（见 04 号 demo）。
  • React 18 使用 createRoot 开启新的根，自动批处理覆盖事件、定时器、Promise 等更多异步来源；需要同步刷新 DOM 时才考虑 flushSync ，不要把它当常规状态更新手段。

面试考点

  • 为什么连续调用 10 次对象式 setState 只加 1？ 结论是：因为 10 次更新都基于同一个旧 this.state.count 算出 count: 1 。原因是 setState 进入队列后由 React 批处理，对象式更新不会自动串联读取前一次更新结果；计数器这类依赖旧值的场景要改用函数式更新。
  • setState 是不是 Promise，能不能 await？ 结论是：class 的 setState 不是 Promise，不能依赖 await this.setState(...) 得到最新值。原因是 React 没有把它设计成返回更新完成的 Promise；需要在提交后执行逻辑时，用 setState(updater, callback) 或生命周期方法。
  • 对象式 setState 和函数式 setState 怎么选？ 结论是：新值不依赖旧 state 时对象式足够，依赖旧 state 或 props 时用函数式。原因是函数式 updater 会按队列顺序拿到最新的 prevState ；边界是返回对象仍然是浅合并，不要直接改嵌套对象。
  • React 18 自动批处理对 class setState 有什么影响？ 结论是：React 18 在 createRoot 下会把更多来源的更新合并成一次渲染。原因是自动批处理不再局限于 React 事件处理器；迁移老代码时，异步回调里立刻读取 this.state 的逻辑更容易暴露问题。
  • 什么时候需要在更新后读取最新 DOM 或 state？ 结论是：优先把读取逻辑放在提交之后的回调或生命周期中。原因是 render 前读取到的是旧快照；只有极少数必须立刻同步 DOM 的场景才使用 ReactDOM.flushSync ，否则会损失并发与批处理收益。
