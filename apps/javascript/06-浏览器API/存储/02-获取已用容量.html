通过 iterate 统计已用容量

阅读本复习区后，点击下方按钮运行 iterate 估算与 origin 配额对照。

知识点要点

  • localforage.iterate() 会遍历当前 store 的每个 key/value，适合做业务层“我写入了多少数据”的粗略统计，但它不是浏览器底层物理占用统计。
  • 用 TextEncoder 计算字符串 UTF-8 字节数，比直接用 length 更接近真实文本大小；中文、emoji 等字符用 length 会明显低估或误判。
  • 值类型会影响估算策略：Blob 可直接读 size，ArrayBuffer 可读 byteLength，普通对象可 JSON 序列化估算，Date、Map、循环引用需要特殊处理。
  • navigator.storage.estimate() 返回的是整个 origin 的 usage 和 quota，包含 IndexedDB、Cache Storage 等来源；它适合判断总体配额压力，不适合精确归因到某个 localforage store。
  • 全量遍历本身是异步 I/O 和序列化工作，数据量大时可能很慢；实际项目应维护增量元数据、分批统计，或只在设置页/诊断页触发。
  • 运行环境： navigator.storage.estimate() 在多数现代浏览器可用；file:// 下配额数值可能为 0 或与 http 不同。结果仅为估算，非精确物理占用。

面试考点

  • 如何粗略统计 localforage 当前 store 已用容量？ 结论是遍历 store 后按 key 和 value 的序列化字节数累加。原因是 localforage 没有提供单个 store 的精确物理占用 API；项目里可用这个估算给用户展示缓存大小，但不能把它当作浏览器配额判定。
  • 为什么 key.length + JSON.stringify(value).length 不是准确字节数？ 结论是它只是字符数近似。原因是 JavaScript 字符串长度按 UTF-16 code unit 计算，存储层可能按 UTF-8、结构化克隆或浏览器内部格式保存；中文、emoji、Blob 和对象都会让误差变大。
  • StorageManager.estimate() 和业务自算容量有什么区别？ 结论是前者看 origin 总体，后者看业务数据估算。原因是 estimate() 汇总浏览器认为当前源已用和可用的配额，不能告诉你某个 key 或 store 占多少；排查配额压力时两者可以结合。
  • 遍历统计容量在大数据量场景有什么性能风险？ 结论是可能造成明显耗时和额外序列化开销。原因是每条数据都要读取、转换、累计；真实项目应避免在首屏自动运行，改成用户主动触发、分页统计或写入时维护累计元数据。
  • Blob、Date、循环引用等值如何影响容量估算？ 结论是不能统一用 JSON。原因是 Blob 有 size，Date 会被 JSON 转成字符串，循环引用会直接抛错；业务上应按数据类型选择估算策略并捕获异常。
