手写 Promise.all

请打开 DevTools Console 查看输出。

知识点要点

  • Promise.all(iterable) 接收一个可迭代对象，返回聚合 Promise；所有输入 fulfilled 后，聚合 Promise fulfilled 为结果数组。
  • 结果数组按输入顺序写入，而不是按完成顺序写入，所以手写时必须记录每个任务的原始索引：第 3 个先完成也要写到结果数组第 3 位。
  • 任一输入 rejected 时，聚合 Promise 立即以第一个拒绝原因 rejected；这叫 fail-fast。其他任务不会被自动取消，它们可能仍在继续执行，只是聚合结果已经失败。
  • 边界处理很重要：空 iterable 应得到 []；普通值和 thenable 需要通过 Promise.resolve 归一化；非 iterable 输入会让返回的 Promise rejected 为 TypeError。
  • 不要用 retArray.length 判断完成数量，因为数组按索引赋值会形成稀疏数组，length 只是最大索引加 1；应使用独立 doneCount 统计 fulfilled 的输入数量。

面试考点

  • Promise.all 为什么按输入顺序返回结果？ 结论是规范要求 fulfillment values 按 iterable 迭代出的顺序组成数组。原因是调用方需要稳定地把结果映射回输入任务，而异步完成顺序不可预测。项目中并发请求用户、订单、权限时，结果顺序稳定能避免额外排序和配对错误。
  • 手写时为什么要用独立完成计数器？ 结论是计数器表示真正 fulfilled 的元素数，数组 length 不能表示完成数。原因是先写入高索引会把 length 拉大，导致稀疏数组误判全部完成。边界是空 iterable 要单独处理，否则计数器永远不会触发 resolve。
  • Promise.all 和 allSettled 怎么选？ 结论是任务互相依赖、任一失败就无法继续时用 all；需要拿到每个任务最终状态时用 allSettled。原因是 all fail-fast，allSettled 等全部 settled。项目中首屏必要数据可用 all，批量上报或多接口容错更适合 allSettled。
  • 普通值、thenable 和非 iterable 输入怎么处理？ 结论是普通值和 thenable 都应通过 Promise.resolve 归一化，非 iterable 会让聚合 Promise rejected。原因是原生 Promise.all 面向 iterable，并对每个元素执行 Promise 解析过程。边界是手写面试版可简化，但要主动说明和规范实现的差异。
