readonly / shallowReactive / markRaw

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • readonly() 返回原对象的只读代理：通过只读代理写入会被阻止并在开发环境给出警告，但修改源 reactive 对象时，只读视图仍会同步更新。
  • readonly() 是深只读代理，不是深拷贝，也不是安全权限边界；如果外部仍持有源对象引用，仍然可以从源对象修改数据。
  • shallowReactive() 只让根层属性成为响应式，嵌套对象不会继续转换为响应式代理。示例中直接改 shallow.nested.count 不会稳定触发视图，替换 shallow.nested 才是推荐写法。
  • markRaw() 标记对象永远跳过响应式转换，适合图表实例、编辑器实例、地图对象、复杂类实例等不应被代理的第三方对象。
  • shallow 与 raw API 都是在创建响应式边界：它们能减少代理成本和意外副作用，但也会让“为什么视图不更新”更难排查，应该只放在边界清晰的对象上。

面试考点

  • readonly 为什么不是深拷贝？ 结论是它返回只读代理，不复制数据。原因是代理仍指向同一份源对象，所以源对象更新会反映到只读视图。项目里常用它向外暴露不可直接写的状态，但不能替代权限控制或不可变数据持久化。
  • shallowReactive 的嵌套字段为什么不更新？ 结论是它只追踪根层属性访问与赋值。原因是 Vue 不会把嵌套对象递归转成响应式代理。实际项目里如果用了 shallow API，更新嵌套数据时应整体替换根层字段，或者改回 reactive。
  • markRaw 适合什么对象？ 结论是适合第三方实例、DOM 相关对象、复杂类实例或大对象缓存。原因是这些对象通常不需要被 Vue 代理，代理还可能破坏实例身份或带来成本。边界是被标记对象内部字段变更不会触发视图，需要额外状态或事件驱动刷新。
  • 三者如何选型？ 结论是 readonly 管“外部不要写”，shallowReactive 管“只追踪第一层”，markRaw 管“完全不要代理”。面试回答要补充边界：它们都不是默认推荐状态建模方式，只有性能、第三方实例或封装边界明确时才使用。
