状态更新后读取 DOM — flushSync 与 useEffect

请打开 DevTools Console 对比普通 +1 与 flushSync +1；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • 函数组件中的 state 是当前 render 的快照。调用 setter 后，同步代码仍读到旧的 count；新 state 要等 React 完成下一次渲染后才体现在 UI 中。
  • 需要在 DOM 已经提交后执行副作用时，优先使用 useEffect 监听相关 state。本 demo 在 count 更新后用 effect 写入提示，表达“commit 后再做事”。
  • ReactDOM.flushSync 会强制 React 同步执行回调内的更新并立即刷新 DOM，适合必须把 DOM 更新交给浏览器或第三方代码前就完成的少数场景。
  • flushSync 会打断 React 的批处理和调度优化，可能降低性能，也可能让 Suspense fallback 更早出现；普通业务状态联动不要用它替代 effect。
  • class 组件的 setState(updater, callback) 可在提交后运行 callback；函数组件没有 setter callback，通常用 useEffect 或 useLayoutEffect 表达提交后的副作用。

面试考点

  • setter 后为什么不能立刻读到新 state？ 结论是当前函数闭包里的 state 是旧快照。原因是 React 先排队更新，再渲染和提交；项目里同步读 state 做下一步逻辑容易读错，应使用 updater、effect 或事件中的原始值。
  • 为什么提交后逻辑优先用 useEffect？ 结论是 effect 在浏览器 DOM 提交后运行，适合日志、订阅、请求、读取已更新 DOM 等副作用。原因是它和渲染阶段分离；边界是布局测量若必须阻塞绘制，应考虑 useLayoutEffect。
  • flushSync 适合什么场景？ 结论是只适合必须同步看到 DOM 更新的少数集成场景。原因是它强制 React 立即提交；例如第三方库马上要测量 DOM，或浏览器 API 回调要求 DOM 已经更新。
  • 为什么 flushSync 要慎用？ 结论是它是逃生口，不是常规状态管理手段。原因是强制同步会削弱批处理和并发调度收益；高频点击、输入、列表更新中滥用会带来性能风险。
  • class setState callback 与 useEffect 如何对应？ 结论是 class 的 callback 在 state 提交后执行；函数组件用 useEffect(() => {...}, [state]) 表达“state 变化且 DOM 已提交后”的副作用。原因是 Hooks 没有 setter 第二参数；需要同步阻塞布局时用 useLayoutEffect。
