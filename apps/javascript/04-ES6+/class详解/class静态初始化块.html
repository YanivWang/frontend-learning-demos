静态初始化块 static { ... }

请打开 DevTools Console 查看输出。

知识点要点

  • static { ... } 是类体内的静态初始化块，在类定义被求值时同步执行，不会等到 new 实例。
  • 静态字段初始化器和多个 static 块按源码顺序交错执行；父类静态初始化会先于子类静态初始化。
  • 块内的 this 指向当前类构造器本身，所以可以读写公共静态字段，也可以访问类中声明的私有静态字段。
  • 静态块内部声明的 var 、 let 、 const 都不会泄漏到块外；这和普通块里 var 会提升到函数作用域不同。
  • 静态块只能同步执行，不能使用 await 、 yield 、 arguments ，也不能调用 super() ；需要异步初始化时应暴露显式初始化方法或返回 Promise。
  • 它适合把复杂静态配置、缓存预处理、私有静态字段初始化封装在类内部，避免暴露一个只为初始化服务的公共 init() 方法。

面试考点

  • static 初始化块什么时候执行？ 结论是类求值时执行，不是创建实例时执行。原因是它属于类的静态初始化流程，会和静态字段按声明顺序同步运行；项目里不能把它当成实例构造逻辑。
  • static 块里的 this 指向谁？ this 指向当前类构造器，因此可访问 this.staticField 和已声明的私有静态字段。边界是实例字段还没有实例可挂载，所以在静态块里读 this.type 读到的是类上的静态属性，而不是实例字段。
  • static 块适合解决什么问题？ 它适合封装复杂的同步静态初始化，例如从多个值推导缓存或初始化私有静态状态。原因是类外函数无法访问私有静态字段，而公开 init() 又会泄漏实现细节；异步初始化仍应放到显式 API 中处理失败路径。
  • 引用下方静态字段有什么坑？ 公共静态字段在执行到声明位置前还没有被设置，读取通常得到 undefined ；私有静态字段如果尚未初始化，访问会抛错。项目里应把被依赖的静态字段写在静态块之前，或者在块内完成统一赋值。
