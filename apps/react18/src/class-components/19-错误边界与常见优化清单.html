Error Boundary 与常见优化清单

请打开 DevTools Console；点击"触发渲染错误"观察 Error Boundary 降级 UI，点击"触发事件错误"验证事件 handler 中的错误不会被捕获；下方 #root 为 React 挂载/交互演示区。

知识点要点

  • Error Boundary 是 React 渲染树的错误兜底机制。React 18 中错误边界仍通过 class 组件实现，使用 static getDerivedStateFromError 渲染降级 UI，并用 componentDidCatch 记录错误信息。
  • 错误边界能捕获子组件在渲染、生命周期和构造函数中的错误；不能捕获事件处理函数、异步回调、服务端渲染错误，也不能捕获错误边界自身抛出的错误。
  • 实际项目中不要只在根节点放一个边界。常见做法是在路由页、关键业务卡片、第三方组件区域分层放置，避免一个局部错误卸载整个应用，同时把错误上报到日志系统。
  • React 性能优化要先定位瓶颈，再选择手段。常见方向包括稳定 key、拆分组件和状态、避免不必要的 Context value 变化、memo、useMemo、useCallback、代码分割和虚拟列表。
  • memo 和缓存 Hook 不是默认必加项。它们会增加比较和维护成本，只有当子组件渲染昂贵、props 引用可稳定、或 DevTools 证明重复渲染是瓶颈时才值得使用。

面试考点

  • Error Boundary 能捕获哪些错误，不能捕获哪些错误？ 结论是能捕获子树渲染、生命周期和构造函数中的错误，不能捕获事件、异步、SSR 和自身错误。原因是它工作在 React 渲染提交流程里；事件和异步已经脱离这次渲染调用栈。
  • 为什么 React 18 仍然需要 class 组件实现错误边界？ 结论是官方 API 仍要求错误边界使用 class 生命周期。原因是 getDerivedStateFromError 和 componentDidCatch 目前是 class API；函数组件通常通过第三方封装或外层 class 边界使用。
  • getDerivedStateFromError 和 componentDidCatch 分别做什么？ 前者返回 state 以渲染降级 UI，后者接收错误和组件栈，适合日志上报。项目里不要在 getDerivedStateFromError 做副作用。
  • 常见 React 性能优化为什么不能先上 memo？ 结论是要先定位瓶颈。原因是 memo 也有比较成本，错误的缓存还会制造过期依赖问题；项目里先用 Profiler、日志和用户路径找出真正慢点。
  • 错误边界和性能优化在项目里如何分层落地？ 结论是错误边界按路由页/业务卡片分层放置，性能优化按瓶颈分层：先拆 state 和组件，再稳定 key 和 Context，最后才加 memo/虚拟列表。原因是局部错误不应拖垮整站，性能手段也有维护成本，不应无证据地全局套用。
