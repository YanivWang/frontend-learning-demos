Notification 桌面通知

请阅读本复习区，并在下方演示区点击按钮；需人工验证：Notification 要求安全上下文（HTTPS/localhost），file:// 通常不可用，且需用户授权。

知识点要点

  • Notification 用于从页面创建系统级通知；它暴露在安全上下文中，且必须先通过 Notification.requestPermission() 获得用户授权。
  • Notification.permission 有 granted、denied、default 三态：已允许才能发送，已拒绝应停止请求，默认态可以在用户手势中发起一次请求。
  • 通知内容可设置 body、icon、tag 等选项，但展示样式、声音、关闭行为会受浏览器和操作系统策略影响，不能当作可靠 UI 渠道。
  • 本页的循环通知用于演示节奏控制；真实项目要限制频率、提供关闭入口，并避免在用户拒绝或页面不可见时持续骚扰。
  • 普通页面 Notification 只能在页面活跃时直接创建；离线或后台推送通常需要 Service Worker、Push API 和服务端推送订阅配合。

面试考点

  • Notification API 为什么必须先请求权限？ 结论是系统通知会打断用户，所以浏览器要求明确授权。原因是通知可能跨出页面可视区域，滥用会造成骚扰和钓鱼风险。项目里应在用户完成订阅、下单提醒等明确动作后请求，而不是首屏自动弹权限。
  • granted、denied、default 三种权限状态如何处理？ 结论是 granted 直接发送，denied 不再请求，default 可在用户手势中请求一次。原因是 denied 往往代表用户明确拒绝，反复触发会损害体验。业务上要提供站内提醒或设置页说明作为降级。
  • 页面 Notification 和 Push API 后台推送有什么区别？ 结论是 Notification 负责展示通知，Push API 负责把服务端消息送到 Service Worker。原因是页面脚本关闭后不能直接执行普通通知逻辑。需要离线到达、后台提醒或消息订阅时，应使用 Service Worker + Push，再由 Service Worker 展示通知。
  • 循环通知或营销提醒要注意哪些体验和安全边界？ 结论是必须限频、可关闭、可解释，并尊重权限和浏览器策略。原因是通知属于高打扰通道，系统可能静默、折叠或拦截滥用通知。项目里应把它用于用户明确期待的提醒，并记录发送失败或权限拒绝路径。
