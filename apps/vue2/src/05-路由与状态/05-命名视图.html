命名视图（同级多个 router-view）

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • 命名视图允许同一级页面放多个 <router-view>，每个出口通过 name 区分，例如默认出口、侧栏出口、底部出口。
  • 路由记录要使用复数 components，其中 default 填充无名出口，a、b 等 key 填充 name="a"、name="b" 的出口。
  • 本页一个 / 路径同时渲染首页、课程、底部三个组件，展示的是“同一路由状态驱动多个布局区域”。
  • 命名视图和嵌套路由不同：命名视图解决同级多出口，嵌套路由解决父子层级。复杂页面可以组合使用，但不要把普通静态布局都塞进路由配置。
  • 常见坑是把 component 写成单数、components 的 key 和 router-view name 不一致，或者忘了给无名出口配置 default。

面试考点

  • 命名视图和嵌套路由分别解决什么问题？ 结论是命名视图解决同级多个出口，嵌套路由解决父子层级出口。原因是命名视图用 name 映射组件，嵌套路由用 children 和父组件内部的 router-view 形成层级。项目里同一 URL 同时刷新主内容和辅助栏可用命名视图。
  • components 对象里的 default 有什么特殊含义？ 结论是 default 对应没有 name 的 router-view。原因是无名出口也需要一个稳定键参与映射。边界是如果只写 component，它等价于只给默认出口渲染一个组件。
  • 一个路径如何同时渲染顶部、内容和底部区域？ 结论是在模板放多个命名出口，在路由记录里写 components 对象。原因是一次路由匹配可以携带多个组件映射。项目里要注意这些区域都由同一个路由状态控制，局部独立切换更适合普通组件状态或嵌套路由。
  • 什么时候不用命名视图而用普通组件组合？ 结论是静态布局或只属于某个页面内部的 UI 组合，普通组件更简单。原因是命名视图会把布局细节推到路由配置里，过度使用会让路由表膨胀。只有多个区域确实需要随 URL 同步切换时才值得使用。
