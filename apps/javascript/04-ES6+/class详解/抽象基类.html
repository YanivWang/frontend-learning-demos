用 new.target 实现抽象基类

请打开 DevTools Console 查看输出。

知识点要点

  • JavaScript 没有原生 abstract class 关键字；这里的“抽象基类”是运行时约束，用构造器里的 new.target 阻止直接实例化。
  • 在 class 构造器中， new.target 指向被 new 直接调用的构造器： new Animal() 时是 Animal ， new Dog() 且 Dog extends Animal 时在 Animal 构造器里仍是 Dog 。
  • if (new.target === Animal) throw 只能防止直接创建基类实例，不能在语法层面强制子类实现某个方法；需要在基类方法里抛错、在测试中覆盖，或用 TypeScript 的 abstract 做编译期约束。
  • 派生类如果写 constructor，必须先调用 super() ，父类构造器执行完之后才能访问 this 并设置子类字段。
  • 运行时检查适合发布到纯 JS 环境的库；内部 TypeScript 项目更常用 abstract class 或接口表达约束，再辅以必要的运行时错误提示。

面试考点

  • new.target 在基类构造器里为什么能识别直接实例化？ 结论是它保存“这次 new 直接调用的构造器”。原因是子类通过 super() 进入父类构造器时， new.target 仍是子类；所以只有直接 new Animal() 时才等于 Animal 。
  • 用 new.target 模拟抽象基类有什么限制？ 它能在运行时阻止直接实例化基类，但不能像 TypeScript 那样在编译期要求子类实现抽象方法。项目中如果要保证接口完整性，应结合类型系统、单元测试或基类默认方法抛错。
  • 运行时抽象基类和 TypeScript abstract 如何分工？ TypeScript 负责开发期约束，编译到 JS 后抽象标记不存在； new.target 负责运行时保护。发布给纯 JS 用户的库，如果不希望基类被直接实例化，通常需要保留运行时检查。
  • 派生类里 super() 和 new.target 有什么关系？ 派生类 constructor 必须先调用 super() 才能拿到 this ；父类构造器内部的 new.target 仍指向子类。这个机制让父类能区分“直接被 new”还是“作为父类被子类调用”。
