computed / watch / watchEffect

在浏览器中操作下方交互演示；复习要点与面试答法见本复习区。

知识点要点

  • computed 用来声明“由状态推导出的值”，它会缓存结果，只有依赖变化时才重新计算；本页 normalizedKeyword 由输入关键词裁剪并转小写得到。点“触发重渲染”时若 keyword 未变，重算次数不会增加，控制台也不会再打印 [computed] 重算。
  • computed 与模板里直接调方法的区别在于：前者按依赖缓存，后者每次渲染都会重新执行。生产里的派生展示、过滤结果应优先 computed，只有确实需要“每次渲染都跑一遍”时才用方法。
  • watch 用来明确监听某个源，回调能拿到新旧值，适合发请求、写日志、同步外部系统这类命令式副作用。本页故意不加 { immediate: true }，因此首次进入页面时 watch 日志为空，改输入后才出现。
  • watch 第三参数常见选项：immediate 挂载即执行；deep: true 深度监听对象；flush: 'post' 在 DOM 更新后触发（适合读 DOM 尺寸）。需要监听对象某个字段时，也常用 getter 形式 watch(() => obj.x, cb)。
  • watchEffect 会在创建后立即执行一次，并自动追踪同步执行期间读取到的响应式依赖；依赖少且关系直观时更省代码，复杂时显式 watch 更好读。它返回 stop 函数，本页按钮可手动停止副作用。
  • watch 与 watchEffect 的执行时机对比：本页加载后 effectMessage 立刻有值，而 logs 仍为空，这就是 lazy watch 与 eager watchEffect 的直观差异。
  • 副作用回调要避免读写同一个依赖集合，否则容易造成循环触发或噪声日志；本页让 watchEffect 只写独立的 effectMessage，不再追加它自己可能追踪的日志数组。
  • 异步副作用要考虑失败、竞态和清理：关键词搜索、订阅事件、定时器都应在下一次副作用前取消或忽略过期结果。完整可运行示例见同章 06-onWatcherCleanup与watch清理。

面试考点

  • computed 和 watch 的核心区别是什么？ 结论是 computed 产出可复用的派生值，watch 响应变化并执行副作用。原因是 computed 有缓存且应保持纯计算，watch 的回调用来处理异步、日志或外部同步。项目里展示价格、过滤列表用 computed，关键词请求和埋点用 watch。
  • computed 和方法调用有什么本质区别？ 结论是 computed 按依赖缓存，方法每次渲染都会执行。原因是 computed 把“派生结果”当成可复用状态，方法只是普通函数调用。模板里多次读同一 computed 只算一次；多次调同一方法会重复计算。本页“触发重渲染”按钮就是验证缓存的入口。
  • watch 与 watchEffect 如何选型？默认执行时机有何不同？ 先结论：依赖明确、需要新旧值或精确控制时用 watch；依赖简单、想自动收集时用 watchEffect。执行时机上，watch 默认 lazy，watchEffect 创建后立即跑；需要首屏就跑 watch 时传 { immediate: true }。原因是 watchEffect 隐式依赖更省代码，但复杂后不易排查；接口请求通常更偏向显式 watch。
  • watchEffect 自动收集依赖有什么边界？ 结论是它只追踪同步执行期间读取到的响应式依赖。原因是异步 await 之后读取的值不一定被纳入首次依赖收集。边界是回调里不要顺手读取无关响应式对象，否则会引入意外触发。
  • 异步副作用为什么需要清理或防竞态？ 结论是旧请求可能比新请求晚返回并覆盖新状态。原因是用户输入、路由变化和网络延迟都可能造成竞态。项目中要用取消控制器、清理回调或请求序号忽略过期结果，并处理失败提示；具体写法见 06-onWatcherCleanup与watch清理。
