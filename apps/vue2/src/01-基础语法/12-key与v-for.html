key 复用控制 + v-for 遍历数组/对象

先在无 key 区输入内容再切换 loginType 观察输入保留；再在有 key 区重复对比； 点「切换 v-if」观察 template 分组渲染。

知识点要点

  • v-for 遍历数组时常写 (item, index) in items，遍历对象时常写 (value, key, index) in object。本页同时演示数组和对象两种来源。
  • Vue 2 更新列表时默认采用就地复用策略：如果数据顺序变化，它会尽量 patch 当前位置的元素，而不是无条件销毁重建。这对性能有利，但表单输入、组件内部状态等临时状态可能被保留。
  • :key 是节点身份提示。给不同登录输入框设置不同 key 后，Vue 会把它们视为不同元素，切换时销毁旧 input 并创建新 input，因此旧输入不会保留。
  • 列表 key 应使用稳定的字符串或数字，例如数据库 id。不要用对象或数组做 key；当列表会插入、删除、排序时，也不推荐用 index，因为 index 会随位置变化，可能导致状态错位。
  • Vue 2 中同一元素上 v-for 优先级高于 v-if，复杂过滤应先用 computed 得到列表，再渲染 v-for。Vue 2 已 EOL，新项目应优先 Vue 3。

面试考点

  • v-for 中 key 的作用是什么？ 结论是 key 帮助 Vue 识别每个节点的稳定身份。原因是列表更新时 Vue 需要判断哪些节点可以复用、移动或需要重建。项目里渲染用户列表、商品列表、表单行时应使用稳定 id 作为 key。
  • 为什么切换两个 input 时加 key 会清空旧输入？ 结论是不同 key 会让 Vue 把它们视为不同元素。原因是无 key 时 Vue 可能复用同一个 input 节点，浏览器保留的输入值也会跟着留下；加 key 后旧节点被销毁，新节点重新创建。项目里登录方式切换、动态表单切换经常需要这个行为。
  • v-for 遍历数组和对象的参数顺序是什么？ 结论是数组通常是 (item, index)，对象通常是 (value, key, index)。原因是对象遍历多了属性名这一层。边界是对象属性枚举顺序遵循 JavaScript 的对象枚举规则，不要把普通对象遍历当成严格排序列表。
  • 为什么列表重排时不推荐用 index 当 key？ 结论是 index 代表位置，不代表数据身份。原因是插入或删除后，同一个 index 可能对应不同数据，导致组件状态或输入值错位。边界是完全静态、不会重排也没有内部状态的简单列表，用 index 风险较低，但仍优先稳定 id。
  • v-if 和 v-for 放在一起时为什么要谨慎？ 结论是同元素混用时 v-for 优先级更高，条件与列表渲染耦合。原因是 Vue 2 会先进入循环再判断条件，既影响可读性也可能带来不必要遍历。项目里通常用 computed 先过滤数组，再 v-for 渲染结果。
