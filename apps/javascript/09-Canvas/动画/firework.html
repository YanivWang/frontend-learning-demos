Canvas 烟花动画

知识点要点

  • requestAnimationFrame 会在浏览器准备重绘前调用回调，回调是一次性的，连续动画必须在每帧重新注册；页面在后台标签页时多数浏览器会暂停它，能减少无效计算。
  • 本例没有用 clearRect() 立刻清屏，而是先把 globalCompositeOperation 设为 destination-out ，再用半透明矩形逐步擦除旧像素，所以轨迹会残留几帧后淡出。
  • lighter 合成模式会把重叠像素的颜色相加，适合烟花、光点、粒子发光这类视觉效果；绘制普通 UI 或图表时通常保持默认的 source-over ，避免颜色被意外叠亮。
  • 烟花对象负责从底部飞向目标点，抵达后创建粒子；粒子对象保存位置、历史坐标、角度、速度、摩擦、重力和透明度，每帧先更新状态再绘制，透明度衰减到阈值后从数组移除。
  • Canvas 动画的常见风险是粒子过多、画布过大、窗口 resize 后尺寸不匹配，以及高刷新率屏幕上如果不用时间差计算会运动变快；项目中通常要限制发射频率、控制粒子上限，并在销毁页面时取消动画。 高 DPI 缩放、窗口 resize 与后台标签页 rAF 暂停 本页未完整实现， 需人工在浏览器验证 （缩放窗口、切换标签页、DevTools 节流观察）。

面试考点

  • 为什么 Canvas 动画优先用 requestAnimationFrame？ 结论是动画主循环优先用 requestAnimationFrame ，因为它和浏览器重绘节奏对齐，后台页通常会暂停，性能和电量表现更稳定。原因是 setInterval 只按计时器排队，不保证和绘制同步，主线程繁忙时还会堆积或抖动。边界是 rAF 的回调只执行一次，高刷新率屏幕上还应使用回调时间戳或时间差计算位移，否则动画速度可能随刷新率变化。
  • Canvas 烟花拖尾为什么用 destination-out 而不是每帧 clearRect？ 结论是 clearRect() 会直接清空旧帧，而本例用 destination-out 配合半透明填充逐步削弱旧像素，视觉上形成拖尾。原因是合成模式决定新绘制内容如何影响已有像素，半透明擦除会让历史轨迹按帧衰减。项目里如果要画精确图表、签名板或截图导出，就不应保留残影，应改为明确清屏或分层画布。
  • 粒子系统如何避免点击过快或页面长时间运行导致卡顿？ 结论是粒子系统要限流、限量并及时清理失效粒子。原因是每个粒子每帧都要 update 和 draw，复杂度近似随粒子数量线性增长，点击连发会迅速增加 CPU/GPU 压力。本例用 limiterTick 限制鼠标按下时的发射频率，并在透明度衰减后 splice 移除粒子；真实项目还常加最大粒子数、暂停不可见动画和性能降级策略。
  • 高 DPI、窗口 resize 和后台标签页会给 Canvas 动画带来什么边界？ 结论是 Canvas 动画不能只按初始化尺寸和固定像素绘制。原因是 canvas.width/height 是绘图缓冲区尺寸，CSS 尺寸、设备像素比和窗口变化会影响清晰度与命中坐标；后台标签页 rAF 暂停后恢复也可能出现时间跨度。项目中要在 resize 时重设缓冲区并按 devicePixelRatio 缩放上下文，使用时间差时还要限制最大步长。
