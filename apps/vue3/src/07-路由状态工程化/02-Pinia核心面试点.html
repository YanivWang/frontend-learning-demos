Pinia 核心面试点

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • Pinia 是 Vue 官方推荐的状态管理库，defineStore 可以写成 Options Store 或 Setup Store；本页代码片段演示 Options Store 的 state、getters 和 actions。
  • state 应由函数返回，避免跨实例共享；getters 类似 store 级 computed，适合派生只读数据；actions 可以同步也可以异步，适合封装登录、请求和事务性状态变更。
  • Pinia 没有 Vuex 中必须经过 mutations 的限制，组件可以直接写 store.count++，也可以用 $patch 批量更新；面试时要说明这是 API 设计差异，不是“没有规范”。
  • 从 store 解构 state 或 getters 时应使用 storeToRefs(store)，否则普通解构会拿到当前值而不是响应式引用；actions 是方法，可以直接解构。
  • 持久化通常通过插件或订阅写入 storage，但 token、用户画像、权限缓存要区分敏感性和时效性；SSR 场景还要为每个请求创建独立 Pinia，避免状态串用户。

面试考点

  • Pinia 和 Vuex 的核心差异是什么？ 结论是 Pinia 更贴近组合式 API，去掉 mutations，类型推导和模块组织更自然。原因是每个 store 本身就是独立模块，actions 可直接处理异步和状态变更；边界是老 Vue 2/Vuex 项目维护时不必强迁，新 Vue 3 项目通常优先 Pinia。
  • getters、actions、computed、composable 如何分工？ 结论是跨组件共享的派生状态放 getters，共享业务动作放 actions，组件局部派生用 computed，非全局状态复用用 composable。原因是 store 是全局状态容器，过度把局部 UI 状态塞进 store 会增加耦合；项目里可按“是否跨页面共享、是否需要持久化或调试”判断。
  • 为什么解构 Pinia store 要用 storeToRefs？ 结论是 state/getters 解构要用 storeToRefs 保持响应式。原因是普通对象解构会复制当前属性值，丢掉 proxy/getter 访问链；边界是 actions 是函数方法，不需要转成 ref，可以直接解构调用。
  • 登录态持久化应该注意什么？ 结论是可以持久化必要的登录标识或刷新后恢复所需状态，但不要把高敏感或长期有效的凭据随意写入 localStorage。原因是本地存储可被同源脚本读取，XSS 风险会放大；项目里还要配合后端过期、刷新、权限接口和退出清理。
