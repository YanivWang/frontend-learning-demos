React 合成事件机制

在浏览器中操作下方交互演示；复习要点与面试答法见上方区块。

知识点要点

  • React DOM 事件名使用小驼峰，例如 onClick、onChange；处理函数接收的是 SyntheticEvent，不是直接暴露的浏览器事件对象。
  • SyntheticEvent 提供与原生事件相似的一致接口，仍可通过 event.nativeEvent 访问底层事件；React 17 起事件对象不再池化，异步读取事件属性不需要 persist()。
  • 表单提交、链接跳转这类默认行为必须显式调用 event.preventDefault()；React 不支持用 return false 同时阻止默认行为和冒泡。
  • 冒泡控制使用 event.stopPropagation()。当 React 事件和原生 addEventListener 混用时，要分清监听器注册位置与捕获/冒泡阶段，避免误以为所有监听都会被拦截。
  • React 17+ 将事件委托从 document 调整到 root 容器附近，多个 React root 或逐步升级旧页面时，事件隔离比旧版本更清晰。

面试考点

  • SyntheticEvent 和浏览器原生事件是什么关系？ 结论是 React 给处理器传入合成事件，它封装了原生事件并提供一致接口。原因是 React 需要统一不同浏览器行为并接入自己的调度与事件系统。需要原生细节时读取 nativeEvent，但不要依赖未声明的映射细节。
  • React 中为什么 return false 不能阻止默认提交？ 结论是必须调用 preventDefault()。原因是 React 事件处理器不会把返回值当作取消信号。本 demo 的表单提交调用 event.preventDefault() 后，页面不会刷新，失败路径是忘记调用时浏览器会执行默认提交。
  • stopPropagation 有什么边界？ 它能阻止当前 React 事件继续冒泡到上层 React 处理器。边界是原生捕获监听、不同 root 或已经触发的监听器不一定按直觉被拦截；复杂页面里应明确监听阶段和挂载位置。
  • React 17+ 事件委托位置变化有什么意义？ 结论是事件委托从全局 document 更靠近 root 容器。原因是这样更利于多版本 React、微前端和逐步迁移。项目里如果混用多个 root，要测试跨 root 点击、弹窗和全局快捷键行为。
