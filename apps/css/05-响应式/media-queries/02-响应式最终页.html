基于 @media 断点的旧式页面响应式改造

缩窄浏览器窗口观察 980px、650px、480px 附近的变化，并和上一页的固定宽度基线对比。

知识点要点

  • 本页先加载 style.css 提供桌面端基线，再加载 media-queries.css 覆盖窄屏样式；后加载的同等优先级规则会在级联中胜出，这是断点改造能生效的前提。
  • @media screen and (max-width: 980px) 把外层容器改为 95%，内容区和侧栏改成百分比宽度，并让嵌入视频宽度跟随容器，解决固定 980px 在平板宽度下溢出的问题。
  • 650px 断点取消 #content 与 #sidebar 的浮动，把多列布局改成单列；同时把导航、搜索框和站点标题从绝对定位调回普通文档流，避免小屏重叠。
  • 480px 断点进一步压缩导航字号和点击区域，并关闭旧 WebKit 的文字自动放大。现代项目通常不会全局禁用文字缩放，应优先保证可读字号和无障碍缩放。
  • 响应式不只改容器：图片、iframe、object/embed 等替换元素也要限制最大宽度，并按设计稿给媒体区域准备稳定尺寸。示例中的 HTTP 视频、Flash object 和第三方脚本属于历史写法，真实项目要准备加载失败、混合内容和移动端播放限制的兜底。

面试考点

  • 这个页面的三个 max-width 断点分别解决了什么布局问题？ 结论是 980px 处理固定桌面容器溢出，650px 把多列改为单列，480px 继续压缩导航和文字细节。原因是响应式改造应按真实破版点逐步覆盖，而不是随便凑设备宽度。项目里断点通常来自内容何时失衡，再映射到设计稿和主流设备区间。
  • @media 规则为什么要放在基础样式之后加载？ 结论是为了让断点规则在级联顺序上覆盖基础规则。原因是这些选择器大多与基础样式优先级相同，如果先加载断点文件，后面的基础样式会把宽度、浮动和定位覆盖回去。边界是也可以用更高优先级或层叠层管理，但普通项目里保持加载顺序更直观。
  • max-width 桌面优先和 min-width 移动优先怎么取舍？ 结论是旧站改造常用 max-width 桌面优先，新项目更推荐移动优先。原因是旧站通常已有完整桌面 CSS，逐步向小屏覆盖成本低；新项目从单列、低复杂度开始，再用 min-width 增强大屏，通常 CSS 更少。边界是团队设计流程、流量设备和历史代码会影响选择。
  • 响应式页面里的图片、iframe 和第三方内容为什么要单独处理？ 结论是它们有自身固有尺寸或外部加载状态，不会天然遵守父容器。原因是替换元素、嵌入媒体和第三方脚本可能撑破布局、加载失败或被浏览器拦截。项目中应设置 max-width: 100%、明确宽高或外层固定比例容器，并提供 HTTPS、错误占位和懒加载策略。
