给 Worker 命名 + 用消息体分发不同任务

请打开 DevTools Console 查看输出。

知识点要点

  • new Worker("work.js", { name: "hardWork1" }) 会给 worker 上下文命名，方便在 DevTools、错误日志或多 worker 场景里区分任务来源；它不是线程池，也不会自动调度多个任务。
  • 本页约定主线程发送 { task, data }，worker 根据 task 分发到不同处理函数，再返回 { task, data, state }。这种显式协议比只传裸值更容易扩展、取消和排查错误。
  • CPU 密集循环放在主线程会阻塞 UI；示例中的 startCount() 只保留为对比，真实大任务应由 worker 执行，主线程只负责按钮响应、状态展示和终止任务。
  • 消息入口要把 task、data、任务规模和未知分支都当作边界处理。来自 UI 或网络的数据不能默认可信，否则可能触发超大循环、内存膨胀或无法停止的后台任务。
  • terminate() 会立即结束当前 worker，同一个 Worker 实例不能恢复执行；如果用户再次启动任务，主线程必须重新创建 worker 并重新绑定消息、错误处理函数。
  • Worker 适合持续或重 CPU 的后台计算；如果只是把轻量任务让出一次事件循环，可用 setTimeout 或 queueMicrotask，如果是空闲期低优先级工作，可考虑 requestIdleCallback 但要注意兼容性。

面试考点

  • Worker 的 name 选项解决什么调试问题？ 结论是它用于给 worker 全局作用域命名，帮助调试时识别哪个后台脚本在运行。原因是多个 worker 同时存在时，只有脚本路径往往不足以定位任务。项目里通常把业务含义写进 name，例如 image-compressor 或 search-index-builder。
  • 多任务 worker 为什么要设计消息协议，而不是只传裸值？ 结论是消息协议能让主线程和 worker 对任务类型、输入、进度、完成和错误形成稳定契约。原因是 postMessage 本身只负责传数据，不知道业务语义。实际项目会把 task、requestId、payload、state、error 都标准化，方便并发任务回包归属。
  • 如何避免 CPU 密集任务卡住 UI 或耗尽浏览器资源？ 结论是重计算放进 worker，并给任务规模、取消、错误和重复点击设置边界。原因是 worker 不阻塞主线程，但仍会消耗 CPU 和内存，任务过大一样会影响设备。项目里常配合按钮禁用、进度状态、超时取消和 terminate() 做保护。
  • Worker 与 setTimeout 分片、requestIdleCallback 的选型边界是什么？ 结论是 worker 用于真正需要并行执行的重任务，分片和 idle callback 只是把主线程任务拆开排队。原因是后两者仍在主线程执行，不能避免总计算量对页面响应的影响。轻量分批渲染可用分片，后台索引、压缩、解析等更适合 worker。
