defineProperty 访问器属性 get/set

请打开 DevTools Console 查看输出。

知识点要点

  • 访问器属性没有自己的 value，它由 get 和 set 函数描述；同一个属性描述符不能同时写 get/set 与 value/writable。
  • 读取属性时触发 getter，getter 没有参数；写入属性时触发 setter，setter 只接收一个新值参数。
  • 访问器函数里的 this 通常是本次属性访问的接收者，因此可以组合对象上的其他字段返回派生值。
  • 最常见的坑是在 get fullName() 或 set fullName() 内部继续读写 this.fullName，这会再次触发同一个访问器并造成递归。
  • 访问器适合单个属性的派生、校验、兼容 Vue 2 风格响应式；如果要拦截新增属性、删除属性、in、枚举等对象级操作，现代写法通常用 Proxy。

面试考点

  • 访问器属性和数据属性为什么不能混用 value/writable？ 结论是两类描述符互斥：数据属性保存值，访问器属性通过函数计算或接收写入。原因是规范会按描述符类型定义属性，混用会让同一属性既有存储语义又有函数语义。项目里要么用 value/writable 表示普通字段，要么用 get/set 表示派生或受控字段。
  • getter 和 setter 的参数、this、触发时机分别是什么？ 结论是 getter 在读属性时运行且无参数，setter 在写属性时运行且只有一个新值参数，this 是本次访问的接收者。原因是属性访问语义只需要“读”或“写入的新值”。继承或 Reflect.get 改变 receiver 时，this 也会随之变化。
  • 为什么在 get/set 里读写同名属性会导致递归？ 结论是同名读写会再次命中同一个访问器。原因是访问器本身就是该属性的读取和写入入口。实际项目应使用内部字段、闭包或私有字段保存真实数据，例如本 demo 的 firstName、lastName。
  • defineProperty 访问器和 Proxy 拦截对象操作时如何取舍？ 结论是单个已知属性用访问器更直接，对象级元操作用 Proxy 更完整。原因是 defineProperty 只能定义已有目标上的具体属性，而 Proxy 可拦截读取、写入、删除、枚举、原型等操作。边界是 Proxy 不能被完全 polyfill，旧环境兼容仍可能需要访问器方案。
