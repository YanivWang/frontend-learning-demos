ref / reactive / 解构丢响应

在浏览器中操作下方交互演示；复习要点与面试答法见本复习区。

知识点要点

  • ref() 用 .value 保存响应式值，适合基本类型，也可包装对象；模板顶层会自动解包，本页 {{ count }} 无需写 .value。点“count +1”时 count 与下方派生文案同步更新，控制台会打印 [ref] count。
  • reactive() 返回对象 Proxy，只能接收对象类型；读写属性走代理 getter/setter，本页 user.name 会随按钮更新。应保持同一个代理引用，不要用新普通对象整体替换 user 变量本身。
  • 直接解构 const { name: nameSnapshot } = user 会得到当前值的普通变量快照；之后改 user.name 时 nameSnapshot 不变，本页“错误解构”一行就是验证入口。
  • toRefs(user) 把已有属性转成与源对象联动的 ref；本页 name（来自 toRefs）会跟着 user.name 变化。批量转换细节见同章 03-toRef-toRefs。
  • ref({ ... }) 整体替换 .value 会触发更新，适合“整块数据换新”的场景；reactive 对象应改属性或属性引用，而不是给变量赋一个全新的非代理对象。三种 API 对比见 07-ref-reactive-shallowRef对比。
  • 组合式函数若返回 reactive 对象，调用方常见会解构；应返回 ...toRefs(state) 或多个独立 ref，避免调用方拿到非响应式快照。
  • 本页“操作日志”记录每次按钮触发的状态；打开控制台可同时看到 [ref] / [reactive] 分组日志，便于对照模板显示与 JS 读写方式。

面试考点

  • ref 和 reactive 如何选型？ 结论是基本类型、需要整体替换的值优先 ref，成组对象状态可用 reactive。原因是 ref 通过 .value 替换清晰，reactive 通过 Proxy 拦截属性。项目里表单对象常用 reactive，计数、开关、当前 id 常用 ref。
  • reactive 直接解构为什么会丢响应？ 先结论：解构取出的是当前值，后续读写不再经过原代理。原因是依赖收集发生在代理 getter 上。需要解构时用 toRefs，见 03-toRef-toRefs。
  • 为什么不要整体替换 reactive 变量本身？ 结论是替换变量会断开已有引用。原因是其他地方持有的是旧代理，重新赋值只改变局部变量指向。应改代理属性，或一开始用 ref({}) 承载可整体替换的数据。
  • composable 返回对象时如何避免解构丢响应？ 结论是返回多个 ref，或对 reactive 使用 toRefs 后返回。原因是调用方常会对象解构。边界是也可返回整个 reactive 并文档化不要解构。
  • ref 包裹对象与 reactive 在整体替换上有什么差异？ 结论是 ref 换 .value 会触发更新，reactive 变量本身不应被赋为新普通对象。原因是 ref 追踪的是容器 .value 的替换，reactive 追踪的是固定代理上的属性访问。
